# tracemalloc — подготовка к собеседованию

## Что это и зачем

`tracemalloc` — встроенный модуль для трассировки выделения памяти в Python. Он отслеживает, *где* (в какой строке кода) и *сколько* памяти было выделено, и позволяет сравнивать снимки (snapshots) состояния памяти во времени, чтобы находить утечки и «прожорливые» места.

Зачем нужен:
- **Поиск утечек памяти.** Когда объём растёт со временем — tracemalloc покажет, какие строки выделяют память, что не освобождается.
- **Профилирование памяти, а не времени.** В отличие от cProfile (время), tracemalloc отвечает на вопрос «куда уходит память».
- **Привязка к исходному коду.** Каждое выделение связывается с traceback — видно конкретную строку (и стек вызовов).

Это инструмент *стандартной библиотеки*, не требующий сторонних пакетов (в отличие от `memory_profiler`, `pympler`, `objgraph`, которые дополняют его).

## Ключевые концепции

- **Трассировка выделений.** После `tracemalloc.start()` каждое новое выделение памяти Python-аллокатором запоминается вместе с traceback.
- **Snapshot (снимок)** — мгновенное состояние всех отслеживаемых выделений. Снимки можно сохранять и сравнивать.
- **Statistics** — агрегированная статистика по снимку, сгруппированная по строке (`'lineno'`), файлу (`'filename'`) или полному traceback (`'traceback'`).
- **Diff снимков** — `snapshot.compare_to(old, ...)` показывает *прирост* памяти между двумя моментами — ключ к поиску утечек.
- **Глубина traceback (nframe)** — сколько кадров стека сохранять для каждого выделения; влияет на детализацию и накладные расходы.
- **Накладные расходы.** Трассировка замедляет программу и сама потребляет память (на хранение traceback'ов).

## Основные функции/классы/методы

### Базовый замер: текущая и пиковая память

```python
import tracemalloc

tracemalloc.start()              # включить трассировку

data = [b"x" * 1000 for _ in range(10_000)]   # что-то выделяем

current, peak = tracemalloc.get_traced_memory()
print(f"текущая память: {current / 1024:.1f} КБ")
print(f"пиковая память: {peak / 1024:.1f} КБ")

tracemalloc.stop()               # выключить и освободить накладные структуры
```

`get_traced_memory()` возвращает `(current, peak)` в байтах: текущий объём отслеживаемых выделений и пик с момента старта (или последнего `reset_peak()`).

### Snapshot и топ выделений

```python
import tracemalloc

tracemalloc.start()

def make_data():
    return {i: str(i) * 100 for i in range(10_000)}

d = make_data()

snapshot = tracemalloc.take_snapshot()

# Группировка по строке кода: где выделено больше всего
top_stats = snapshot.statistics("lineno")

print("[ Топ-5 выделений ]")
for stat in top_stats[:5]:
    # stat.size — суммарный размер, stat.count — число блоков
    print(stat)
    # пример: app.py:7: size=1024 KiB, count=10000, average=104 B
```

Группировки в `statistics(key_type)`:
- `"lineno"` — по строке (файл:номер). Самая частая.
- `"filename"` — по файлу.
- `"traceback"` — по полному стеку (нужен `nframe>1` при старте).

### Сравнение снимков — поиск утечки

Главный приём: сделать снимок «до», выполнить подозрительную работу, снимок «после», и сравнить *прирост*.

```python
import tracemalloc

tracemalloc.start()

snapshot1 = tracemalloc.take_snapshot()

# Подозрительный код, который может «течь»
leaked = []
for i in range(100_000):
    leaked.append(str(i))        # намеренно держим ссылки

snapshot2 = tracemalloc.take_snapshot()

# Сравнить второй снимок с первым по строкам
diff = snapshot2.compare_to(snapshot1, "lineno")

print("[ Топ прироста памяти ]")
for stat in diff[:10]:
    # size_diff — прирост размера, count_diff — прирост числа блоков
    print(stat)
    # пример: app.py:12: size=3500 KiB (+3500 KiB), count=100000 (+100000)
```

Положительный `size_diff`/`count_diff`, который растёт от итерации к итерации, — признак утечки.

### Полный traceback выделения

Чтобы видеть не только строку, но и *кто её вызвал*, увеличивают глубину стека:

```python
import tracemalloc

# Сохранять до 10 кадров стека для каждого выделения
tracemalloc.start(10)

data = build_something()

snapshot = tracemalloc.take_snapshot()
top = snapshot.statistics("traceback")

stat = top[0]                              # самое крупное выделение
print(f"{stat.count} блоков, {stat.size / 1024:.1f} КБ")
for line in stat.traceback.format():       # печать всего стека вызовов
    print(line)
```

### Фильтрация снимков

Снимки часто «зашумлены» выделениями самого Python/импортов. Фильтры помогают сфокусироваться:

```python
import tracemalloc

snapshot = tracemalloc.take_snapshot()

# Исключить служебные модули и сам tracemalloc
snapshot = snapshot.filter_traces((
    tracemalloc.Filter(False, "<frozen importlib._bootstrap>"),
    tracemalloc.Filter(False, "<unknown>"),
    tracemalloc.Filter(False, tracemalloc.__file__),
    # Включить ТОЛЬКО свой код:
    tracemalloc.Filter(True, "/app/*"),
))

for stat in snapshot.statistics("lineno")[:5]:
    print(stat)
```

`Filter(inclusive, pattern)`: `inclusive=True` — оставить совпадающие, `False` — исключить.

### Управление пиком и сохранение снимков

```python
import tracemalloc

tracemalloc.start()

phase1()
_, peak1 = tracemalloc.get_traced_memory()
tracemalloc.reset_peak()         # обнулить пик, чтобы мерить следующую фазу

phase2()
_, peak2 = tracemalloc.get_traced_memory()

# Снимки можно сериализовать на диск для анализа позже:
snap = tracemalloc.take_snapshot()
snap.dump("memory.snapshot")
# Загрузка:
loaded = tracemalloc.Snapshot.load("memory.snapshot")
```

### Включение из окружения / при старте интерпретатора

```bash
# Запустить с уже включённой трассировкой (глубина 25 кадров)
python -X tracemalloc=25 script.py

# Через переменную окружения
PYTHONTRACEMALLOC=25 python script.py
```

Это полезно, чтобы поймать выделения, происходящие *до* первого вызова `start()` в коде (например, на этапе импортов).

## Частые вопросы на собеседовании

**Q1. Что делает tracemalloc и чем отличается от cProfile?**
tracemalloc трассирует *выделения памяти*: где (строка/стек) и сколько байт выделено, и позволяет сравнивать снимки. cProfile профилирует *время* выполнения функций. Один отвечает «куда уходит память», другой — «куда уходит время».

**Q2. Что такое snapshot и зачем сравнивать снимки?**
Snapshot — мгновенное состояние всех отслеживаемых выделений. Сравнение двух снимков (`compare_to`) показывает *прирост* памяти между двумя моментами времени. Это основной способ найти утечку: растущий `size_diff` на конкретной строке от прогона к прогону указывает на код, удерживающий объекты.

**Q3. Как найти утечку памяти с помощью tracemalloc?**
Сделать снимок «до», выполнить подозрительный участок (желательно в цикле несколько раз), снимок «после», вызвать `snapshot2.compare_to(snapshot1, "lineno")` и посмотреть строки с наибольшим положительным приростом. Стабильно растущее `count_diff`/`size_diff` — утечка.

**Q4. Что возвращает get_traced_memory()?**
Кортеж `(current, peak)` в байтах: текущий объём отслеживаемых выделений и пиковое значение с момента старта или последнего `reset_peak()`. Удобно для быстрой оценки потребления и пика без детального снимка.

**Q5. Зачем параметр nframe (глубина) при start()?**
Он задаёт, сколько кадров стека сохранять для каждого выделения. `nframe=1` (по умолчанию) даёт только строку выделения; больше кадров (`start(10)`) позволяют группировать по `"traceback"` и видеть, *кто* вызвал выделяющий код. Цена — больше памяти и оверхеда на трассировку.

**Q6. Каковы накладные расходы tracemalloc и где это критично?**
Трассировка замедляет программу (каждое выделение перехватывается) и сама расходует память на хранение traceback'ов (тем больше, чем выше nframe). Поэтому в высоконагруженном проде её обычно не держат включённой постоянно; включают точечно для диагностики.

**Q7. Зачем нужны фильтры снимков?**
Снимок содержит выделения интерпретатора, импортов, самого tracemalloc — это шум. `filter_traces` с `Filter(False, pattern)` исключает служебные модули, а `Filter(True, "/app/*")` оставляет только свой код, делая отчёт читаемым.

**Q8. Как поймать выделения, происходящие до запуска вашего кода (на импортах)?**
Включить трассировку при старте интерпретатора: `python -X tracemalloc=25 script.py` или `PYTHONTRACEMALLOC=25`. Тогда выделения на этапе импорта тоже попадут в снимки, чего не даст вызов `start()` внутри программы.

**Q9. tracemalloc показывает все объекты в памяти?**
Нет. Он трассирует только выделения, сделанные Python-аллокатором *после* старта (или с момента запуска при `-X tracemalloc`). Память, выделенная C-расширениями напрямую (минуя Python-аллокатор), и объекты, созданные до старта, могут не учитываться. Для C-памяти и графа ссылок нужны другие инструменты.

**Q10. Чем tracemalloc дополняют другие инструменты?**
`gc` — найти циклические ссылки и недостижимые объекты; `objgraph` — визуализировать графы ссылок и понять, *что держит* объект; `memory_profiler` — построчное потребление RSS процесса; `pympler` — детальный учёт объектов. tracemalloc хорош для «где выделено и что растёт», но не всегда объясняет, *почему* объект жив (кто на него ссылается).

**Q11. Утечка ли это, если current не падает после удаления данных?**
Не обязательно: Python-аллокатор и пулы (arena) могут *не отдавать* память ОС сразу, кэшируя её для повторного использования. tracemalloc считает отслеживаемые блоки, но RSS процесса может оставаться высоким из-за фрагментации/пулов. Растущий `compare_to` важнее единичного `current`.

**Q12. Как профилировать пиковое потребление по фазам программы?**
Использовать `reset_peak()` между фазами: замерить `get_traced_memory()` пик после фазы 1, вызвать `reset_peak()`, выполнить фазу 2 и снова прочитать пик. Так получают пик каждой фазы отдельно.

## Подводные камни (gotchas)

- **Не учитывает выделения до start().** Если важны импорты/инициализация, включайте через `-X tracemalloc`/`PYTHONTRACEMALLOC`, а не только `tracemalloc.start()` в середине кода.
- **C-память невидима.** Выделения в C-расширениях, идущие в обход Python-аллокатора (часть numpy, нативные библиотеки), tracemalloc не видит. Для них нужны нативные инструменты (valgrind, profilers уровня ОС).
- **Накладные расходы.** Трассировка замедляет код и потребляет память на traceback'и (растёт с nframe). Не держите включённой в горячем проде постоянно.
- **current != RSS.** tracemalloc меряет отслеживаемые блоки Python-объектов, а не RSS процесса. Аллокатор кэширует освобождённую память в пулах, поэтому RSS может не уменьшаться — это не всегда утечка.
- **Шум снимков.** Без фильтров топ забит выделениями интерпретатора/импортов. Используйте `filter_traces`, чтобы видеть свой код.
- **nframe=1 скрывает источник.** С одной рамкой видно только строку выделения (часто это обёртка типа `str()`), но не кто её вызвал. Для диагностики поднимайте глубину и группируйте по `"traceback"`.
- **Не объясняет, кто держит объект.** tracemalloc показывает, *где выделено*, но не *почему живёт*. Для поиска удерживающих ссылок нужен `gc.get_referrers`/`objgraph`.
- **Снимок дорог по памяти.** Хранение многих снимков с глубоким traceback само раздувает память. Берите снимки точечно, при необходимости сериализуйте (`dump`) и анализируйте офлайн.

## Лучшие практики

- Для поиска утечек повторяйте подозрительную операцию в цикле и сравнивайте снимки (`compare_to`) — растущий прирост красноречивее единичного замера.
- Включайте трассировку как можно раньше (через `-X tracemalloc=N`), если интересуют выделения на старте/импортах.
- Фильтруйте снимки (`filter_traces`), оставляя только свой код, чтобы не тонуть в служебном шуме.
- Поднимайте `nframe` (например, 10–25) для группировки по `"traceback"`, когда строка выделения неинформативна (обёртки, фабрики).
- Используйте `reset_peak()` для пофазного измерения пика.
- Сериализуйте снимки (`dump`/`load`) для анализа продовых инцидентов офлайн, не нагружая процесс анализом.
- Не оставляйте tracemalloc включённым в высоконагруженном проде — включайте точечно для диагностики и отключайте (`stop`) после.
- Дополняйте tracemalloc инструментами поиска ссылок (`gc`, `objgraph`), когда нужно понять, *кто держит* утекающий объект.

## Шпаргалка

```python
import tracemalloc

tracemalloc.start()                 # включить (nframe=1)
tracemalloc.start(25)               # хранить 25 кадров стека

cur, peak = tracemalloc.get_traced_memory()   # (текущая, пик) в байтах
tracemalloc.reset_peak()            # обнулить пик (пофазный замер)

snap = tracemalloc.take_snapshot()  # снимок

# Топ выделений
for s in snap.statistics("lineno")[:10]:
    print(s)                        # file:line: size=..., count=...

# Сравнение -> поиск утечки
diff = snap2.compare_to(snap1, "lineno")
for s in diff[:10]:
    print(s)                        # ... (+size_diff), count (+count_diff)

# Полный traceback
for s in snap.statistics("traceback")[:1]:
    print("\n".join(s.traceback.format()))

# Фильтрация
snap = snap.filter_traces((
    tracemalloc.Filter(False, "<frozen importlib._bootstrap>"),
    tracemalloc.Filter(True, "/app/*"),
))

# Сохранение/загрузка
snap.dump("mem.snapshot")
loaded = tracemalloc.Snapshot.load("mem.snapshot")

tracemalloc.stop()                  # выключить
```
