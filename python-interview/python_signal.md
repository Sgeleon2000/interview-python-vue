# signal — подготовка к собеседованию

## Что это и зачем

Модуль `signal` позволяет обрабатывать **сигналы операционной системы** — асинхронные уведомления, которые ОС или другие процессы посылают вашему процессу. Сигналы — это базовый механизм межпроцессного оповещения в Unix.

Зачем нужен:
- **Graceful shutdown** — корректное завершение сервиса по `SIGTERM`/`SIGINT` (закрыть соединения, сбросить буферы, дождаться задач).
- **Перезагрузка конфигурации** на лету по `SIGHUP` без рестарта.
- **Реакция на Ctrl+C** (`SIGINT`), таймеры (`SIGALRM`), завершение дочерних процессов (`SIGCHLD`).

Типичная задача на собеседовании: «как корректно остановить сервис в контейнере». Ответ почти всегда про обработку `SIGTERM` (его шлёт `docker stop`, Kubernetes, systemd).

## Ключевые концепции

### Что такое сигнал
Программное прерывание: ОС асинхронно «доставляет» процессу номер сигнала. У каждого сигнала есть **действие по умолчанию**: завершить процесс, завершить с дампом, проигнорировать, остановить/продолжить. Программа может **переопределить** обработку (кроме `SIGKILL` и `SIGSTOP`).

### Частые сигналы
```
┌──────────┬──────┬──────────────────────────┬───────────────────────┐
│ Сигнал   │ №    │ Когда                    │ Действие по умолчанию │
├──────────┼──────┼──────────────────────────┼───────────────────────┤
│ SIGINT   │ 2    │ Ctrl+C в терминале       │ завершить (KeyboardInterrupt)│
│ SIGTERM  │ 15   │ kill / docker stop / k8s │ завершить (можно перехв.)│
│ SIGKILL  │ 9    │ kill -9                  │ завершить (НЕЛЬЗЯ перехв.)│
│ SIGHUP   │ 1    │ закрыт терминал / reload │ завершить               │
│ SIGALRM  │ 14   │ signal.alarm(), таймер   │ завершить               │
│ SIGCHLD  │ 17   │ дочерний процесс завершён│ игнорировать            │
│ SIGSTOP  │ 19   │ приостановка             │ остановить (НЕЛЬЗЯ перехв.)│
│ SIGUSR1/2│ 10/12│ пользовательские         │ завершить               │
└──────────┴──────┴──────────────────────────┴───────────────────────┘
```
**`SIGKILL` (9) и `SIGSTOP` (19) нельзя перехватить, заблокировать или проигнорировать** — ими ОС гарантированно убивает/останавливает процесс.

### Обработчик сигнала (signal handler)
Функция, вызываемая при доставке сигнала. Принимает два аргумента: номер сигнала и текущий стек-фрейм. В CPython обработчик исполняется **в главном потоке** между байт-кодами интерпретатора (не строго в момент сигнала), что накладывает ограничения.

## Основные функции/классы/методы

### Регистрация обработчика
```python
import signal
import sys

def handler(signum, frame):
    # signum — номер сигнала, frame — стек-фрейм прерванного кода
    print(f"Получен сигнал {signum}, завершаюсь корректно")
    sys.exit(0)

# Перехватываем SIGTERM (от docker stop / kill) и SIGINT (Ctrl+C)
signal.signal(signal.SIGTERM, handler)
signal.signal(signal.SIGINT, handler)
```

### Специальные значения обработчика
```python
signal.signal(signal.SIGINT, signal.SIG_IGN)   # игнорировать сигнал
signal.signal(signal.SIGINT, signal.SIG_DFL)   # вернуть действие по умолчанию
old = signal.getsignal(signal.SIGINT)            # узнать текущий обработчик
```

### Graceful shutdown через флаг (рекомендуемый паттерн)
```python
import signal
import time

shutdown = False

def request_shutdown(signum, frame):
    # В обработчике делаем МИНИМУМ — только ставим флаг
    global shutdown
    shutdown = True
    print("Запрошено завершение...")

signal.signal(signal.SIGTERM, request_shutdown)
signal.signal(signal.SIGINT, request_shutdown)

# Основной цикл проверяет флаг и завершается в безопасной точке
while not shutdown:
    # ... полезная работа ...
    time.sleep(0.5)

print("Закрываю ресурсы, сбрасываю буферы, выхожу")
```

### Таймер: signal.alarm и SIGALRM
```python
import signal

class TimeoutError(Exception):
    pass

def on_alarm(signum, frame):
    raise TimeoutError("операция превысила лимит времени")

signal.signal(signal.SIGALRM, on_alarm)
signal.alarm(5)            # доставить SIGALRM через 5 секунд
try:
    long_blocking_operation()   # если не успеет за 5с — прервётся исключением
finally:
    signal.alarm(0)        # отменить таймер
```

### Отправка сигнала из Python
```python
import os
import signal

os.kill(pid, signal.SIGTERM)     # послать сигнал процессу по PID
signal.raise_signal(signal.SIGINT)  # послать сигнал самому себе (3.8+)
```

### Интеграция с asyncio
```python
import asyncio
import signal

async def main():
    loop = asyncio.get_running_loop()
    stop = asyncio.Event()

    # Безопасная регистрация обработчиков в event loop
    for sig in (signal.SIGTERM, signal.SIGINT):
        loop.add_signal_handler(sig, stop.set)

    print("Работаю, жду сигнала...")
    await stop.wait()                # ждём, пока придёт сигнал
    print("Завершаюсь корректно")

asyncio.run(main())
```

### Контекстный менеджер для временной обработки
```python
import signal
from contextlib import contextmanager

@contextmanager
def timeout(seconds):
    def handler(signum, frame):
        raise TimeoutError()
    old = signal.signal(signal.SIGALRM, handler)   # сохраняем старый обработчик
    signal.alarm(seconds)
    try:
        yield
    finally:
        signal.alarm(0)                            # отменяем таймер
        signal.signal(signal.SIGALRM, old)         # восстанавливаем обработчик

with timeout(3):
    slow_function()   # прервётся, если дольше 3 секунд
```

## Частые вопросы на собеседовании

**Q1. Какие сигналы нельзя перехватить?**
`SIGKILL` (9) и `SIGSTOP` (19). Их нельзя перехватить, заблокировать или проигнорировать — ОС гарантированно убивает/останавливает процесс. Поэтому `kill -9` нельзя «обработать».

**Q2. В чём разница между `SIGTERM` и `SIGKILL`?**
`SIGTERM` (15) — вежливая просьба завершиться: процесс может перехватить её и выполнить graceful shutdown (закрыть соединения, сохранить состояние). `SIGKILL` (9) — немедленное принудительное завершение ядром без возможности реакции и очистки.

**Q3. Что происходит при Ctrl+C?**
Терминал шлёт процессу `SIGINT`. По умолчанию Python преобразует его в исключение `KeyboardInterrupt` в главном потоке. Можно переопределить обработчик через `signal.signal(SIGINT, ...)`.

**Q4. Как реализовать graceful shutdown сервиса?**
Зарегистрировать обработчик `SIGTERM` (и `SIGINT`), который ставит флаг/событие завершения. Основной цикл периодически проверяет флаг и при его установке завершает текущую работу, закрывает соединения, сбрасывает буферы и выходит. В контейнерах это критично: `docker stop`/k8s сначала шлют `SIGTERM`, а через grace period — `SIGKILL`.

**Q5. Какие ограничения у обработчиков сигналов в Python?**
(1) Регистрировать обработчики можно только из **главного потока** главного интерпретатора. (2) Обработчик вызывается **между байт-кодами**, а не точно в момент сигнала, поэтому есть задержка. (3) В обработчике нельзя делать сложную/блокирующую работу — рекомендуется только ставить флаг. (4) Большинство сигналов на Windows не поддерживается.

**Q6. Почему в обработчике нужно делать минимум?**
Обработчик прерывает выполнение в произвольной точке. Сложные операции, блокировки, ввод-вывод могут привести к взаимоблокировкам или повреждению состояния (не все функции async-signal-safe). Безопасный паттерн — установить флаг/событие и обработать его в основном коде.

**Q7. Как сделать таймаут на блокирующую операцию через signal?**
`signal.signal(SIGALRM, handler)` + `signal.alarm(seconds)`: через заданное время придёт `SIGALRM`, обработчик бросит исключение, прервав операцию. После — `signal.alarm(0)` для отмены. Ограничение: работает только в главном потоке и только на Unix.

**Q8. Как обрабатывать сигналы в asyncio?**
Через `loop.add_signal_handler(sig, callback)` — это безопаснее, чем `signal.signal`, потому что колбэк выполняется в контексте event loop, а не прерывает произвольную точку. Обычно колбэк устанавливает `asyncio.Event`, которое ждёт основная корутина.

**Q9. Работают ли сигналы в дочерних потоках?**
Обработчики сигналов всегда выполняются в **главном потоке**. Регистрировать их (`signal.signal`) можно только из главного потока, иначе `ValueError`. Доставка сигнала в конкретный поток не гарантируется — это особенность POSIX.

**Q10. Что такое `SIGHUP` и зачем используется?**
Изначально — «обрыв связи» терминала. По соглашению многие демоны перехватывают `SIGHUP` для **перечитывания конфигурации** без перезапуска (`nginx -s reload`, перезагрузка конфигов).

## Подводные камни (gotchas)

- **`SIGKILL`/`SIGSTOP` неперехватываемы** — нельзя сделать graceful shutdown по `kill -9`.
- **Обработчики только из главного потока** — `signal.signal` из другого потока бросает `ValueError`.
- **Задержка доставки** — обработчик вызывается между байт-кодами, а не мгновенно; долгий C-вызов может отложить реакцию.
- **Сложная логика в обработчике** опасна (не все функции async-signal-safe) — ставьте флаг/событие.
- **Windows почти не поддерживает сигналы** — доступны лишь `SIGINT`, `SIGTERM` и немногие; `SIGALRM`, `add_signal_handler` и большинство — нет.
- **Перезапись обработчика по умолчанию** (`SIG_DFL`) или игнор (`SIG_IGN`) может незаметно сломать поведение (например, отключить Ctrl+C).
- **`signal.alarm` только Unix и только главный поток**, перезаписывает предыдущий таймер.
- **В asyncio используйте `add_signal_handler`**, а не `signal.signal` — иначе колбэк может выполниться в неподходящий момент относительно цикла.
- **Прерванные системные вызовы** — сигнал может прервать блокирующий вызов с `EINTR`; в современном Python большинство автоматически перезапускаются, но не всегда.

## Лучшие практики

- В обработчике делайте **минимум** — установите флаг или `asyncio.Event`, остальное обработайте в основном коде.
- Для сервисов всегда обрабатывайте **`SIGTERM`** (и `SIGINT`) для graceful shutdown — особенно в контейнерах.
- В **asyncio** используйте **`loop.add_signal_handler`**.
- **Сохраняйте и восстанавливайте** прежний обработчик (`getsignal`), если меняете его временно.
- Не полагайтесь на сигналы для **кросс-платформенного** кода (Windows) — предусмотрите альтернативу.
- **Отменяйте таймеры** (`signal.alarm(0)`) в `finally`.
- Регистрируйте обработчики **только в главном потоке**.
- Помните про **grace period** в Docker/Kubernetes: успейте завершиться до прихода `SIGKILL`.

## Шпаргалка

```python
import signal, os, sys

# --- Регистрация ---
signal.signal(signal.SIGTERM, handler)   # handler(signum, frame)
signal.signal(signal.SIGINT, handler)    # Ctrl+C
signal.signal(signal.SIGINT, signal.SIG_IGN)   # игнорировать
signal.signal(signal.SIGINT, signal.SIG_DFL)   # действие по умолчанию
old = signal.getsignal(signal.SIGINT)          # текущий обработчик

# --- Graceful shutdown (паттерн с флагом) ---
stop = False
def h(s, f):
    global stop; stop = True
signal.signal(signal.SIGTERM, h)
while not stop: ...                       # проверяем флаг в цикле

# --- Таймаут через alarm (Unix, главный поток) ---
signal.signal(signal.SIGALRM, on_alarm)  # on_alarm бросает исключение
signal.alarm(5)                          # через 5с придёт SIGALRM
signal.alarm(0)                          # отменить

# --- Отправка сигнала ---
os.kill(pid, signal.SIGTERM)
signal.raise_signal(signal.SIGINT)       # самому себе (3.8+)

# --- asyncio ---
loop.add_signal_handler(signal.SIGTERM, stop_event.set)

# Нельзя перехватить: SIGKILL (9), SIGSTOP (19)
# Обработчики — только в главном потоке
```
