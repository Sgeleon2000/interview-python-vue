# doctest — подготовка к собеседованию

## Что это и зачем

`doctest` — встроенный модуль, который ищет в docstring-ах фрагменты, похожие на сессию интерактивного интерпретатора (`>>> ...`), выполняет их и сверяет фактический вывод с ожидаемым, записанным сразу под строкой ввода.

Зачем нужен:
- **Документация = тесты.** Примеры в документации гарантированно актуальны: если код изменился и пример сломался, doctest это покажет.
- **Дешёвые smoke-тесты.** Минимум усилий, чтобы зафиксировать базовое поведение функции.
- **Обучаемость.** Читатель видит, как функцию вызывать и что она вернёт.

Ограничения: doctest не заменяет полноценный фреймворк (нет фикстур, параметризации, сложной настройки). Это инструмент для *иллюстративных* примеров, а не для тяжёлого тестирования.

## Ключевые концепции

- **Пример (example)** — пара «ввод после `>>>` (и `...` для продолжений) + ожидаемый вывод».
- **Совпадение вывода** сравнивается *буквально* по строкам (с учётом пробелов), кроме случаев, изменённых директивами.
- **Пустая строка** завершает ожидаемый вывод примера. Чтобы показать пустую строку в выводе, используют `<BLANKLINE>`.
- **Директивы (directives)** — комментарии `# doctest: +FLAG` рядом с примером, меняющие поведение сравнения (например, игнор многоточием).
- doctest можно держать как в docstring модуля/функции/класса, так и в отдельных текстовых файлах.

## Основные функции/классы/методы

### Doctest в docstring функции

```python
def factorial(n):
    """Возвращает факториал n.

    >>> factorial(0)
    1
    >>> factorial(5)
    120
    >>> factorial(3) * 2
    12

    Исключение тоже можно проверить:

    >>> factorial(-1)
    Traceback (most recent call last):
        ...
    ValueError: n должно быть >= 0
    """
    if n < 0:
        raise ValueError("n должно быть >= 0")
    result = 1
    for i in range(2, n + 1):
        result *= i
    return result


if __name__ == "__main__":
    import doctest
    # Запускает все doctest текущего модуля
    doctest.testmod()
```

### Запуск

```bash
# Способ 1: из командной строки (verbose необязателен)
python -m doctest mymodule.py
python -m doctest -v mymodule.py        # подробно: показывает каждый пример

# Способ 2: внутри модуля через testmod() и прямой запуск
python mymodule.py                       # если есть testmod() в __main__

# Способ 3: doctest для отдельного текстового файла
python -m doctest examples.txt -v
```

По умолчанию (без `-v`) при УСПЕХЕ doctest *ничего не выводит* — «тишина = успех». Падения печатаются с diff ожидаемого и фактического.

### testmod и testfile

```python
import doctest

# Тестировать docstring текущего модуля:
doctest.testmod()
doctest.testmod(verbose=True)            # подробный вывод

# Тестировать конкретный модуль:
import mymodule
doctest.testmod(mymodule)

# Тестировать примеры из текстового файла:
doctest.testfile("examples.txt")

# Вернуть статистику:
results = doctest.testmod()
print(results.failed, results.attempted) # сколько упало / сколько всего
```

### Многострочный ввод и продолжения

```python
def greet_all(names):
    """
    >>> for name in ["Аня", "Боря"]:
    ...     print(f"Привет, {name}!")
    Привет, Аня!
    Привет, Боря!

    >>> d = {
    ...     "a": 1,
    ...     "b": 2,
    ... }
    >>> sorted(d.items())
    [('a', 1), ('b', 2)]
    """
```

`...` — продолжение многострочной конструкции (тело цикла, многострочный литерал).

### Пустые строки в выводе: <BLANKLINE>

```python
def show():
    r"""
    >>> print("первая\n\nтретья")
    первая
    <BLANKLINE>
    третья
    """
```

Без `<BLANKLINE>` пустая строка интерпретировалась бы как конец ожидаемого вывода, и тест бы сломался.

### Директивы

Директивы пишутся как комментарий к строке ввода: `# doctest: +FLAG` или `-FLAG`.

```python
def make_set():
    """
    Порядок множества/словаря не гарантирован — ELLIPSIS спасает:

    >>> sorted({3, 1, 2})
    [1, 2, 3]

    ELLIPSIS — многоточие ... сопоставляется с любым текстом:

    >>> print("Очень длинная строка с деталями")  # doctest: +ELLIPSIS
    Очень...деталями

    Объекты с адресами в памяти:

    >>> object()  # doctest: +ELLIPSIS
    <object object at 0x...>

    NORMALIZE_WHITESPACE — все пробелы/переводы строк считаются одним пробелом:

    >>> print("a   b\tc")  # doctest: +NORMALIZE_WHITESPACE
    a b c

    SKIP — пример показывается, но НЕ выполняется (например, недетерминированный):

    >>> import random; random.random()  # doctest: +SKIP
    0.123456789

    IGNORE_EXCEPTION_DETAIL — игнорирует текст/модуль исключения, сверяя только тип:

    >>> raise ValueError("любой текст")  # doctest: +IGNORE_EXCEPTION_DETAIL
    Traceback (most recent call last):
    ValueError: что угодно
    """
```

Основные флаги:
- `+ELLIPSIS` — `...` в ожидаемом выводе матчит произвольный фрагмент.
- `+NORMALIZE_WHITESPACE` — нормализует пробельные символы (удобно для таблиц/длинных строк).
- `+SKIP` — пропустить пример (не выполнять).
- `+IGNORE_EXCEPTION_DETAIL` — сверять только тип исключения, игнорируя сообщение и путь модуля.
- `+DONT_ACCEPT_TRUE_FOR_1`, `+DONT_ACCEPT_BLANKLINE` — более строгое сравнение.
- `+REPORT_ONLY_FIRST_FAILURE`, `+REPORT_NDIFF`, `+REPORT_UDIFF` — формат отчёта о падениях.

Глобально флаги задаются через `optionflags`:

```python
import doctest
doctest.testmod(optionflags=doctest.ELLIPSIS | doctest.NORMALIZE_WHITESPACE)
```

Или директивой в начале файла-текста: `# doctest: +ELLIPSIS`.

### Интеграция с unittest

doctest можно «упаковать» в `TestSuite` и запускать вместе с обычными тестами:

```python
import doctest
import unittest
import mymodule


def load_tests(loader, tests, ignore):
    # Стандартный хук unittest: добавляем doctest в общий набор
    tests.addTests(doctest.DocTestSuite(mymodule))
    tests.addTests(doctest.DocFileSuite("examples.txt"))
    return tests


if __name__ == "__main__":
    unittest.main()
```

pytest также умеет собирать doctest: `pytest --doctest-modules` (docstring) и `pytest --doctest-glob='*.txt'`.

## Частые вопросы на собеседовании

**Q1. Что такое doctest и где он берёт тесты?**
Это модуль, выполняющий примеры из docstring-ов (и текстовых файлов), оформленные как сессия REPL (`>>> ввод` + ожидаемый вывод под ним), и сверяющий реальный вывод с ожидаемым. Документация становится исполняемым тестом.

**Q2. Как doctest сравнивает вывод?**
Буквально, построчно, с учётом пробелов, по строковому представлению (то, что напечатал бы интерпретатор / `print`). Пустая строка завершает ожидаемый вывод. Поведение можно ослабить директивами (`ELLIPSIS`, `NORMALIZE_WHITESPACE`).

**Q3. Как протестировать, что функция бросает исключение?**
Записать `Traceback (most recent call last):`, затем `...` (опускаем тело трейсбека), затем строку с типом и сообщением: `ValueError: текст`. Для устойчивости к тексту используют `# doctest: +IGNORE_EXCEPTION_DETAIL`.

**Q4. Зачем нужен <BLANKLINE>?**
Пустая строка в ожидаемом выводе означала бы «конец примера». Чтобы реально проверить пустую строку в выводе, на её месте пишут `<BLANKLINE>`.

**Q5. Что делает директива ELLIPSIS?**
Позволяет `...` в ожидаемом выводе совпадать с любым фрагментом текста. Незаменима для недетерминированных частей: адресов объектов (`at 0x...`), длинных/изменчивых строк.

**Q6. Как обойти недетерминированный порядок (множества, словари до 3.7, hash)?**
Сделать вывод детерминированным: сортировать (`sorted(...)`), приводить к каноничному виду; либо применить `+ELLIPSIS`/`+NORMALIZE_WHITESPACE`; либо пометить пример `+SKIP`, если детерминизм недостижим.

**Q7. Что выводит doctest при успехе?**
Без `-v`/`verbose=True` — ничего (тишина означает успех). С verbose печатается каждый выполненный пример и итоговая статистика.

**Q8. Чем doctest отличается от unittest/pytest и когда его выбирать?**
doctest проще и привязан к документации, но не имеет фикстур, параметризации, сложной настройки и плохо масштабируется. Его выбирают для иллюстративных примеров и smoke-тестов; для серьёзного покрытия используют unittest/pytest. doctest можно встроить в их suite.

**Q9. Как запускать doctest?**
`python -m doctest file.py [-v]`, либо `doctest.testmod()`/`doctest.testfile()` внутри модуля, либо через `DocTestSuite`/`DocFileSuite` в unittest, либо `pytest --doctest-modules`.

**Q10. Главные подводные камни doctest?**
Хрупкость к пробелам и представлению объектов; недетерминированный вывод (порядок словарей/множеств, адреса, время, float-погрешность); смешение примеров с состоянием (один пример опирается на переменные другого). Решается директивами, сортировкой и аккуратной независимостью примеров.

## Подводные камни (gotchas)

- **Чувствительность к пробелам.** Лишний/недостающий пробел в ожидаемом выводе ломает тест. Помогает `+NORMALIZE_WHITESPACE`.
- **Repr объектов.** Вывод вроде `<object at 0x10ab>` нестабилен (адрес меняется). Используйте `+ELLIPSIS` (`at 0x...`) или печатайте детерминированные поля.
- **Порядок словарей/множеств.** До Python 3.7 порядок словарей не гарантирован; множества не упорядочены всегда. Сортируйте вывод (`sorted`).
- **Float-погрешность.** `0.1 + 0.2` даёт `0.30000000000000004`. Округляйте (`round(...)`) или форматируйте строкой.
- **Пустые строки.** Пустая строка обрывает ожидаемый вывод — используйте `<BLANKLINE>`.
- **Состояние между примерами.** Примеры в одном docstring делят пространство имён по порядку; перестановка/удаление может сломать последующие. Делайте примеры по возможности независимыми.
- **Текущее время / случайность.** Недетерминированно — `+SKIP` или мок (что в doctest неудобно). Часто это сигнал, что нужен полноценный тест.
- **Юникод и экранирование.** В docstring с `\n`, `\t` используйте raw-строку (`r"""..."""`), иначе экранирование исказит пример.
- **Тяжёлая логика в примерах.** doctest не для сложных сценариев — нет setUp/teardown, сложно мокать. Не превращайте документацию в тест-фреймворк.

## Лучшие практики

- Используйте doctest для *коротких иллюстративных* примеров «как вызвать и что получить», а не для покрытия краевых случаев.
- Делайте вывод детерминированным: `sorted`, `round`, каноничные представления.
- Включайте `ELLIPSIS` и `NORMALIZE_WHITESPACE` глобально через `optionflags`, если в проекте много примеров с длинным/изменчивым выводом.
- Держите примеры независимыми; не полагайтесь на переменные из соседнего примера без необходимости.
- Запускайте doctest в CI (через `pytest --doctest-modules` или `DocTestSuite`), чтобы документация не «загнивала».
- Для исключений предпочитайте `+IGNORE_EXCEPTION_DETAIL`, чтобы тест не ломался от изменения текста сообщения.
- Не дублируйте серьёзное тестирование в doctest — это вторая, более слабая, система; основное покрытие держите в unittest/pytest.

## Шпаргалка

```python
def f(x):
    """
    >>> f(2)          # ввод
    4                 # ожидаемый вывод
    >>> for i in [1, 2]:   # многострочно
    ...     print(i)
    1
    2
    >>> raise ValueError("boom")    # исключение
    Traceback (most recent call last):
        ...
    ValueError: boom
    >>> print("a\\n\\nb")   # пустая строка в выводе
    a
    <BLANKLINE>
    b
    """
    return x * x
```

```
Директивы (# doctest: +FLAG рядом со строкой >>>):
  +ELLIPSIS                  ... матчит любой текст
  +NORMALIZE_WHITESPACE      пробелы как один пробел
  +SKIP                      не выполнять пример
  +IGNORE_EXCEPTION_DETAIL   сверять только тип исключения
  +REPORT_NDIFF / +REPORT_UDIFF   формат diff при падении
```

```bash
python -m doctest mymodule.py        # запуск (тишина = успех)
python -m doctest -v mymodule.py     # подробно
pytest --doctest-modules             # через pytest
```

import doctest
doctest.testmod(verbose=True,
