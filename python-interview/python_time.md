# time — подготовка к собеседованию

## Что это и зачем

`time` — это встроенный модуль стандартной библиотеки Python, который предоставляет низкоуровневый доступ к функциям работы со временем операционной системы. По сути это тонкая обёртка над системными вызовами C (`time()`, `clock_gettime()`, `nanosleep()` и т.д.).

Модуль решает несколько разных по своей природе задач, которые важно не путать:

1. **Метки времени (timestamps)** — получить текущий момент как число секунд от начала эпохи: `time()`, `time_ns()`.
2. **Измерение длительности (бенчмарки)** — посчитать, сколько прошло между двумя точками кода: `perf_counter()`, `monotonic()`, `process_time()`.
3. **Задержки** — приостановить выполнение: `sleep()`.
4. **Форматирование и парсинг** — преобразовать момент времени в строку и обратно: `strftime()`, `strptime()`, `struct_time`, `gmtime()`, `localtime()`.

Ключевая мысль для собеседования: **`time` — это про измерения, метки времени и взаимодействие с ОС. Для арифметики дат, таймзон и человекочитаемой работы с датами есть `datetime`.**

```python
import time

# Текущий момент как Unix timestamp (число секунд с плавающей точкой)
print(time.time())        # 1750000000.123456

# Измерение длительности участка кода
start = time.perf_counter()
sum(range(10_000_000))
print(time.perf_counter() - start)  # ~0.15 (секунды)
```

## Ключевые концепции

### Эпоха (epoch)

**Эпоха** — это точка отсчёта, от которой считается время. В Unix-системах (и в Python на всех основных платформах) эпоха — это **00:00:00 UTC 1 января 1970 года**. Unix timestamp — это количество секунд, прошедшее с этого момента (без учёта високосных секунд — Unix-время их «не замечает»).

```python
import time

# Узнать, какая эпоха используется на текущей платформе
print(time.gmtime(0))
# time.struct_time(tm_year=1970, tm_mon=1, tm_mday=1, tm_hour=0,
#                  tm_min=0, tm_sec=0, tm_wday=3, tm_yday=1, tm_isdst=0)
```

### Виды часов: wall clock vs монотонные vs процессорные

Это самая частая ловушка собеседований. Нужно чётко понимать разницу:

| Часы | Функция | Идут назад? | Что измеряют |
|------|---------|-------------|--------------|
| Wall clock (стенные) | `time()` | Да (NTP, ручной перевод) | Реальное календарное время |
| Монотонные | `monotonic()` | Нет | Время с произвольной точки, только вперёд |
| Высокоточные | `perf_counter()` | Нет | То же, но с максимальным разрешением |
| Процессорные | `process_time()` | — | Время CPU, потраченное процессом |

- **Wall clock** (`time()`) может «прыгнуть» назад, если систему синхронизировали по NTP или пользователь перевёл часы. Использовать для измерения длительности **нельзя**.
- **Монотонные часы** (`monotonic`, `perf_counter`) гарантированно никогда не идут назад — они идеальны для измерения интервалов. Их абсолютное значение не имеет смысла (точка отсчёта произвольна), важна только разница между двумя замерами.
- **Процессорное время** (`process_time`) считает только время, когда CPU реально выполнял ваш процесс. Сон (`sleep`) и ожидание ввода-вывода в него **не входят**.

```python
import time

start_wall = time.monotonic()
start_cpu = time.process_time()

time.sleep(1)            # спим 1 секунду — CPU простаивает
sum(range(5_000_000))    # а тут CPU работает

print("Реальное время:", time.monotonic() - start_wall)   # ~1.1 c
print("Время CPU:",       time.process_time() - start_cpu) # ~0.1 c (без сна!)
```

### `*_ns` варианты: точность в наносекундах

Функции с суффиксом `_ns` (`time_ns`, `perf_counter_ns`, `monotonic_ns`, `process_time_ns`) возвращают **целое число наносекунд** вместо `float` секунд. Они появились в Python 3.7.

Зачем они нужны: `float` (double) имеет 53 бита мантиссы. Для больших timestamp'ов (миллиарды секунд от эпохи) точность float падает до ~微секунд, и наносекундное разрешение теряется. Целочисленный `_ns` сохраняет полную точность.

```python
import time

print(time.time_ns())          # 1750000000123456789 (int, наносекунды)
print(time.perf_counter_ns())  # 123456789012 (int)

# Бенчмарк без потери точности на сложении float
t0 = time.perf_counter_ns()
work()
elapsed_ns = time.perf_counter_ns() - t0
print(f"{elapsed_ns / 1e6:.3f} мс")
```

### `struct_time` — разложенное время

`struct_time` — это именованный кортеж (`time.struct_time`), описывающий момент как набор полей: год, месяц, день, часы и т.д. Его возвращают `gmtime()` и `localtime()`, а принимают `mktime()`, `strftime()`, `asctime()`.

| Индекс | Атрибут | Значение |
|--------|---------|----------|
| 0 | `tm_year` | Год (например, 2026) |
| 1 | `tm_mon` | Месяц [1, 12] |
| 2 | `tm_mday` | День месяца [1, 31] |
| 3 | `tm_hour` | Часы [0, 23] |
| 4 | `tm_min` | Минуты [0, 59] |
| 5 | `tm_sec` | Секунды [0, 61] (60/61 для високосных секунд) |
| 6 | `tm_wday` | День недели [0, 6], понедельник = 0 |
| 7 | `tm_yday` | День года [1, 366] |
| 8 | `tm_isdst` | Флаг летнего времени: 1, 0 или -1 (неизвестно) |

```python
import time

t = time.localtime()
print(t.tm_year, t.tm_mon, t.tm_mday)   # доступ по имени
print(t[0], t[1], t[2])                  # или по индексу — это кортеж
```

### timezone, altzone, daylight, tzname

Модуль хранит информацию о настроенной в системе таймзоне:

- `time.timezone` — смещение локальной (не-DST) зоны от UTC в секундах, **на запад положительное**. Для Москвы (UTC+3) это `-10800`.
- `time.altzone` — то же, но для летнего времени (DST).
- `time.daylight` — ненулевое, если в таймзоне определено летнее время.
- `time.tzname` — кортеж из двух строк: имена зимней и летней зоны.

```python
import time

print(time.timezone)   # -10800 для МСК (UTC+3); знак инвертирован!
print(time.daylight)   # 0, если DST не используется (как в России с 2014)
print(time.tzname)     # ('MSK', 'MSK')
```

## Основные функции/классы/методы

### `time.time() -> float`

Возвращает текущее wall-clock время как Unix timestamp в секундах (UTC-эпоха). Самый частый способ получить «текущий момент» для логов, БД, API.

```python
import time
ts = time.time()
print(ts)  # 1750000000.123456
```

### `time.time_ns() -> int`

То же, но целое число наносекунд. Точнее на больших значениях.

```python
import time
print(time.time_ns())  # 1750000000123456789
```

### `time.sleep(secs)`

Приостанавливает текущий **поток** на `secs` секунд (можно дробное). Это не «занятое ожидание» — поток отдаёт управление ОС.

**Важно для собеседования:** `time.sleep()` **освобождает GIL**. Пока поток спит, другие потоки Python могут выполняться. Поэтому `sleep` в одном потоке не блокирует остальные. Реальное время сна может быть чуть больше запрошенного (планировщик ОС, точность таймера).

```python
import time
import threading

def worker(name):
    print(f"{name} стартовал")
    time.sleep(2)            # отдаёт GIL — другой поток работает параллельно
    print(f"{name} проснулся")

# Оба потока спят «одновременно», а не последовательно
t1 = threading.Thread(target=worker, args=("A",))
t2 = threading.Thread(target=worker, args=("B",))
t1.start(); t2.start()
t1.join();  t2.join()
# Суммарное время ~2 c, а не 4 c
```

### `time.perf_counter() -> float` и `time.perf_counter_ns() -> int`

Высокоточный таймер **специально для бенчмарков**. Включает время сна. Имеет наивысшее доступное разрешение. Монотонный (не идёт назад). Точка отсчёта не определена — осмысленна только разница.

```python
import time

start = time.perf_counter()
result = sum(i * i for i in range(1_000_000))
elapsed = time.perf_counter() - start
print(f"Заняло {elapsed:.6f} c")
```

Это рекомендованный выбор по умолчанию для измерения «сколько работал кусок кода».

### `time.monotonic() -> float` и `time.monotonic_ns() -> int`

Монотонные часы, которые **гарантированно не идут назад** даже при переводе системных часов или NTP-синхронизации. Используются для таймаутов, измерения интервалов, где критична надёжность.

```python
import time

deadline = time.monotonic() + 5.0   # дедлайн через 5 секунд
while time.monotonic() < deadline:
    # делаем работу; даже если NTP переведёт часы — цикл не сломается
    do_chunk()
```

Разница `perf_counter` vs `monotonic`: `perf_counter` обычно имеет более высокое разрешение и идеален для бенчмарков; `monotonic` — более «общий» инструмент для таймаутов. На многих платформах под капотом они близки.

### `time.process_time() -> float` и `time.process_time_ns() -> int`

Сумма системного и пользовательского **процессорного** времени текущего процесса. **Не включает** время сна и ожидания I/O. Полезно, когда нужно измерить именно нагрузку на CPU, отфильтровав ожидания.

```python
import time

t0 = time.process_time()
time.sleep(3)          # это НЕ попадёт в измерение
heavy_computation()    # а это попадёт
print("CPU time:", time.process_time() - t0)
```

Есть ещё `time.thread_time()` — то же, но для текущего потока.

### `time.gmtime([secs]) -> struct_time`

Преобразует timestamp в `struct_time` в **UTC**. Без аргумента — текущий момент.

```python
import time
print(time.gmtime(0))  # эпоха: 1970-01-01 00:00:00 UTC
```

### `time.localtime([secs]) -> struct_time`

То же, но в **локальной** таймзоне системы (с учётом DST).

```python
import time
lt = time.localtime()
print(lt.tm_hour)  # час по местному времени
```

### `time.mktime(struct_time) -> float`

Обратная к `localtime`: из `struct_time` (трактуется как **локальное** время) получает Unix timestamp. Учитывает локальную таймзону.

```python
import time
t = time.localtime()
ts = time.mktime(t)       # struct_time -> timestamp
print(ts)
```

Парная функция для UTC-`struct_time` — `calendar.timegm()` (в модуле `calendar`, не `time`).

### `time.strftime(format[, struct_time]) -> str`

Форматирует `struct_time` в строку по шаблону с директивами. Без второго аргумента берёт `localtime()`.

```python
import time
print(time.strftime("%Y-%m-%d %H:%M:%S", time.localtime()))
# 2026-06-20 14:30:00
```

### `time.strptime(string, format) -> struct_time`

Парсит строку в `struct_time` по шаблону. Бросает `ValueError`, если строка не совпала с форматом.

```python
import time
t = time.strptime("2026-06-20 14:30:00", "%Y-%m-%d %H:%M:%S")
print(t.tm_year, t.tm_mon)  # 2026 6
```

### Основные директивы форматирования

| Директива | Значение | Пример |
|-----------|----------|--------|
| `%Y` | Год, 4 цифры | 2026 |
| `%y` | Год, 2 цифры | 26 |
| `%m` | Месяц [01, 12] | 06 |
| `%d` | День месяца [01, 31] | 20 |
| `%H` | Часы, 24ч [00, 23] | 14 |
| `%I` | Часы, 12ч [01, 12] | 02 |
| `%M` | Минуты [00, 59] | 30 |
| `%S` | Секунды [00, 61] | 00 |
| `%p` | AM/PM | PM |
| `%j` | День года [001, 366] | 171 |
| `%a` / `%A` | День недели (кратко/полно) | Sat / Saturday |
| `%b` / `%B` | Месяц (кратко/полно) | Jun / June |
| `%w` | День недели как число [0=Вс, 6] | 6 |
| `%z` | Смещение UTC | +0300 |
| `%Z` | Имя таймзоны | MSK |
| `%%` | Литерал `%` | % |

```python
import time
t = time.localtime()
print(time.strftime("Сегодня %A, %d %B %Y, %H:%M", t))
```

### `time.asctime([struct_time]) -> str` и `time.ctime([secs]) -> str`

Готовое человекочитаемое представление в фиксированном формате `'Sat Jun 20 14:30:00 2026'`.

- `asctime` принимает `struct_time`.
- `ctime` принимает timestamp (по умолчанию текущий) — это сокращение для `asctime(localtime(secs))`.

```python
import time
print(time.ctime())                 # 'Sat Jun 20 14:30:00 2026'
print(time.asctime(time.gmtime()))  # то же, но из UTC struct_time
```

## Частые вопросы на собеседовании

**Вопрос:** Чем `time.time()` отличается от `time.monotonic()` и какой использовать для измерения длительности?

**Ответ:** `time.time()` — это wall-clock (стенные часы), реальное календарное время от эпохи. Оно может скакать вперёд и **назад** из-за NTP-синхронизации или ручного перевода часов. `time.monotonic()` гарантированно никогда не идёт назад. Для измерения длительности нужно использовать `monotonic()` или ещё лучше `perf_counter()` — иначе при синхронизации часов можно получить отрицательную или искажённую длительность. `time.time()` подходит только для меток времени (логи, БД), а не для измерений.

---

**Вопрос:** Что такое эпоха (epoch) и какое значение у неё в Python?

**Ответ:** Эпоха — точка отсчёта времени. В Python на основных платформах это 00:00:00 UTC 1 января 1970 года. Unix timestamp — число секунд от этого момента. Проверить можно через `time.gmtime(0)`, которое вернёт 1 января 1970. Важная деталь: Unix-время игнорирует високосные секунды.

---

**Вопрос:** В чём разница между `perf_counter()` и `process_time()`?

**Ответ:** `perf_counter()` измеряет **реальное (wall) время** с высоким разрешением, включая время сна и ожидания I/O. `process_time()` измеряет только **процессорное время** — время, когда CPU реально выполнял наш процесс; сон и I/O в него не входят. Если нужно понять «сколько по часам заняла операция» — `perf_counter`. Если «сколько CPU потратил на вычисления, без учёта ожиданий» — `process_time`.

```python
import time
t_real, t_cpu = time.perf_counter(), time.process_time()
time.sleep(1)
print(time.perf_counter() - t_real)    # ~1.0 (включает сон)
print(time.process_time() - t_cpu)     # ~0.0 (сон не считается)
```

---

**Вопрос:** Зачем нужны функции с суффиксом `_ns` (например, `time_ns`, `perf_counter_ns`)?

**Ответ:** Они возвращают целое число наносекунд вместо `float` секунд. У `float` (double) 53 бита мантиссы, и для больших timestamp'ов (миллиарды секунд от эпохи) разрешение падает до микросекунд — наносекундная точность теряется при делении/сложении. Целочисленные `_ns` сохраняют полную точность таймера и избегают ошибок округления float. Появились в Python 3.7.

---

**Вопрос:** Освобождает ли `time.sleep()` GIL? Что это значит на практике?

**Ответ:** Да, `time.sleep()` освобождает GIL на время сна. Это значит, что пока один поток спит, другие Python-потоки могут выполнять байткод. Поэтому если запустить несколько потоков, каждый из которых спит 2 секунды, суммарное время будет ~2 секунды, а не 2×N. Это делает `sleep` пригодным для реализации простого «дросселирования» в многопоточном коде, не блокируя остальные потоки.

---

**Вопрос:** Когда использовать `time`, а когда `datetime`?

**Ответ:** `time` — низкоуровневый модуль: метки времени (timestamps), измерение длительности (`perf_counter`, `monotonic`), задержки (`sleep`), взаимодействие с ОС. Он оперирует числами и `struct_time`. `datetime` — высокоуровневый: объекты `datetime`/`date`/`timedelta` для **арифметики дат** (прибавить день, найти разницу), работы с **таймзонами** (`tzinfo`, `zoneinfo`), удобного форматирования. Правило: для измерений и таймстампов — `time`; для манипуляций с календарными датами и таймзонами — `datetime`. Они interоперабельны: `datetime.fromtimestamp(time.time())`, `dt.timestamp()`.

---

**Вопрос:** `time.gmtime()` vs `time.localtime()` — в чём разница, и как из `struct_time` получить обратно timestamp?

**Ответ:** `gmtime()` даёт `struct_time` в UTC, `localtime()` — в локальной таймзоне системы (с учётом DST). Обратное преобразование: для локального `struct_time` — `time.mktime()`; для UTC `struct_time` — `calendar.timegm()` (важно: в `time` нет прямой обратной к `gmtime`, поэтому используют `calendar`).

---

**Вопрос:** Что вернёт `time.strptime("31/02/2026", "%d/%m/%Y")`?

**Ответ:** Бросит `ValueError`, потому что 31 февраля не существует — `strptime` валидирует дату. Вообще `strptime` кидает `ValueError`, если строка не соответствует формату или дата некорректна. Это важно оборачивать в try/except при парсинге пользовательского ввода.

---

**Вопрос:** Почему `time.timezone` для Москвы равен `-10800`, хотя Москва это UTC+3?

**Ответ:** Потому что `time.timezone` хранит смещение **на запад от UTC в секундах**, то есть знак инвертирован относительно привычного «UTC+3». Москва UTC+3 → смещение к западу −3 часа → `-10800` секунд. Это классическая ловушка. Также `time.altzone` хранит смещение для летнего времени, а `time.daylight` показывает, используется ли DST вообще.

---

**Вопрос:** Гарантирует ли `time.sleep(1)` ровно одну секунду сна?

**Ответ:** Нет. `sleep` гарантирует **минимум** запрошенное время, но реальная задержка может быть больше из-за планировщика ОС, разрешения системного таймера и нагрузки. Сон может быть прерван раньше сигналом (в этом случае поведение зависит от версии Python; в современных версиях `sleep` обычно досыпает остаток). Для точных таймаутов лучше проверять `monotonic()` в цикле.

---

**Вопрос:** Что такое поле `tm_isdst` в `struct_time` и зачем значение -1?

**Ответ:** `tm_isdst` — флаг летнего времени: `1` — действует DST, `0` — не действует, `-1` — неизвестно. Значение `-1` важно при передаче `struct_time` в `mktime()`: оно говорит библиотеке самой определить, было ли летнее время в этот момент. Если выставить флаг неверно, `mktime` может ошибиться на час около переходов на летнее/зимнее время.

---

**Вопрос:** Можно ли использовать `perf_counter()` для записи «когда произошло событие» (например, в лог с датой)?

**Ответ:** Нельзя. У `perf_counter()` и `monotonic()` точка отсчёта произвольна и не привязана к календарю — осмысленна только **разница** между двумя замерами. Для абсолютной метки времени нужен `time.time()` / `time.time_ns()` или `datetime`. Монотонные часы — только для интервалов.

## Подводные камни (gotchas)

```python
import time

# 1) НЕ измеряйте длительность через time() — он может пойти назад при NTP
start = time.time()
work()
bad = time.time() - start          # может быть отрицательным!
# Правильно:
start = time.perf_counter()
work()
good = time.perf_counter() - start
```

- **`time()` для бенчмарков** — частая ошибка. Wall-clock не монотонен. Используйте `perf_counter`/`monotonic`.
- **Абсолютное значение `perf_counter`/`monotonic` бессмысленно** — точка отсчёта произвольна. Только разница.
- **Знак `time.timezone`** инвертирован: к западу от UTC — положительный. UTC+3 даёт `-10800`.
- **`time.timezone` фиксируется при импорте** на основе переменной окружения `TZ`. Изменение `TZ` во время работы требует вызова `time.tzset()` (только на Unix), иначе значения устаревают.
- **Потеря точности float** на больших timestamp'ах: `time()` возвращает float, и наносекунды теряются. Для высокой точности — `time_ns()`.
- **`mktime` и `tm_isdst`**: неверный флаг летнего времени даёт ошибку на час около переходов DST.
- **`strptime` медленный** — он не кэширует разбор формата эффективно; в горячем цикле парсинг строк дат может стать узким местом.
- **`sleep` не гарантирует точность** — только минимальную задержку. Не стройте на нём точные таймеры реального времени.
- **`struct_time` неизменяем как кортеж**, но и не предназначен для арифметики — для расчётов переходите в `datetime`/`timedelta`.

```python
import time
# Ловушка с большими значениями float:
ns = time.time_ns()
sec = ns / 1e9
back_ns = int(sec * 1e9)
print(ns - back_ns)   # обычно != 0 — потеряли точность на float
```

## Лучшие практики

- **Для бенчмарков** используйте `time.perf_counter()` (или `perf_counter_ns()` для максимальной точности). Для повторяемых микробенчмарков — модуль `timeit`.
- **Для таймаутов и интервалов** в коде — `time.monotonic()`, чтобы перевод часов не ломал логику.
- **Для меток времени** (логи, БД, API) — `time.time()` / `time.time_ns()`, а для дальнейшей работы конвертируйте в `datetime` с таймзоной UTC.
- **Для измерения нагрузки на CPU** (без учёта ожиданий) — `time.process_time()`.
- **Не используйте `time.time()` для измерения длительности** — он не монотонен.
- **Работайте в UTC**: храните timestamps и `gmtime`-значения в UTC, конвертируйте в локальное время только для отображения.
- **Для арифметики дат и таймзон** переходите на `datetime` + `zoneinfo`, не пытайтесь считать вручную через `struct_time`.
- **Оборачивайте `strptime` в try/except `ValueError`** при разборе внешнего ввода.
- **Предпочитайте `_ns`-версии**, если измеряете очень короткие интервалы или работаете близко к разрешению таймера.

```python
import time
import datetime

# Хороший паттерн: метка времени в UTC через time -> datetime
ts = time.time()
dt_utc = datetime.datetime.fromtimestamp(ts, tz=datetime.timezone.utc)
print(dt_utc.isoformat())

# Хороший паттерн: контекстный таймер на perf_counter
from contextlib import contextmanager

@contextmanager
def timer(label):
    start = time.perf_counter()
    try:
        yield
    finally:
        print(f"{label}: {time.perf_counter() - start:.6f} c")

with timer("обработка"):
    sum(range(1_000_000))
```

## Шпаргалка

```python
import time

# --- Метки времени (timestamps) ---
time.time()              # float, секунды от эпохи (UTC, 1970-01-01)
time.time_ns()           # int, наносекунды (полная точность)

# --- Измерение длительности (бенчмарки) ---
time.perf_counter()      # float, высокоточный, монотонный, включает сон
time.perf_counter_ns()   # int, наносекунды
time.monotonic()         # float, монотонный, для таймаутов
time.monotonic_ns()      # int
time.process_time()      # float, только CPU-время (без сна/IO)
time.process_time_ns()   # int
time.thread_time()       # CPU-время текущего потока

# --- Задержка ---
time.sleep(1.5)          # спит >= 1.5 c; ОСВОБОЖДАЕТ GIL

# --- struct_time <-> timestamp ---
time.gmtime(ts)          # timestamp -> struct_time (UTC)
time.localtime(ts)       # timestamp -> struct_time (локальная TZ)
time.mktime(st)          # локальный struct_time -> timestamp
# calendar.timegm(st)    # UTC struct_time -> timestamp (модуль calendar)

# --- Форматирование / парсинг ---
time.strftime("%Y-%m-%d %H:%M:%S", time.localtime())   # struct_time -> str
time.strptime("2026-06-20", "%Y-%m-%d")                # str -> struct_time
time.ctime(ts)           # 'Sat Jun 20 14:30:00 2026'
time.asctime(st)         # то же из struct_time

# --- Таймзона (заполняется при импорте из TZ) ---
time.timezone            # смещение зимней TZ, секунды, к западу +  (UTC+3 -> -10800)
time.altzone             # смещение летней TZ
time.daylight            # != 0, если есть DST
time.tzname              # ('MSK', 'MSK')
# time.tzset()           # перечитать TZ (только Unix)

# --- Эпоха ---
time.gmtime(0)           # 1970-01-01 00:00:00 UTC

# Директивы strftime/strptime:
# %Y год(4)  %m месяц  %d день  %H час(24)  %M мин  %S сек
# %y год(2)  %I час(12) %p AM/PM  %j день года  %z смещение  %Z имя TZ
# %a/%A день недели  %b/%B месяц словом  %w номер дня недели  %% литерал %
```

**Правило выбора часов:**
- Метка времени для лога/БД → `time()` / `time_ns()` → лучше сразу в `datetime` (UTC).
- «Сколько заняло по часам» → `perf_counter()`.
- Таймаут, устойчивый к переводу часов → `monotonic()`.
- «Сколько CPU потратил» → `process_time()`.
- Арифметика дат, таймзоны → модуль `datetime` (+ `zoneinfo`), не `time`.
