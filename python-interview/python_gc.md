# GC (сборка мусора) — подготовка к собеседованию

## Что это и зачем

В Python управление памятью автоматическое: вы не вызываете `free()`/`delete`. За освобождение объектов отвечают два механизма, работающих вместе:

1. **Подсчёт ссылок (reference counting)** — основной механизм. У каждого объекта есть счётчик ссылок; когда он падает до нуля, объект немедленно уничтожается.
2. **Сборщик мусора циклических ссылок (cyclic garbage collector)** — модуль `gc`. Он нужен, потому что подсчёт ссылок **не умеет** освобождать объекты, замкнутые в цикл ссылок (A ссылается на B, B на A — счётчики никогда не дойдут до нуля).

Модуль `gc` даёт интерфейс именно к **циклическому** сборщику: его можно включать/выключать, запускать вручную, настраивать пороги, искать утечки.

Зачем это знать на собеседовании:
- Понимать, **почему** объекты освобождаются (и почему иногда нет).
- Уметь находить и предотвращать утечки памяти (циклы, кэши, `__del__`).
- Понимать влияние GC на производительность (паузы, когда отключать).
- Это фундаментальная тема: показывает глубину понимания внутренностей CPython.

Важная оговорка: всё ниже — про **CPython** (эталонную реализацию). В других реализациях (PyPy, Jython) механизм другой: например, PyPy не использует подсчёт ссылок, а полагается на полноценный трассирующий GC. Поэтому код, полагающийся на немедленное уничтожение по reference counting, непортируемый.

## Ключевые концепции

### 1. Reference counting (подсчёт ссылок)

У каждого объекта CPython в заголовке (`PyObject`) есть поле `ob_refcnt` — целое число, сколько ссылок на объект существует. Счётчик:

- **увеличивается** (+1), когда создаётся новая ссылка: присваивание переменной, добавление в список/словарь, передача аргументом в функцию, и т.д.;
- **уменьшается** (−1), когда ссылка исчезает: переменная выходит из области видимости, `del`, переприсваивание, удаление из контейнера.

Когда `ob_refcnt` достигает **нуля**, объект уничтожается **немедленно** (вызывается `__del__`, освобождается память).

```python
import sys

a = []                  # refcount = 1
print(sys.getrefcount(a))  # покажет на 1 больше, т.к. getrefcount сам создаёт временную ссылку (аргумент)
b = a                   # refcount = 2
del b                   # refcount = 1
a = None                # последняя ссылка ушла -> объект [] уничтожается немедленно
```

`sys.getrefcount(obj)` всегда показывает **на 1 больше**, чем «реально», потому что сам факт передачи `obj` аргументом в `getrefcount` создаёт временную ссылку.

**Плюсы подсчёта ссылок:**
- Детерминированность: объект освобождается сразу, как только перестал быть нужен (предсказуемо, без задержек).
- Память освобождается равномерно, без больших пауз.
- Простота реализации.

**Минусы:**
- Накладные расходы: при каждой операции со ссылкой надо обновлять счётчик (это, в частности, одна из причин, почему GIL тяжело убрать — счётчики надо защищать).
- **Не справляется с циклами.** Это главная проблема, ради которой существует модуль `gc`.
- Память под счётчик в каждом объекте.

### 2. Циклические ссылки — почему нужен GC

Рассмотрим цикл:

```python
a = {}
b = {}
a["b"] = b      # a ссылается на b
b["a"] = a      # b ссылается на a
del a           # внешняя ссылка на a убрана, но b["a"] всё ещё держит -> refcount(a) = 1
del b           # внешняя ссылка на b убрана, но a["b"] всё ещё держит -> refcount(b) = 1
# Оба объекта недостижимы из программы, но refcount у обоих = 1.
# Подсчёт ссылок их НИКОГДА не освободит -> это «мусорный цикл».
```

После `del a; del b` ни одна переменная не указывает на эти словари — программа до них уже не доберётся. Но их счётчики ссылок = 1 (они держат друг друга). Reference counting бессилен. Именно такие циклы вычищает **циклический сборщик** (`gc`).

Самый частый реальный пример — объект, ссылающийся сам на себя или на родителя:

```python
class Node:
    def __init__(self):
        self.parent = None
        self.children = []

root = Node()
child = Node()
root.children.append(child)
child.parent = root     # цикл: root <-> child
del root, child         # недостижимы, но держат друг друга -> работа для gc
```

### 3. Как работает циклический сборщик: mark-and-sweep + поколения

Циклический GC в CPython — это трассирующий сборщик, отслеживающий **только контейнерные объекты** (те, что могут ссылаться на другие: list, dict, set, instances пользовательских классов, кортежи с контейнерами и т.п.). Объекты, которые не могут участвовать в циклах (int, str, float), GC не отслеживает вовсе — их полностью обслуживает подсчёт ссылок.

Алгоритм упрощённо:
1. GC берёт множество отслеживаемых объектов.
2. Делает «пробную» копию счётчиков и вычитает внутренние ссылки (ссылки между объектами внутри множества). Если после этого у объекта эффективный счётчик > 0 — значит, на него есть ссылка извне (он достижим).
3. Объекты, достижимые от «извне-достижимых», помечаются как живые (mark).
4. Всё, что осталось непомеченным, — недостижимый мусор (в т.ч. циклы), он уничтожается (sweep).

#### Поколения (generations)

GC использует **поколенческую** стратегию, основанную на гипотезе: «большинство объектов умирает молодыми». Объекты делятся на 3 поколения (0, 1, 2):

- Новые объекты попадают в **поколение 0**.
- Если объект пережил сборку своего поколения — он «повышается» в следующее поколение.
- **Поколение 0** собирается часто (дёшево, мало объектов).
- **Поколение 1** — реже.
- **Поколение 2** (старое, долгоживущие объекты) — редко (дорого, много объектов).

Это балансирует «как часто проверяем» против «сколько объектов проверяем»: молодые проверяем часто и быстро, старые — редко.

#### Пороги (thresholds)

Сборка поколения 0 запускается, когда **(число аллокаций − число деаллокаций)** превышает `threshold0`. По умолчанию пороги `(700, 10, 10)`:

```python
import gc
gc.get_threshold()   # (700, 10, 10)
```

- `threshold0 = 700`: сборка поколения 0 после ~700 «чистых» новых аллокаций.
- `threshold1 = 10`: после 10 сборок поколения 0 запускается сборка поколения 1.
- `threshold2 = 10`: после 10 сборок поколения 1 запускается сборка поколения 2.

Можно настроить: `gc.set_threshold(1000, 15, 15)`. Установка `gc.set_threshold(0)` фактически отключает автоматическую сборку.

### 4. `__del__` (финализатор) и его опасности

`__del__` — метод-финализатор, вызывается при уничтожении объекта (когда refcount → 0 или при сборке цикла). Он **не** аналог деструктора C++ и его использование рискованно:

```python
class Resource:
    def __del__(self):
        print("освобождаю ресурс")

r = Resource()
del r            # печатает "освобождаю ресурс" — refcount упал до 0
```

Проблемы `__del__`:
- **Время вызова недетерминировано** для объектов в циклах (зависит от GC).
- До Python 3.4 объекты с `__del__`, попавшие в цикл, **вообще не собирались** (попадали в `gc.garbage`). После PEP 442 (Python 3.4+) это исправлено: GC умеет финализировать циклы с `__del__`, но порядок вызова `__del__` в цикле не определён.
- Исключения в `__del__` подавляются (печатаются в stderr, но не пробрасываются).
- `__del__` может «воскресить» объект (создать новую ссылку на себя) — источник трудных багов.

Поэтому для управления ресурсами используют **контекстные менеджеры** (`with`) или явный `.close()`, а не `__del__`.

### 5. `weakref` — слабые ссылки

Слабая ссылка (`weakref`) указывает на объект, **не увеличивая** его счётчик. Это главный инструмент для **разрыва циклов** и построения кэшей, которые не держат объекты:

```python
import weakref

class Node:
    def __init__(self):
        self.children = []
        self._parent = None

    @property
    def parent(self):
        return self._parent() if self._parent else None  # weakref вызывается как функция

    @parent.setter
    def parent(self, node):
        self._parent = weakref.ref(node) if node else None

root = Node()
child = Node()
root.children.append(child)
child.parent = root      # слабая ссылка вверх -> НЕТ сильного цикла

del root                 # child.parent теперь возвращает None, root освобождён сразу
```

Если объект, на который указывает слабая ссылка, уничтожен, `weakref` начинает возвращать `None` (или у `weakref.proxy` — бросать `ReferenceError`). `weakref.WeakValueDictionary`/`WeakKeyDictionary` — готовые кэши, не удерживающие объекты.

## Основные функции/классы/методы

```python
import gc

# --- Управление сборщиком ---
gc.enable()          # включить автоматическую циклическую сборку (по умолчанию вкл.)
gc.disable()         # выключить автоматическую сборку (reference counting продолжает работать!)
gc.isenabled()       # True/False

# --- Ручной запуск ---
gc.collect()         # запустить полную сборку (все поколения), вернёт число найденных недостижимых объектов
gc.collect(0)        # собрать только поколение 0
gc.collect(2)        # собрать поколения 0,1,2 (полная)

# --- Пороги поколений ---
gc.get_threshold()              # (700, 10, 10) по умолчанию
gc.set_threshold(1000, 20, 20)  # настроить
gc.get_count()                  # текущие счётчики (gen0, gen1, gen2) — сколько накопилось

# --- Интроспекция ---
gc.get_objects()           # список ВСЕХ отслеживаемых объектов (дорого!)
gc.get_referrers(obj)      # кто ссылается на obj (ищет «держателя» утечки)
gc.get_referents(obj)      # на кого ссылается obj
gc.get_stats()             # статистика по поколениям (число сборок, собранные объекты)
gc.is_tracked(obj)         # отслеживается ли объект циклическим GC

# --- Утечки и неубираемое ---
gc.garbage                 # список объектов, которые GC нашёл, но не смог освободить
gc.set_debug(gc.DEBUG_LEAK)  # включить отладочный режим (логирует утечки)

# --- Финализаторы ---
gc.freeze()                # «заморозить» текущие объекты (исключить из сборки) — оптимизация (3.7+)
```

### Пример: демонстрация работы с циклом

```python
import gc

class Node:
    def __init__(self, name):
        self.name = name
        self.ref = None
    def __del__(self):
        print(f"уничтожен {self.name}")

gc.disable()   # отключим авто-сборку, чтобы контролировать момент

a = Node("A")
b = Node("B")
a.ref = b
b.ref = a      # цикл A <-> B

del a, b       # __del__ НЕ вызывается: refcount остался 1 у каждого (цикл)
print("после del — объекты ещё живы")

n = gc.collect()   # запускаем циклический GC вручную
# теперь печатается "уничтожен A" / "уничтожен B" (порядок не определён)
print(f"собрано недостижимых объектов: {n}")
```

### Пример: поиск утечки через get_referrers

```python
import gc

leaked = SomeObject()
# ... подозреваем, что кто-то держит leaked, мешая освобождению

referrers = gc.get_referrers(leaked)
for r in referrers:
    print(type(r), id(r))   # смотрим, кто держит ссылку — список? словарь? кэш?
```

### Пример: измерение влияния GC на производительность

```python
import gc, time

def build_many():
    data = [{"i": i, "self": None} for i in range(2_000_000)]
    for d in data:
        d["self"] = d   # создаём циклы намеренно
    return data

# с включённым GC
start = time.perf_counter()
build_many()
print("GC on:", time.perf_counter() - start)

# с выключенным GC во время массового создания
gc.disable()
start = time.perf_counter()
build_many()
print("GC off:", time.perf_counter() - start)
gc.enable()
# Часто "GC off" заметно быстрее на массовых аллокациях — авто-сборка не дёргается
```

### Пример: weakref для кэша без удержания

```python
import weakref

class Image:
    def __init__(self, path):
        self.path = path

# WeakValueDictionary не мешает сборке значений
cache = weakref.WeakValueDictionary()

def load(path):
    img = cache.get(path)
    if img is None:
        img = Image(path)
        cache[path] = img   # храним слабо
    return img

img = load("a.png")
del img            # больше нет сильных ссылок -> объект освобождается, из кэша исчезает сам
```

## Частые вопросы на собеседовании

**Q1: Как в Python управляется память?**
A: Двумя механизмами. Основной — подсчёт ссылок (reference counting): объект уничтожается немедленно, когда счётчик ссылок падает до нуля. Дополнительный — циклический сборщик мусора (модуль `gc`), который вычищает объекты, замкнутые в циклы ссылок, недостижимые подсчётом ссылок.

**Q2: Что такое reference counting и когда объект уничтожается?**
A: У каждого объекта есть счётчик ссылок `ob_refcnt`. Он растёт при создании ссылки (присваивание, добавление в контейнер, передача аргументом) и падает при удалении ссылки (`del`, выход из области видимости, переприсваивание). Когда достигает нуля — объект уничтожается немедленно, детерминированно.

**Q3: Почему недостаточно подсчёта ссылок, зачем нужен GC?**
A: Подсчёт ссылок не освобождает циклические ссылки. Если A ссылается на B, а B на A, их счётчики никогда не дойдут до нуля, даже когда они недостижимы из программы. Циклический GC находит и удаляет такие недостижимые циклы.

**Q4: Как работает циклический сборщик?**
A: Это трассирующий mark-and-sweep сборщик для контейнерных объектов. Он временно вычитает внутренние (между отслеживаемыми объектами) ссылки из счётчиков; объекты с оставшимся положительным счётчиком достижимы извне и помечаются живыми вместе со всем, до чего дотягиваются; непомеченное — мусор и удаляется. Работает по поколениям.

**Q5: Что такое поколения и зачем они?**
A: GC делит объекты на 3 поколения (0, 1, 2). Новые — в поколении 0; пережившие сборку повышаются. Поколение 0 собирается часто и дёшево, поколение 2 — редко. Основано на гипотезе, что большинство объектов умирает молодыми, что снижает накладные расходы.

**Q6: Что такое пороги (700, 10, 10)?**
A: 700 — сборка поколения 0 запускается, когда (аллокации − деаллокации) превысит 700. 10 — после 10 сборок поколения 0 запускается поколение 1; ещё 10 — поколение 2. Настраиваются через `gc.set_threshold`.

**Q7: Что делает `gc.disable()` — отключает всю сборку мусора?**
A: Нет. Отключается только **циклический** сборщик. Подсчёт ссылок продолжает работать, и объекты без циклов освобождаются как обычно. Отключают `gc` обычно для оптимизации (убрать паузы при массовых аллокациях), но рискуют накопить циклический мусор.

**Q8: Когда полезно отключать GC?**
A: При массовом создании множества долгоживущих объектов (загрузка большого датасета, прогрев кэша) — авто-сборка иначе многократно проверяет растущую кучу, тратя время. Также в latency-критичных секциях, чтобы избежать паузы. Важно потом включить обратно или вызвать `gc.collect()` вручную.

**Q9: Что такое утечка памяти в Python и как она возникает?**
A: Объекты, которые не освобождаются, хотя уже не нужны. Причины: непреднамеренное удержание ссылок (глобальные списки/кэши, растущие без очистки), циклы с объектами, имеющими `__del__` (до 3.4), замыкания/обработчики событий, держащие большие объекты, незакрытые ресурсы. Цикл сам по себе не утечка (GC его уберёт), но мешает детерминированному освобождению.

**Q10: Чем опасен `__del__`?**
A: Время его вызова непредсказуемо для объектов в циклах; исключения в нём подавляются; он может «воскресить» объект; до Python 3.4 циклы с `__del__` не собирались вовсе (уходили в `gc.garbage`). Поэтому для ресурсов используют `with`/контекстные менеджеры или явный `.close()`, а не `__del__`.

**Q11: Как разорвать цикл ссылок, чтобы не зависеть от GC?**
A: Использовать слабые ссылки (`weakref`). Слабая ссылка не увеличивает счётчик, поэтому, например, ссылка «ребёнок → родитель» через `weakref` не создаёт сильного цикла, и объекты освобождаются подсчётом ссылок сразу. Также есть `WeakValueDictionary`/`WeakKeyDictionary` для кэшей.

**Q12: Какие объекты отслеживает циклический GC?**
A: Только контейнерные — те, что могут ссылаться на другие объекты: list, dict, set, экземпляры пользовательских классов, кортежи с контейнерами и т.п. Простые неизменяемые объекты (int, str, float) не могут участвовать в циклах, GC их не отслеживает; их полностью обслуживает подсчёт ссылок. Проверить: `gc.is_tracked(obj)`.

**Q13: Как найти, кто держит объект (мешает освобождению)?**
A: `gc.get_referrers(obj)` показывает объекты, ссылающиеся на `obj`; `gc.get_referents(obj)` — на кого ссылается сам `obj`. Также помогают `tracemalloc` (трекинг аллокаций), `objgraph` (визуализация ссылок), `gc.set_debug(gc.DEBUG_LEAK)`.

**Q14: Почему `sys.getrefcount(x)` возвращает «лишнюю» единицу?**
A: Передача `x` аргументом в функцию `getrefcount` сама создаёт временную ссылку, поэтому результат всегда на 1 больше реального числа удерживающих ссылок.

**Q15: GC в CPython и PyPy — одинаковы?**
A: Нет. CPython использует подсчёт ссылок + циклический сборщик. PyPy не использует подсчёт ссылок, а полагается на полноценный трассирующий GC. Поэтому код, рассчитывающий на немедленное уничтожение объекта по reference counting (например, закрытие файла «само собой»), непортируем.

## Подводные камни (gotchas)

1. **Расчёт на немедленное уничтожение.** `open(...).read()` без явного закрытия «работает» в CPython, потому что временный файловый объект уничтожается по refcount сразу. В PyPy/Jython файл может остаться открытым надолго. Всегда `with open(...)`.

2. **Цикл с `__del__` (исторически).** До Python 3.4 такой цикл не собирался и копился в `gc.garbage`. Сейчас собирается, но порядок вызова `__del__` в цикле не определён — не полагайтесь на него.

3. **`gc.disable()` без последующей очистки.** Отключив авто-сборку и создавая циклы, можно накопить большой объём недостижимого мусора. Включайте обратно или вызывайте `gc.collect()` вручную.

4. **Глобальные кэши/списки как источник «утечек».** Это не утечка в смысле GC (есть ссылка → объект жив правомерно), но память растёт. Решение — ограничить размер (`functools.lru_cache(maxsize=...)`), использовать `weakref`-словари, чистить вручную.

5. **Замыкания держат больше, чем кажется.** Лямбда/вложенная функция, захватившая `self` или большой объект, продлевает его жизнь. Частая причина утечек в колбэках/обработчиках событий — забытая отписка.

6. **`sys.getrefcount` показывает +1.** Не пугайтесь «лишней» ссылки — это аргумент функции.

7. **Исключения держат traceback, traceback держит локальные переменные.** Сохранённый объект исключения (или `sys.exc_info()`) удерживает весь стек кадров и все локальные переменные в них — может неожиданно продлить жизнь крупным объектам. Не храните исключения дольше необходимого.

8. **GC может вносить паузы.** На приложениях с огромной кучей долгоживущих объектов сборка поколения 2 бывает заметной. Лечится `gc.freeze()` (заморозить стартовые объекты), настройкой порогов или временным отключением.

9. **`gc.get_objects()` дорогой и сам влияет на состояние.** Это полный обход всех отслеживаемых объектов — не вызывайте в горячем пути.

10. **Воскрешение в `__del__`.** Если `__del__` создаст новую ссылку на `self`, объект «оживёт» — `__del__` больше не вызовется повторно при следующем обнулении (в CPython). Источник трудноуловимых багов.

11. **fork и GC.** При `os.fork()` (например, в multiprocessing на Linux) дочерний процесс наследует объекты; GC, трогая счётчики ссылок, провоцирует copy-on-write копирование страниц памяти. Поэтому в pre-fork серверах иногда вызывают `gc.freeze()` перед форком, чтобы стабильные объекты не копировались.

## Лучшие практики

- **Управляйте ресурсами через `with`/`contextlib`, а не `__del__`.** Файлы, сокеты, блокировки — закрывайте детерминированно.
- **Избегайте сильных циклов там, где можно.** Для обратных ссылок (child → parent, наблюдатель → субъект) используйте `weakref`.
- **Ограничивайте кэши.** `functools.lru_cache(maxsize=N)`, `WeakValueDictionary`, ручная очистка — чтобы память не росла бесконечно.
- **Не злоупотребляйте `gc.disable()`.** Отключайте GC только осознанно (массовая инициализация, latency-критичный участок) и обязательно включайте обратно/вызывайте `collect()`.
- **Для пакетной загрузки больших данных** временно отключите GC, загрузите, включите и вызовите `gc.collect()` один раз — это часто ускоряет на десятки процентов.
- **Ищите утечки инструментами:** `tracemalloc` (стандартный, показывает, где аллоцировано), `objgraph` (графы ссылок), `gc.get_referrers`, `gc.set_debug(gc.DEBUG_LEAK)`.
- **Не храните исключения и traceback дольше нужного** — они держат кадры стека и локальные переменные.
- **Не полагайтесь на порядок и время вызова `__del__`**, особенно для объектов в циклах.
- **В pre-fork серверах** рассмотрите `gc.freeze()` перед форком воркеров — снижает copy-on-write нагрузку.
- **Помните о переносимости:** код не должен зависеть от немедленного уничтожения по refcount (PyPy ведёт себя иначе).

## Шпаргалка

```python
import gc, sys, weakref

# === Reference counting ===
sys.getrefcount(obj)        # текущий счётчик ссылок (+1 от аргумента!)
# refcount растёт: присваивание, добавление в контейнер, аргумент функции
# refcount падает: del, выход из scope, переприсваивание, удаление из контейнера
# refcount == 0  -> объект уничтожается НЕМЕДЛЕННО (вызывается __del__)

# === Циклический GC (только для циклов!) ===
gc.collect()                # ручной запуск, возвращает число собранных объектов
gc.collect(0|1|2)           # собрать конкретные поколения
gc.disable() / gc.enable()  # выкл/вкл только ЦИКЛИЧЕСКИЙ сборщик (refcount работает всегда)
gc.isenabled()

# === Поколения и пороги ===
gc.get_threshold()          # (700, 10, 10) по умолчанию
gc.set_threshold(700,10,10)
gc.get_count()              # текущие счётчики поколений
gc.get_stats()              # статистика сборок
# Идея: gen0 часто/дёшево, gen2 редко/дорого ("объекты умирают молодыми")

# === Интроспекция / поиск утечек ===
gc.get_referrers(obj)       # КТО держит obj (ищем утечку)
gc.get_referents(obj)       # на кого ссылается obj
gc.is_tracked(obj)          # отслеживается ли GC (int/str — нет; list/dict/instance — да)
gc.get_objects()            # все отслеживаемые объекты (ДОРОГО)
gc.garbage                  # неубираемый мусор
gc.set_debug(gc.DEBUG_LEAK)
# + стандартный tracemalloc, сторонние objgraph

# === Разрыв циклов ===
weakref.ref(obj)            # слабая ссылка (НЕ увеличивает refcount), вызывается как r()
weakref.proxy(obj)          # прокси (ReferenceError после смерти)
weakref.WeakValueDictionary()  # кэш, не удерживающий значения
weakref.WeakKeyDictionary()

# === Главное, что нужно сказать на собеседовании ===
# 1) Основной механизм — подсчёт ссылок (детерминированно, refcount==0 -> уничтожение).
# 2) Циклический GC нужен ТОЛЬКО для недостижимых циклов ссылок.
# 3) Поколения (0,1,2) + пороги (700,10,10) для эффективности.
# 4) Ресурсы — через with, а не __del__.
# 5) Циклы разрывать weakref; кэши ограничивать.
# 6) Это про CPython; PyPy без refcount (непортируемо полагаться на немедленное уничтожение).
```
