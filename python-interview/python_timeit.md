# timeit — подготовка к собеседованию

## Что это и зачем

`timeit` — встроенный модуль для точного измерения времени выполнения небольших фрагментов кода (микробенчмаркинг). Он автоматически устраняет распространённые ошибки наивного замера через `time.time()`: учитывает накладные расходы, отключает сборщик мусора, выполняет код много раз и выбирает лучший результат.

Зачем нужен:
- **Сравнить варианты реализации.** Что быстрее: list comprehension или `map`, конкатенация строк через `+` или `join`.
- **Объективность.** Один запуск ненадёжен из-за шума ОС; timeit усредняет/повторяет.
- **Минимум кода.** Замер в одну строку из CLI или одной функцией из Python.

Важно: timeit измеряет *wall-clock* (реальное прошедшее время) самым точным доступным таймером (`time.perf_counter`), а не процессорное время.

## Ключевые концепции

- **stmt** — измеряемый код (строка или вызываемый объект).
- **setup** — код подготовки, выполняемый *один раз* перед серией замеров и НЕ входящий в измеряемое время (импорты, создание данных).
- **number** — сколько раз выполнить `stmt` в одном «прогоне» (одном замере). Время суммируется по всем number итерациям.
- **repeat** — сколько раз повторить весь замер (получить серию результатов).
- **Минимум, а не среднее.** Для серии берут *минимум*, потому что внешние помехи (ОС, другой процесс) только *замедляют*, а не ускоряют; минимум — наименее зашумлённая оценка «чистого» времени.
- **Отключение GC.** На время замера сборщик мусора по умолчанию выключается, чтобы его срабатывания не искажали результат.

## Основные функции/классы/методы

### Командная строка

```bash
# Базовый замер: timeit сам подберёт number автоматически
python -m timeit "sum(range(100))"

# setup через -s (выполняется один раз, не измеряется)
python -m timeit -s "data = list(range(1000))" "sum(data)"

# Явное число итераций -n и повторов -r
python -m timeit -n 1000 -r 5 "'-'.join(str(i) for i in range(50))"

# Несколько строк setup — повторить -s
python -m timeit -s "import math" -s "x = 2.0" "math.sqrt(x)"
```

Вывод примерно такой:
```
500000 loops, best of 5: 412 nsec per loop
```
`loops` — подобранное number, `best of 5` — минимум из repeat, далее время на одну итерацию.

Если не указать `-n`, timeit сам увеличивает number (1, 10, 100, ...), пока суммарное время прогона не станет достаточно большим для надёжности.

### Программный интерфейс: timeit.timeit

```python
import timeit

# stmt и setup как строки. Возвращает СУММАРНОЕ время number итераций (в секундах)
t = timeit.timeit(
    stmt="'-'.join(str(i) for i in range(100))",
    number=10000,
)
print(t)                          # например 0.85 (секунд на 10000 прогонов)
print(t / 10000)                  # время на одну итерацию

# setup для импортов/данных:
t = timeit.timeit(
    stmt="sorted(data)",
    setup="import random; data = [random.random() for _ in range(1000)]",
    number=1000,
)
```

### Замер вызываемых объектов (без строк)

Строки удобны для простого кода, но для замера готовых функций лучше передавать callable:

```python
import timeit


def build_with_join():
    return "-".join(str(i) for i in range(100))


def build_with_concat():
    s = ""
    for i in range(100):
        s += str(i) + "-"
    return s


# stmt — вызываемый объект; setup не нужен, имена берутся из замыкания
t_join = timeit.timeit(build_with_join, number=10000)
t_concat = timeit.timeit(build_with_concat, number=10000)
print(f"join:   {t_join:.4f}")
print(f"concat: {t_concat:.4f}")
```

Если измеряемый код использует имена из текущего модуля при строковом `stmt`, передайте `globals`:

```python
import timeit

def my_func():
    return sum(range(100))

# globals=globals() даёт доступ к my_func из строки stmt
t = timeit.timeit("my_func()", globals=globals(), number=10000)
```

### repeat — серия замеров и минимум

```python
import timeit

# Возвращает СПИСОК из repeat значений (каждое — суммарное время number итераций)
results = timeit.repeat(
    stmt="sum(range(1000))",
    number=10000,
    repeat=5,
)
print(results)                      # [0.21, 0.20, 0.22, 0.20, 0.21]

# Правильная метрика — минимум (наименее зашумлённый), делённый на number
best_per_iter = min(results) / 10000
print(f"{best_per_iter * 1e6:.3f} мкс на итерацию")
```

### Класс Timer

`timeit.timeit`/`timeit.repeat` — обёртки над классом `Timer`. Прямое использование даёт больше контроля и метод автоподбора number:

```python
import timeit

timer = timeit.Timer(
    stmt="sorted(data)",
    setup="data = [3, 1, 2] * 100",
)

# autorange: сам подбирает number, чтобы прогон длился >= 0.2 c
loops, total = timer.autorange()
print(f"подобрано {loops} итераций, суммарно {total:.4f} c")

# repeat возвращает серию
series = timer.repeat(repeat=5, number=loops)
print(min(series) / loops)
```

### Удобный замер в Jupyter / IPython

```python
# Магические команды (не входят в стандартный timeit, но используют ту же логику):
# %timeit sum(range(1000))            -> один statement
# %%timeit                            -> целая ячейка
#   data = list(range(1000))
#   sorted(data)
```

`%timeit` сам подбирает number и repeat и печатает среднее ± стандартное отклонение.

## Частые вопросы на собеседовании

**Q1. Почему timeit лучше, чем замер через time.time()?**
`time.time()` имеет низкое разрешение и подвержен скачкам системных часов; единичный замер зашумлён планировщиком ОС. timeit использует высокоточный `perf_counter`, выполняет код много раз (`number`), повторяет серию (`repeat`), отключает GC и берёт минимум — это даёт стабильную, воспроизводимую оценку.

**Q2. Чем отличаются number и repeat?**
`number` — сколько раз выполнить stmt внутри *одного* замера (время суммируется). `repeat` — сколько раз повторить весь замер, получив серию результатов. Итого выполнений = number × repeat. number борется с разрешением таймера, repeat — со случайным шумом между прогонами.

**Q3. Почему берут минимум, а не среднее по repeat?**
Внешние помехи (другой процесс, прерывания ОС, GC) могут только *замедлить* код, но не ускорить. Поэтому минимум серии — наиболее «чистая» оценка времени самого кода; среднее и максимум загрязнены шумом окружения.

**Q4. Что попадает в setup и почему он не измеряется?**
В `setup` кладут подготовку: импорты, создание тестовых данных. Он выполняется один раз перед серией и НЕ входит в измеряемое время, чтобы замерять именно целевую операцию, а не разовую подготовку. Например, генерацию случайного списка для сортировки.

**Q5. Отключает ли timeit сборщик мусора и зачем?**
Да, по умолчанию GC выключается на время замера, чтобы его срабатывания не искажали результат недетерминированными паузами. Если важно учитывать GC (реалистичный сценарий с активным выделением), его можно вернуть: `setup="gc.enable()"` (с `import gc`).

**Q6. Как замерить функцию, а не строку кода?**
Передать сам callable: `timeit.timeit(my_func, number=...)`. Для строкового stmt, использующего внешние имена, нужно передать `globals=globals()`, иначе timeit не найдёт функцию (он исполняет код в изолированном пространстве имён).

**Q7. Что возвращают timeit() и repeat()?**
`timeit()` возвращает *суммарное* время всех `number` итераций (в секундах), поэтому для времени на одну операцию делят на number. `repeat()` возвращает *список* таких суммарных времён длиной `repeat`.

**Q8. Что такое autorange и когда он полезен?**
`Timer.autorange()` автоматически подбирает `number` так, чтобы прогон длился достаточно долго (>= 0.2 c) для надёжности, и возвращает `(loops, total_time)`. Это то, что делает CLI без `-n`. Полезно, когда не хочется вручную угадывать number для очень быстрого/медленного кода.

**Q9. Какие главные подводные камни микробенчмаркинга?**
Слишком короткий код (тонет в накладных расходах и разрешении таймера), кеш-эффекты и прогрев, влияние других процессов, нерепрезентативные данные, измерение не той части (включили setup в stmt), экстраполяция микробенчмарка на реальную систему. Микро-выигрыш часто незначим на фоне I/O.

**Q10. timeit или cProfile — что когда?**
`timeit` — точное измерение *времени* небольшого фрагмента и сравнение альтернатив. `cProfile` — *профилирование* программы: где именно тратится время по функциям, кто кого вызывает. timeit отвечает «сколько», cProfile — «где». Их используют на разных этапах оптимизации.

## Подводные камни (gotchas)

- **Слишком быстрый код.** Если операция занимает наносекунды, накладные расходы цикла и разрешение таймера доминируют. Увеличивайте `number` (или используйте autorange) и сравнивайте относительные, а не абсолютные значения.
- **setup попал в измерение.** Создание данных внутри `stmt` (а не `setup`) исказит результат — будете мерить и подготовку. Всё, что не должно измеряться, — в setup.
- **Имена не видны.** Строковый stmt исполняется в изолированном namespace: внешние функции/переменные не видны без `setup="from __main__ import ..."` или `globals=globals()`.
- **GC отключён по умолчанию.** Результат может быть оптимистичнее реального, где GC работает. Для реалистичности включайте GC в setup.
- **Нерепрезентативные данные.** Сортировка уже отсортированного списка, словарь с одним ключом, пустые коллекции — не отражают прод. Готовьте данные, похожие на боевые.
- **Кеш и прогрев.** Первые прогоны могут быть медленнее (импорт, кеш процессора). timeit частично сглаживает это повторами, но для JIT/кеширующего кода учитывайте прогрев.
- **Сравнение в разных условиях.** Замеры на нагруженной машине, с разным питанием (ноутбук на батарее throttle-ит CPU), в разных интерпретаторах несравнимы. Сравнивайте в идентичных условиях за один сеанс.
- **Микрооптимизация без профиля.** Ускорять код, который не является узким местом, — пустая трата. Сначала профиль (cProfile), потом timeit на горячем участке.
- **Среднее вместо минимума.** Брать `mean(repeat())` менее корректно, чем `min(...)`, из-за односторонней природы шума.

## Лучшие практики

- Готовьте данные и импорты в `setup`, измеряйте в `stmt` только целевую операцию.
- Для серьёзных сравнений используйте `repeat>=5` и берите `min(...) / number`.
- Предпочитайте передачу callable + `globals=globals()` строковым stmt со скрытыми зависимостями.
- Сравнивайте альтернативы в одном запуске, на одинаковых данных и при стабильной нагрузке системы.
- Используйте `autorange`/CLI-автоподбор number вместо ручного угадывания.
- Для реалистичных сценариев включайте GC, если он влияет на нагрузку.
- Не делайте выводов из микро-разницы (наносекунды) — она часто незначима против I/O и реальной нагрузки; профилируйте всю систему.
- Документируйте условия замера (версия Python, машина), чтобы результаты были воспроизводимы.

## Шпаргалка

```bash
# CLI
python -m timeit "code"                          # автоподбор number
python -m timeit -s "setup" "code"               # с подготовкой
python -m timeit -n 1000 -r 5 "code"             # number=1000, repeat=5
python -m timeit -s "import math" "math.sqrt(2)" # импорт в setup
# Вывод: "N loops, best of R: T per loop"
```

```python
import timeit

# Суммарное время number итераций (секунды)
timeit.timeit(stmt="code", setup="setup", number=10000)

# Серия из repeat значений -> берём минимум
res = timeit.repeat(stmt="code", number=10000, repeat=5)
best = min(res) / 10000          # время на одну итерацию

# Callable вместо строки
timeit.timeit(my_func, number=10000)
timeit.timeit("my_func()", globals=globals(), number=10000)

# Класс Timer + autorange
t = timeit.Timer("code", setup="setup")
loops, total = t.autorange()
series = t.repeat(repeat=5, number=loops)
```

```
Памятка:
  number  — повторов внутри одного замера (борьба с разрешением таймера)
  repeat  — число замеров в серии (борьба с шумом)
  метрика — min(series) / number
  setup   — НЕ измеряется (импорты, данные)
  GC      — отключён по умолчанию
  Jupyter — %timeit  /  %%timeit
```
