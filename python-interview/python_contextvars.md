# contextvars — подготовка к собеседованию

## Что это и зачем

`contextvars` — это стандартный модуль Python (добавлен в **Python 3.7**, [PEP 567](https://peps.python.org/pep-0567/)), который предоставляет механизм хранения данных, **локальных для текущего контекста выполнения**. Главная его задача — переносить «неявный» контекст (например, идентификатор запроса, текущего пользователя, locale) сквозь стек вызовов, **не пробрасывая его явно через аргументы каждой функции**, и при этом корректно работать как с потоками, так и с **корутинами и тасками `asyncio`**.

Зачем он нужен:

- **Неявный сквозной контекст.** Часто нужно «дотащить» какое-то значение из верхнего уровня (middleware HTTP-сервера) до самого низа (форматтер логов), при этом промежуточные функции про это значение ничего не знают и не должны знать. Прокидывать `request_id` параметром через 20 функций — это шум и протечка абстракций.
- **Корректность в асинхронном коде.** Старое решение — `threading.local` — привязывает данные к **потоку ОС**. В `asyncio` тысячи корутин выполняются в **одном** потоке, чередуясь на каждом `await`. Поэтому `threading.local` не способен изолировать данные между конкурентными корутинами — они все «видят» одно и то же состояние потока. `contextvars` решает это: контекст логически связан с конкретной задачей/цепочкой выполнения, а не с потоком ОС.
- **Изоляция между задачами.** Каждая `asyncio.Task` запускается в **копии** текущего контекста. Изменения переменных внутри задачи **не протекают** в родительский контекст и в соседние задачи.

Главная мысль для собеседования: *contextvars — это «правильный thread-local для эпохи async/await», работающий на уровне логического контекста выполнения, а не потока ОС.*

## Ключевые концепции

1. **ContextVar** — объект-«переменная контекста». Создаётся **один раз на уровне модуля** (как глобальная константа), а его значение зависит от текущего контекста. У него есть имя (для отладки) и опциональное значение по умолчанию.

2. **Context (контекст)** — это, по сути, неизменяемое отображение `ContextVar -> значение`. В любой момент времени выполнения есть «текущий контекст». Все операции `get`/`set` работают именно с ним.

3. **get() / set() / reset()** — чтение, запись и откат значения переменной в текущем контексте.

4. **Token** — объект, который возвращает `set()`. Он хранит предыдущее значение переменной и позволяет позже сделать `reset(token)`, вернув всё ровно как было. Это как «undo».

5. **copy_context()** — создаёт **снимок (копию)** текущего контекста. Копия — поверхностная по значениям (shallow): копируется отображение «переменная → значение», но не сами объекты-значения.

6. **Context.run(callable, ...)** — запускает функцию **внутри указанного контекста**. Все изменения переменных, сделанные внутри `run`, остаются в этом контексте и не протекают наружу.

7. **default (значение по умолчанию)** — если переменная не была установлена в текущем контексте, `get()` вернёт `default` (заданный либо при создании `ContextVar`, либо переданный в сам `get()`). Если default нигде не задан — `get()` бросит `LookupError`.

8. **Наследование контекста задачами.** `asyncio.create_task()` (а также `loop.call_soon` и т. п.) автоматически делают `copy_context()` и запускают корутину/колбэк в этой копии. Поэтому задачи **наследуют** значения на момент создания, но изменения внутри них **изолированы**.

9. **Отличие от threading.local.** `threading.local` — состояние на поток ОС. `contextvars` — состояние на логический контекст выполнения (которое умеет правильно копироваться/изолироваться для тасок). В многопоточном коде contextvars тоже работает: каждый поток стартует со своим (пустым/дефолтным) контекстом.

## Основные функции/классы/методы

### `ContextVar` — создание и значение по умолчанию

```python
import contextvars

# Переменную контекста принято объявлять ОДИН РАЗ на уровне модуля,
# как глобальную "константу". Имя ('request_id') используется только для отладки/repr.

# Вариант 1: без значения по умолчанию
request_id_var: contextvars.ContextVar[str] = contextvars.ContextVar("request_id")

# Вариант 2: со значением по умолчанию
locale_var: contextvars.ContextVar[str] = contextvars.ContextVar("locale", default="en_US")

# get() с дефолтом, заданным при создании -> вернёт 'en_US'
print(locale_var.get())  # en_US

# get() без установленного значения и без дефолта -> LookupError
try:
    request_id_var.get()
except LookupError as e:
    print("LookupError:", e)  # переменная не установлена и нет default

# get() можно передать "запасной" дефолт прямо в вызове.
# Он имеет приоритет над отсутствием значения (но переменную НЕ устанавливает).
print(request_id_var.get("<no-request-id>"))  # <no-request-id>
```

Важно: `ContextVar` **нельзя** создавать «на лету» внутри функции при каждом вызове — это создаст новые объекты и сломает логику. Объявляйте их как глобальные.

### `set()` и `Token`

```python
import contextvars

user_var: contextvars.ContextVar[str] = contextvars.ContextVar("user")

# set() возвращает Token, который хранит предыдущее состояние переменной
token = user_var.set("alice")
print(user_var.get())  # alice

# Token знает старое значение. Если значения не было — это "missing"-маркер.
print(token.old_value)  # MISSING (специальный маркер contextvars.Token.MISSING)
print(token.var is user_var)  # True — токен помнит, к какой переменной относится
```

### `reset()` — откат к предыдущему значению

```python
import contextvars

user_var: contextvars.ContextVar[str] = contextvars.ContextVar("user")

token = user_var.set("alice")
print(user_var.get())  # alice

token2 = user_var.set("bob")
print(user_var.get())  # bob

# reset(token2) возвращает значение к тому, что было ДО set("bob") -> "alice"
user_var.reset(token2)
print(user_var.get())  # alice

# reset(token) возвращает к состоянию ДО первого set -> переменная снова "не установлена"
user_var.reset(token)
try:
    user_var.get()
except LookupError:
    print("снова не установлена")  # печатается

# Идиома "временно установить значение" через контекстный менеджер:
from contextlib import contextmanager

@contextmanager
def use_user(name: str):
    token = user_var.set(name)
    try:
        yield
    finally:
        user_var.reset(token)  # гарантированный откат, даже при исключении

with use_user("charlie"):
    print(user_var.get())  # charlie
# здесь user снова не установлен
```

Нюанс: `reset(token)` можно вызвать **только в том же контексте**, где был сделан `set()`, и только один раз для конкретного токена. Иначе будет `ValueError`/`RuntimeError`.

### `copy_context()` и `Context.run()`

```python
import contextvars

var: contextvars.ContextVar[int] = contextvars.ContextVar("var", default=0)


def increment_and_show():
    # Внутри run() меняем переменную
    var.set(var.get() + 100)
    print("внутри run:", var.get())


var.set(1)
print("до:", var.get())  # 1

# copy_context() делает СНИМОК текущего контекста
ctx = contextvars.copy_context()

# Context.run() выполняет функцию ВНУТРИ копии контекста.
# Все set() внутри остаются в ctx и НЕ протекают наружу.
ctx.run(increment_and_show)  # внутри run: 101

print("после:", var.get())  # 1 — снаружи значение не изменилось!

# Изменение "застряло" внутри ctx. Можно убедиться, перебрав контекст:
for ctx_var, value in ctx.items():
    if ctx_var is var:
        print("в копии контекста:", value)  # 101
```

`Context.run()` синхронен и не реентерабелен: нельзя запустить уже исполняющийся контекст повторно (рекурсивно) — будет `RuntimeError`.

### Контекст в `asyncio`: наследование и изоляция задач

```python
import asyncio
import contextvars

request_id_var: contextvars.ContextVar[str] = contextvars.ContextVar("request_id", default="-")


async def worker(name: str):
    # Каждая задача стартует с КОПИЕЙ контекста родителя на момент create_task().
    # Поэтому здесь видно значение, которое было установлено ДО создания задачи.
    print(f"[{name}] унаследовал request_id =", request_id_var.get())

    # set() внутри задачи меняет ТОЛЬКО её собственную копию контекста.
    request_id_var.set(f"req-from-{name}")
    await asyncio.sleep(0.1)
    print(f"[{name}] после set видит:", request_id_var.get())  # своё значение


async def main():
    request_id_var.set("REQ-ROOT")

    # Обе задачи создаются, когда request_id == "REQ-ROOT" -> обе его наследуют.
    t1 = asyncio.create_task(worker("A"))
    t2 = asyncio.create_task(worker("B"))
    await asyncio.gather(t1, t2)

    # КЛЮЧЕВОЕ: set() внутри задач НЕ протёк наружу.
    print("[main] после задач request_id =", request_id_var.get())  # REQ-ROOT


asyncio.run(main())
# Вывод (порядок A/B может меняться):
# [A] унаследовал request_id = REQ-ROOT
# [B] унаследовал request_id = REQ-ROOT
# [A] после set видит: req-from-A
# [B] после set видит: req-from-B
# [main] после задач request_id = REQ-ROOT
```

### `Context.run()` для асинхронного запуска в нужном контексте

```python
import asyncio
import contextvars

var: contextvars.ContextVar[str] = contextvars.ContextVar("var", default="default")


async def show():
    print("корутина видит:", var.get())


async def main():
    var.set("главный-контекст")

    # Снимем контекст, изменим в нём переменную и запустим задачу в этом контексте.
    ctx = contextvars.copy_context()
    ctx.run(var.set, "контекст-для-задачи")

    # create_task принимает context=... (Python 3.11+),
    # чтобы запустить задачу в заранее подготовленном контексте.
    task = asyncio.create_task(show(), context=ctx)
    await task  # корутина видит: контекст-для-задачи

    print("main видит:", var.get())  # главный-контекст


asyncio.run(main())
```

## Частые вопросы на собеседовании

**Q1. Что такое `contextvars` и какую проблему он решает?**
A. Это модуль для хранения данных, локальных для текущего контекста выполнения. Решает задачу неявной передачи «сквозного» контекста (request id, текущий пользователь, locale) через стек вызовов без явных аргументов, и делает это **корректно в asyncio**, где много корутин живут в одном потоке. По сути это «async-aware замена `threading.local`».

**Q2. В чём принципиальное отличие `contextvars` от `threading.local`?**
A. `threading.local` хранит данные **на поток ОС**: разные потоки видят разные значения, но всё, что выполняется в одном потоке, делит общее состояние. `contextvars` хранит данные на **логический контекст выполнения**, который умеет копироваться и изолироваться для каждой `asyncio.Task`. В асинхронном коде тысячи корутин крутятся в одном потоке, поэтому `threading.local` не может их разделить, а `contextvars` — может. При этом в многопоточной программе `contextvars` тоже работает: каждый поток стартует со своим контекстом.

**Q3. Почему `threading.local` ломается в `asyncio`? Приведи пример.**
A. Потому что переключение между корутинами (`await`) происходит **внутри одного потока**, а `threading.local` различает только потоки. Две конкурентные корутины делят одно thread-local состояние и затирают данные друг друга:

```python
import asyncio
import threading

_local = threading.local()


async def handle(req_id: str):
    _local.request_id = req_id      # пишем "своё" значение
    await asyncio.sleep(0.1)        # тут управление уходит другой корутине,
                                    # которая перезапишет _local.request_id
    # после возврата мы видим ЧУЖОЙ request_id — данные перемешались
    print(f"ожидал {req_id}, а в thread-local: {_local.request_id}")


async def main():
    await asyncio.gather(handle("REQ-1"), handle("REQ-2"))


asyncio.run(main())
# Вывод (значения перепутаны, потому что поток один):
# ожидал REQ-1, а в thread-local: REQ-2
# ожидал REQ-2, а в thread-local: REQ-2
```

С `contextvars` каждая задача имеет свою копию контекста, и значения не перемешиваются.

**Q4. Зачем `set()` возвращает `Token` и что делает `reset()`?**
A. `Token` хранит предыдущее значение переменной (или маркер `MISSING`, если значения не было). `reset(token)` восстанавливает это предыдущее состояние — то есть «откатывает» один конкретный `set()`. Это важно для middleware/контекстных менеджеров: установили значение на время обработки, в `finally` откатили, чтобы не «загрязнять» контекст. `reset` нужно вызывать в том же контексте, где был `set`, и один раз на токен.

**Q5. Как `asyncio.create_task()` работает с контекстом? Протекают ли изменения наружу?**
A. `create_task()` делает `copy_context()` и запускает корутину в **копии** текущего контекста. Поэтому задача **наследует** значения переменных на момент её создания. Но любые `set()` внутри задачи меняют только её копию и **не протекают** ни в родителя, ни в соседние задачи. Можно явно передать готовый контекст через `create_task(coro, context=ctx)` (Python 3.11+).

**Q6. Что делает `copy_context()` — глубокую или поверхностную копию?**
A. Поверхностную (shallow): копируется отображение `ContextVar -> значение`, но не сами объекты-значения. Если значение — изменяемый объект (список, dict), то и оригинал, и копия ссылаются на **один и тот же** объект, и мутация (`list.append`) будет видна в обоих. Изолировано только присваивание (`var.set(...)`), а не мутация содержимого.

**Q7. Что вернёт `get()`, если переменная не установлена?**
A. Зависит от наличия дефолта. Приоритет: (1) `default`, переданный в сам `get(default)`; (2) `default`, заданный при создании `ContextVar(name, default=...)`; (3) если нигде дефолта нет — бросается `LookupError`. Маркер «значение не установлено» — это не `None`; `None` может быть валидным установленным значением.

**Q8. Где объявлять `ContextVar` — внутри функции или на уровне модуля?**
A. На уровне модуля, как глобальный объект, один раз. Создание `ContextVar` внутри функции при каждом вызове породит новые объекты-переменные, и значения «потеряются»: каждая такая переменная — отдельная сущность. Это распространённая ошибка.

**Q9. Что такое `Context.run()` и чем он отличается от прямого вызова функции?**
A. `Context.run(fn, *args)` выполняет `fn` **внутри указанного контекста**: все изменения переменных остаются в этом контексте и не протекают в текущий. Прямой вызов меняет текущий контекст. `run()` нельзя вызывать рекурсивно для уже исполняющегося контекста (будет `RuntimeError`), и он синхронный (для корутин нужен другой механизм, например `create_task(..., context=ctx)`).

**Q10. Наследуют ли контекст обычные функции в `run_in_executor` / потоки?**
A. Новый поток (`threading.Thread`) **не** наследует контекст автоматически — он стартует со своим пустым/дефолтным контекстом. Если нужно перенести контекст в поток, передайте его явно: снимите `ctx = copy_context()` и выполните в потоке `ctx.run(func)`. Для `loop.run_in_executor` стандартный контекст также не переносится автоматически — оборачивайте вызов через `functools.partial` + `ctx.run`.

**Q11. Можно ли использовать `contextvars` как замену передачи параметров везде?**
A. Не стоит злоупотреблять. Это «неявное» состояние, и при избытке оно усложняет отладку (откуда взялось значение?). Хорошие кейсы: сквозные технические данные — request/correlation id, текущий пользователь/tenant, locale, трассировка. Бизнес-данные лучше передавать явно через аргументы.

**Q12. Как реализовать correlation id в логах через `contextvars`?**
A. Объявить `ContextVar`, выставлять его в middleware на входящий запрос, а в логировании — через `logging.Filter`/адаптер подмешивать значение в каждую запись (см. раздел ниже). Так каждая строка лога автоматически получает id запроса без явной передачи в каждый `logger.info`.

**Q13. Что произойдёт со значением переменной после завершения задачи/контекста?**
A. Контекст задачи живёт вместе с задачей. После её завершения её копия контекста отбрасывается, и установленные внутри значения нигде «не сохраняются» (они и так не протекали наружу). В родительском контексте всё остаётся как было до создания задачи.

**Q14. Безопасны ли `ContextVar` для конкурентного доступа из разных потоков?**
A. Сам объект `ContextVar` — глобальный и общий, но **значения** изолированы по контекстам, а у каждого потока свой текущий контекст. Поэтому потоки не «дерутся» за одно значение. Операции `get/set/reset` не требуют ручных блокировок для корректной изоляции по контекстам. (Это не значит, что разделяемый изменяемый объект-значение защищён — за его потокобезопасность отвечаете вы.)

## Подводные камни (gotchas)

- **`get()` без дефолта бросает `LookupError`, а не возвращает `None`.** Всегда либо задавайте `default`, либо явно передавайте дефолт в `get()`, либо ловите `LookupError`.

- **`ContextVar`, созданный внутри функции, — это новый объект каждый раз.** Объявляйте переменные на уровне модуля.

- **`copy_context()` поверхностна.** Мутация изменяемого значения (`mylist.append(x)`) видна во всех контекстах, ссылающихся на тот же объект. Изолируется только переприсваивание через `set()`.

- **`threading.Thread` не наследует контекст.** В новом потоке контекст пуст/дефолтен. Переносите явно через `copy_context().run(...)`.

- **`reset(token)` привязан к контексту и одноразов.** Нельзя сделать `reset` чужим токеном или в другом контексте, нельзя дважды — будет ошибка. Используйте паттерн `try/finally` или контекстный менеджер.

- **Изменения внутри задачи не видны снаружи.** Если вы рассчитывали, что задача «вернёт» значение через `ContextVar` в родителя, — этого не произойдёт. Возвращайте значения нормально (через результат корутины).

- **Долгоживущая задача и «устаревший» контекст.** Задача видит снимок контекста на момент `create_task()`. Если родитель потом поменял переменную — задача этого не увидит. Это by design, но иногда удивляет.

- **Голые корутины без `create_task` делят контекст вызывающего.** Если вы просто делаете `await some_coro()` (без оборачивания в Task), корутина выполняется в **текущем** контексте, и её `set()` **протекут** в вызывающий код. Изоляция появляется именно у задач, а не у любой корутины.

- **Не использовать как глобальный mutable-стейт «по лени».** Избыток неявного контекста превращает код в трудноотлаживаемый. Применяйте точечно.

## Лучшие практики

- **Объявляйте `ContextVar` глобально, на уровне модуля, с говорящим именем** (имя видно в `repr`, помогает в отладке).

- **Задавайте `default`, когда «не установлено» — валидное состояние.** Это избавляет от `LookupError` и упрощает чтение.

- **Откатывайте `set()` через `reset(token)` в `finally`** или используйте контекстный менеджер. Особенно в middleware и в коде, который вызывается многократно в одном контексте.

- **Используйте `contextvars` для технического сквозного контекста**: correlation/request id, текущий пользователь/tenant, locale, данные трассировки. Бизнес-параметры передавайте явно.

- **Для переноса контекста в потоки/executor используйте `copy_context().run(...)`** (или `functools.partial`), не полагайтесь на автоматику.

- **Интегрируйте с логированием через `logging.Filter`/адаптер**, чтобы каждая запись автоматически получала correlation id (см. ниже).

- **Помните про shallow-копирование.** Если кладёте в `ContextVar` изменяемый объект и хотите изоляции — кладите неизменяемые значения или создавайте новую копию при изменении.

### Реалистичный пример: correlation id в логах (asyncio-веб-сервис)

```python
import asyncio
import logging
import uuid
import contextvars
from contextlib import contextmanager

# 1) Глобальная переменная контекста для correlation id запроса.
correlation_id_var: contextvars.ContextVar[str] = contextvars.ContextVar(
    "correlation_id", default="-"
)


# 2) Фильтр логирования: подмешивает correlation_id в КАЖДУЮ запись.
class CorrelationIdFilter(logging.Filter):
    def filter(self, record: logging.LogRecord) -> bool:
        # get() безопасен: есть default="-"
        record.correlation_id = correlation_id_var.get()
        return True


# Настройка логгера с форматом, использующим correlation_id
handler = logging.StreamHandler()
handler.setFormatter(
    logging.Formatter("%(asctime)s [cid=%(correlation_id)s] %(levelname)s %(message)s")
)
handler.addFilter(CorrelationIdFilter())

logger = logging.getLogger("app")
logger.setLevel(logging.INFO)
logger.addHandler(handler)
logger.propagate = False


# 3) Контекстный менеджер: устанавливает cid на время обработки и откатывает.
@contextmanager
def correlation_id(value: str):
    token = correlation_id_var.set(value)
    try:
        yield value
    finally:
        correlation_id_var.reset(token)  # гарантированный откат


# 4) "Глубокая" бизнес-логика НЕ знает про correlation id —
#    он подмешивается в логи автоматически.
async def fetch_user(user_id: int) -> dict:
    logger.info("Загружаю пользователя %s", user_id)
    await asyncio.sleep(0.05)
    return {"id": user_id, "name": "Alice"}


async def handle_request(user_id: int):
    # Имитация middleware: на каждый запрос — свой correlation id.
    cid = uuid.uuid4().hex[:8]
    with correlation_id(cid):
        logger.info("Начало обработки запроса")
        user = await fetch_user(user_id)
        logger.info("Готово, пользователь=%s", user["name"])


async def main():
    # Несколько конкурентных запросов — у каждого СВОЙ cid в логах,
    # потому что каждая задача работает в своей копии контекста.
    await asyncio.gather(
        asyncio.create_task(handle_request(1)),
        asyncio.create_task(handle_request(2)),
        asyncio.create_task(handle_request(3)),
    )


asyncio.run(main())
# В логах строки разных запросов имеют РАЗНЫЕ cid и не перемешиваются:
# ... [cid=ab12cd34] INFO Начало обработки запроса
# ... [cid=ef56gh78] INFO Начало обработки запроса
# ... [cid=ab12cd34] INFO Загружаю пользователя 1
# ...
```

## Шпаргалка

```python
import contextvars

# --- Создание (на уровне модуля!) ---
var = contextvars.ContextVar("var")                 # без дефолта
var2 = contextvars.ContextVar("var2", default=0)    # с дефолтом

# --- Чтение ---
var2.get()            # 0 (дефолт)
var.get("fallback")   # явный дефолт в вызове
# var.get()           # LookupError, если не установлено и нет дефолта

# --- Запись / откат ---
token = var.set("значение")   # возвращает Token
var.reset(token)              # откат к предыдущему состоянию
token.old_value               # прежнее значение или Token.MISSING
token.var                     # переменная, к которой относится токен

# --- Снимок и запуск в контексте ---
ctx = contextvars.copy_context()   # поверхностная копия текущего контекста
ctx.run(some_func, *args)          # выполнить some_func внутри ctx (изменения не протекают)
for v, value in ctx.items():       # перебор переменных контекста
    ...
ctx[var2]                          # доступ к значению (KeyError если нет)
var2 in ctx                        # проверка наличия

# --- asyncio ---
import asyncio
task = asyncio.create_task(coro())             # копирует текущий контекст
task = asyncio.create_task(coro(), context=ctx) # явный контекст (Python 3.11+)
```

| Свойство | `threading.local` | `contextvars.ContextVar` |
|---|---|---|
| Привязка | к потоку ОС | к логическому контексту выполнения |
| Работает в asyncio | нет (корутины делят поток) | да (изоляция по задачам) |
| Наследование в дочернюю «единицу» | нет | да (`create_task` копирует контекст) |
| Откат значения | вручную | через `Token` + `reset()` |
| Перенос в новый поток | нет | через `copy_context().run()` |

**Запомнить главное:**
- `ContextVar` объявляем глобально, читаем `get()`, пишем `set()` (получаем `Token`), откатываем `reset(token)`.
- `copy_context()` + `Context.run()` — изолированное выполнение; копия поверхностная.
- `asyncio.create_task()` копирует контекст: задачи наследуют значения, но их `set()` не протекает наружу.
- `contextvars` — это «async-aware thread-local»: работает там, где `threading.local` ломается между корутинами.
```
