# queue — подготовка к собеседованию

## Что это и зачем

`queue` — это модуль стандартной библиотеки Python, реализующий **потокобезопасные (thread-safe) очереди**. Главное назначение — безопасный обмен данными между несколькими потоками (threads), которые работают параллельно.

Когда у вас есть несколько потоков, которые одновременно пишут и читают данные из общей структуры, обычные коллекции (`list`, `dict`) приводят к **состоянию гонки (race condition)**: два потока могут одновременно прочитать/записать элемент и испортить данные. Чтобы этого избежать, нужна синхронизация через блокировки (`Lock`, `Condition`). Модуль `queue` уже инкапсулирует всю эту синхронизацию внутри себя — вы просто вызываете `put()` и `get()`, не думая о блокировках.

Типичные сценарии использования:

- **Producer-Consumer (производитель-потребитель)** — одни потоки кладут задачи в очередь, другие их разбирают.
- **Пул воркеров (worker pool)** — N потоков обрабатывают поток заданий.
- **Конвейер (pipeline)** — данные проходят несколько стадий обработки, каждая стадия — отдельные потоки, соединённые очередями.
- **Буферизация** — сглаживание неравномерной нагрузки между быстрым производителем и медленным потребителем.

В модуле четыре класса очередей:

| Класс | Порядок выдачи | Описание |
|-------|----------------|----------|
| `Queue` | FIFO | First In, First Out — кто первый пришёл, тот первый вышел |
| `LifoQueue` | LIFO | Last In, First Out — как стек |
| `PriorityQueue` | по приоритету | сначала выдаётся наименьший элемент |
| `SimpleQueue` | FIFO | упрощённая, без `maxsize` и `task_done()`, появилась в Python 3.7 |

> **Важно:** `queue.Queue` предназначена **только для потоков (threading), а не для процессов (multiprocessing)**. Для процессов есть отдельный класс `multiprocessing.Queue`. Подробности — ниже.

## Ключевые концепции

### Потокобезопасность (thread-safety)

Все методы `Queue` (`put`, `get`, `task_done`, `join`) атомарны относительно других потоков. Внутри используется `threading.Lock` и условные переменные `threading.Condition`. Это значит, что можно безопасно вызывать `put`/`get` из любого количества потоков одновременно — данные не потеряются и не задвоятся.

### Блокирующие операции, block и timeout

`put` и `get` по умолчанию **блокирующие**:

- `get(block=True, timeout=None)` — если очередь пуста, поток засыпает и ждёт, пока появится элемент.
- `put(item, block=True, timeout=None)` — если очередь полна (достигнут `maxsize`), поток ждёт, пока освободится место.

Параметр `timeout` ограничивает время ожидания: если за `timeout` секунд ничего не произошло — выбрасывается исключение (`Empty` для `get`, `Full` для `put`).

### maxsize и backpressure (обратное давление)

`Queue(maxsize=N)` ограничивает размер очереди. Когда очередь заполнена, `put` блокируется. Это механизм **backpressure**: медленный потребитель «придерживает» быстрого производителя, не давая памяти разрастись бесконечно. Если `maxsize <= 0` (по умолчанию `0`) — очередь не ограничена.

### task_done() и join()

Эти два метода работают в паре и реализуют **отслеживание завершённости задач**:

- Внутри очереди есть счётчик незавершённых задач. Каждый `put` увеличивает его на 1.
- `task_done()` уменьшает счётчик на 1 — потребитель сигнализирует «я обработал один элемент».
- `join()` блокирует поток, пока счётчик не станет равным 0 — то есть пока все задачи, положенные в очередь, не будут обработаны.

Это позволяет главному потоку дождаться полной обработки всех задач, не зная заранее, сколько времени это займёт.

### Sentinel и graceful shutdown (корректное завершение)

Воркеры обычно крутятся в бесконечном цикле `while True: q.get()`. Чтобы их корректно остановить, в очередь кладут специальный маркер-«отравленную таблетку» (poison pill / sentinel), обычно `None`. Получив `None`, воркер понимает: «работа окончена» — и завершает цикл.

## Основные функции/классы/методы

### Queue — базовая FIFO-очередь

```python
import queue

# Создаём очередь с ограничением в 10 элементов
q = queue.Queue(maxsize=10)

# Кладём элементы (FIFO — First In, First Out)
q.put("первый")
q.put("второй")

# Забираем — в том же порядке, в котором клали
print(q.get())  # "первый"
print(q.get())  # "второй"

# Размер и проверки состояния
print(q.qsize())   # приблизительный размер (см. gotchas)
print(q.empty())   # True, если пусто (не надёжно в многопоточности!)
print(q.full())    # True, если достигнут maxsize
```

### put / get с block и timeout

```python
import queue

q = queue.Queue(maxsize=2)

# Блокирующий put с таймаутом: ждём не более 1 секунды свободного места
q.put("a", block=True, timeout=1)
q.put("b", block=True, timeout=1)

try:
    # Очередь полна — ждём 1 сек, потом исключение Full
    q.put("c", block=True, timeout=1)
except queue.Full:
    print("Очередь переполнена, не смогли положить за 1 секунду")

# Блокирующий get с таймаутом
try:
    while True:
        item = q.get(timeout=0.5)  # ждём элемент не более 0.5 сек
        print("Получили:", item)
        q.task_done()
except queue.Empty:
    print("Очередь пуста, элементов больше нет")
```

### put_nowait / get_nowait — неблокирующие варианты

```python
import queue

q = queue.Queue(maxsize=1)

# put_nowait(x) — это эквивалент put(x, block=False)
q.put_nowait("единственный")

try:
    q.put_nowait("лишний")  # места нет — сразу исключение, без ожидания
except queue.Full:
    print("Full: места нет, ждать не стали")

# get_nowait() — эквивалент get(block=False)
print(q.get_nowait())  # "единственный"

try:
    q.get_nowait()  # пусто — сразу исключение
except queue.Empty:
    print("Empty: очередь пуста, ждать не стали")
```

### LifoQueue — стек (LIFO)

```python
import queue

stack = queue.LifoQueue()
stack.put(1)
stack.put(2)
stack.put(3)

# Last In, First Out — последний положенный выходит первым
print(stack.get())  # 3
print(stack.get())  # 2
print(stack.get())  # 1
```

### PriorityQueue — очередь с приоритетами

Сначала выдаётся **наименьший** элемент (по сравнению `<`). Элементы должны быть сравнимы между собой.

```python
import queue

pq = queue.PriorityQueue()

# Кладём кортежи (приоритет, значение). Меньший приоритет — выше в очереди.
pq.put((3, "низкий приоритет"))
pq.put((1, "высокий приоритет"))
pq.put((2, "средний приоритет"))

print(pq.get())  # (1, 'высокий приоритет')
print(pq.get())  # (2, 'средний приоритет')
print(pq.get())  # (3, 'низкий приоритет')
```

#### Приём (priority, counter, item) — защита от сравнения несравнимых объектов

Проблема: если у двух элементов **одинаковый приоритет**, Python начнёт сравнивать вторые элементы кортежа. Если это объекты, которые не поддерживают `<` (например, два словаря или два экземпляра класса без `__lt__`), вы получите `TypeError`.

```python
import queue

pq = queue.PriorityQueue()

# Плохо: при равном приоритете Python попытается сравнить словари -> TypeError
# pq.put((1, {"task": "A"}))
# pq.put((1, {"task": "B"}))  # TypeError: '<' not supported between dicts
```

Решение — вставить между приоритетом и объектом **монотонно растущий счётчик (counter)**. Счётчики всегда уникальны, поэтому до сравнения самого объекта дело никогда не дойдёт. Бонус: при равном приоритете сохраняется FIFO-порядок (что положили раньше — выйдет раньше).

```python
import queue
import itertools

pq = queue.PriorityQueue()
counter = itertools.count()  # бесконечный счётчик 0, 1, 2, ...

def put_task(priority, item):
    # (приоритет, уникальный счётчик, сам объект)
    # Если приоритеты равны — сравнятся счётчики (всегда разные),
    # до сравнения item дело не дойдёт.
    pq.put((priority, next(counter), item))

# Объекты, которые НЕ умеют сравниваться через "<"
put_task(1, {"task": "A"})
put_task(1, {"task": "B"})  # тот же приоритет — но TypeError не возникнет
put_task(0, {"task": "срочное"})

while not pq.empty():
    priority, _, item = pq.get()
    print(priority, item)
# 0 {'task': 'срочное'}
# 1 {'task': 'A'}
# 1 {'task': 'B'}  -> A раньше B, т.к. counter у A меньше
```

Альтернатива — использовать `dataclass` с `field(compare=False)`:

```python
from dataclasses import dataclass, field
from typing import Any
import queue

@dataclass(order=True)
class PrioritizedItem:
    priority: int
    # item исключён из сравнения — сравниваются только priority
    item: Any = field(compare=False)

pq = queue.PriorityQueue()
pq.put(PrioritizedItem(1, {"task": "A"}))
pq.put(PrioritizedItem(1, {"task": "B"}))  # TypeError не будет
pq.put(PrioritizedItem(0, {"task": "срочное"}))
```

### SimpleQueue — упрощённая очередь (Python 3.7+)

`SimpleQueue` — это облегчённая FIFO-очередь без `maxsize`, без `task_done()`/`join()` и без методов `qsize`/`empty` гарантий. Зато она быстрее и `put()` у неё **никогда не блокируется** (нет ограничения размера). Безопасна даже для вызова из обработчиков сигналов и деструкторов `__del__`.

```python
import queue

sq = queue.SimpleQueue()
sq.put("a")
sq.put("b")
print(sq.get())  # "a"
print(sq.get())  # "b"
# sq.task_done()  # AttributeError — такого метода нет
```

### task_done() и join()

```python
import queue
import threading

q = queue.Queue()

def worker():
    while True:
        item = q.get()         # ждём задачу
        print(f"Обрабатываю {item}")
        q.task_done()          # ОБЯЗАТЕЛЬНО сигналим о завершении

threading.Thread(target=worker, daemon=True).start()

for i in range(5):
    q.put(i)                   # счётчик незавершённых задач = 5

q.join()                       # блокируемся, пока счётчик не станет 0
print("Все задачи обработаны")
```

## Частые вопросы на собеседовании

**Q1. Чем `queue.Queue` отличается от `collections.deque`?**

`collections.deque` — это структура данных (двусторонняя очередь) с быстрыми операциями `append`/`popleft`. Её отдельные методы (`append`, `popleft`, `appendleft`, `pop`) атомарны благодаря GIL, поэтому `deque` можно использовать как простую потокобезопасную очередь. Но `deque` **не умеет блокировать** поток при ожидании элемента: если очередь пуста, `popleft()` сразу бросает `IndexError`. Нет ни `timeout`, ни `task_done/join`, ни ограничения `maxsize` с блокировкой производителя. `queue.Queue` — это полноценный инструмент межпоточной коммуникации поверх `deque` плюс блокировки, условные переменные, backpressure и отслеживание задач. Правило: для координации потоков берите `queue.Queue`, для локальной структуры данных — `deque`.

**Q2. Почему `queue.Queue` нельзя использовать между процессами?**

`queue.Queue` хранит данные в обычной памяти процесса и синхронизируется через `threading.Lock` / `Condition`. При создании нового процесса (через `multiprocessing` или `fork`) у дочернего процесса своя, отдельная копия памяти. Объект очереди скопируется, но это будет **две независимые очереди** — изменения в одной не видны в другой. Кроме того, потоковые блокировки не работают между процессами. Для межпроцессного обмена нужен `multiprocessing.Queue`, который под капотом использует пайпы (pipe) и сериализацию (pickle) для передачи данных через границу процессов.

**Q3. В чём разница между `multiprocessing.Queue` и `queue.Queue`?**

`queue.Queue` — для потоков, данные лежат в общей памяти, передаются по ссылке без копирования. `multiprocessing.Queue` — для процессов, данные **сериализуются через pickle**, передаются через пайп и десериализуются на другой стороне. Отсюда следствия: (1) объекты в `multiprocessing.Queue` должны быть picklable; (2) есть накладные расходы на сериализацию; (3) у `multiprocessing.Queue` нет методов `task_done()`/`join()` (для этого есть `JoinableQueue`); (4) `qsize()` может бросать `NotImplementedError` на некоторых платформах (macOS).

**Q4. Что делают `task_done()` и `join()` и зачем они нужны?**

Очередь ведёт счётчик незавершённых задач. `put()` увеличивает счётчик. `task_done()` уменьшает его — потребитель сообщает «эту задачу я доделал». `join()` блокирует вызывающий поток, пока счётчик не обнулится. Это позволяет дождаться завершения **всех** задач, не считая их вручную и не зная время обработки. Важно: число вызовов `task_done()` должно строго соответствовать числу `put()`, иначе либо `join()` зависнет навсегда, либо вылетит `ValueError` (если `task_done` вызвали лишний раз).

**Q5. Как корректно остановить воркеры (graceful shutdown)?**

Воркеры обычно крутятся в `while True: q.get()`. Чтобы их остановить, применяют паттерн **sentinel / poison pill**: после всех реальных задач в очередь кладут N маркеров `None` (по одному на каждый воркер). Получив `None`, воркер делает `task_done()` (или сначала кладёт sentinel обратно для соседей) и выходит из цикла через `break`. Альтернатива — `threading.Event` как флаг остановки в сочетании с `get(timeout=...)`. Нельзя просто «убить» поток — корректное завершение делается только сигналом.

**Q6. Что произойдёт при `get()` на пустой очереди и при `put()` на полной?**

По умолчанию (`block=True`, `timeout=None`) оба метода **блокируют поток бесконечно**: `get()` ждёт появления элемента, `put()` — освобождения места (при заданном `maxsize`). Если задать `timeout`, то по истечении времени `get()` бросит `queue.Empty`, а `put()` — `queue.Full`. Неблокирующие варианты `get_nowait()` / `put_nowait()` (то же, что `block=False`) бросают эти исключения немедленно, не дожидаясь.

**Q7. Надёжны ли `qsize()`, `empty()`, `full()` в многопоточном коде?**

Нет. Эти методы возвращают состояние «на момент вызова», но между проверкой и последующим действием другой поток может изменить очередь (классическая TOCTOU-гонка — Time Of Check, Time Of Use). Например, `if not q.empty(): q.get()` небезопасно: после `empty()` другой поток может забрать последний элемент, и ваш `get()` заблокируется или бросит исключение. Правильный подход — просто вызывать `get(timeout=...)` и ловить `Empty`, а не проверять `empty()` заранее.

**Q8. Как реализовать backpressure (обратное давление)?**

Задать `maxsize` при создании очереди: `Queue(maxsize=100)`. Когда очередь заполнится, блокирующий `put()` приостановит производителя, пока потребители не разгребут накопившееся. Так память не разрастается бесконечно, а скорость производителя автоматически подстраивается под скорость потребителя. Без `maxsize` (значение 0) быстрый производитель при медленном потребителе может исчерпать всю память.

**Q9. Как работает `PriorityQueue` и как избежать `TypeError` при равных приоритетах?**

`PriorityQueue` хранит элементы в куче (heap) и выдаёт наименьший по оператору `<`. Если кладёте кортежи `(priority, item)` и у двух элементов равный `priority`, Python начнёт сравнивать `item`. Если `item` не поддерживает сравнение (словарь, произвольный объект) — будет `TypeError`. Решение: вставить уникальный монотонный счётчик — `(priority, next(counter), item)`. Счётчики всегда различны, поэтому до сравнения `item` дело не дойдёт; заодно при равных приоритетах сохраняется FIFO. Альтернатива — `dataclass(order=True)` с `field(compare=False)` на поле `item`.

**Q10. В чём разница между `Queue` и `SimpleQueue`?**

`SimpleQueue` (Python 3.7+) — упрощённая неограниченная FIFO-очередь. У неё нет `maxsize`, нет `task_done()`/`join()`, она быстрее и её `put()` никогда не блокируется. Она безопасна для вызова из reentrant-контекстов (обработчики сигналов, `__del__`). `Queue` — полнофункциональная: поддерживает `maxsize`, backpressure, отслеживание задач через `task_done/join`. Берите `SimpleQueue`, когда нужна простая быстрая передача без учёта завершённости и ограничений размера.

**Q11. Освобождает ли GIL потокам параллельно обрабатывать задачи из очереди?**

GIL не даёт двум потокам исполнять Python-байткод одновременно, поэтому для **CPU-bound** задач многопоточность с очередью почти не даёт ускорения — лучше `multiprocessing` или `ProcessPoolExecutor`. Но для **I/O-bound** задач (сеть, диск, БД) поток освобождает GIL во время ожидания ввода-вывода, и другие потоки работают — здесь паттерн «очередь + пул потоков» очень эффективен.

**Q12. Можно ли потерять элемент, если воркер упал между `get()` и `task_done()`?**

Да. Если воркер забрал элемент через `get()`, но упал с исключением до завершения обработки и до `task_done()`, то: (а) элемент уже извлечён из очереди и потерян; (б) `task_done()` не вызван, значит счётчик не уменьшится и `join()` может зависнуть навсегда. Поэтому обработку оборачивают в `try/finally`, и `task_done()` вызывают в `finally` — гарантированно. Для гарантии «не потерять задачу» нужна более сложная схема: подтверждение обработки и повторная постановка задачи в очередь при ошибке (как в брокерах сообщений).

**Q13. Как дождаться окончания всех задач — через `join()` очереди или `join()` потоков?**

Два разных `join`. `queue.join()` ждёт, пока все **задачи** обработаны (счётчик незавершённых = 0). `thread.join()` ждёт, пока **поток** завершится. Часто используют оба: сначала `q.join()` — дождаться обработки всех данных, затем разослать sentinel-маркеры, чтобы воркеры вышли из циклов, и затем `thread.join()` для каждого потока — дождаться их фактического завершения и не оставить «висящих» потоков.

## Подводные камни (gotchas)

### Забытый `task_done()` приводит к вечному `join()`

Самая частая ошибка. Каждый `put()` увеличивает счётчик незавершённых задач. Если потребитель забыл вызвать `task_done()` после обработки, счётчик не дойдёт до нуля, и `q.join()` **заблокируется навсегда** — программа зависнет.

```python
import queue, threading

q = queue.Queue()

def bad_worker():
    while True:
        item = q.get()
        print("обработал", item)
        # ЗАБЫЛИ q.task_done() !!!

threading.Thread(target=bad_worker, daemon=True).start()
q.put(1)
q.join()  # ВИСНЕТ НАВСЕГДА: счётчик незавершённых задач остался = 1
```

Правильно — вызывать `task_done()` всегда, в `finally`:

```python
def good_worker():
    while True:
        item = q.get()
        try:
            print("обработал", item)
        finally:
            q.task_done()  # вызовется даже при исключении в обработке
```

### Лишний `task_done()` бросает `ValueError`

Если вызвать `task_done()` больше раз, чем было `put()`, очередь бросит `ValueError: task_done() called too many times`. Число `task_done()` должно строго равняться числу `put()`.

### `queue.Queue` не работает между процессами

Очередь живёт в памяти одного процесса. Передать её в `multiprocessing.Process` и ожидать общих данных нельзя — у дочернего процесса будет своя копия, синхронизация не сработает. Для процессов используйте `multiprocessing.Queue` (объекты должны быть picklable).

```python
import multiprocessing as mp
import queue

# НЕПРАВИЛЬНО:
# q = queue.Queue()           # потоковая очередь
# mp.Process(target=worker, args=(q,)).start()  # дочерний получит копию -> данные не разделяются

# ПРАВИЛЬНО для процессов:
q = mp.Queue()                # межпроцессная очередь
# mp.Process(target=worker, args=(q,)).start()
```

### `empty()` / `full()` / `qsize()` — приблизительны и небезопасны

В многопоточной среде значение может устареть сразу после возврата. Не стройте логику на `if not q.empty(): q.get()`. Используйте блокирующий `get(timeout=...)` с ловлей `Empty`.

### Бесконечная память без `maxsize`

Если производитель быстрее потребителя, а `maxsize=0` (без ограничения), очередь будет расти, пока не исчерпается память. Для долгоживущих систем почти всегда задавайте `maxsize`.

### Sentinel должен пройти через очередь, а не миновать её

Если положить `None` сразу при старте, он может быть забран раньше реальных задач (особенно у `PriorityQueue`/`LifoQueue`!). Кладите sentinel **после** всех задач. Для `LifoQueue` sentinel вообще выйдет первым — там паттерн poison pill требует другого подхода (например, `Event`).

### Потерянная задача при падении воркера

Между `get()` и `task_done()` элемент уже извлечён. Если воркер упал — задача потеряна, а `join()` может зависнуть. Оборачивайте обработку в `try/finally`.

### `get()` без timeout может «подвесить» программу при завершении

Воркер в `q.get()` без таймаута и без sentinel никогда не проснётся, если новых данных нет. Если поток не `daemon`, программа не завершится. Используйте sentinel или `daemon=True` + продуманный shutdown.

## Лучшие практики

- **Всегда вызывайте `task_done()` в `finally`** — гарантирует корректный счётчик даже при исключениях.
- **Задавайте `maxsize`** для долгоживущих систем — это защита от разрастания памяти (backpressure).
- **Не полагайтесь на `empty()`/`full()`/`qsize()`** для управления потоком — это гонки. Ловите `Empty`/`Full`.
- **Используйте sentinel (`None`) для graceful shutdown** — по одному на каждый воркер, кладите после всех задач.
- **Оборачивайте обработку элемента в `try/except/finally`** — ошибка одного элемента не должна ронять весь воркер.
- **Для CPU-bound** задач берите процессы (`multiprocessing`/`ProcessPoolExecutor`), а не потоки — из-за GIL.
- **В `PriorityQueue` используйте `(priority, counter, item)`** или `dataclass(order=True)` с `field(compare=False)`.
- **Рассмотрите `concurrent.futures.ThreadPoolExecutor`** — он часто заменяет ручную возню с очередью и потоками для пула воркеров.
- **Делайте воркеры `daemon=True`** только если допустимо их прерывание при завершении программы; для гарантированной доработки задач используйте sentinel + `thread.join()`.

## Полный рабочий пример: producer-consumer с пулом воркеров

Ниже законченный пример паттерна «производитель-потребитель» с несколькими воркерами, использованием `task_done()`/`join()` и корректным завершением через sentinel (`None`).

```python
import queue
import threading
import time
import random

# Очередь с ограничением размера -> включаем backpressure
task_queue: "queue.Queue" = queue.Queue(maxsize=20)

# Очередь для сбора результатов (тоже потокобезопасна)
result_queue: "queue.Queue" = queue.Queue()

NUM_WORKERS = 4          # количество потоков-потребителей
SENTINEL = None          # маркер завершения (poison pill)


def producer(num_tasks: int) -> None:
    """Производитель: генерирует задачи и кладёт их в очередь."""
    for i in range(num_tasks):
        item = f"task-{i}"
        task_queue.put(item)          # блокируется, если очередь полна (maxsize)
        print(f"[producer] положил {item}")
        time.sleep(random.uniform(0, 0.05))
    print("[producer] все задачи отправлены")


def consumer(worker_id: int) -> None:
    """Потребитель: забирает задачи из очереди и обрабатывает их."""
    while True:
        item = task_queue.get()       # ждём задачу (блокирующий get)
        try:
            if item is SENTINEL:
                # Получили маркер завершения -> выходим из цикла.
                print(f"[worker {worker_id}] получил sentinel, завершаюсь")
                break

            # Имитация полезной работы
            time.sleep(random.uniform(0.05, 0.2))
            result = item.upper()
            result_queue.put((worker_id, result))
            print(f"[worker {worker_id}] обработал {item} -> {result}")
        finally:
            # ВАЖНО: task_done() в finally — вызовется даже для sentinel
            # и даже если обработка упала с исключением.
            task_queue.task_done()


def main() -> None:
    # 1. Запускаем пул воркеров-потребителей
    workers = []
    for wid in range(NUM_WORKERS):
        t = threading.Thread(target=consumer, args=(wid,))
        t.start()
        workers.append(t)

    # 2. Запускаем производителя (в отдельном потоке)
    prod = threading.Thread(target=producer, args=(30,))
    prod.start()

    # 3. Ждём, пока производитель положит все задачи
    prod.join()

    # 4. Ждём, пока ВСЕ задачи будут обработаны (счётчик незавершённых -> 0).
    #    join() очереди разблокируется, когда на каждый put() пришёл task_done().
    task_queue.join()
    print("[main] все задачи обработаны (queue.join разблокирован)")

    # 5. Graceful shutdown: кладём по одному sentinel на каждого воркера.
    #    Делаем это ПОСЛЕ task_queue.join(), чтобы sentinel не учитывался
    #    раньше реальных задач.
    for _ in range(NUM_WORKERS):
        task_queue.put(SENTINEL)

    # 6. Ждём фактического завершения всех потоков-воркеров.
    for t in workers:
        t.join()
    print("[main] все воркеры завершены")

    # 7. Собираем результаты
    results = []
    while not result_queue.empty():
        results.append(result_queue.get())
    print(f"[main] собрано результатов: {len(results)}")


if __name__ == "__main__":
    main()
```

Ключевые моменты этого примера:

1. `maxsize=20` включает backpressure — производитель притормозит, если воркеры не успевают.
2. `task_done()` вызывается в `finally` — счётчик корректен даже при ошибках и для sentinel.
3. `task_queue.join()` дожидается обработки **всех** реальных задач.
4. Sentinel кладётся **после** `join()` — по одному на воркера, чтобы каждый получил свой маркер и вышел.
5. `thread.join()` для каждого воркера гарантирует, что все потоки реально завершились.

## Шпаргалка

```python
import queue

# --- Создание ---
q  = queue.Queue(maxsize=0)      # FIFO, 0 = без ограничения
lq = queue.LifoQueue()           # LIFO (стек)
pq = queue.PriorityQueue()       # по приоритету (меньший первым)
sq = queue.SimpleQueue()         # упрощённая FIFO (3.7+), без maxsize/join

# --- Положить ---
q.put(x)                         # блокирующий, ждёт места при maxsize
q.put(x, timeout=2)              # ждёт max 2 сек -> queue.Full
q.put(x, block=False)            # не ждёт -> queue.Full
q.put_nowait(x)                  # == put(x, block=False)

# --- Забрать ---
item = q.get()                   # блокирующий, ждёт элемент
item = q.get(timeout=2)          # ждёт max 2 сек -> queue.Empty
item = q.get(block=False)        # не ждёт -> queue.Empty
item = q.get_nowait()            # == get(block=False)

# --- Учёт задач ---
q.task_done()                    # сигнал «задача обработана» (число == числу put)
q.join()                         # ждать, пока все задачи обработаны

# --- Состояние (приблизительно, НЕ для гонок!) ---
q.qsize()                        # примерный размер
q.empty()                        # примерно пусто
q.full()                         # примерно полно

# --- Исключения ---
# queue.Empty  -> get на пустой (с timeout/nowait)
# queue.Full   -> put на полной (с timeout/nowait)

# --- PriorityQueue: защита от несравнимых объектов ---
import itertools
counter = itertools.count()
pq.put((priority, next(counter), item))   # counter уникален -> item не сравнивается

# --- Graceful shutdown ---
SENTINEL = None
# для каждого воркера: task_queue.put(SENTINEL)
# в воркере: if item is SENTINEL: break

# --- Память ---
# queue.Queue          -> ТОЛЬКО для потоков (threading), общая память
# multiprocessing.Queue -> для процессов, через pickle + pipe
# collections.deque     -> структура данных, без блокировки/timeout/join
```

| Класс | Порядок | maxsize | task_done/join | put блокируется |
|-------|---------|---------|----------------|-----------------|
| `Queue` | FIFO | да | да | да (при maxsize) |
| `LifoQueue` | LIFO | да | да | да (при maxsize) |
| `PriorityQueue` | приоритет | да | да | да (при maxsize) |
| `SimpleQueue` | FIFO | нет | нет | нет (никогда) |
