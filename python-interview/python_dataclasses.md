# Dataclasses — подготовка к собеседованию

## Что это и зачем

Модуль `dataclasses` (появился в Python 3.7, PEP 557) — это инструмент для автоматической генерации «служебных» методов классов, которые хранят данные: `__init__`, `__repr__`, `__eq__`, при желании `__lt__/__le__/__gt__/__ge__`, `__hash__` и др. Вы описываете только поля (с типами), а декоратор `@dataclass` дописывает за вас весь шаблонный код.

Зачем это нужно:

- **Убрать бойлерплейт.** Класс на 5 полей без dataclass требует руками написать `__init__`, `__repr__`, `__eq__` — это десятки строк, которые легко рассинхронизировать при добавлении поля.
- **Читаемость и декларативность.** Класс превращается в «схему данных»: видно, какие поля есть, какие у них типы и значения по умолчанию.
- **Меньше ошибок.** Сгенерированные методы консистентны: `__eq__` сравнивает ровно те же поля, что и `__repr__`, ничего не забыто.
- **Совместимость с типизацией.** Поля описываются через аннотации типов, mypy/pyright их понимают.

Сравните «руками» и через dataclass:

```python
# Классический способ — много шаблонного кода
class PointManual:
    def __init__(self, x: int, y: int = 0):
        self.x = x
        self.y = y

    def __repr__(self) -> str:
        return f"PointManual(x={self.x!r}, y={self.y!r})"

    def __eq__(self, other) -> bool:
        if other.__class__ is not self.__class__:
            return NotImplemented
        return (self.x, self.y) == (other.x, other.y)


# То же самое через dataclass — три строки
from dataclasses import dataclass

@dataclass
class Point:
    x: int
    y: int = 0
```

`Point` уже умеет: создаваться (`Point(1, 2)`), печататься (`repr`), сравниваться (`==`).

Важно: `@dataclass` — это **обычный декоратор класса**, а не метакласс и не базовый класс. Он смотрит на аннотации полей в теле класса и динамически генерирует методы. Это значит, что dataclass можно наследовать, миксовать, и он не ломает MRO.

## Ключевые концепции

### Поля определяются аннотациями

dataclass распознаёт поле, только если у атрибута есть **аннотация типа**:

```python
@dataclass
class Example:
    a: int           # поле (есть аннотация)
    b: int = 5       # поле со значением по умолчанию
    c = 10           # НЕ поле! Просто атрибут класса (нет аннотации)
```

`c` не попадёт ни в `__init__`, ни в `__repr__`, ни в `__eq__`. Это частая ловушка на собеседовании.

Тип в аннотации dataclass **не проверяет** — это подсказка для статических анализаторов. `x: int = "строка"` не вызовет ошибку в рантайме.

### Параметры декоратора @dataclass

Полная сигнатура (Python 3.11+):

```python
@dataclass(
    init=True,          # генерировать __init__
    repr=True,          # генерировать __repr__
    eq=True,            # генерировать __eq__
    order=False,        # генерировать __lt__, __le__, __gt__, __ge__
    unsafe_hash=False,  # принудительно сгенерировать __hash__
    frozen=False,       # сделать инстансы неизменяемыми
    match_args=True,    # сгенерировать __match_args__ (для match/case), 3.10+
    kw_only=False,      # все поля только по ключевым словам, 3.10+
    slots=False,        # сгенерировать __slots__, 3.10+
    weakref_slot=False, # добавить __weakref__ в slots, 3.11+
)
class C: ...
```

### Поле и функция field()

Для тонкой настройки отдельного поля используют `field()`:

```python
from dataclasses import dataclass, field

@dataclass
class Config:
    name: str
    tags: list = field(default_factory=list)   # изменяемое значение по умолчанию
    _cache: dict = field(default_factory=dict, repr=False, compare=False)
    id: int = field(default=0, metadata={"unit": "px"})
```

Параметры `field()`:

- `default` — обычное значение по умолчанию.
- `default_factory` — фабрика (вызывается без аргументов), даёт новое значение для каждого инстанса. Нужна для изменяемых значений (см. ниже про общий gotcha).
- `init` — включать ли поле в `__init__` (по умолчанию `True`).
- `repr` — показывать ли в `__repr__`.
- `compare` — участвует ли в `__eq__`/`__lt__` и др.
- `hash` — участвует ли в `__hash__` (по умолчанию следует за `compare`).
- `kw_only` — сделать это конкретное поле только-keyword (3.10+).
- `metadata` — произвольный словарь, dataclass его не использует, но он доступен через `fields()`. Удобно для библиотек (валидаторы, сериализаторы).

### Изменяемые значения по умолчанию запрещены напрямую

```python
@dataclass
class Bad:
    items: list = []   # TypeError при определении класса!
```

dataclass специально бросает `ValueError: mutable default ... is not allowed`, чтобы вы не наступили на классическую ловушку Python (один общий список на все инстансы). Используйте `default_factory`:

```python
@dataclass
class Good:
    items: list = field(default_factory=list)   # у каждого инстанса свой список
```

(Технически dataclass запрещает значения с `__hash__ is None`, то есть нехешируемые: list, dict, set. Но логика та же — «не используйте изменяемое».)

### Порядок полей и значения по умолчанию

Как и в обычных функциях, поля без значения по умолчанию не могут идти после полей со значением:

```python
@dataclass
class Bad:
    a: int = 0
    b: int        # TypeError: non-default argument 'b' follows default argument
```

Обойти можно через `kw_only=True` (3.10+), тогда порядок не важен.

## Основные функции/классы/методы

### Генерируемые методы

#### `__init__`

Генерируется из полей по порядку:

```python
@dataclass
class User:
    name: str
    age: int = 18

# Эквивалентно сгенерированному:
#   def __init__(self, name, age=18):
#       self.name = name
#       self.age = age

u = User("Аня", 30)
u2 = User(name="Боб")        # age=18
```

#### `__repr__`

```python
@dataclass
class Point:
    x: int
    y: int

print(repr(Point(1, 2)))   # Point(x=1, y=2)
```

Использует `!r` (repr) для каждого поля. Поля с `field(repr=False)` пропускаются.

#### `__eq__`

Сравнивает кортеж полей. Сравнение происходит **только при совпадении класса**:

```python
@dataclass
class Point:
    x: int
    y: int

Point(1, 2) == Point(1, 2)   # True
Point(1, 2) == Point(1, 3)   # False

@dataclass
class Vector:
    x: int
    y: int

Point(1, 2) == Vector(1, 2)  # False! Разные классы, __eq__ вернёт NotImplemented
```

#### Сравнения с `order=True`

```python
@dataclass(order=True)
class Version:
    major: int
    minor: int
    patch: int

Version(1, 2, 0) < Version(1, 3, 0)   # True (сравнивает кортеж (major, minor, patch))
sorted([Version(2, 0, 0), Version(1, 9, 9)])   # сортировка работает
```

`order=True` генерирует все четыре метода (`__lt__`, `__le__`, `__gt__`, `__ge__`). Они сравнивают кортежи полей лексикографически. Поля с `compare=False` исключаются.

**Частый приём** — отделить «ключ сортировки» от «полезной нагрузки»:

```python
@dataclass(order=True)
class PriorityTask:
    priority: int
    # описание не должно влиять на порядок в очереди с приоритетом:
    description: str = field(compare=False)

import heapq
q = []
heapq.heappush(q, PriorityTask(2, "помыть посуду"))
heapq.heappush(q, PriorityTask(1, "потушить пожар"))
heapq.heappop(q)   # PriorityTask(priority=1, ...) — сравнивается только priority
```

#### `__hash__` и hashability

Логика хеширования зависит от комбинации `eq` и `frozen`:

| `eq`  | `frozen` | `__hash__`                              | Можно класть в set/dict как ключ |
|-------|----------|------------------------------------------|----------------------------------|
| True  | False    | `None` (объект нехешируемый)             | Нет                              |
| True  | True     | сгенерирован                             | Да                               |
| False | (любой)  | наследуется от объекта (по id)           | Да                               |

То есть **обычный dataclass с `eq=True, frozen=False` нехешируем** — это сделано намеренно: если объект изменяемый, его хеш может «поплыть», и он сломает set/dict. Чтобы получить хешируемый dataclass — делайте его `frozen=True`:

```python
@dataclass(frozen=True)
class Coord:
    x: int
    y: int

{Coord(1, 2), Coord(1, 2)}   # {Coord(x=1, y=2)} — работает, дубликат схлопнулся
```

`unsafe_hash=True` форсирует генерацию `__hash__` даже для изменяемого класса — «unsafe», потому что вы берёте ответственность за то, что не будете мутировать поля, влияющие на хеш.

### frozen — неизменяемость

```python
@dataclass(frozen=True)
class Money:
    amount: int
    currency: str

m = Money(100, "USD")
m.amount = 200   # FrozenInstanceError: cannot assign to field 'amount'
```

Как это работает: dataclass генерирует `__setattr__` и `__delattr__`, которые бросают `FrozenInstanceError`. Это **не настоящая** глубокая неизменяемость — если поле содержит список, сам список менять можно:

```python
@dataclass(frozen=True)
class Box:
    items: list = field(default_factory=list)

b = Box()
b.items.append(1)   # OK! сам список изменяем
b.items = []        # FrozenInstanceError — переприсваивание атрибута запрещено
```

Замена полей для frozen-объекта делается через `replace()` (создаёт новый объект):

```python
from dataclasses import replace
m2 = replace(m, amount=200)   # Money(amount=200, currency='USD')
```

### `__post_init__` — пост-инициализация

Если определить метод `__post_init__`, dataclass вызовет его в конце сгенерированного `__init__`. Используется для валидации, вычисляемых полей, нормализации:

```python
@dataclass
class Rectangle:
    width: float
    height: float
    area: float = field(init=False)   # не в __init__, вычисляется

    def __post_init__(self):
        if self.width < 0 or self.height < 0:
            raise ValueError("стороны должны быть неотрицательными")
        self.area = self.width * self.height

r = Rectangle(3, 4)
print(r.area)   # 12
```

Для `frozen=True` обычное присваивание в `__post_init__` не сработает (сработает `FrozenInstanceError`). Нужно идти в обход через `object.__setattr__`:

```python
@dataclass(frozen=True)
class Circle:
    radius: float
    area: float = field(init=False)

    def __post_init__(self):
        # обход запрета на присваивание у frozen
        object.__setattr__(self, "area", 3.14159 * self.radius ** 2)
```

#### `InitVar` — параметры только для инициализации

`InitVar[T]` — поле, которое попадает в `__init__` и передаётся в `__post_init__`, но **не сохраняется** как атрибут инстанса и не участвует в repr/eq:

```python
from dataclasses import dataclass, field, InitVar

@dataclass
class User:
    name: str
    password_hash: str = field(init=False)
    password: InitVar[str] = ""    # только для __init__

    def __post_init__(self, password: str):   # InitVar-поля приходят аргументами
        self.password_hash = f"hashed:{password}"

u = User("Аня", password="secret")
print(u)               # User(name='Аня', password_hash='hashed:secret')
# у u нет атрибута password
```

### slots — оптимизация памяти

`@dataclass(slots=True)` (3.10+) генерирует `__slots__` из имён полей. Это убирает `__dict__` у каждого инстанса — экономит память и чуть ускоряет доступ к атрибутам:

```python
@dataclass(slots=True)
class Point:
    x: int
    y: int

p = Point(1, 2)
p.z = 5   # AttributeError: 'Point' object has no attribute 'z' (нет __dict__)
```

Важная деталь реализации: `slots=True` фактически **создаёт новый класс** (нельзя добавить `__slots__` к уже созданному классу), поэтому декораторы/ссылки на «старый» класс могут вести себя неожиданно. Также со `slots=True` ломается `default` через `field(default=...)` в некоторых старых версиях из-за конфликта имени слота и атрибута класса — в современных версиях это поправлено, но `default_factory` всегда надёжнее.

До 3.10 `slots` руками:

```python
@dataclass
class Point:
    __slots__ = ("x", "y")   # объявляем вручную
    x: int
    y: int
```

### Вспомогательные функции модуля

```python
from dataclasses import (
    fields, asdict, astuple, replace, is_dataclass, make_dataclass, MISSING
)

@dataclass
class P:
    x: int
    y: int

p = P(1, 2)

fields(P)          # кортеж объектов Field с метаданными о полях
asdict(p)          # {'x': 1, 'y': 2} — рекурсивно, глубокое копирование
astuple(p)         # (1, 2)
replace(p, x=10)   # P(x=10, y=2) — новый объект
is_dataclass(p)    # True (работает и на классе, и на инстансе)
is_dataclass(P)    # True
```

`asdict` рекурсивно разворачивает вложенные dataclass, списки, словари и делает **глубокую копию** значений — для больших структур это может быть дорого.

`make_dataclass` создаёт класс динамически:

```python
Point3D = make_dataclass("Point3D", ["x", "y", ("z", int, field(default=0))])
Point3D(1, 2)   # Point3D(x=1, y=2, z=0)
```

`MISSING` — синглтон-сентинел, означающий «значение не задано». Используется, чтобы отличить «по умолчанию None» от «по умолчанию ничего». Полезен при работе с `fields()`:

```python
for f in fields(P):
    has_default = f.default is not MISSING
```

### Наследование dataclass

Поля наследуются и объединяются по MRO. Поля базового класса идут первыми в `__init__`:

```python
@dataclass
class Base:
    a: int
    b: int = 2

@dataclass
class Child(Base):
    c: int = 3
    # __init__(self, a, b=2, c=3)
```

Проблема порядка: если базовый класс имеет поле со значением по умолчанию, а наследник добавляет поле без значения — будет ошибка «non-default follows default». Решается через `kw_only=True`:

```python
@dataclass(kw_only=True)
class Base:
    a: int = 0

@dataclass(kw_only=True)
class Child(Base):
    b: int   # ок, потому что все поля keyword-only
```

### kw_only — поля только по ключевым словам

```python
@dataclass
class C:
    a: int
    b: int = field(kw_only=True)   # b можно передать только по имени

C(1, b=2)   # ок
C(1, 2)     # TypeError
```

Или для всего класса: `@dataclass(kw_only=True)`. Это решает проблему «обязательное поле после поля с дефолтом».

### match_args и сопоставление с образцом (3.10+)

`@dataclass` генерирует `__match_args__`, что позволяет использовать позиционные паттерны в `match`:

```python
@dataclass
class Point:
    x: int
    y: int

match Point(0, 5):
    case Point(0, y):       # благодаря __match_args__ = ('x', 'y')
        print(f"на оси Y, y={y}")
    case Point(x, y):
        print(f"общая точка {x},{y}")
```

## Сравнение: dataclass vs namedtuple vs NamedTuple vs attrs vs Pydantic

Это любимый вопрос на собеседовании. Держите в голове таблицу и нюансы.

### `collections.namedtuple`

```python
from collections import namedtuple
Point = namedtuple("Point", ["x", "y"])
p = Point(1, 2)
p.x          # 1
p[0]         # 1 — индексация как у кортежа
x, y = p     # распаковка
p.x = 5      # AttributeError — неизменяемый (это tuple)
```

- Это **подкласс tuple**: неизменяемый, итерируемый, индексируемый, распаковываемый, хешируемый, занимает мало памяти.
- Минусы: нельзя добавить методы красиво (нужен подкласс), нет аннотаций типов, нет значений по умолчанию (только через `defaults=`), «случайно» ведёт себя как кортеж (`(1,2) == Point(1,2)` → True, что может быть нежелательно).

### `typing.NamedTuple`

Типизированная и удобная версия namedtuple, с поддержкой методов и значений по умолчанию:

```python
from typing import NamedTuple

class Point(NamedTuple):
    x: int
    y: int = 0

    def dist(self) -> float:
        return (self.x ** 2 + self.y ** 2) ** 0.5

p = Point(1)
p.x          # 1
p[0]         # 1 — всё ещё кортеж под капотом
p.dist()     # методы можно
```

- Под капотом всё тот же кортеж: неизменяемый, хешируемый, итерируемый/индексируемый, легче по памяти, чем dataclass.
- Минусы: нельзя сделать изменяемым; наследование ограничено (нельзя наследовать другой класс кроме NamedTuple); сравнивается с обычными кортежами как кортеж.

### `dataclass`

- Обычный класс (не tuple): по умолчанию изменяемый, неитерируемый, не индексируется.
- Гибкость: `frozen`, `order`, `slots`, `__post_init__`, `InitVar`, `field` с метаданными, наследование как у обычных классов.
- Нет автоматической валидации/приведения типов (типы только для статического анализа).
- Хороший дефолт «по умолчанию», когда не нужна сериализация/валидация.

### `attrs` (сторонняя библиотека)

Предшественник и «старший брат» dataclasses (dataclasses во многом вдохновлён attrs). Современный синтаксис:

```python
from attrs import define, field

@define
class Point:
    x: int
    y: int = 0
    tags: list = field(factory=list)
```

- Больше возможностей: валидаторы (`@x.validator`), конвертеры (`converter=`), `__slots__` по умолчанию (в `@define`), более гибкий контроль над генерацией, поддержка старых Python.
- Активно развивается, иногда быстрее dataclasses.
- Минус: внешняя зависимость; для простых случаев dataclasses достаточно.

### `Pydantic` (сторонняя, v1/v2)

Это уже про **валидацию данных в рантайме**, а не просто генерацию методов:

```python
from pydantic import BaseModel, field_validator

class User(BaseModel):
    name: str
    age: int

User(name="Аня", age="30")   # age приведётся к int=30 (в v1) / зависит от режима в v2
User(name="Аня", age="abc")  # ValidationError
```

- **Проверяет и приводит типы при создании объекта** (ключевое отличие). Идеально для границ системы: API (FastAPI), парсинг JSON/конфигов, валидация входных данных.
- Богатые возможности: валидаторы, сериализация в/из JSON, схемы JSON Schema, вложенные модели.
- Минусы: тяжелее по производительности и памяти (хотя v2 на Rust сильно ускорена), внешняя зависимость, своя «магия». Для чисто внутренних структур данных это оверкилл.
- У Pydantic есть `@pydantic.dataclasses.dataclass` — совместимый со стандартным dataclass декоратор с валидацией.

| Это tuple (индекс/итерация)| Да       | Да         | Нет              | Нет          | Нет             |
| Аннотации типов           | Нет       | Да         | Да               | Да           | Да              |
| Значения по умолчанию     | Ограничено| Да         | Да               | Да           | Да              |
| Валидация в рантайме      | Нет       | Нет        | Нет (только __post_init__) | Да (валидаторы) | Да (главная фича) |
| Приведение типов          | Нет       | Нет        | Нет              | Конвертеры   | Да              |
| Хешируемость              | Да        | Да         | frozen → да      | Настраивается| Да              |
| Память                    | Очень мало| Очень мало | slots → мало     | slots → мало | Больше          |
| Внешняя зависимость       | Нет       | Нет        | Нет              | Да           | Да              |
| Когда выбирать            | лёгкий неизменяемый кортеж-запись | то же + методы/типы | внутренняя модель данных по умолчанию | продвинутая валидация без тяжести | границы системы, парсинг/валидация |

**Краткая эвристика для ответа:** «namedtuple/NamedTuple — когда нужен лёгкий неизменяемый кортеж-запись; dataclass — стандартный выбор для внутренних структур данных; Pydantic — когда нужна валидация и парсинг внешних данных (API, конфиги); attrs — когда нужны фичи attrs или поддержка старого Python».

## Частые вопросы на собеседовании

**Q1: Что генерирует `@dataclass` по умолчанию?**
A: `__init__`, `__repr__`, `__eq__`. Не генерирует по умолчанию методы сравнения (`order=False`) и `__hash__` (для изменяемого класса он становится `None`).

**Q2: Почему нельзя `field: list = []` и как правильно?**
A: dataclass бросает ошибку при изменяемом значении по умолчанию, потому что иначе один список разделялся бы всеми инстансами (классическая ловушка Python). Правильно: `field(default_factory=list)` — фабрика вызывается для каждого инстанса.

**Q3: Чем отличается `default` от `default_factory`?**
A: `default` — одно фиксированное значение (подходит для неизменяемых: числа, строки, кортежи). `default_factory` — функция без аргументов, вызывается при создании каждого инстанса, даёт свежий объект (нужна для list, dict, set и любых изменяемых/динамических значений). Указывать оба одновременно нельзя.

**Q4: Можно ли положить обычный dataclass в `set` или использовать как ключ `dict`?**
A: По умолчанию нет — при `eq=True, frozen=False` `__hash__` устанавливается в `None`, объект нехешируем. Чтобы сделать хешируемым: `frozen=True` (рекомендуется) или `unsafe_hash=True` (на свой риск).

**Q5: Что делает `frozen=True` и это глубокая неизменяемость?**
A: Генерирует `__setattr__`/`__delattr__`, бросающие `FrozenInstanceError`, плюс делает объект хешируемым. Это «поверхностная» неизменяемость: переприсвоить атрибут нельзя, но если поле — список, его содержимое менять можно. Для изменения создавайте копию через `dataclasses.replace`.

**Q6: Зачем `__post_init__` и как присвоить поле во frozen-классе внутри него?**
A: `__post_init__` вызывается после `__init__` для валидации/вычисляемых полей. Во frozen-классе обычное присваивание бросит ошибку, поэтому используют `object.__setattr__(self, "field", value)`.

**Q7: Что такое `InitVar`?**
A: Псевдо-поле, которое передаётся в `__init__` и пробрасывается в `__post_init__` как аргумент, но не сохраняется как атрибут и не участвует в repr/eq. Полезно для входных данных, нужных только на этапе инициализации (например, сырой пароль для хеширования).

**Q8: dataclass vs NamedTuple — когда что?**
A: NamedTuple — неизменяемый кортеж-запись: легче по памяти, индексируется, итерируется, хешируем, но не изменяем и плохо наследуется. dataclass — обычный класс: изменяемый (или frozen), гибче, поддерживает `__post_init__`, slots, метаданные. NamedTuple для лёгких неизменяемых записей; dataclass — дефолт для внутренних моделей.

**Q9: dataclass vs Pydantic?**
A: dataclass генерирует методы, но не валидирует типы в рантайме (аннотации — для статического анализа). Pydantic валидирует и приводит типы при создании объекта — это его главная задача. Pydantic для границ системы (API, парсинг JSON, конфиги); dataclass для внутренних структур, где валидация не нужна.

**Q10: Что даёт `slots=True` и какие подводные камни?**
A: Генерирует `__slots__`, убирает `__dict__` у инстансов — экономит память и ускоряет доступ к полям. Нельзя добавлять новые атрибуты «на лету». Подводный камень: `slots=True` создаёт новый объект класса (не модифицирует существующий), что может ломать ссылки/декораторы, рассчитанные на исходный класс; также бывают конфликты с `default` в старых версиях.

**Q11: Как сделать класс сортируемым и положить в очередь с приоритетом?**
A: `@dataclass(order=True)` генерирует `__lt__` и др., сравнивающие кортеж полей. Чтобы по приоритету сортировать, но «полезную нагрузку» исключить из сравнения — пометьте её `field(compare=False)`. Тогда объекты можно класть в `heapq`.

**Q12: Что вернёт `Point(1,2) == OtherClass(1,2)`?**
A: `False`. Сгенерированный `__eq__` сначала проверяет, что класс совпадает (`other.__class__ is self.__class__`), иначе возвращает `NotImplemented`, и Python в итоге даёт `False`. Это отличает dataclass от namedtuple, который сравнивается с любым кортежем по значениям.

**Q13: Наследуются ли поля и каков порядок в `__init__`?**
A: Да, поля собираются по MRO; поля базовых классов идут первыми. Проблема: поле без значения по умолчанию в наследнике не может идти после поля с дефолтом в базе — решается `kw_only=True`.

**Q14: Чем `asdict()` отличается от `vars()`/`__dict__`?**
A: `asdict()` рекурсивно разворачивает вложенные dataclass/списки/словари и делает глубокую копию — дорого для больших структур. `vars(obj)`/`obj.__dict__` даёт плоский словарь атрибутов конкретного инстанса без рекурсии (и не работает со `slots=True`, где нет `__dict__`).

**Q15: Проверяет ли dataclass типы полей в рантайме?**
A: Нет. Аннотации нужны только статическим анализаторам (mypy/pyright) и для самого распознавания полей. `Point(x="строка", y=None)` создастся без ошибок. Для рантайм-валидации нужен `__post_init__`, attrs-валидаторы или Pydantic.

## Подводные камни (gotchas)

1. **Атрибут без аннотации не становится полем.** `c = 10` (без `: int`) — это атрибут класса, его не будет в `__init__`/`__repr__`/`__eq__`. Легко не заметить и удивиться.

2. **Изменяемые значения по умолчанию.** `field(default=[])` или просто `= []` запрещены/опасны. Всегда `default_factory` для list/dict/set.

3. **`default_factory` и аргументы.** Фабрика вызывается **без аргументов**. Если нужен список с зависимостью от других полей — делайте это в `__post_init__`, а не в factory.

4. **frozen — не глубокая неизменяемость.** Содержимое изменяемых полей можно менять. И хеш такого «frozen» объекта может стать ненадёжным, если внутри изменяемая структура.

5. **frozen + `__post_init__`.** Прямое присваивание бросит `FrozenInstanceError`. Нужен `object.__setattr__`.

6. **Порядок default-полей при наследовании.** «non-default argument follows default argument» — частая ошибка. Решение: `kw_only=True`.

7. **`asdict()` дорогая и делает deepcopy.** На больших/вложенных структурах это заметная просадка. Если нужен плоский словарь без копирования — `dataclasses.fields` + ручной сбор или `vars()`.

8. **`==` строго по классу.** Два разных dataclass с одинаковыми полями не равны. Это плюс по сравнению с namedtuple, но иногда удивляет.

9. **slots ломает `__dict__`-зависимый код.** Нельзя добавлять атрибуты на лету, не работает `vars()`, могут ломаться сторонние библиотеки, ожидающие `__dict__`. Также `weakref` требует `weakref_slot=True`.

10. **Поля типа `ClassVar` и `InitVar` не становятся обычными полями.** `x: ClassVar[int] = 0` — это переменная класса, dataclass её игнорирует как поле инстанса (не в `__init__`, не в repr/eq).

11. **Хеш и `unsafe_hash`.** Если используете `unsafe_hash=True` на изменяемом классе и потом мутируете поля, объект «потеряется» в set/dict (хеш изменился) — источник трудноуловимых багов.

12. **`field()` нельзя использовать без присваивания типу.** `field()` должен стоять справа от аннотации: `x: int = field(...)`. Просто вызвать `field()` в теле без аннотации бессмысленно.

13. **Сравнение float-полей.** `__eq__` сравнивает напрямую (`==`), для float это может давать неожиданности из-за погрешностей. dataclass тут ничем не отличается от ручного сравнения.

## Лучшие практики

- **dataclass как дефолт** для внутренних структур данных вместо ручных классов или словарей «с магическими ключами».
- **Используйте `frozen=True`** для объектов-значений (value objects), которые логически неизменяемы: координаты, деньги, конфиги. Это даёт хешируемость и защиту от случайных мутаций.
- **`default_factory` для всего изменяемого**: list, dict, set, и для динамических значений (`field(default_factory=datetime.now)` — будьте осторожны, вызовется при каждом создании).
- **`slots=True` для классов, создаваемых в больших количествах** (десятки/сотни тысяч инстансов) — заметная экономия памяти.
- **Валидацию выносите в `__post_init__`** (или используйте Pydantic/attrs, если валидации много). dataclass типы не проверяет.
- **`field(repr=False)` для больших/секретных полей** (пароли, бинарные блобы, кэши) — чтобы не светить их в логах и не раздувать repr.
- **`field(compare=False)` для полей, не определяющих идентичность** объекта (кэши, метки времени), чтобы `==` и сортировка работали по «сути».
- **`kw_only=True` для классов с многими полями** — улучшает читаемость вызовов и снимает проблему порядка default-полей при наследовании.
- **Не злоупотребляйте `asdict()`** в горячих путях — он рекурсивно копирует. Для сериализации больших объёмов лучше специализированные сериализаторы.
- **Для границ системы (API/конфиги) — Pydantic**, для внутренней модели — dataclass. Не тащите Pydantic туда, где валидация не нужна.

## Шпаргалка

```python
from dataclasses import (
    dataclass, field, fields, asdict, astuple,
    replace, is_dataclass, InitVar, MISSING, FrozenInstanceError
)
from typing import ClassVar

# --- Базовый dataclass ---
@dataclass
class Point:
    x: int
    y: int = 0                       # значение по умолчанию

# --- Все основные опции ---
@dataclass(
    frozen=True,    # неизменяемый + хешируемый
    order=True,     # __lt__/__le__/__gt__/__ge__
    slots=True,     # __slots__, экономия памяти (3.10+)
    kw_only=True,   # все поля только по ключевым словам (3.10+)
)
class Full:
    a: int
    b: list = field(default_factory=list)            # изменяемый дефолт
    c: str = field(default="x", repr=False, compare=False)  # скрыть из repr, не сравнивать
    counter: ClassVar[int] = 0                        # переменная КЛАССА, НЕ поле
    secret: InitVar[str] = ""                         # только для __init__/__post_init__

    def __post_init__(self, secret: str):
        # для frozen — присваивание только через object.__setattr__
        object.__setattr__(self, "a", self.a + len(secret))

# --- Параметры field() ---
field(
    default=...,            # фиксированное значение по умолчанию
    default_factory=list,   # фабрика (для изменяемых) — НЕ вместе с default
    init=True,              # включать в __init__
    repr=True,              # показывать в __repr__
    compare=True,           # участвовать в __eq__/__lt__
    hash=None,              # участвовать в __hash__ (по умолчанию = compare)
    kw_only=False,          # только по ключевому слову (3.10+)
    metadata={"k": "v"},    # произвольные метаданные
)

# --- Вспомогательные функции ---
fields(Point)              # кортеж Field-объектов
asdict(p)                  # -> dict, рекурсивно + deepcopy
astuple(p)                 # -> tuple, рекурсивно
replace(p, x=10)           # новый объект с изменёнными полями (для frozen)
is_dataclass(Point)        # True

# --- Хешируемость ---
# eq=True,  frozen=False -> __hash__ = None  (нехешируемый)  [по умолчанию]
# eq=True,  frozen=True  -> __hash__ сгенерирован             (хешируемый)
# eq=False               -> __hash__ по id объекта
# unsafe_hash=True       -> форсировать __hash__ (на свой риск)

# --- Правила порядка ---
# Поля без default не могут идти после полей с default (если не kw_only=True)
# Изменяемые значения по умолчанию запрещены -> используйте default_factory

# --- Когда что выбирать ---
# namedtuple / NamedTuple : лёгкий неизменяемый кортеж-запись
# dataclass               : дефолт для внутренних моделей данных
# attrs                   : продвинутые фичи (валидаторы/конвертеры) без тяжести
# Pydantic                : валидация и парсинг ВНЕШНИХ данных (API, конфиги)
```
