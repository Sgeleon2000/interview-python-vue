# pickle — подготовка к собеседованию

## Что это и зачем

`pickle` — стандартный модуль Python для **сериализации** (маршалинга) и **десериализации** произвольных Python-объектов в поток байтов и обратно. Сериализация (`pickling`) превращает объект в последовательность байтов, которую можно сохранить в файл, передать по сети или положить в кэш. Десериализация (`unpickling`) восстанавливает объект из байтов.

Ключевая особенность: pickle понимает **почти любые** Python-объекты — вложенные структуры, экземпляры пользовательских классов, циклические ссылки, разделяемые объекты. Это бинарный, **Python-специфичный** формат: его невозможно прочитать из других языков.

Когда применять:
- кэширование вычисленных объектов на диск;
- передача объектов между процессами (`multiprocessing` использует pickle под капотом);
- сохранение состояния (чекпоинты ML-моделей, например `joblib` основан на pickle);
- очереди задач (Celery, RQ при дефолтных настройках).

Когда НЕ применять:
- обмен данными с внешним миром / другими языками — используйте JSON, Protocol Buffers, MessagePack;
- хранение данных, которые получены из **недоверенного источника** (см. раздел про безопасность).

## Ключевые концепции

- **Pickle-протокол** — версия бинарного формата. Сейчас актуальны протоколы 0–5.
- **Опкоды** — pickle на самом деле является небольшим стековым «языком»; десериализация = выполнение программы из опкодов виртуальной pickle-машиной. Именно поэтому unpickling недоверенных данных опасен.
- **`__reduce__`** — главный хук, описывающий, как пересоздать объект.
- **`__getstate__` / `__setstate__`** — хуки управления состоянием экземпляра.
- **Picklable** — объект, который можно сериализовать. Не всё picklable (файлы, сокеты, лямбды, локальные функции).
- **persistent_id / persistent_load** — механизм внешних ссылок (не сохранять объект, а сохранить ссылку-идентификатор).
- **Memo** — таблица уже сериализованных объектов; обеспечивает сохранение разделяемых ссылок и обработку циклов.

## Основные функции/классы/методы

### dump / dumps / load / loads

```python
import pickle

data = {"name": "Андрей", "ids": [1, 2, 3], "nested": {"ok": True}}

# dumps -> bytes (сериализация в память)
raw: bytes = pickle.dumps(data)
print(type(raw))  # <class 'bytes'>

# loads -> объект (десериализация из памяти)
restored = pickle.loads(raw)
print(restored == data)  # True

# dump -> запись в файл (открывать ОБЯЗАТЕЛЬНО в бинарном режиме 'wb')
with open("data.pkl", "wb") as f:
    pickle.dump(data, f)

# load -> чтение из файла ('rb')
with open("data.pkl", "rb") as f:
    restored2 = pickle.load(f)
```

Памятка по именам: суффикс `s` = «string/bytes» (работа с байтами в памяти), без `s` = работа с файловым объектом.

### Протоколы

```python
import pickle

# Доступные протоколы
print(pickle.HIGHEST_PROTOCOL)  # 5 (на современных Python)
print(pickle.DEFAULT_PROTOCOL)  # обычно 4 или 5 в зависимости от версии

# Явно указываем протокол при сериализации
raw0 = pickle.dumps(data, protocol=0)   # текстовый, читаемый ASCII, медленный
raw5 = pickle.dumps(data, protocol=5)   # самый компактный и быстрый
print(len(raw0), len(raw5))
```

Краткая характеристика протоколов:
- **0** — древний ASCII-протокол, человекочитаемый, обратная совместимость. Очень неэффективен.
- **1** — старый бинарный.
- **2** (Python 2.3+) — эффективная сериализация new-style классов.
- **3** (Python 3.0+) — поддержка `bytes`; **не читается** Python 2.
- **4** (Python 3.4+) — большие объекты (>4 ГБ), больше типов, более компактен. Долго был дефолтным.
- **5** (Python 3.8+, PEP 574) — **out-of-band данные** и zero-copy для буферов (`PickleBuffer`), важно для больших numpy-массивов и `multiprocessing` с разделяемой памятью.

Правило: чем выше протокол, тем компактнее и быстрее, но тем выше требование к минимальной версии Python при чтении.

### Что можно и что нельзя сериализовать

Можно:
- `None`, `bool`, `int`, `float`, `complex`, `str`, `bytes`, `bytearray`;
- кортежи, списки, множества, словари из picklable-объектов (рекурсивно);
- функции и классы, **определённые на верхнем уровне модуля** (сериализуется по сути ссылка `module.qualname`, а не код);
- экземпляры классов, чьё состояние picklable.

Нельзя (по умолчанию):
- лямбды и вложенные/локальные функции (`<locals>`);
- объекты ОС: открытые файлы, сокеты, соединения с БД, потоки, замки;
- генераторы, итераторы, корутины;
- многие C-объекты без поддержки pickle.

```python
import pickle

# Лямбда не сериализуется
try:
    pickle.dumps(lambda x: x + 1)
except (pickle.PicklingError, AttributeError) as e:
    print("Ошибка:", e)  # Can't pickle local object ...

# Функция верхнего уровня сериализуется (по ИМЕНИ, не по коду!)
def top_level_func(x):
    return x * 2

raw = pickle.dumps(top_level_func)
print(pickle.loads(raw)(10))  # 20
# ВАЖНО: при загрузке функция должна существовать в том же модуле под тем же именем
```

### `__reduce__` — низкоуровневый протокол восстановления

`__reduce__` возвращает либо строку (имя глобали), либо кортеж из 2–6 элементов: `(callable, args[, state[, listitems[, dictitems[, state_setter]]]])`. При десериализации pickle вызовет `callable(*args)`, затем применит state.

```python
import pickle

class Temperature:
    def __init__(self, celsius):
        self.celsius = celsius

    def __reduce__(self):
        # Говорим pickle: чтобы восстановить — вызови Temperature(self.celsius)
        return (self.__class__, (self.celsius,))

t = Temperature(36.6)
raw = pickle.dumps(t)
t2 = pickle.loads(raw)
print(t2.celsius)  # 36.6
```

`__reduce_ex__(protocol)` — версия, получающая номер протокола; вызывается раньше `__reduce__` и по умолчанию делегирует ему.

### `__getstate__` / `__setstate__` — управление состоянием

Эти хуки удобнее, когда нужно исключить непиклимые атрибуты (соединения, кэши, локи).

```python
import pickle
import threading

class Connection:
    def __init__(self, host):
        self.host = host
        self._socket = object()        # имитация непиклимого ресурса
        self._lock = threading.Lock()  # лок нельзя сериализовать

    def __getstate__(self):
        # Возвращаем то, ЧТО сохранять. Копируем __dict__ и выкидываем ресурсы.
        state = self.__dict__.copy()
        del state["_socket"]
        del state["_lock"]
        return state

    def __setstate__(self, state):
        # Восстанавливаем сохранённое состояние и ПЕРЕсоздаём ресурсы.
        self.__dict__.update(state)
        self._socket = object()        # переоткрываем соединение
        self._lock = threading.Lock()  # создаём новый лок

conn = Connection("db.local")
raw = pickle.dumps(conn)
conn2 = pickle.loads(raw)
print(conn2.host)        # db.local
print(conn2._lock)       # новый лок есть
```

С Python 3.11 пустой `__getstate__` по умолчанию возвращает `__dict__` (или `None`), что меняет некоторые edge-cases. По умолчанию (без хуков) сериализуется `self.__dict__`, а `__setstate__` по умолчанию делает `self.__dict__.update(state)`.

### `__slots__` и pickle

Для классов со `__slots__` нет `__dict__`, поэтому состояние сериализуется как кортеж `(None, {slot: value})`. Если есть и `__dict__`, и `__slots__`, состояние — кортеж `(dict_state, slots_state)`.

```python
import pickle

class Point:
    __slots__ = ("x", "y")
    def __init__(self, x, y):
        self.x, self.y = x, y

p = pickle.loads(pickle.dumps(Point(1, 2)))
print(p.x, p.y)  # 1 2  — со slots работает «из коробки» (protocol >= 2)
```

### Кастомизация через Pickler / Unpickler

```python
import pickle
import io

# Ограничивающий Unpickler (см. раздел безопасности)
class RestrictedUnpickler(pickle.Unpickler):
    ALLOWED = {("builtins", "list"), ("builtins", "dict")}
    def find_class(self, module, name):
        if (module, name) in self.ALLOWED:
            return super().find_class(module, name)
        raise pickle.UnpicklingError(f"Запрещён доступ к {module}.{name}")

def safe_loads(b):
    return RestrictedUnpickler(io.BytesIO(b)).load()
```

## Частые вопросы на собеседовании

**Q1. В чём разница между `dump`/`load` и `dumps`/`loads`?**
`dump`/`load` работают с файловым объектом (нужно открывать в бинарном режиме `wb`/`rb`). `dumps`/`loads` работают с `bytes` в памяти. Суффикс `s` ≈ «string/bytes».

**Q2. Чем pickle принципиально отличается от json?**
pickle — бинарный, Python-специфичный, сериализует почти любые объекты (включая экземпляры классов и циклические ссылки), но **небезопасен** при загрузке чужих данных и непереносим между языками. json — текстовый, кросс-языковой, безопасный, но поддерживает только базовые типы (dict/list/str/number/bool/null) и читается людьми. См. таблицу ниже.

**Q3. Почему загружать чужой pickle опасно?**
`pickle.loads` выполняет опкоды как программу и может вызывать произвольные callable через `__reduce__`/`find_class`. Специально сформированный pickle при загрузке выполнит произвольный код (RCE). Никогда не делайте `loads` данных из недоверенного источника.

**Q4. Можно ли сериализовать лямбду или локальную функцию?**
Нет (по умолчанию). pickle сохраняет функции по ссылке `module.qualname`, а у лямбд/локальных функций нет адресуемого глобального имени. Решения: вынести функцию на верхний уровень модуля, использовать `dill`/`cloudpickle`, либо обернуть в picklable-объект.

**Q5. Что произойдёт, если сериализовать объект, изменить определение класса, а потом десериализовать?**
pickle хранит только данные и ссылку на класс, а не его код. При загрузке используется **текущее** определение класса. Если убрали атрибут — `__setstate__`/код может упасть; если добавили — старый pickle не содержит нового атрибута. Версионирование состояния лежит на разработчике (часто добавляют поле `version` в state).

**Q6. Зачем нужен `__getstate__`/`__setstate__`?**
Чтобы исключить из сериализации непиклимые/тяжёлые ресурсы (сокеты, локи, кэши, дескрипторы файлов) и пересоздать их при восстановлении. `__getstate__` определяет, что сохранить; `__setstate__` — как восстановить.

**Q7. Как pickle обрабатывает циклические ссылки и разделяемые объекты?**
Через таблицу `memo`: каждый уже сериализованный объект запоминается по `id`. Повторная ссылка кодируется как ссылка на memo, а не дублируется. Поэтому циклы и общий объект восстанавливаются корректно (с сохранением идентичности).

**Q8. Какой протокол использовать и почему не всегда самый высокий?**
Высокий протокол компактнее и быстрее, но требует более новой версии Python для чтения. Если pickle читается другим процессом со старым Python — берите минимально достаточный протокол. По умолчанию подходит `pickle.DEFAULT_PROTOCOL`; для совместимости можно зафиксировать конкретное число.

**Q9. Что такое `__reduce__` и когда его переопределять?**
`__reduce__` возвращает рецепт восстановления: callable и аргументы. Переопределяют для C-расширений, объектов без обычного `__dict__`, или когда нужна нестандартная логика конструирования. Для большинства классов достаточно `__getstate__`/`__setstate__`.

**Q10. Потокобезопасен ли pickle? Детерминирован ли вывод?**
`pickle.dumps` не гарантирует **байт-в-байт** одинаковый результат между версиями Python/протоколами; порядок может зависеть от порядка обхода словарей/множеств. Не используйте pickle-байты как стабильный хэш/ключ кэша между версиями.

**Q11. Чем pickle отличается от `copy.deepcopy`?**
`deepcopy` использует тот же протокол (`__reduce_ex__`), но копирует в памяти без перехода в байты. pickle — это сериализация в переносимый поток. Многие объекты, ломающие pickle, ломают и deepcopy.

## Подводные камни (gotchas)

### БЕЗОПАСНОСТЬ — главный gotcha

> **Никогда не делайте `pickle.load`/`loads` данных, которым вы не доверяете полностью.** Это эквивалент `eval()` над произвольным кодом.

Демонстрация (НЕ запускать с реальной нагрузкой — иллюстрация механизма):

```python
import pickle
import os

class Exploit:
    def __reduce__(self):
        # При десериализации pickle ВЫЗОВЕТ os.system('...')
        return (os.system, ("echo ВЫПОЛНЕН ПРОИЗВОЛЬНЫЙ КОД",))

payload = pickle.dumps(Exploit())
# Любая система, делающая pickle.loads(payload), выполнит команду:
pickle.loads(payload)  # печатает "ВЫПОЛНЕН ПРОИЗВОЛЬНЫЙ КОД"
```

Почему так: при загрузке pickle вызывает результат `__reduce__` — то есть `os.system(...)`. Атакующему достаточно подсунуть байты.

Меры защиты:
- не принимайте pickle из сети / от пользователей / из недоверенных файлов;
- для обмена данными используйте JSON/MessagePack/Protobuf;
- если pickle неизбежен — подписывайте данные **HMAC** и проверяйте подпись ДО загрузки:

```python
import hmac, hashlib, pickle

SECRET = b"super-secret-key"

def sign(payload: bytes) -> bytes:
    mac = hmac.new(SECRET, payload, hashlib.sha256).digest()
    return mac + payload

def verify_and_load(blob: bytes):
    mac, payload = blob[:32], blob[32:]
    expected = hmac.new(SECRET, payload, hashlib.sha256).digest()
    if not hmac.compare_digest(mac, expected):  # защита от timing-атак
        raise ValueError("Подпись не совпадает — данные подделаны")
    return pickle.loads(payload)
```

- либо ограничивайте `find_class` через кастомный `Unpickler` (whitelist классов), как показано выше. Но это **не** полная защита: безопаснее вообще не использовать pickle для недоверенных данных.

### Прочие подводные камни

- **Бинарный режим обязателен.** Открытие файла в текстовом режиме (`'w'`/`'r'`) приведёт к ошибке/порче данных.
- **Хрупкость при рефакторинге.** Переименование/перемещение класса или функции ломает загрузку старых pickle (`ModuleNotFoundError`, `AttributeError`).
- **Версии Python/протокола.** pickle, созданный новым протоколом, не прочитается старым Python.
- **`pickle` ≠ `cPickle`.** В Python 3 это объединено; в Python 2 был отдельный быстрый `cPickle`.
- **Большие объекты в памяти.** `dumps` держит весь объект и результат в памяти; для гигабайтных данных используйте `dump` в файл и протокол 5 с буферами.
- **numpy/pandas.** Pickle работает, но для совместимости версий numpy лучше специализированные форматы; protocol 5 даёт zero-copy для буферов.
- **Изменяемые дефолты в `__reduce__`** и разделяемое состояние могут давать неожиданные алиасы после загрузки.

## Лучшие практики

1. **Только доверенные источники.** Pickle — для внутренних, контролируемых данных. Для внешнего обмена — JSON/Protobuf.
2. **Подписывайте HMAC**, если pickle передаётся/хранится там, куда возможен внешний доступ.
3. **Фиксируйте протокол** явно (`protocol=`) для воспроизводимости и совместимости.
4. **Версионируйте состояние** классов (поле `version` в `__getstate__`), обрабатывайте миграции в `__setstate__`.
5. **Исключайте ресурсы** (сокеты, локи, файлы) через `__getstate__`, пересоздавайте в `__setstate__`.
6. **Файлы — в бинарном режиме** (`wb`/`rb`).
7. **Не используйте pickle-байты как стабильный хэш** между версиями Python.
8. Для лямбд/замыканий/сложных функций берите `cloudpickle`/`dill`, а не самопальные обёртки.

## Шпаргалка

```python
import pickle

# --- В память ---
b = pickle.dumps(obj, protocol=pickle.HIGHEST_PROTOCOL)  # объект -> bytes
obj = pickle.loads(b)                                    # bytes  -> объект

# --- В файл (ТОЛЬКО бинарный режим) ---
with open("x.pkl", "wb") as f:
    pickle.dump(obj, f, protocol=5)
with open("x.pkl", "rb") as f:
    obj = pickle.load(f)

# --- Константы протоколов ---
pickle.HIGHEST_PROTOCOL   # 5
pickle.DEFAULT_PROTOCOL   # дефолт текущего Python

# --- Хуки класса ---
# __reduce__(self) -> (callable, args[, state, listitems, dictitems])
# __getstate__(self) -> что сохранить
# __setstate__(self, state) -> как восстановить

# --- Нельзя сериализовать: лямбды, локальные функции, сокеты, файлы, локи, генераторы

# --- БЕЗОПАСНОСТЬ ---
# pickle.loads(недоверенные_байты)  ==  выполнение произвольного кода (RCE)!
# Защита: HMAC-подпись ДО load, либо вообще не использовать pickle для чужих данных.
```

### pickle vs json — сводная таблица

| Критерий | pickle | json |
|---|---|---|
| Формат | бинарный | текстовый |
| Кросс-язык | нет (только Python) | да |
| Читаемость человеком | нет | да |
| Поддерживаемые типы | почти любые объекты, классы, циклы | dict/list/str/number/bool/null |
| Безопасность загрузки | **опасно** (RCE) | безопасно |
| Скорость/компактность | обычно быстрее/компактнее | медленнее на сложных данных |
| datetime/Decimal/set | сериализуются | нужны кастомные энкодеры/недоступны |
| Применение | внутренний кэш, IPC, чекпоинты | API, конфиги, обмен данными |
