# asyncio — подготовка к собеседованию

## Что это и зачем

`asyncio` — стандартная библиотека Python для написания **конкурентного** кода с помощью синтаксиса `async`/`await`. Это реализация **кооперативной многозадачности** в рамках **одного потока** и **одного процесса**.

Главная идея: пока одна корутина **ждёт** ввод-вывод (сеть, диск, БД, таймер), управление возвращается в **event loop**, который запускает другие корутины. Так один поток обслуживает тысячи одновременных соединений без блокировки.

Зачем нужен:
- **I/O-bound задачи** — веб-серверы, HTTP-клиенты, работа с БД, очередями, WebSocket. Большую часть времени программа ждёт ответа сети/диска, а не считает на CPU.
- **Высокая конкурентность дёшево** — корутина весит сотни байт против ~8 МБ стека у потока ОС. Можно держать десятки/сотни тысяч одновременных задач.
- **Отсутствие гонок данных в привычном смысле** — переключение происходит только в точках `await`, между ними код выполняется атомарно (в одном потоке).

Когда НЕ нужен:
- **CPU-bound задачи** (тяжёлые вычисления, обработка изображений, ML-инференс) — здесь нужен `multiprocessing` или вынос в `ProcessPoolExecutor`, потому что asyncio работает в одном потоке и не обходит GIL.

```
Сравнение моделей конкурентности:
┌────────────────┬───────────────┬──────────────┬─────────────────┐
│ Модель         │ Параллелизм   │ Накладные    │ Для чего        │
├────────────────┼───────────────┼──────────────┼─────────────────┤
│ asyncio        │ нет (1 поток) │ минимальные  │ I/O-bound       │
│ threading      │ нет (GIL)*    │ средние      │ I/O-bound,      │
│                │               │              │ блокирующие API │
│ multiprocessing│ да            │ большие      │ CPU-bound       │
└────────────────┴───────────────┴──────────────┴─────────────────┘
* до Python 3.13 (в 3.13+ есть экспериментальный free-threaded режим)
```

## Ключевые концепции

### Event loop (цикл событий)
Сердце asyncio. Это бесконечный цикл, который:
1. Берёт готовые к выполнению задачи и выполняет их код **до ближайшего `await`**.
2. Регистрирует ожидаемые события (готовность сокета, истечение таймера) в системном механизме мультиплексирования (`epoll`/`kqueue`/`select` через модуль `selectors`).
3. Когда событие наступает — «будит» соответствующую корутину и продолжает её выполнение.

В каждый момент времени **выполняется ровно одна** корутина. Конкурентность достигается за счёт быстрого переключения в точках ожидания.

### Coroutine (корутина)
Функция, объявленная через `async def`. Её **вызов не выполняет тело**, а возвращает **объект корутины** (coroutine object). Чтобы код выполнился, корутину нужно либо `await`-нуть, либо передать в event loop (через `asyncio.run`, `create_task`, `gather` и т.п.).

```python
import asyncio

async def hello():
    print("начало")
    await asyncio.sleep(1)   # отдаём управление циклу на 1 секунду
    print("конец")
    return 42

# Просто вызов НЕ запускает код — будет RuntimeWarning: coroutine was never awaited
coro = hello()          # это объект корутины, тело ещё не выполнялось
result = asyncio.run(hello())  # вот так запускается, result == 42
```

### Awaitable (ожидаемый объект)
Объект, который можно использовать с `await`. Три вида:
- **корутины** (`async def`),
- **Task** (обёртка над корутиной, запланированная в цикле),
- **Future** (низкоуровневый объект-обещание результата).

### Future
Низкоуровневый объект, представляющий **результат, который появится в будущем**. У него есть состояния: pending → finished (с результатом или исключением). Обычно прикладной код не создаёт Future руками — это делают библиотеки. `Task` — это подкласс `Future`.

### Task (задача)
**Обёртка над корутиной**, которая планирует её выполнение в event loop **немедленно и конкурентно**. Именно Task даёт настоящую конкурентность: создав несколько задач, вы запускаете несколько корутин «параллельно» (с точки зрения ожидания I/O).

```python
async def main():
    # create_task сразу планирует корутину в цикл — она начнёт выполняться
    task = asyncio.create_task(hello())
    print("задача запущена, но мы не ждём её здесь")
    await task   # дожидаемся завершения и получаем результат
```

## Основные функции/классы/методы

### asyncio.run()
Точка входа в асинхронную программу. Создаёт **новый event loop**, выполняет корутину, по завершении корректно закрывает loop (отменяет недоделанные задачи, завершает асинхронные генераторы).

```python
import asyncio

async def main():
    print("работаю в event loop")
    await asyncio.sleep(0.5)

asyncio.run(main())  # нельзя вызывать, если loop уже запущен!
```

Важно: `asyncio.run()` **нельзя** вызывать внутри уже запущенного цикла (например, в Jupyter, где loop уже работает). Должна быть **одна** точка входа на программу.

### await
Приостанавливает текущую корутину до завершения awaitable, **отдавая управление event loop**. Пока ждём — цикл может выполнять другие задачи.

```python
async def fetch(name, delay):
    await asyncio.sleep(delay)   # имитация сетевого запроса
    return f"{name} готов за {delay}с"

async def main():
    # ПОСЛЕДОВАТЕЛЬНО — каждый await ждёт завершения предыдущего (всего ~3с)
    a = await fetch("A", 1)
    b = await fetch("B", 2)
    print(a, b)
```

### asyncio.create_task()
Планирует корутину как Task — она начинает выполняться **конкурентно**, не дожидаясь `await`. Возвращает объект Task.

```python
async def main():
    # Обе задачи стартуют сразу и выполняются конкурентно (всего ~2с, а не 3)
    t1 = asyncio.create_task(fetch("A", 1))
    t2 = asyncio.create_task(fetch("B", 2))
    print(await t1)
    print(await t2)
```

ВАЖНО: нужно сохранять ссылку на Task. Сборщик мусора может уничтожить задачу, на которую нет «сильных» ссылок, и она прервётся. Поэтому держим список задач:
```python
background_tasks = set()

def spawn(coro):
    task = asyncio.create_task(coro)
    background_tasks.add(task)
    task.add_done_callback(background_tasks.discard)  # убираем по завершении
    return task
```

### asyncio.gather()
Запускает несколько awaitable **конкурентно** и ждёт их **все**, возвращая список результатов **в порядке аргументов** (не в порядке завершения).

```python
async def main():
    results = await asyncio.gather(
        fetch("A", 1),
        fetch("B", 2),
        fetch("C", 1.5),
    )
    print(results)  # ['A готов...', 'B готов...', 'C готов...'] — порядок аргументов
```

Поведение при ошибке:
- По умолчанию (`return_exceptions=False`): первое исключение **пробрасывается** немедленно вызывающему, но **остальные задачи НЕ отменяются** автоматически и продолжают работать в фоне (источник утечек!).
- С `return_exceptions=True`: исключения **возвращаются как элементы списка** наравне с результатами, gather не падает.

```python
async def boom():
    raise ValueError("ошибка")

async def main():
    # Исключения попадут в результат как объекты, а не выбросятся
    res = await asyncio.gather(fetch("A", 1), boom(), return_exceptions=True)
    # res == ['A готов за 1с', ValueError('ошибка')]
```

### asyncio.TaskGroup (Python 3.11+) — современная замена gather
Структурированная конкурентность: контекстный менеджер, который ждёт все задачи и **отменяет оставшиеся при ошибке** в любой из них. Ошибки собираются в `ExceptionGroup`.

```python
async def main():
    async with asyncio.TaskGroup() as tg:
        t1 = tg.create_task(fetch("A", 1))
        t2 = tg.create_task(fetch("B", 2))
    # Выход из блока произойдёт только когда ВСЕ задачи завершатся.
    # Если одна упадёт — остальные будут отменены, ошибки -> ExceptionGroup.
    print(t1.result(), t2.result())
```

Рекомендуется предпочитать `TaskGroup` вместо `gather` в новом коде (3.11+) — безопаснее по отмене.

### asyncio.wait_for()
Ограничивает время ожидания awaitable. При таймауте **отменяет** задачу и бросает `asyncio.TimeoutError` (в 3.11+ это псевдоним `TimeoutError`).

```python
async def main():
    try:
        # Ждём не больше 1с; fetch требует 5с -> таймаут и отмена
        result = await asyncio.wait_for(fetch("slow", 5), timeout=1.0)
    except asyncio.TimeoutError:
        print("не уложились в таймаут, задача отменена")
```

### asyncio.timeout() (Python 3.11+) — контекстный менеджер таймаута
```python
async def main():
    try:
        async with asyncio.timeout(1.0):
            await fetch("slow", 5)
    except TimeoutError:
        print("таймаут истёк")
```

### asyncio.wait()
Низкоуровневое ожидание набора задач с гибкими условиями (`FIRST_COMPLETED`, `FIRST_EXCEPTION`, `ALL_COMPLETED`). Возвращает два множества: `done` и `pending`. **Не пробрасывает** исключения сам — их нужно доставать из задач. Принимает только Task/Future (не голые корутины с 3.11+).

```python
async def main():
    tasks = [asyncio.create_task(fetch(str(i), i)) for i in range(1, 4)]
    done, pending = await asyncio.wait(tasks, return_when=asyncio.FIRST_COMPLETED)
    for t in done:
        print("завершилась:", t.result())
    for t in pending:
        t.cancel()  # отменяем незавершённые вручную
```

### asyncio.as_completed()
Итерация по задачам **в порядке их завершения** (а не в порядке передачи).

```python
async def main():
    coros = [fetch("A", 3), fetch("B", 1), fetch("C", 2)]
    for coro in asyncio.as_completed(coros):
        result = await coro
        print("готово:", result)   # B, потом C, потом A
```

### asyncio.sleep()
Неблокирующая «спячка»: отдаёт управление циклу на заданное время. **Это не `time.sleep()`** — `time.sleep` заблокировал бы весь поток и весь event loop.

```python
await asyncio.sleep(2)       # пауза, но цикл обслуживает другие задачи
await asyncio.sleep(0)       # «yield» — даёт циклу шанс переключиться, не засыпая
```

### asyncio.shield()
Защищает awaitable от отмены извне.

```python
async def main():
    # Если внешний код отменит main, critical всё равно попытается доработать
    await asyncio.shield(critical_operation())
```

### asyncio.to_thread() (Python 3.9+) — спасение от блокирующего кода
Выполняет **блокирующую** функцию в отдельном потоке (через пул), возвращая awaitable. Так блокирующий вызов не замораживает event loop.

```python
import time

def blocking_io():
    time.sleep(3)            # блокирующая операция (например, requests, тяжёлая БД-библиотека)
    return "готово"

async def main():
    # Выполняем в потоке, цикл остаётся свободным
    result = await asyncio.to_thread(blocking_io)
```

### Низкоуровневое: loop.run_in_executor()
Тот же смысл, что `to_thread`, но можно указать пул (потоков или процессов).

```python
from concurrent.futures import ProcessPoolExecutor

async def main():
    loop = asyncio.get_running_loop()
    # CPU-bound -> в пул процессов
    with ProcessPoolExecutor() as pool:
        result = await loop.run_in_executor(pool, cpu_heavy_func, arg)
```

### Async context managers (`async with`)
Объект с методами `__aenter__` / `__aexit__` — асинхронный вход/выход. Нужен, когда установка/закрытие ресурса само асинхронно (открытие соединения, транзакция).

```python
class AsyncResource:
    async def __aenter__(self):
        await asyncio.sleep(0.1)   # асинхронное подключение
        print("ресурс открыт")
        return self

    async def __aexit__(self, exc_type, exc, tb):
        await asyncio.sleep(0.1)   # асинхронное закрытие
        print("ресурс закрыт")

async def main():
    async with AsyncResource() as r:
        print("работаем с ресурсом")
```

### Async iterators / generators (`async for`)
Объект с `__aiter__` / `__anext__`, либо `async def` с `yield`. Позволяет получать элементы асинхронно (например, читать чанки из сети по мере поступления).

```python
async def stream(n):
    for i in range(n):
        await asyncio.sleep(0.5)   # имитация ожидания следующего элемента
        yield i                    # асинхронный генератор

async def main():
    async for value in stream(3):
        print("получено:", value)

    # асинхронный генератор-выражение
    result = [x async for x in stream(3)]
```

### asyncio.Lock — взаимная блокировка
Гарантирует, что критическую секцию выполняет только одна корутина одновременно. Нужен, когда между `await`-ами есть инвариант, который нельзя нарушать.

```python
lock = asyncio.Lock()

async def critical():
    async with lock:                 # вход в критическую секцию
        # пока держим lock, другие корутины ждут здесь
        await asyncio.sleep(1)       # await внутри безопасен — lock держится
```

Зачем нужен, если поток один? Между двумя `await` другая корутина может вклиниться и нарушить инвариант (например, прочитать-изменить-записать общую переменную). Lock сериализует доступ через границы `await`.

### asyncio.Semaphore — ограничение параллелизма
Разрешает одновременный доступ не более чем N корутинам. Классика для ограничения числа одновременных запросов.

```python
async def main():
    sem = asyncio.Semaphore(3)       # максимум 3 одновременно

    async def worker(i):
        async with sem:              # не больше 3 воркеров внутри одновременно
            await fetch(str(i), 1)

    await asyncio.gather(*(worker(i) for i in range(20)))
```

### asyncio.Event и asyncio.Queue
```python
# Event — сигнал между корутинами
event = asyncio.Event()
async def waiter():
    await event.wait()               # ждём, пока кто-то вызовет event.set()
    print("событие наступило")

# Queue — потокобезопасная (в рамках цикла) очередь producer/consumer
async def producer(q):
    for i in range(5):
        await q.put(i)               # ждёт, если очередь полна (maxsize)
    await q.put(None)                # сигнал завершения

async def consumer(q):
    while True:
        item = await q.get()         # ждёт, если очередь пуста
        if item is None:
            break
        print("обработали", item)
        q.task_done()
```

### Отмена задач (cancellation)
`task.cancel()` бросает внутрь корутины `asyncio.CancelledError` в ближайшей точке `await`. Корректный код может перехватить её для очистки, но **должен пробросить дальше**.

```python
async def cancellable():
    try:
        await asyncio.sleep(10)
    except asyncio.CancelledError:
        print("отменяемся, освобождаем ресурсы")
        raise          # обязательно пробросить, иначе отмена не сработает!

async def main():
    task = asyncio.create_task(cancellable())
    await asyncio.sleep(1)
    task.cancel()
    try:
        await task
    except asyncio.CancelledError:
        print("задача отменена")
```

## Частые вопросы на собеседовании

**Q1. Чем asyncio отличается от threading и multiprocessing?**
asyncio — **кооперативная** многозадачность в одном потоке: переключение только в точках `await`, нет настоящего параллелизма, минимальные накладные расходы, идеален для I/O-bound. threading — **вытесняющая** многозадачность ОС, но из-за GIL Python-код не выполняется по-настоящему параллельно; подходит для I/O и блокирующих библиотек. multiprocessing — отдельные процессы, настоящий параллелизм (обходит GIL), для CPU-bound, но дорог по памяти и накладным расходам на IPC.

**Q2. В чём разница между корутиной и Task?**
Корутина — объект, полученный вызовом `async def`-функции; сам по себе он **не выполняется**, пока его не `await`-нут. Task — обёртка, которая **планирует корутину в event loop немедленно** и даёт конкурентное выполнение. `await coro` выполняет последовательно; `create_task(coro)` запускает конкурентно.

**Q3. Что произойдёт, если вызвать `async def`-функцию без `await`?**
Вернётся объект корутины, тело **не выполнится**, и при сборке мусора Python выдаст `RuntimeWarning: coroutine '...' was never awaited`.

**Q4. Почему нельзя использовать `time.sleep()` в корутине?**
`time.sleep()` блокирует **весь поток**, а значит и event loop целиком — все остальные корутины замирают. Нужно `await asyncio.sleep()`, которое отдаёт управление циклу.

**Q5. В чём разница между `gather` и `wait`?**
`gather` возвращает результаты в порядке аргументов и (по умолчанию) пробрасывает первое исключение; удобен, когда нужны все результаты. `wait` возвращает множества `done`/`pending`, не пробрасывает исключения (их достаёшь из задач), позволяет ждать по условию (`FIRST_COMPLETED` и т.д.). В новом коде (3.11+) обоих часто заменяет `TaskGroup`.

**Q6. Что делает `return_exceptions=True` в `gather`?**
Заставляет gather **не падать** при первом исключении, а вернуть исключения как элементы итогового списка наравне с обычными результатами. Без него первое исключение пробрасывается, а остальные задачи продолжают работать в фоне (потенциальная утечка).

**Q7. Зачем нужен `asyncio.Lock`, если поток один и нет настоящего параллелизма?**
Потому что между двумя `await` управление может уйти другой корутине, которая изменит общее состояние. Если инвариант должен сохраняться через границы `await` (например, паттерн read-modify-write), Lock сериализует доступ.

**Q8. Чем `Semaphore` отличается от `Lock`?**
Lock — частный случай семафора с лимитом 1 (только одна корутина в секции). Semaphore(N) разрешает до N корутин одновременно — используется для ограничения степени параллелизма (например, не более 10 одновременных HTTP-запросов).

**Q9. Как запустить блокирующий код (например, `requests` или тяжёлую БД-библиотеку), не заморозив event loop?**
Вынести его в пул потоков: `await asyncio.to_thread(func, *args)` (3.9+) или `loop.run_in_executor(executor, func, *args)`. Для CPU-bound — в `ProcessPoolExecutor`. Идеально — заменить на нативно-асинхронную библиотеку (`aiohttp`, `asyncpg`).

**Q10. Что такое event loop и сколько их может быть?**
Event loop — цикл, который выполняет корутины, регистрирует и обрабатывает события I/O и таймеры через `selectors`. На один поток — один активный loop. `asyncio.run()` создаёт свой loop, выполняет корутину и закрывает его. Запускать `run()` внутри уже работающего loop нельзя.

**Q11. Как работает отмена задачи?**
`task.cancel()` запрашивает отмену: при следующем `await` внутри задачи бросается `CancelledError`. Корутина может перехватить её для очистки ресурсов, но **обязана пробросить дальше**, иначе отмена «проглатывается». Защитить участок от отмены можно через `asyncio.shield()`.

**Q12. Что такое `asyncio.Future` и чем отличается от Task?**
Future — низкоуровневый объект-«обещание» результата с состояниями pending/finished. Task — это подкласс Future, который дополнительно **оборачивает корутину** и сам управляет её выполнением. Future обычно создают библиотеки/драйверы, Task — прикладной код.

**Q13. Что такое `TaskGroup` и почему он лучше `gather`?**
`TaskGroup` (3.11+) реализует структурированную конкурентность: гарантирует, что все запущенные внутри `async with` задачи завершатся к выходу из блока, а при ошибке в любой задаче **отменяет остальные** и собирает ошибки в `ExceptionGroup`. У `gather` при ошибке остальные задачи остаются «висеть», что ведёт к утечкам.

**Q14. Что выведет этот код и почему?**
```python
async def main():
    print("1")
    task = asyncio.create_task(say(2))
    print("3")
    await task
async def say(x):
    print(x)
```
Выведет `1`, `3`, `2`. `create_task` лишь **планирует** корутину, но `say` не выполнится, пока `main` не дойдёт до `await` (точки переключения). До `await task` управление не возвращается циклу, поэтому сначала печатается `3`, и только на `await` цикл запускает `say`, печатая `2`.

**Q15. В чём разница `await asyncio.sleep(0)` и обычной паузы?**
`asyncio.sleep(0)` — это «yield»: корутина мгновенно отдаёт управление event loop, давая другим задачам шанс выполниться, и сразу же ставится обратно в очередь готовых. Используется, чтобы не монополизировать цикл в длинном вычислительном цикле.

**Q16. Можно ли использовать `async for` / `async with` в обычной функции?**
Нет. `async for`, `async with`, `await` допустимы только внутри `async def`. В синхронной функции это синтаксическая ошибка.

**Q17. Почему задача, на которую нет ссылки, может «исчезнуть»?**
event loop хранит только **слабую** ссылку на Task. Если в коде нет сильной ссылки на объект Task, сборщик мусора может удалить его, и задача оборвётся в произвольный момент. Решение — хранить задачи в множестве/списке и удалять через `add_done_callback`.

## Подводные камни (gotchas)

- **Блокирующий вызов замораживает всё.** `time.sleep`, синхронный `requests`, тяжёлая CPU-петля, синхронные драйверы БД блокируют event loop, и вся конкурентность исчезает. Решение: `asyncio.to_thread` / `run_in_executor` / нативные async-библиотеки.
- **Забытый `await`.** Вызов корутины без `await` ничего не выполняет, только создаёт объект и даёт `RuntimeWarning`. Особенно коварно с функциями, возвращающими корутину.
- **Перемешивание sync и async API.** `asyncio.run()` внутри уже запущенного цикла бросает `RuntimeError`. В Jupyter loop уже работает — нужно `await` напрямую или `nest_asyncio`.
- **`gather` не отменяет соседей при ошибке.** Упавшая задача пробрасывает исключение, остальные продолжают работать в фоне → утечки. Используйте `TaskGroup` или `return_exceptions=True` с ручной очисткой.
- **Потеря ссылки на Task** → задача собирается сборщиком мусора и обрывается.
- **Проглатывание `CancelledError`.** Если перехватить её и не пробросить, отмена не сработает, задача «зависнет».
- **`asyncio.wait` с голыми корутинами.** С 3.11+ принимает только Task/Future — оборачивайте корутины в `create_task`.
- **Не закрытые ресурсы при исключении.** Открытые соединения/файлы нужно закрывать через `async with` или `try/finally`, иначе утечки дескрипторов.
- **CPU-bound в asyncio.** Долгие вычисления без `await` монополизируют цикл, конкурентность пропадает. Выносите в процессы.
- **Создание Lock/Event/Queue вне работающего loop.** В старых версиях привязка к loop происходила при создании — создавайте примитивы синхронизации внутри корутины, а не на уровне модуля.

## Лучшие практики

- Используйте **`asyncio.run()`** как единственную точку входа; не управляйте loop вручную без необходимости.
- Предпочитайте **`asyncio.TaskGroup`** (3.11+) вместо `gather`/`wait` — безопасная отмена и сбор ошибок.
- Для таймаутов — **`asyncio.timeout()`** (3.11+) или `wait_for`.
- **Не блокируйте loop**: любой синхронный долгий вызов — в `to_thread` (I/O) или `ProcessPoolExecutor` (CPU).
- Выбирайте **нативно-асинхронные** библиотеки: `aiohttp`/`httpx` вместо `requests`, `asyncpg`/`aiomysql` вместо синхронных драйверов, `aiofiles` для файлов.
- **Ограничивайте параллелизм** через `Semaphore`, чтобы не перегрузить внешние сервисы и не упереться в лимиты дескрипторов.
- **Храните ссылки** на фоновые задачи и снимайте их по завершении через `add_done_callback`.
- **Всегда обрабатывайте `CancelledError`** корректно: очистка + `raise`.
- Закрывайте ресурсы через **`async with`** / `try/finally`.
- Не смешивайте несколько event loop в одном потоке; для интеграции с sync-кодом из другого потока — `loop.call_soon_threadsafe` / `asyncio.run_coroutine_threadsafe`.
- Включайте **debug-режим** (`asyncio.run(main(), debug=True)` или `PYTHONASYNCIODEBUG=1`) при отладке — он предупреждает о медленных колбэках и забытых await.

## Шпаргалка

```python
import asyncio

# --- Запуск ---
asyncio.run(main())                       # точка входа, создаёт и закрывает loop

# --- Корутина и await ---
async def f(): ...                        # объявление корутины
await coro                                # последовательное ожидание (отдаёт управление)

# --- Конкурентность ---
task = asyncio.create_task(coro)          # запустить конкурентно (вернёт Task)
results = await asyncio.gather(c1, c2)     # ждать все, результаты по порядку аргументов
await asyncio.gather(*cs, return_exceptions=True)  # не падать на ошибках
async with asyncio.TaskGroup() as tg:      # 3.11+ структурированная конкурентность
    tg.create_task(coro)
for c in asyncio.as_completed(cs):         # итерация по мере завершения
    await c
done, pending = await asyncio.wait(tasks)  # гибкие условия ожидания

# --- Таймауты ---
await asyncio.wait_for(coro, timeout=5)    # таймаут + отмена
async with asyncio.timeout(5):             # 3.11+ таймаут-контекст
    await coro

# --- Паузы / yield ---
await asyncio.sleep(secs)                  # неблокирующая пауза
await asyncio.sleep(0)                     # отдать управление циклу

# --- Блокирующий код ---
await asyncio.to_thread(blocking_fn, *a)   # 3.9+ в пул потоков
loop = asyncio.get_running_loop()
await loop.run_in_executor(pool, fn, *a)   # в указанный пул (потоки/процессы)

# --- Синхронизация ---
async with asyncio.Lock(): ...             # взаимное исключение
async with asyncio.Semaphore(N): ...       # лимит параллелизма
ev = asyncio.Event(); await ev.wait(); ev.set()
q = asyncio.Queue(); await q.put(x); await q.get()

# --- Async-протоколы ---
async with ctx_manager: ...                # __aenter__/__aexit__
async for item in async_iter: ...          # __aiter__/__anext__
async def gen():  yield x                  # асинхронный генератор

# --- Отмена ---
task.cancel()                              # запросить отмену (бросит CancelledError)
await asyncio.shield(coro)                 # защитить от отмены извне
```
