# threading — подготовка к собеседованию

## Что это и зачем

Модуль `threading` предоставляет высокоуровневый API для работы с **потоками** (threads) внутри одного процесса. Потоки — это параллельные ветви исполнения, которые разделяют **общее адресное пространство** (общую память, переменные, файловые дескрипторы). Это и сила, и опасность потоков: общими данными удобно обмениваться, но именно из-за общего доступа возникают race conditions и deadlock'и.

Зачем нужны потоки в Python:

- **I/O-bound задачи** (сетевые запросы, чтение/запись файлов, обращения к БД, ожидание ответа от внешних сервисов). Пока один поток ждёт ответа от сети, интерпретатор отдаёт управление другому потоку. Здесь потоки дают реальный выигрыш.
- **Отзывчивость интерфейса / фоновые задачи** — например, фоновая выгрузка логов, пока основной поток обслуживает запросы.
- **Упрощение архитектуры** там, где нужно много одновременно ожидающих операций, но не хочется писать асинхронный код на `asyncio`.

Чего потоки **НЕ** дают в CPython:

- **Параллельного исполнения Python-байткода на нескольких ядрах CPU.** Из-за GIL (см. ниже) в любой момент времени Python-байткод выполняет только один поток. Поэтому для **CPU-bound** задач (вычисления, обработка чисел, сжатие, шифрование на чистом Python) потоки выигрыша по скорости почти не дают — нужен `multiprocessing` или нативные расширения.

```python
import threading

def worker(name):
    print(f"Привет из потока {name}")

# Создаём и запускаем поток
t = threading.Thread(target=worker, args=("A",))
t.start()   # запускает worker в отдельном потоке
t.join()    # ждём завершения потока
print("Поток завершён")
```

## Ключевые концепции

### GIL (Global Interpreter Lock) — главная тема собеседования

**GIL — это глобальный мьютекс (блокировка) интерпретатора CPython, который гарантирует, что в каждый момент времени байткод Python исполняет ровно один поток.**

Подробно:

1. **Зачем он нужен.** Управление памятью в CPython основано на **подсчёте ссылок** (reference counting). Каждый объект хранит счётчик ссылок; при достижении нуля объект удаляется. Операции увеличения/уменьшения счётчика не атомарны. Если бы два потока одновременно меняли счётчик одного объекта без блокировки, счётчик мог бы испортиться → утечки памяти или преждевременное удаление объекта (use-after-free, краш). GIL — это самое простое решение: один глобальный замок защищает всё внутреннее состояние интерпретатора. Это делает большинство встроенных операций (например, добавление в список) эффективно атомарными и упрощает написание C-расширений.

2. **Как он работает.** Чтобы исполнять байткод, поток должен **держать** GIL. Поток отдаёт GIL в двух случаях:
   - **Добровольно**, когда уходит в блокирующую операцию ввода-вывода (сеть, диск), в `time.sleep()`, или вызывает C-функцию, которая явно освобождает GIL.
   - **Принудительно**, по истечении интервала (в Python 3 по умолчанию каждые ~5 миллисекунд, регулируется `sys.setswitchinterval()`). Интерпретатор просит текущий поток отпустить GIL, чтобы дать шанс другим потокам.

3. **Следствие для CPU-bound.** Если у вас два потока считают числа в цикле на чистом Python, они **не** ускорятся на двух ядрах — они будут по очереди отбирать друг у друга GIL. Из-за накладных расходов на переключение и борьбу за замок многопоточная CPU-bound программа может работать даже **медленнее** однопоточной.

4. **Следствие для I/O-bound.** Когда поток ждёт ответа из сети, он **отпускает GIL**, и другой поток может работать. Поэтому для I/O-bound задач потоки масштабируются хорошо: пока один ждёт, другие работают.

5. **Что GIL НЕ делает.** GIL **не** делает ваш код потокобезопасным на уровне логики. Атомарны отдельные байткод-операции, но составные операции (`counter += 1` — это чтение, сложение, запись) — не атомарны. Между ними может вклиниться переключение потока. Поэтому race conditions в Python вполне реальны (см. ниже).

6. **GIL и C-расширения / numpy.** Многие тяжёлые библиотеки (numpy, pandas, операции с BLAS, сжатие zlib) выполняют вычисления в C-коде и **отпускают GIL** на время этих вычислений. Поэтому numpy-операции из нескольких потоков могут реально параллелиться.

7. **Будущее GIL (важно для senior).** В Python 3.12 появился **per-interpreter GIL** (PEP 684) — субинтерпретаторы могут иметь свои GIL. В Python 3.13 появилась **экспериментальная сборка без GIL** (free-threaded, PEP 703, флаг сборки `--disable-gil`, "free-threading"). Это пока опционально и не дефолт. На собеседовании достаточно знать, что направление развития — отказ от единого GIL, но в стандартной сборке CPython 3.13 GIL по-прежнему есть.

```python
import time
import threading

def cpu_bound(n):
    # Чисто вычислительная нагрузка — упирается в GIL
    total = 0
    for i in range(n):
        total += i * i
    return total

N = 30_000_000

# Однопоточно
start = time.perf_counter()
cpu_bound(N)
cpu_bound(N)
print("Последовательно:", time.perf_counter() - start)

# Два потока — из-за GIL НЕ быстрее, часто медленнее
start = time.perf_counter()
t1 = threading.Thread(target=cpu_bound, args=(N,))
t2 = threading.Thread(target=cpu_bound, args=(N,))
t1.start(); t2.start()
t1.join(); t2.join()
print("Два потока:", time.perf_counter() - start)
# Вывод покажет, что два потока НЕ дают ускорения для CPU-bound
```

### Race condition (состояние гонки)

Ситуация, когда результат зависит от порядка/времени переключения потоков, обращающихся к общим данным без синхронизации.

```python
import threading

counter = 0

def increment():
    global counter
    for _ in range(1_000_000):
        counter += 1   # НЕ атомарно: чтение -> +1 -> запись

threads = [threading.Thread(target=increment) for _ in range(4)]
for t in threads:
    t.start()
for t in threads:
    t.join()

print(counter)  # Почти НИКОГДА не равно 4_000_000 из-за гонки
```

### Deadlock (взаимная блокировка)

Два (и более) потока ждут ресурсы, удерживаемые друг другом, и никто не может продолжить. Классический пример — два замка, захватываемые в разном порядке.

### Daemon threads (демоны)

Поток-демон автоматически завершается, когда завершается главный поток программы. Недемонические потоки удерживают процесс «живым» до своего завершения. Демоны удобны для фоновых задач, но **их нельзя join'ить надёжно** — они могут быть прерваны в любой момент без shutdown-логики.

## Основные функции/классы/методы

### Thread

```python
import threading
import time

def task(name, delay):
    for i in range(3):
        time.sleep(delay)
        print(f"[{name}] шаг {i}, поток id={threading.get_ident()}")

# Способ 1: передать target
t = threading.Thread(target=task, args=("worker", 0.1), name="MyWorker")
t.start()
print("Имя:", t.name, "Жив?", t.is_alive())
t.join(timeout=5)      # ждём максимум 5 секунд
print("Жив после join?", t.is_alive())

# Способ 2: наследование от Thread
class MyThread(threading.Thread):
    def __init__(self, n):
        super().__init__()
        self.n = n
        self.result = None
    def run(self):                 # переопределяем run, НЕ start
        self.result = sum(range(self.n))

th = MyThread(1000)
th.start()
th.join()
print("Результат:", th.result)
```

Полезные функции модуля:

```python
threading.current_thread()        # текущий объект Thread
threading.get_ident()             # числовой id потока
threading.active_count()          # число живых потоков
threading.enumerate()             # список живых потоков
threading.main_thread()           # главный поток
```

### Lock — простой мьютекс

```python
import threading

counter = 0
lock = threading.Lock()

def safe_increment():
    global counter
    for _ in range(1_000_000):
        with lock:           # захват/освобождение через контекстный менеджер
            counter += 1     # критическая секция защищена

threads = [threading.Thread(target=safe_increment) for _ in range(4)]
for t in threads: t.start()
for t in threads: t.join()
print(counter)  # Теперь ВСЕГДА 4_000_000

# Ручной захват (with предпочтительнее — освобождает даже при исключении):
lock.acquire()
try:
    ...  # критическая секция
finally:
    lock.release()

# Неблокирующая попытка
if lock.acquire(blocking=False):
    try:
        ...
    finally:
        lock.release()
else:
    print("Замок занят, делаем что-то другое")
```

`Lock` **нельзя** захватить повторно тем же потоком — это приведёт к deadlock самого с собой. Для рекурсии нужен `RLock`.

### RLock — реентрантный (рекурсивный) замок

Может быть захвачен **одним и тем же потоком** многократно; освобождается столько же раз. Нужен, когда метод под замком вызывает другой метод, который тоже захватывает тот же замок.

```python
import threading

class Account:
    def __init__(self):
        self._lock = threading.RLock()
        self._balance = 0

    def deposit(self, amount):
        with self._lock:
            self._balance += amount

    def transfer_in(self, amount):
        with self._lock:        # уже под замком
            self.deposit(amount) # deposit снова берёт ТОТ ЖЕ замок -> RLock спасает
```

### Semaphore / BoundedSemaphore — счётчик разрешений

Ограничивает число потоков, одновременно входящих в секцию. Удобен для пула ресурсов (например, не более N одновременных соединений).

```python
import threading
import time

# Не более 3 потоков одновременно «качают»
semaphore = threading.Semaphore(3)

def download(i):
    with semaphore:
        print(f"Скачивает {i}")
        time.sleep(1)
        print(f"Готово {i}")

threads = [threading.Thread(target=download, args=(i,)) for i in range(10)]
for t in threads: t.start()
for t in threads: t.join()
```

`BoundedSemaphore` дополнительно бросает `ValueError`, если `release()` вызван больше раз, чем `acquire()` — защита от ошибок.

### Event — флаг-сигнал между потоками

```python
import threading
import time

event = threading.Event()   # изначально не установлен

def waiter():
    print("Жду сигнала...")
    event.wait()            # блокируется, пока не set()
    print("Сигнал получен, продолжаю")

def setter():
    time.sleep(2)
    print("Устанавливаю событие")
    event.set()             # будит всех ожидающих

threading.Thread(target=waiter).start()
threading.Thread(target=setter).start()

# event.clear()   — сбросить флаг
# event.is_set()  — проверить
# event.wait(timeout=5)  — ждать с таймаутом, вернёт True/False
```

### Condition — условная переменная (для сложной координации)

Позволяет потокам ждать выполнения условия и будить друг друга. Под капотом — замок + механизм ожидания/уведомления.

```python
import threading
import collections

class Buffer:
    def __init__(self, size):
        self.size = size
        self.items = collections.deque()
        self.cond = threading.Condition()

    def put(self, item):
        with self.cond:
            while len(self.items) >= self.size:
                self.cond.wait()          # ждём, пока освободится место
            self.items.append(item)
            self.cond.notify()            # будим потребителя

    def get(self):
        with self.cond:
            while not self.items:
                self.cond.wait()          # ждём появления элемента
            item = self.items.popleft()
            self.cond.notify()            # будим производителя
            return item
```

Важно: условие проверяют в цикле `while`, а не `if`, из-за возможных «ложных пробуждений» (spurious wakeups) и того, что между notify и реальным захватом замка состояние могло измениться.

### Barrier — барьер синхронизации

Заставляет N потоков дождаться друг друга в одной точке, прежде чем все продолжат.

```python
import threading
import time
import random

barrier = threading.Barrier(3)   # ждём 3 потока

def worker(i):
    time.sleep(random.random())
    print(f"Поток {i} достиг барьера")
    barrier.wait()               # блок, пока все 3 не дойдут
    print(f"Поток {i} прошёл барьер")

threads = [threading.Thread(target=worker, args=(i,)) for i in range(3)]
for t in threads: t.start()
for t in threads: t.join()
```

### local — thread-local данные

Каждый поток видит свою копию атрибутов объекта.

```python
import threading

data = threading.local()

def show():
    # У каждого потока свой data.value
    print(threading.current_thread().name, getattr(data, "value", "не задано"))

def init(v):
    data.value = v
    show()

threading.Thread(target=init, args=(1,)).start()
threading.Thread(target=init, args=(2,)).start()
```

## Частые вопросы на собеседовании

**Q1. Что такое GIL и зачем он нужен?**
GIL — глобальный мьютекс CPython, гарантирующий исполнение Python-байткода только одним потоком одновременно. Нужен для защиты подсчёта ссылок и внутреннего состояния интерпретатора; упрощает реализацию и C-расширения. Следствие: потоки не ускоряют CPU-bound код, но эффективны для I/O-bound, так как при блокирующем вводе-выводе GIL отпускается.

**Q2. Дают ли потоки в Python настоящий параллелизм?**
Для чистого Python-байткода — нет (из-за GIL, исполняется один поток за раз; это конкурентность, а не параллелизм). Настоящий параллелизм на ядрах дают: `multiprocessing`, C-расширения, отпускающие GIL (numpy), или free-threaded сборка 3.13+.

**Q3. В чём разница между потоками и процессами?**
Потоки разделяют общую память одного процесса (легко обмениваться данными, но нужна синхронизация, и есть GIL). Процессы имеют изолированную память (обмен через IPC/pickling, дороже создание, но обходят GIL и дают параллелизм на ядрах).

**Q4. Когда выбирать threading, а когда multiprocessing?**
threading — для I/O-bound (сеть, диск, БД, ожидание). multiprocessing — для CPU-bound (вычисления). Для I/O часто ещё лучше `asyncio`.

**Q5. Что такое race condition и как её предотвратить?**
Состояние гонки — некорректный результат из-за неконтролируемого порядка доступа потоков к общим данным. Предотвращают синхронизацией: `Lock`/`RLock`, атомарными структурами (`queue.Queue`), неизменяемостью данных, или избеганием общего изменяемого состояния.

**Q6. Чем Lock отличается от RLock?**
`Lock` нельзя захватить повторно тем же потоком (будет deadlock). `RLock` — реентрантный: один поток может захватить его несколько раз (счётчик), должен освободить столько же раз. Нужен при вложенных вызовах под одним замком.

**Q7. Что такое deadlock и как его избежать?**
Взаимная блокировка — потоки бесконечно ждут ресурсы друг друга. Избегают: единым порядком захвата замков, таймаутами на `acquire`, минимизацией числа удерживаемых одновременно замков, использованием более высокоуровневых примитивов (`queue.Queue`).

**Q8. Что такое daemon thread?**
Поток-демон автоматически уничтожается при завершении главного потока, не блокирует выход программы. Подходит для фоновых задач, но его нельзя корректно «дождаться» — он может оборваться в любой момент, поэтому не используйте демонов там, где нужна гарантированная финализация.

**Q9. Зачем нужен join()?**
`join()` блокирует вызывающий поток, пока целевой не завершится. Используется, чтобы дождаться результата/завершения перед продолжением. Без join главный поток может завершиться раньше рабочих (а недемонические потоки задержат выход процесса).

**Q10. Является ли `counter += 1` атомарной операцией?**
Нет. Это три шага: чтение значения, инкремент, запись. Между ними возможно переключение потоков → потеря обновлений. GIL не спасает, потому что переключение происходит между байткод-инструкциями.

**Q11. Почему `list.append()` потокобезопасен, а `+=` нет?**
Отдельные методы встроенных типов реализованы на C и выполняются под GIL как одна операция, поэтому `append` атомарен. `+=` — это составная операция (чтение + операция + присваивание) на уровне байткода, и между шагами поток может быть прерван.

**Q12. Как поделиться результатом работы потока?**
Через общий объект под замком, через `queue.Queue` (потокобезопасна), через атрибут собственного подкласса `Thread`, или удобнее — через `concurrent.futures.ThreadPoolExecutor`, где `future.result()` вернёт результат и пробросит исключение.

**Q13. Можно ли «убить» поток в Python?**
Нет штатного безопасного способа принудительно завершить поток. Правильный подход — кооперативная остановка: поток периодически проверяет `threading.Event` (флаг остановки) и завершается сам.

**Q14. Влияет ли количество ядер CPU на скорость многопоточной CPU-bound программы в CPython?**
В стандартной сборке практически нет: из-за GIL дополнительные ядра не используются для чистого Python-байткода. Выигрыш появится только с процессами, GIL-освобождающими расширениями или free-threaded сборкой.

## Подводные камни (gotchas)

- **`counter += 1` не атомарен.** Защищайте составные операции замком, даже несмотря на GIL.
- **GIL ≠ потокобезопасность логики.** GIL защищает интерпретатор, а не вашу инвариантность данных.
- **CPU-bound потоки не ускоряются**, а могут замедляться из-за борьбы за GIL. Берите `multiprocessing`.
- **Lock, захваченный дважды одним потоком, → deadlock.** Нужен `RLock`.
- **Разный порядок захвата замков → deadlock.** Захватывайте замки в едином глобальном порядке.
- **Демонические потоки могут оборваться** в любой момент без выполнения `finally`/закрытия ресурсов. Не используйте для важной финализации.
- **Исключение в потоке не «всплывает» в главный поток** — оно по умолчанию печатается и теряется. Используйте `ThreadPoolExecutor`/`future.result()` или ловите исключения вручную внутри потока.
- **`Condition.wait()` нужно вызывать в цикле `while`**, проверяя условие, а не одним `if` (ложные пробуждения).
- **Не передавайте `target=task()` со скобками** — это вызовет функцию сразу в главном потоке. Нужно `target=task, args=(...)`.
- **Слишком много потоков** — память на стек каждого и накладные расходы на переключение. Используйте пулы (`ThreadPoolExecutor`).
- **`time.sleep` в цикле блокировки** не отменяется мгновенно — для отменяемого ожидания используйте `Event.wait(timeout)`.

## Лучшие практики

- Предпочитайте **высокоуровневые API**: `concurrent.futures.ThreadPoolExecutor` и `queue.Queue` вместо ручного управления потоками и замками.
- **Минимизируйте общее изменяемое состояние.** Лучший замок — отсутствие разделяемых данных; передавайте данные через очереди.
- Захватывайте замки через `with lock:` — гарантирует освобождение при исключении.
- Держите **критические секции короткими**, не выполняйте под замком блокирующий ввод-вывод.
- Используйте **единый порядок** захвата нескольких замков во всей программе.
- Для остановки потоков применяйте **кооперативную отмену** через `threading.Event`, а не «убийство».
- Получайте результаты и **пробрасывайте исключения** через `Future`/`Queue`, не теряйте ошибки в потоках.
- Для **CPU-bound** переходите на `multiprocessing`/`ProcessPoolExecutor`, а для массового I/O рассмотрите `asyncio`.
- Ограничивайте число одновременных ресурсоёмких операций через `Semaphore`.
- Делайте код **идемпотентным и проверяйте условия в цикле** при работе с `Condition`.

## Шпаргалка

```python
import threading

# Создание и запуск
t = threading.Thread(target=fn, args=(...), kwargs={...}, name="w", daemon=True)
t.start(); t.join(timeout=None)
t.is_alive()

# Замки
lock = threading.Lock()         # нереентрантный мьютекс
rlock = threading.RLock()       # реентрантный (рекурсивный)
with lock: ...                  # предпочтительный способ
ok = lock.acquire(blocking=False, timeout=-1)

# Семафор (ограничение параллелизма)
sem = threading.Semaphore(N)    # BoundedSemaphore — с проверкой release
with sem: ...

# Событие-сигнал
ev = threading.Event()
ev.set(); ev.clear(); ev.is_set(); ev.wait(timeout=None)

# Условная переменная
cond = threading.Condition()
with cond:
    while not predicate: cond.wait()
    cond.notify()  / cond.notify_all()

# Барьер
b = threading.Barrier(N); b.wait()

# Thread-local
tl = threading.local(); tl.x = 1

# Утилиты
threading.current_thread(); threading.get_ident()
threading.active_count(); threading.enumerate(); threading.main_thread()
```

| Сценарий | Что использовать |
|----------|------------------|
| I/O-bound, много ожидания | threading / ThreadPoolExecutor / asyncio |
| CPU-bound, вычисления | multiprocessing / ProcessPoolExecutor |
| Защита счётчика/инварианта | Lock / RLock |
| Ограничить число одновременных операций | Semaphore |
| Сигнал «событие произошло» | Event |
| Сложная координация по условию | Condition |
| Сбор всех в одной точке | Barrier |
| Безопасный обмен данными | queue.Queue |

**Запомнить:** GIL → один поток исполняет байткод за раз → потоки для I/O, процессы для CPU. GIL не делает `+=` атомарным.
