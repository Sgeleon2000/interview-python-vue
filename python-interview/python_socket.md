# socket — подготовка к собеседованию

## Что это и зачем

Модуль `socket` — низкоуровневый интерфейс к **сетевому API операционной системы** (Berkeley sockets / BSD sockets). Сокет — это абстракция **конечной точки сетевого соединения**, через которую процессы обмениваются данными по сети (или локально).

Зачем нужен:
- Прямая работа с **TCP** и **UDP** без надстроек.
- Понимание, как «под капотом» устроены HTTP-серверы, БД-драйверы, мессенджеры, игры.
- Реализация собственных протоколов, прокси, сетевых утилит.

На сокетах построено практически всё сетевое в Python: `http`, `urllib`, `requests`, фреймворки, БД-клиенты.

## Ключевые концепции

### Семейства адресов (address family)
- **`AF_INET`** — IPv4 (адрес = `(host, port)`).
- **`AF_INET6`** — IPv6.
- **`AF_UNIX`** — локальные Unix-сокеты (адрес = путь к файлу), быстрый IPC на одной машине.

### Типы сокетов (socket type)
- **`SOCK_STREAM`** — потоковый, надёжный, с установлением соединения → **TCP**. Гарантирует доставку и порядок байтов.
- **`SOCK_DGRAM`** — датаграммный, без соединения → **UDP**. Быстрый, но без гарантий доставки/порядка.
- **`SOCK_RAW`** — сырые сокеты (нужны привилегии), для своих протоколов поверх IP.

### TCP против UDP
```
┌──────────────┬───────────────────────┬─────────────────────────┐
│              │ TCP (SOCK_STREAM)     │ UDP (SOCK_DGRAM)        │
├──────────────┼───────────────────────┼─────────────────────────┤
│ Соединение   │ да (handshake)        │ нет                     │
│ Надёжность   │ гарантия доставки      │ нет гарантий            │
│ Порядок      │ сохраняется            │ может нарушаться        │
│ Границы сообщ│ потоковый (нет границ) │ датаграммы (есть границы)│
│ Скорость     │ медленнее (overhead)   │ быстрее                 │
│ Применение   │ HTTP, БД, файлы       │ DNS, видео, игры, метрики│
└──────────────┴───────────────────────┴─────────────────────────┘
```

### Потоковая природа TCP
TCP — это **поток байтов без границ сообщений**. Один `send()` на одной стороне может прийти как несколько `recv()` на другой и наоборот. Поэтому **приложение само** обозначает границы сообщений (разделитель, например `\n`, или префикс длины). Это классический источник багов на собеседовании.

## Основные функции/классы/методы

### Создание сокета
```python
import socket

# TCP/IPv4
tcp_sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
# UDP/IPv4
udp_sock = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
```

### Серверные методы
- **`bind((host, port))`** — привязать сокет к адресу/порту.
- **`listen(backlog)`** — перевести в режим прослушивания, `backlog` — длина очереди ожидающих соединений.
- **`accept()`** — заблокироваться и дождаться клиента; возвращает **новый сокет** для общения с этим клиентом и его адрес.

### Клиентские методы
- **`connect((host, port))`** — установить соединение с сервером.

### Обмен данными (TCP)
- **`send(bytes)`** — отправить, возвращает число реально отправленных байт (может быть меньше!).
- **`sendall(bytes)`** — отправить **всё**, повторяя send в цикле.
- **`recv(bufsize)`** — принять до `bufsize` байт; возвращает `b""` (пустые байты) при **закрытии соединения** удалённой стороной.

### Обмен данными (UDP)
- **`sendto(bytes, (host, port))`** — отправить датаграмму конкретному адресу.
- **`recvfrom(bufsize)`** — принять датаграмму, возвращает `(data, addr)`.

### Закрытие
- **`close()`** — закрыть сокет.
- **`shutdown(how)`** — закрыть направление (`SHUT_RD`, `SHUT_WR`, `SHUT_RDWR`).

### TCP-сервер (эхо)
```python
import socket

HOST, PORT = "127.0.0.1", 9000

server = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
# Разрешаем переиспользовать адрес сразу после рестарта (избегаем "Address already in use")
server.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
server.bind((HOST, PORT))          # привязка к адресу
server.listen(5)                   # очередь до 5 соединений
print(f"Слушаю {HOST}:{PORT}")

while True:
    conn, addr = server.accept()   # блокируемся до подключения клиента
    print("Подключился:", addr)
    with conn:                     # отдельный сокет для этого клиента
        while True:
            data = conn.recv(1024) # читаем до 1024 байт
            if not data:           # b"" => клиент закрыл соединение
                break
            conn.sendall(data)     # эхо обратно (гарантированно всё)
    print("Клиент отключился:", addr)
```

### TCP-клиент
```python
import socket

with socket.socket(socket.AF_INET, socket.SOCK_STREAM) as client:
    client.connect(("127.0.0.1", 9000))   # подключаемся к серверу
    client.sendall(b"Hello, server!")     # отправляем всё
    response = client.recv(1024)          # ждём ответ
    print("Ответ:", response.decode())
```

### UDP-сервер
```python
import socket

server = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
server.bind(("127.0.0.1", 9001))   # listen/accept НЕ нужны для UDP
print("UDP-сервер готов")

while True:
    data, addr = server.recvfrom(1024)    # принимаем датаграмму + адрес отправителя
    print(f"От {addr}: {data.decode()}")
    server.sendto(data, addr)             # отвечаем тому же адресу
```

### UDP-клиент
```python
import socket

client = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
client.sendto(b"ping", ("127.0.0.1", 9001))  # connect не обязателен
data, addr = client.recvfrom(1024)
print("Ответ:", data.decode())
client.close()
```

### Корректное чтение целого сообщения по длине
```python
def recv_exactly(sock, n):
    """Читать ровно n байт (TCP не гарантирует, что всё придёт за один recv)."""
    buf = bytearray()
    while len(buf) < n:
        chunk = sock.recv(n - len(buf))
        if not chunk:                      # соединение закрыто раньше времени
            raise ConnectionError("соединение закрыто до получения всех данных")
        buf.extend(chunk)
    return bytes(buf)

import struct
def send_message(sock, payload: bytes):
    # Префикс длины (4 байта, big-endian) + сами данные
    sock.sendall(struct.pack(">I", len(payload)) + payload)

def recv_message(sock) -> bytes:
    raw_len = recv_exactly(sock, 4)        # сначала длина
    (length,) = struct.unpack(">I", raw_len)
    return recv_exactly(sock, length)      # потом ровно столько данных
```

### Блокирующие, неблокирующие сокеты и таймауты
```python
sock.setblocking(True)    # по умолчанию: recv/accept блокируются до данных
sock.setblocking(False)   # неблокирующий: при отсутствии данных бросает BlockingIOError
sock.settimeout(5.0)      # блокирующий, но с таймаутом -> socket.timeout
sock.settimeout(None)     # снова блокирующий без таймаута
```

Неблокирующие сокеты обычно используют вместе с мультиплексированием (`select`/`selectors`) для обслуживания многих соединений в одном потоке:
```python
import socket

sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
sock.setblocking(False)
try:
    data = sock.recv(1024)     # если данных нет — сразу исключение
except BlockingIOError:
    pass                       # данных пока нет, займёмся другими сокетами
```

### Полезные функции
```python
socket.gethostname()                       # имя хоста
socket.gethostbyname("example.com")        # DNS -> IPv4
socket.getaddrinfo("example.com", 443)     # универсальное разрешение (IPv4/IPv6)
socket.inet_aton("127.0.0.1")              # IP-строка -> 4 байта
```

## Частые вопросы на собеседовании

**Q1. В чём разница между TCP и UDP?**
TCP (`SOCK_STREAM`) — с установлением соединения, надёжный, сохраняет порядок, потоковый (без границ сообщений), но медленнее. UDP (`SOCK_DGRAM`) — без соединения, без гарантий доставки и порядка, датаграммы с границами, быстрее. TCP для HTTP/БД/файлов, UDP для DNS/видео/игр/метрик.

**Q2. Что означают `AF_INET` и `SOCK_STREAM`?**
`AF_INET` — семейство адресов IPv4 (адрес — пара host/port). `SOCK_STREAM` — потоковый тип сокета поверх TCP. Вместе `socket(AF_INET, SOCK_STREAM)` создаёт TCP/IPv4-сокет.

**Q3. Какова последовательность вызовов на сервере и клиенте?**
Сервер: `socket()` → `bind()` → `listen()` → `accept()` (в цикле) → `recv`/`send` → `close()`. Клиент: `socket()` → `connect()` → `send`/`recv` → `close()`.

**Q4. Что возвращает `accept()`?**
Кортеж `(conn, addr)`: **новый** сокет `conn` для общения именно с этим клиентом и адрес клиента. Исходный слушающий сокет продолжает принимать новые соединения.

**Q5. В чём разница между `send()` и `sendall()`?**
`send()` может отправить **меньше** байт, чем передано, и возвращает фактическое число — нужно вызывать в цикле. `sendall()` сам повторяет отправку, пока не отправит **все** данные (или не возникнет ошибка).

**Q6. Что значит, если `recv()` вернул пустые байты `b""`?**
Удалённая сторона **корректно закрыла** соединение (EOF). Это сигнал завершить чтение/цикл, а не «нет данных».

**Q7. Почему одно сообщение может прийти частями?**
TCP — поток байтов **без границ сообщений**. Один `send` может разбиться на несколько `recv` и наоборот. Нужно самому выделять сообщения: разделителем (`\n`) или префиксом длины.

**Q8. Что такое `backlog` в `listen()`?**
Максимальная длина очереди соединений, которые приняты ОС, но ещё не обработаны `accept()`. При переполнении новые попытки соединения могут отклоняться или ждать.

**Q9. Зачем `SO_REUSEADDR`?**
Позволяет сразу переиспользовать адрес/порт после перезапуска сервера, не дожидаясь, пока ОС освободит сокет из состояния `TIME_WAIT`. Без него часто получаешь `Address already in use`.

**Q10. Чем отличаются блокирующие и неблокирующие сокеты?**
Блокирующий (по умолчанию): `recv`/`accept`/`connect` приостанавливают поток до готовности. Неблокирующий (`setblocking(False)`): сразу бросают `BlockingIOError`, если операция не может выполниться немедленно; используются с `select`/`selectors`/`epoll` для обслуживания многих соединений одним потоком.

**Q11. Чем `settimeout()` отличается от `setblocking()`?**
`settimeout(t)` оставляет сокет блокирующим, но ограничивает ожидание временем `t` — по истечении бросает `socket.timeout`. `setblocking(False)` эквивалентен `settimeout(0)` — полностью неблокирующий режим.

**Q12. Как обслужить много клиентов в TCP-сервере?**
Варианты: поток/процесс на клиента (`threading`/`multiprocessing` — просто, но дорого при тысячах), мультиплексирование `select`/`selectors`/`epoll` (один поток, много неблокирующих сокетов), либо asyncio (`asyncio.start_server`).

**Q13. Зачем нужен `shutdown()`, если есть `close()`?**
`close()` закрывает сокет целиком и освобождает дескриптор. `shutdown(how)` закрывает только одно направление (чтение/запись), позволяя, например, сообщить «я больше не буду писать» (`SHUT_WR`), но продолжать читать ответ. Часто `shutdown` делают перед `close` для аккуратного завершения.

## Подводные камни (gotchas)

- **TCP не сохраняет границы сообщений** — нельзя полагаться, что один `send` = один `recv`. Используйте разделитель или префикс длины.
- **`send` может отправить не всё** — используйте `sendall` или цикл.
- **`recv()` == `b""` означает закрытие**, а не «данных пока нет».
- **Забыли `SO_REUSEADDR`** → `Address already in use` после рестарта.
- **Работа с `str` вместо `bytes`** — сокеты передают только байты; кодируйте/декодируйте (`.encode()`/`.decode()`).
- **UDP не гарантирует доставку** — датаграммы могут теряться, дублироваться, приходить не по порядку; не используйте для критичных данных без своей логики подтверждений.
- **Слишком маленький `bufsize` в UDP** — лишняя часть датаграммы **теряется** (в отличие от TCP, где остаток дочитается).
- **Не закрытые сокеты** — утечка дескрипторов; используйте `with` или `try/finally`.
- **Блокирующий `accept`/`recv` в одном потоке** не даёт обслуживать других клиентов — нужны потоки или мультиплексирование.

## Лучшие практики

- Используйте сокеты как **контекстные менеджеры** (`with socket.socket(...) as s:`) — автоматическое закрытие.
- Для отправки всегда **`sendall`**, для приёма цельных сообщений — **протокол с длиной** или разделителем.
- Ставьте **`SO_REUSEADDR`** на серверных сокетах.
- Задавайте **таймауты** (`settimeout`), чтобы не зависать навечно.
- Для масштабируемости — **`selectors`** (мультиплексирование) или **asyncio**, а не поток на каждое соединение при больших нагрузках.
- Явно **кодируйте/декодируйте** текст (`utf-8`), не смешивайте `str` и `bytes`.
- Для UDP закладывайте **достаточный буфер** и логику обработки потерь/повторов, если данные важны.
- Используйте **высокоуровневые** библиотеки (`socketserver`, `asyncio`, `http`), когда не нужен низкий уровень.

## Шпаргалка

```python
import socket

# --- Создание ---
s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)   # TCP
u = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)    # UDP

# --- Сервер TCP ---
s.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
s.bind((host, port)); s.listen(5)
conn, addr = s.accept()        # новый сокет на клиента
data = conn.recv(1024)         # b"" => закрыто
conn.sendall(b"...")

# --- Клиент TCP ---
s.connect((host, port))
s.sendall(b"...")
resp = s.recv(1024)

# --- UDP ---
u.bind((host, port))           # сервер
data, addr = u.recvfrom(1024)
u.sendto(b"...", (host, port)) # клиент/ответ

# --- Режимы ---
s.setblocking(False)           # неблокирующий
s.settimeout(5.0)              # таймаут -> socket.timeout
s.shutdown(socket.SHUT_RDWR)   # закрыть направление
s.close()

# --- DNS / адреса ---
socket.gethostbyname("host")
socket.getaddrinfo("host", port)
```
