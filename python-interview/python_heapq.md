# heapq — подготовка к собеседованию

## Что это и зачем

`heapq` — это модуль стандартной библиотеки Python, реализующий **алгоритм двоичной (бинарной) кучи** поверх обычного списка. Куча — это специализированная древовидная структура данных, которая позволяет очень быстро получать и извлекать **минимальный** (в случае `heapq` — всегда min-heap) элемент.

Главная ценность кучи в том, что она даёт компромисс между полностью отсортированным списком и неупорядоченным набором:

| Операция | Несортированный список | Отсортированный список | Куча (heap) |
|---|---|---|---|
| Вставка элемента | O(1) | O(n) | **O(log n)** |
| Получить минимум | O(n) | O(1) | **O(1)** |
| Извлечь минимум | O(n) | O(n) (с конца — O(1)) | **O(log n)** |

Куча НЕ хранит элементы в полностью отсортированном виде. Она поддерживает лишь частичный порядок — гарантируется только то, что минимальный элемент всегда находится в корне (`heap[0]`). Это и есть причина, почему вставка стоит всего `O(log n)`, а не `O(n)`.

**Где применяется на практике:**
- Приоритетные очереди (планировщики задач, диспетчеры, обработка событий по времени).
- Алгоритм Дейкстры, A\*, алгоритм Прима (поиск кратчайших путей / MST).
- Задачи "top-K" (k самых больших/частых/близких элементов).
- Слияние множества отсортированных потоков (external merge sort, обработка логов).
- Поддержание медианы скользящего потока данных.
- Кодирование Хаффмана.

Особенность Python: в `heapq` нет отдельного класса `Heap`. Вы работаете с обычным `list`, а функции модуля поддерживают в нём инвариант кучи. Это даёт нулевые накладные расходы на обёртки, но требует дисциплины — список нельзя менять "руками", иначе инвариант сломается.

## Ключевые концепции

### Бинарная min-куча и инвариант кучи

**Бинарная куча** — это *полное (complete) бинарное дерево*, заполняемое по уровням слева направо. "Полное" означает, что все уровни заполнены, кроме, возможно, последнего, который заполняется слева направо без "дыр". Именно это свойство позволяет компактно хранить дерево в массиве без указателей.

**Инвариант min-кучи:** для каждого узла его значение **меньше либо равно** значениям обоих его потомков. Формально:

```
heap[i] <= heap[2*i + 1]   # левый потомок
heap[i] <= heap[2*i + 2]   # правый потомок
```

Из инварианта следует ключевое свойство: **в корне (heap[0]) всегда лежит минимум** всего набора. Но это НЕ означает полной сортировки — соседние элементы в списке могут идти в любом порядке. Например, `[1, 3, 2, 7, 4]` — корректная куча, хотя список не отсортирован.

### Представление в виде списка: индексы родителя и детей

Полное бинарное дерево хранится в плоском списке. Для узла с индексом `i`:

```python
# Для узла с индексом i:
left_child  = 2 * i + 1   # левый потомок
right_child = 2 * i + 2   # правый потомок
parent      = (i - 1) // 2  # родитель
```

Визуализация для кучи `[1, 3, 2, 7, 8, 4, 5]`:

```
Индексы:           0
                 /   \
               1       2
              / \     / \
             3   4   5   6

Значения:          1            (индекс 0)
                 /   \
                3     2          (индексы 1, 2)
               / \   / \
              7   8 4   5        (индексы 3, 4, 5, 6)
```

Проверим инвариант: `heap[0]=1 <= heap[1]=3` и `heap[1]=3 <= heap[3]=7`, `heap[1]=3 <= heap[4]=8` и т.д. Всё корректно.

### Операции просеивания: sift-up и sift-down

Внутри `heapq` две фундаментальные операции восстанавливают инвариант:

- **sift-up (просеивание вверх)** — используется при вставке. Новый элемент кладётся в конец списка (последний лист), затем "всплывает" вверх, меняясь местами с родителем, пока меньше него. Высота дерева `log n` → `O(log n)`.
- **sift-down (просеивание вниз)** — используется при извлечении. Корень удаляется, на его место ставится последний элемент, который затем "тонет" вниз, меняясь местами с меньшим из детей, пока не займёт корректную позицию. Тоже `O(log n)`.

В CPython внутренние функции называются `_siftup` и `_siftdown` (их имена немного контринтуитивны относительно классической терминологии, но суть та же).

### Стабильность и сравнение элементов

`heapq` сравнивает элементы через оператор `<`. Если элементы — кортежи, сравнение идёт лексикографически. Это критично для приоритетных очередей: при равных приоритетах Python начнёт сравнивать сами объекты, что может привести к `TypeError`, если объекты несравнимы (см. раздел "Подводные камни").

## Основные функции/классы/методы

### `heapq.heapify(x)` — превратить список в кучу за O(n)

Преобразует произвольный список в корректную кучу **на месте** (in-place), за линейное время.

```python
import heapq

data = [5, 3, 8, 1, 9, 2, 7]
heapq.heapify(data)          # модифицирует список на месте
print(data)                  # => [1, 3, 2, 5, 9, 8, 7]
print(data[0])               # => 1  (минимум всегда в корне)
```

**Сложность: O(n)**, а НЕ `O(n log n)`. Это частый вопрос на собеседовании. Наивная оценка "n вставок по `log n`" даёт `O(n log n)`, но `heapify` использует метод Флойда (build-heap): просеивает вниз начиная с последнего нелистового узла к корню. Узлов на глубоких уровнях много, но их высота мала, и сумма `Σ (узлов на уровне × высота)` сходится к `O(n)`.

```python
# Эквивалент того, что делает heapify под капотом (упрощённо):
def heapify_manual(arr):
    n = len(arr)
    # Начинаем с последнего узла, у которого есть дети, и идём к корню
    for i in reversed(range(n // 2)):
        heapq._siftup(arr, i)   # просеиваем вниз
```

### `heapq.heappush(heap, item)` — добавить элемент

Добавляет `item` в кучу, сохраняя инвариант (sift-up).

```python
import heapq

heap = []
heapq.heappush(heap, 5)
heapq.heappush(heap, 1)
heapq.heappush(heap, 3)
print(heap)        # => [1, 5, 3]
print(heap[0])     # => 1
```

**Сложность: O(log n).**

### `heapq.heappop(heap)` — извлечь минимум

Удаляет и возвращает наименьший элемент. Инвариант восстанавливается через sift-down.

```python
import heapq

heap = [1, 3, 2, 7, 4]
heapq.heapify(heap)
print(heapq.heappop(heap))   # => 1
print(heapq.heappop(heap))   # => 2
print(heap)                  # => [3, 4, 7]
```

**Сложность: O(log n).** Извлечение из пустой кучи бросает `IndexError`.

Полная сортировка через кучу (heapsort) — `O(n log n)`:

```python
import heapq

def heapsort(iterable):
    h = list(iterable)
    heapq.heapify(h)                      # O(n)
    return [heapq.heappop(h) for _ in range(len(h))]  # n * O(log n)

print(heapsort([3, 1, 4, 1, 5, 9, 2, 6]))  # => [1, 1, 2, 3, 4, 5, 6, 9]
```

### `heapq.heappushpop(heap, item)` — push, затем pop (за один проход)

Эффективно: сначала кладёт `item`, потом извлекает минимум. Но реализовано **умнее**, чем два отдельных вызова.

```python
import heapq

heap = [1, 3, 5]
heapq.heapify(heap)
# item=4: сначала "добавим" 4, минимум по-прежнему 1 -> вернётся 1
print(heapq.heappushpop(heap, 4))   # => 1
print(heap)                         # => [3, 4, 5]

# item=0: 0 меньше текущего минимума -> сразу вернётся 0, куча не меняется
heap2 = [2, 3, 5]
heapq.heapify(heap2)
print(heapq.heappushpop(heap2, 0))  # => 0
print(heap2)                        # => [2, 3, 5]
```

**Сложность: O(log n)**, но фактически **один** проход sift-down вместо вставки + извлечения. Если `item <= heap[0]`, функция возвращает `item` мгновенно, не трогая кучу.

### `heapq.heapreplace(heap, item)` — pop, затем push

Извлекает минимум и **затем** добавляет `item`. Порядок операций обратный `heappushpop`.

```python
import heapq

heap = [1, 3, 5]
heapq.heapify(heap)
# Сначала извлекается 1, потом добавляется 4
print(heapq.heapreplace(heap, 4))   # => 1
print(heap)                         # => [3, 4, 5]

# Даже если item меньше минимума — минимум ВСЁ РАВНО извлекается первым
heap2 = [2, 3, 5]
heapq.heapify(heap2)
print(heapq.heapreplace(heap2, 0))  # => 2  (а не 0!)
print(heap2)                        # => [0, 3, 5]
```

**Сложность: O(log n)** (один sift-down). Куча не должна быть пустой.

**Отличие `heappushpop` от `heapreplace` — частый вопрос:**

| | `heappushpop(h, x)` | `heapreplace(h, x)` |
|---|---|---|
| Порядок | push → pop | pop → push |
| Что вернёт при `x < min` | вернёт `x` (куча не меняется) | вернёт старый минимум, `x` войдёт в кучу |
| Размер кучи | не меняется | не меняется |
| Пустая куча | можно (станет с 1 элем., если x не вышел) | **нельзя** — `IndexError` |

Практическое правило для top-K больших: используйте `heappushpop` (он сам отбросит элемент, если он меньше границы). Для скользящего окна фиксированного размера, где вы гарантированно хотите обновить — `heapreplace`.

### `heapq.nlargest(n, iterable, key=None)` и `nsmallest(n, iterable, key=None)`

Возвращают `n` наибольших / наименьших элементов в виде отсортированного списка.

```python
import heapq

nums = [5, 1, 8, 3, 9, 2, 7]
print(heapq.nlargest(3, nums))    # => [9, 8, 7]
print(heapq.nsmallest(3, nums))   # => [1, 2, 3]

# С ключом — как у sorted()
people = [("Аня", 30), ("Боб", 25), ("Вера", 35)]
print(heapq.nlargest(2, people, key=lambda p: p[1]))
# => [('Вера', 35), ('Аня', 30)]
```

**Сложность: O(n log k)**, где `k = n` (запрашиваемое количество). Реализация держит кучу размера `k` и проходит по всем элементам один раз.

**Когда лучше `sorted`, а когда `nlargest/nsmallest`:**
- Если `k` близко к `len(iterable)` или вам нужен полный порядок → используйте `sorted` (`O(N log N)`).
- Если `k` мало по сравнению с `N` → `nlargest/nsmallest` экономят время: `O(N log k)` против `O(N log N)`, и память `O(k)` против `O(N)` для итераторов.
- Для `k == 1` используйте встроенные `max()` / `min()` — они `O(N)` и быстрее.

```python
# Эвристика самого heapq:
# nlargest(1, it)  -> по сути max(it)
# nlargest(n, it) при n >= len(it) -> sorted(it, reverse=True)
```

### `heapq.merge(*iterables, key=None, reverse=False)` — слияние отсортированных потоков

Ленивое слияние нескольких **уже отсортированных** входных итераторов в один отсортированный итератор. Возвращает генератор — не загружает всё в память.

```python
import heapq

a = [1, 4, 7]
b = [2, 3, 8]
c = [0, 5, 9]
print(list(heapq.merge(a, b, c)))
# => [0, 1, 2, 3, 4, 5, 7, 8, 9]

# reverse=True требует, чтобы входы были отсортированы по убыванию
print(list(heapq.merge([5, 3, 1], [6, 2, 0], reverse=True)))
# => [6, 5, 3, 2, 1, 0]
```

**Сложность: O(N log k)**, где `N` — суммарное число элементов, `k` — число потоков. Память — `O(k)` (а не `O(N)`!), потому что merge ленивый. Идеально для external sort и обработки больших отсортированных файлов, которые не помещаются в память.

## Частые вопросы на собеседовании

**Q1: Почему `heapify` работает за O(n), а не за O(n log n)?**

A: Наивный подход — `n` раз вызвать `heappush`, каждый по `O(log n)` → `O(n log n)`. Но `heapify` использует алгоритм Флойда (build-heap snizu vverh): просеивает вниз (sift-down) начиная с последнего нелистового узла к корню. Ключевой момент — большинство узлов находятся у листьев, где высота просеивания мала (или равна нулю). Узлов на высоте `h` примерно `n / 2^(h+1)`, и стоимость sift-down для них — `O(h)`. Сумма `Σ (h * n / 2^(h+1))` для `h` от 0 до `log n` сходится к константе, умноженной на `n`, то есть `O(n)`. Интуиция: половина узлов — листья (0 работы), четверть — на высоте 1 и т.д.

**Q2: В чём разница между `heappushpop` и `heapreplace`?**

A: Оба меняют один элемент и не меняют размер кучи, но порядок операций разный. `heappushpop(h, x)` сначала добавляет `x`, потом извлекает минимум — поэтому если `x` меньше или равен текущему минимуму, он немедленно возвращается, и куча не меняется. `heapreplace(h, x)` наоборот: сначала извлекает минимум (всегда!), потом добавляет `x`. Значит `heapreplace` всегда вернёт старый минимум, даже если `x` ещё меньше, а `x` обязательно попадёт в кучу. Ещё: `heapreplace` падает с `IndexError` на пустой куче, `heappushpop` — нет.

**Q3: Как сделать max-heap, если `heapq` — только min-heap?**

A: Есть три приёма:
1. **Негация значений.** Кладём `-value`, извлекаем и снова негируем. Работает для чисел.
2. **Кортеж `(-priority, item)`** — если значение не само число, а ключ.
3. Для `nlargest`/`nsmallest` есть готовые функции, негация не нужна.

```python
import heapq

# Max-heap через негацию
max_heap = []
for x in [3, 1, 4, 1, 5]:
    heapq.heappush(max_heap, -x)
print(-heapq.heappop(max_heap))   # => 5  (наибольший)
print(-heapq.heappop(max_heap))   # => 4
```

Для дробных и отрицательных тоже работает (`-(-2.5) == 2.5`). Минус подхода — нельзя негировать произвольные объекты (строки, кортежи строк), тогда используют обёртку с переопределённым `__lt__` или `nlargest`.

**Q4: Почему приоритетная очередь на кортежах `(priority, item)` может упасть, и как чинить?**

A: При равных приоритетах Python переходит к сравнению второго элемента кортежа — самого `item`. Если `item` несравним (например, два словаря или кастомных объекта без `__lt__`), получаем `TypeError: '<' not supported`. Решение — вставить **монотонный счётчик** между приоритетом и элементом: `(priority, count, item)`. Счётчик уникален и всегда сравним, до `item` дело не дойдёт. Бонус: счётчик обеспечивает FIFO-стабильность при равных приоритетах.

```python
import heapq, itertools

counter = itertools.count()      # уникальные возрастающие числа
pq = []
heapq.heappush(pq, (2, next(counter), "задача B"))
heapq.heappush(pq, (1, next(counter), "задача A"))
heapq.heappush(pq, (2, next(counter), "задача C"))  # тот же приоритет 2

while pq:
    prio, _, task = heapq.heappop(pq)
    print(prio, task)
# => 1 задача A
# => 2 задача B
# => 2 задача C   (стабильный порядок благодаря счётчику)
```

**Q5: Как использовать `dataclass` для элементов очереди, чтобы избежать проблемы сравнения?**

A: Декорируем датакласс `order=True` (генерирует методы сравнения), а поле с данными помечаем `field(compare=False)`, чтобы оно не участвовало в сравнении. Тогда сравниваться будет только приоритет.

```python
import heapq
from dataclasses import dataclass, field
from typing import Any

@dataclass(order=True)
class PrioritizedItem:
    priority: int
    item: Any = field(compare=False)   # не участвует в сравнении!

pq = []
heapq.heappush(pq, PrioritizedItem(2, {"name": "B"}))
heapq.heappush(pq, PrioritizedItem(1, {"name": "A"}))
heapq.heappush(pq, PrioritizedItem(2, {"name": "C"}))  # равный приоритет — ок

print(heapq.heappop(pq).item)   # => {'name': 'A'}
```

Без `compare=False` при равных приоритетах Python попытался бы сравнить словари и упал бы. Минус: чистый dataclass без счётчика **не гарантирует FIFO** при равных приоритетах (порядок не определён). Если нужна стабильность — добавьте поле-счётчик.

**Q6: Как найти k самых частых элементов? (классика — LeetCode Top K Frequent)**

A: Считаем частоты через `Counter`, затем берём top-k по частоте. `Counter.most_common(k)` под капотом использует `heapq.nlargest`.

```python
import heapq
from collections import Counter

nums = [1, 1, 1, 2, 2, 3]
cnt = Counter(nums)               # {1: 3, 2: 2, 3: 1}

# Вариант 1: готовый most_common -> O(n log k)
print([x for x, _ in cnt.most_common(2)])   # => [1, 2]

# Вариант 2: руками через nlargest
print(heapq.nlargest(2, cnt.keys(), key=cnt.get))   # => [1, 2]
```

Сложность: подсчёт `O(n)`, выбор top-k через кучу размера `k` — `O(n log k)`. Это лучше полной сортировки `O(n log n)`, когда `k << n`.

**Q7: Как слить k отсортированных списков? (LeetCode Merge k Sorted Lists)**

A: Два способа. Самый простой — `heapq.merge`. Классический "ручной" способ для собеседования — куча из `k` указателей, мы всегда извлекаем минимальный головной элемент и добавляем следующий из того же списка.

```python
import heapq

def merge_k_sorted(lists):
    heap = []
    # Кладём первый элемент каждого списка: (значение, индекс списка, индекс внутри)
    for i, lst in enumerate(lists):
        if lst:
            heapq.heappush(heap, (lst[0], i, 0))

    result = []
    while heap:
        val, list_idx, elem_idx = heapq.heappop(heap)
        result.append(val)
        # Если в этом списке есть следующий элемент — добавляем его
        if elem_idx + 1 < len(lists[list_idx]):
            nxt = lists[list_idx][elem_idx + 1]
            heapq.heappush(heap, (nxt, list_idx, elem_idx + 1))
    return result

print(merge_k_sorted([[1, 4, 7], [2, 3, 8], [0, 5, 9]]))
# => [0, 1, 2, 3, 4, 5, 7, 8, 9]
```

Сложность: `O(N log k)`, где `N` — всего элементов, `k` — число списков. Куча держит максимум `k` элементов одновременно → память `O(k)`. Индекс списка во втором поле кортежа важен — он делает кортежи сравнимыми, даже если значения равны.

**Q8: Как находить медиану потока чисел? (LeetCode Find Median from Data Stream)**

A: Две кучи. **Max-heap** (`lo`, через негацию) хранит меньшую половину, **min-heap** (`hi`) — большую. Поддерживаем баланс: размеры отличаются не более чем на 1. Медиана — либо вершина большей кучи, либо среднее двух вершин.

```python
import heapq

class MedianFinder:
    def __init__(self):
        self.lo = []   # max-heap (через негацию): меньшая половина
        self.hi = []   # min-heap: большая половина

    def add(self, num):
        # Сначала кладём в lo (max-heap), затем балансируем
        heapq.heappush(self.lo, -num)
        # Переносим максимум из lo в hi, чтобы все lo <= все hi
        heapq.heappush(self.hi, -heapq.heappop(self.lo))
        # Поддерживаем |lo| >= |hi|
        if len(self.hi) > len(self.lo):
            heapq.heappush(self.lo, -heapq.heappop(self.hi))

    def median(self):
        if len(self.lo) > len(self.hi):
            return -self.lo[0]                 # нечётное число элементов
        return (-self.lo[0] + self.hi[0]) / 2  # чётное -> среднее вершин

mf = MedianFinder()
for x in [5, 2, 8, 1]:
    mf.add(x)
print(mf.median())   # => 3.5  (отсортировано: 1,2,5,8 -> (2+5)/2)
```

Сложность: добавление `O(log n)`, запрос медианы `O(1)`.

**Q9: Можно ли изменить приоритет элемента, уже лежащего в куче (decrease-key)?**

A: Напрямую `heapq` не поддерживает decrease-key/удаление произвольного элемента — это была бы `O(n)` на поиск. Стандартный приём — **ленивое удаление (lazy deletion)**: помечаем старую запись как недействительную (через `dict` соответствий `entry -> [prio, item, valid]`) и добавляем новую запись с новым приоритетом. При `heappop` пропускаем записи, помеченные как удалённые. Это паттерн из документации `heapq` ("Priority Queue Implementation Notes").

```python
import heapq, itertools

REMOVED = object()
pq = []
entry_finder = {}
counter = itertools.count()

def add_task(task, priority=0):
    if task in entry_finder:
        remove_task(task)                  # обновление = удалить + добавить
    entry = [priority, next(counter), task]
    entry_finder[task] = entry
    heapq.heappush(pq, entry)

def remove_task(task):
    entry = entry_finder.pop(task)
    entry[-1] = REMOVED                    # ленивая пометка

def pop_task():
    while pq:
        prio, _, task = heapq.heappop(pq)
        if task is not REMOVED:
            del entry_finder[task]
            return task
    raise KeyError("очередь пуста")

add_task("A", 5)
add_task("B", 2)
add_task("A", 1)        # обновили приоритет A на 1
print(pop_task())       # => A  (новый приоритет 1 — самый высокий)
```

**Q10: Сохраняет ли `heapq` стабильность (порядок равных элементов)?**

A: Сама структура кучи **не стабильна** — при равных ключах порядок извлечения не определён и зависит от внутренних перестановок. Если нужна стабильность (FIFO при равных приоритетах), её обеспечивают искусственно через монотонный счётчик во втором поле кортежа (см. Q4). А вот `heapq.merge` и `nsmallest`/`nlargest` **стабильны** относительно своих гарантий — но не полагайтесь на это для кучи в целом.

**Q11: `heappop` из пустой кучи — что будет, и как защититься?**

A: `heappop([])` бросает `IndexError: index out of range`. Всегда проверяйте `if heap:` перед извлечением в циклах. То же с `heapreplace` на пустой куче. А вот `heappushpop` на пустой куче допустим (он сначала добавляет).

**Q12: Потокобезопасен ли `heapq`? Что использовать в многопоточности?**

A: Функции `heapq` НЕ потокобезопасны — это просто операции над списком, и параллельные `heappush`/`heappop` могут испортить инвариант. Для многопоточных сценариев используйте `queue.PriorityQueue` (обёртка над `heapq` с блокировками) или защищайте доступ собственным `threading.Lock`. `PriorityQueue` удобна для producer-consumer, но имеет накладные расходы на синхронизацию.

## Подводные камни (gotchas)

1. **Сравнение `item` при равных приоритетах.** Кортежи `(priority, item)` падают с `TypeError`, если `priority` равны и `item` несравнимы. Всегда добавляйте счётчик: `(priority, count, item)`.

2. **`heapq` — это только min-heap.** Нет параметра `reverse` для самой кучи (только для `merge`/`nlargest`). Для max-heap негируйте значения или используйте обёртку.

3. **Негация ломается на нечисловых типах.** `-"строка"` невозможно. Для max-heap по строкам нужна обёртка с `__lt__` или `nlargest`.

4. **Нельзя менять список кучи "руками".** Прямое `heap.append(x)`, `heap.sort()`, `heap[i] = y` ломают инвариант. После таких операций вызывайте `heapify` заново.

5. **`heap[0]` — это минимум, но `heap` НЕ отсортирован.** Не итерируйтесь по куче, ожидая порядок. Для сортировки извлекайте через `heappop` или используйте `sorted(heap)`.

6. **`heapreplace` на пустой куче — `IndexError`.** В отличие от `heappushpop`.

7. **`merge` требует уже отсортированные входы.** Если входы не отсортированы — результат будет неверным (мусор), без ошибки. Молчаливый баг.

8. **`nlargest(k, ...)` при `k >= len` не быстрее сортировки.** heapq сам падает на `sorted()` — это нормально, но не ждите выигрыша в этом случае.

9. **Изменяемые объекты как ключи кучи.** Если элемент-кортеж содержит изменяемый объект и его мутируют, инвариант может незаметно сломаться.

10. **`heapq` не потокобезопасен.** В многопоточности — `queue.PriorityQueue` или явный лок.

## Лучшие практики

- **Для приоритетной очереди всегда используйте кортеж с счётчиком** `(priority, count, item)` или `dataclass(order=True)` с `field(compare=False)`. Это убирает целый класс багов.

- **Top-K: выбирайте инструмент по `k`.**
  - `k == 1` → `min()` / `max()` (`O(n)`).
  - `k << n` → `heapq.nlargest`/`nsmallest` (`O(n log k)`).
  - `k ≈ n` или нужен полный порядок → `sorted()` (`O(n log n)`).

- **Для скользящего top-K больших** держите min-heap размера `k` и используйте `heappushpop` — старый минимум автоматически отбрасывается, если новый элемент больше.

```python
import heapq
def top_k_streaming(stream, k):
    heap = []
    for x in stream:
        if len(heap) < k:
            heapq.heappush(heap, x)
        else:
            heapq.heappushpop(heap, x)   # эффективно: один проход
    return sorted(heap, reverse=True)
print(top_k_streaming([5, 1, 8, 3, 9, 2, 7], 3))   # => [9, 8, 7]
```

- **Для слияния больших отсортированных источников** используйте `heapq.merge` — он ленивый и тратит `O(k)` памяти, а не `O(N)`.

- **`heapify` дешевле, чем серия `heappush`.** Если у вас уже есть список — `heapify` (`O(n)`) лучше, чем `n` вставок (`O(n log n)`).

- **В многопоточном коде** — `queue.PriorityQueue`, не голый `heapq`.

- **Для decrease-key / удаления** применяйте паттерн ленивого удаления с `dict` соответствий и маркером `REMOVED`.

## Шпаргалка

```python
import heapq

# --- БАЗОВЫЕ ОПЕРАЦИИ ---
h = [3, 1, 4, 1, 5]
heapq.heapify(h)              # O(n)  -> список становится кучей на месте
heapq.heappush(h, 2)          # O(log n)  добавить
mn = heapq.heappop(h)         # O(log n)  извлечь минимум
peek = h[0]                   # O(1)  посмотреть минимум (не извлекая)

heapq.heappushpop(h, x)       # push, потом pop  (1 проход, не падает на [])
heapq.heapreplace(h, x)       # pop, потом push   (всегда вернёт старый min)

# --- TOP-K ---
heapq.nlargest(3, data)              # O(n log k) -> [самые большие], отсортировано
heapq.nsmallest(3, data)             # O(n log k) -> [самые маленькие]
heapq.nlargest(3, data, key=len)     # с ключом

# --- СЛИЯНИЕ ОТСОРТИРОВАННЫХ ---
list(heapq.merge([1, 4], [2, 3]))    # ленивый итератор -> [1, 2, 3, 4]
heapq.merge(a, b, reverse=True)      # для убывающих входов

# --- MAX-HEAP (через негацию) ---
mh = []
heapq.heappush(mh, -value)
biggest = -heapq.heappop(mh)

# --- ПРИОРИТЕТНАЯ ОЧЕРЕДЬ (безопасная) ---
import itertools
cnt = itertools.count()
pq = []
heapq.heappush(pq, (priority, next(cnt), item))   # счётчик спасает от TypeError
prio, _, item = heapq.heappop(pq)

# --- ИНДЕКСЫ ДЕРЕВА ---
# left = 2*i + 1 ; right = 2*i + 2 ; parent = (i-1)//2
```

| Функция | Сложность | Назначение |
|---|---|---|
| `heapify(x)` | O(n) | список → куча на месте |
| `heappush(h, x)` | O(log n) | вставка |
| `heappop(h)` | O(log n) | извлечь минимум |
| `h[0]` | O(1) | посмотреть минимум |
| `heappushpop(h, x)` | O(log n) | push → pop, 1 проход |
| `heapreplace(h, x)` | O(log n) | pop → push, 1 проход |
| `nlargest(k, it)` | O(n log k) | k наибольших |
| `nsmallest(k, it)` | O(n log k) | k наименьших |
| `merge(*its)` | O(N log k) | слить отсортированные, память O(k) |

**Главное, что нужно запомнить:** `heapq` — это min-heap поверх списка; минимум всегда в `heap[0]`; `heapify` — это `O(n)`; для приоритетной очереди добавляйте счётчик `(prio, count, item)`; max-heap — через негацию; для top-K при малом k используйте `nlargest`/`nsmallest`, а не `sorted`; для слияния потоков — `merge`.
