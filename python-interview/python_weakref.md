# weakref — подготовка к собеседованию

## Что это и зачем

Модуль `weakref` из стандартной библиотеки Python позволяет создавать **слабые ссылки** (weak references) на объекты. Слабая ссылка — это ссылка, которая **не увеличивает счётчик ссылок** (reference count) объекта и поэтому **не препятствует** его удалению сборщиком мусора.

Главная идея: иногда нам нужно «знать» об объекте, но не «удерживать» его в памяти. Если все остальные ссылки на объект исчезнут, объект будет уничтожен, даже если на него есть слабая ссылка. Сама слабая ссылка после этого «протухнет» и начнёт возвращать `None`.

Зачем это нужно на практике:

- **Кэширование**: храним результаты, но не мешаем их удалению, когда они больше нигде не используются (экономия памяти).
- **Разрыв циклов ссылок**: например, в паттерне «родитель ↔ ребёнок» обратная ссылка ребёнка на родителя делается слабой, чтобы не создавать цикл, который удерживает память.
- **Привязка дополнительных данных к чужим объектам** без модификации этих объектов и без удержания их в памяти.
- **Observer (наблюдатель)**: список подписчиков хранит слабые ссылки, чтобы «забытые» подписчики автоматически отписывались и не текли.
- **Надёжная финализация**: `weakref.finalize` — современная и более безопасная альтернатива методу `__del__`.

```python
import sys
import weakref


class Resource:
    pass


obj = Resource()
print(sys.getrefcount(obj))      # => 2  (1 — переменная obj, 1 — временный аргумент getrefcount)

r = weakref.ref(obj)             # создаём слабую ссылку
print(sys.getrefcount(obj))      # => 2  (refcount НЕ вырос — это ключевое свойство)

print(r())                       # => <__main__.Resource object at 0x...>  слабая ссылка жива

del obj                          # удаляем единственную сильную ссылку
print(r())                       # => None  объект уничтожен, слабая ссылка "протухла"
```

## Ключевые концепции

### Подсчёт ссылок (reference counting) в CPython

CPython использует **подсчёт ссылок** как основной механизм управления памятью. У каждого объекта есть счётчик `ob_refcnt`. Когда счётчик доходит до нуля — объект немедленно уничтожается (вызывается деструктор, освобождается память).

```python
import sys

x = []                       # refcount списка = 1
y = x                        # refcount = 2 (x и y ссылаются на тот же список)
print(sys.getrefcount(x))    # => 3  (+1 — временный аргумент функции getrefcount)
del y                        # refcount = 1
del x                        # refcount = 0 -> объект уничтожен
```

**Слабая ссылка специально не трогает этот счётчик.** Поэтому объект может быть собран, пока на него смотрит только слабая ссылка.

### Сборщик мусора (GC) и циклы

Подсчёт ссылок не справляется с **циклическими ссылками**: если A ссылается на B, а B на A, то даже после удаления внешних ссылок их взаимные счётчики остаются равными 1, и объекты не освобождаются. Для таких случаев в CPython есть отдельный **циклический сборщик мусора** (`gc`), который периодически находит и уничтожает недостижимые циклы.

Слабые ссылки — естественный способ **не создавать циклы вообще**: сделав «обратную» ссылку слабой, мы избавляем GC от лишней работы и гарантируем своевременное освобождение через refcount.

### "Протухание" (dead reference)

После уничтожения объекта слабая ссылка `weakref.ref` начинает возвращать `None` при вызове. `weakref.proxy` после смерти объекта бросает `ReferenceError` при любом обращении. Это надо учитывать в коде.

### Какие объекты поддерживают слабые ссылки

Не на каждый объект можно создать слабую ссылку. Для этого у объекта должен быть слот `__weakref__` (внутреннее поле, хранящее список слабых ссылок).

- **Поддерживают**: экземпляры обычных классов (с `__dict__`), множества `set`, и большинство пользовательских типов.
- **НЕ поддерживают**: `int`, `float`, `str`, `bytes`, `tuple`, `list`, `dict` (встроенные базовые типы), а также экземпляры классов с `__slots__`, в которых не указан `__weakref__`.

```python
import weakref

weakref.ref(42)          # => TypeError: cannot create weak reference to 'int' object
weakref.ref("hello")     # => TypeError: cannot create weak reference to 'str' object
weakref.ref([1, 2, 3])   # => TypeError: cannot create weak reference to 'list' object
weakref.ref({1: 2})      # => TypeError: cannot create weak reference to 'dict' object


class C:                 # обычный класс — поддерживает
    pass

weakref.ref(C())         # => OK
```

## Основные функции/классы/методы

### `weakref.ref(object[, callback])` — базовая слабая ссылка

Возвращает callable-объект слабой ссылки. Чтобы получить сам объект, ссылку нужно **вызвать** как функцию: `r()`. Если объект жив — вернётся он, если умер — `None`.

```python
import weakref


class Node:
    def __init__(self, name):
        self.name = name


n = Node("A")
r = weakref.ref(n)

print(r)            # => <weakref at 0x...; to 'Node' at 0x...>
print(r())          # => <__main__.Node object at 0x...>
print(r().name)     # => A

del n
print(r())          # => None
print(r() is None)  # => True
```

**Callback при удалении.** Второй аргумент — функция, которая будет вызвана, когда объект уничтожается. В неё передаётся **сама слабая ссылка** (не объект — объекта уже нет!).

```python
import weakref


class Node:
    def __init__(self, name):
        self.name = name


def on_dead(ref):
    print(f"Объект умер, ссылка: {ref}")


n = Node("A")
r = weakref.ref(n, on_dead)

del n
# => Объект умер, ссылка: <weakref at 0x...; dead>
```

Слабые ссылки можно сравнивать и хешировать (пока живы их объекты), поэтому их удобно складывать в множества и словари.

### `weakref.proxy(object[, callback])` — слабый прокси

Возвращает **прокси-объект**, который ведёт себя как сам объект: к нему можно обращаться напрямую, без вызова `()`. Это удобнее `ref`, когда хочется «прозрачно» работать с объектом. Но после смерти объекта любое обращение бросает `ReferenceError`.

```python
import weakref


class Node:
    def __init__(self, name):
        self.name = name

    def hello(self):
        return f"hello from {self.name}"


n = Node("A")
p = weakref.proxy(n)

print(p.name)        # => A  (обращаемся как к самому объекту, без p())
print(p.hello())     # => hello from A

del n
try:
    print(p.name)    # объект мёртв
except ReferenceError as e:
    print("ReferenceError:", e)
# => ReferenceError: weakly-referenced object no longer exists
```

**`ref` vs `proxy`:**

| | `weakref.ref` | `weakref.proxy` |
|---|---|---|
| Доступ к объекту | через вызов `r()` | напрямую, как к объекту |
| После смерти | возвращает `None` | бросает `ReferenceError` |
| Хешируемость | да (можно класть в set/dict) | нет |
| Когда использовать | проверки, контейнеры, callbacks | прозрачное использование объекта |

### `weakref.WeakValueDictionary` — словарь со слабыми значениями

Словарь, в котором **значения** хранятся через слабые ссылки. Как только на значение не остаётся других (сильных) ссылок — запись автоматически исчезает из словаря. Классический инструмент для **кэша**.

```python
import weakref


class Image:
    def __init__(self, path):
        self.path = path


cache = weakref.WeakValueDictionary()

img = Image("logo.png")
cache["logo"] = img

print(list(cache.keys()))   # => ['logo']
print(cache["logo"].path)   # => logo.png

del img                     # удаляем единственную сильную ссылку на Image
# запись автоматически вычищается, т.к. значение собрано GC
print(list(cache.keys()))   # => []
print("logo" in cache)      # => False
```

Важно: пока есть хоть одна сильная ссылка на значение (например, переменная `img`), запись жива. Кэш не «удерживает» объект, он лишь «подсматривает».

### `weakref.WeakKeyDictionary` — словарь со слабыми ключами

Словарь, в котором **ключи** хранятся слабо. Идеален для того, чтобы **привязать дополнительные данные к чужим объектам**, не модифицируя их и не удерживая в памяти. Когда объект-ключ умирает — запись исчезает.

```python
import weakref


class Widget:
    pass


# хотим хранить метаданные о виджетах, но не мешать их удалению
metadata = weakref.WeakKeyDictionary()

w = Widget()
metadata[w] = {"clicks": 5, "color": "red"}

print(metadata[w])          # => {'clicks': 5, 'color': 'red'}
print(len(metadata))        # => 1

del w                       # виджет уничтожен
print(len(metadata))        # => 0  (запись ушла автоматически)
```

Это лучше, чем обычный `dict[id(obj)] = data`, потому что обычный словарь:
1) удерживает данные вечно (утечка), 2) `id()` может переиспользоваться после смерти объекта (баг).

### `weakref.WeakSet` — множество слабых ссылок

Множество, элементы которого хранятся слабо. Элемент пропадает из множества, как только он уничтожен. Удобно для реестров «всех живых экземпляров» или списков подписчиков.

```python
import weakref


class Plugin:
    _registry = weakref.WeakSet()

    def __init__(self, name):
        self.name = name
        Plugin._registry.add(self)


p1 = Plugin("auth")
p2 = Plugin("logging")
print(len(Plugin._registry))   # => 2
print(sorted(pl.name for pl in Plugin._registry))  # => ['auth', 'logging']

del p1
print(len(Plugin._registry))   # => 1  (auth исчез автоматически)
```

### `weakref.finalize(obj, func, *args, **kwargs)` — надёжный финализатор

Регистрирует функцию, которая будет вызвана, когда `obj` уничтожается (или при выходе из интерпретатора). Это **современная замена `__del__`**: проще, надёжнее, не воскрешает объект, не ломается на циклах.

```python
import weakref


class Connection:
    def __init__(self, host):
        self.host = host
        # finalize "запоминает" нужные данные (host), а не сам объект
        self._finalizer = weakref.finalize(self, self._cleanup, host)

    @staticmethod
    def _cleanup(host):
        print(f"Закрываю соединение с {host}")


conn = Connection("db.local")
del conn
# => Закрываю соединение с db.local
```

`finalize` возвращает объект, у которого есть полезные свойства и методы:

```python
import weakref


class Res:
    pass


r = Res()
f = weakref.finalize(r, print, "очистка выполнена")

print(f.alive)      # => True   объект ещё жив
f()                 # принудительно вызываем финализатор
# => очистка выполнена
print(f.alive)      # => False  повторно НЕ вызовется
f()                 # ничего не печатает — финализатор уже отработал
```

ВАЖНО: в callback нельзя передавать ссылку на сам объект (иначе он не сможет умереть). Передавайте только нужные данные. Поэтому `_cleanup` сделан `staticmethod` и принимает `host`, а не `self`.

### `weakref.WeakMethod` — слабая ссылка на связанный метод

`weakref.ref` плохо работает со **связанными методами** (`obj.method`): такой bound-method — временный объект, он умирает сразу. `WeakMethod` решает проблему: слабо держит и объект, и функцию, а при вызове возвращает живой bound-method (или `None`).

```python
import weakref


class Handler:
    def on_event(self):
        return "событие обработано"


h = Handler()

# Наивно — НЕ работает: bound-method временный, ссылка сразу протухает
bad = weakref.ref(h.on_event)
print(bad())                 # => None  (бывает; bound-method уже уничтожен)

# Правильно — WeakMethod
m = weakref.WeakMethod(h.on_event)
method = m()                 # вернёт живой bound-method
print(method())              # => событие обработано

del h
print(m())                   # => None  (объект умер -> метода нет)
```

`WeakMethod` незаменим в реализации сигналов/событий, где callbacks — это методы объектов, и мы не хотим удерживать эти объекты живыми.

### `weakref.getweakrefcount` и `getweakrefs`

```python
import weakref


class C:
    pass


c = C()
r1 = weakref.ref(c)
r2 = weakref.ref(c)
print(weakref.getweakrefcount(c))   # => 1  (ref дедуплицируются: r1 is r2)
print(r1 is r2)                     # => True
print(weakref.getweakrefs(c))       # => [<weakref ...>]
```

## Частые вопросы на собеседовании

**Q1: Что такое слабая ссылка и чем она отличается от обычной?**

A: Обычная (сильная) ссылка увеличивает счётчик ссылок объекта и удерживает его в памяти. Слабая ссылка (`weakref`) **не увеличивает refcount** и не препятствует уничтожению объекта. Если на объект остались только слабые ссылки, он будет уничтожен сборщиком/refcount-механизмом, а слабые ссылки «протухнут» (`ref()` вернёт `None`, `proxy` бросит `ReferenceError`). Используется для кэшей, разрыва циклов, реестров и финализаторов.

**Q2: Как устроен подсчёт ссылок в CPython и при чём тут weakref?**

A: У каждого объекта есть поле `ob_refcnt`. Присваивание увеличивает его, удаление/выход из области видимости — уменьшает. При достижении нуля объект немедленно уничтожается. Слабые ссылки специально реализованы так, чтобы не трогать этот счётчик — они хранятся в отдельном слоте `__weakref__` объекта. Поэтому объект «не знает» о существовании слабых ссылок в смысле своего времени жизни, а когда умирает — CPython проходит по списку слабых ссылок, обнуляет их и вызывает их callbacks.

**Q3: В чём разница между `weakref.ref` и `weakref.proxy`?**

A: `ref` — это callable: чтобы достать объект, нужно вызвать `r()`, и после смерти объекта он вернёт `None`. `ref` хешируется, его можно класть в множества/словари, удобен для проверок и контейнеров. `proxy` ведёт себя как сам объект (доступ к атрибутам/методам напрямую, без `()`), но после смерти объекта любое обращение к прокси бросает `ReferenceError`, и прокси не хешируется. `proxy` удобен для «прозрачной» подмены, `ref` — для логики и контейнеров.

**Q4: Зачем нужен `WeakValueDictionary` и почему обычный `dict` для кэша опасен?**

A: Обычный `dict` хранит **сильные** ссылки на значения, поэтому пока запись в кэше — объект жив, даже если он больше нигде не нужен. Это утечка/раздувание памяти: кэш растёт неограниченно. `WeakValueDictionary` держит значения слабо: как только на значение не осталось других ссылок, запись автоматически исчезает. Получается «кэш с автоочисткой», который не мешает GC. Минус — нельзя гарантировать, что объект останется в кэше, если им никто не пользуется.

**Q5: Когда брать `WeakKeyDictionary`?**

A: Когда нужно **привязать дополнительные данные/метаданные к чужим объектам**, которые мы не можем или не хотим модифицировать, и при этом не удерживать их в памяти. Например, кэш вычислений по объекту, аннотации, ACL. Когда объект-ключ умирает, запись уходит автоматически. Это безопаснее, чем `dict[id(obj)] = data`: обычный словарь удерживает данные вечно, а `id()` после смерти объекта может быть переиспользован другим объектом — получится перепутанная привязка.

**Q6: Какие объекты НЕ поддерживают слабые ссылки и как это исправить для своего класса?**

A: Не поддерживают многие встроенные типы: `int`, `float`, `str`, `bytes`, `tuple`, `list`, `dict`, `bool`. Причина — у них нет слота `__weakref__` (это экономит память). Обычные пользовательские классы (с `__dict__`) поддерживают по умолчанию. А вот класс с `__slots__` теряет `__weakref__`, если его явно не добавить в слоты:

```python
import weakref


class NoWeak:
    __slots__ = ("x",)


class WithWeak:
    __slots__ = ("x", "__weakref__")   # явно включаем поддержку


try:
    weakref.ref(NoWeak())
except TypeError as e:
    print("Нельзя:", e)   # => Нельзя: cannot create weak reference to 'NoWeak' object

print(weakref.ref(WithWeak()))  # => <weakref ...>  работает
```

**Q7: Как слабые ссылки помогают разорвать циклические ссылки? Приведите пример.**

A: При двунаправленной связи (родитель ↔ ребёнок) обе сильные ссылки образуют цикл: refcount никогда не падает до нуля, освобождение откладывается до цикла GC. Если «обратную» ссылку сделать слабой, цикла нет — объекты освобождаются сразу по refcount, без участия циклического сборщика.

```python
import weakref


class Parent:
    def __init__(self):
        self.children = []


class Child:
    def __init__(self, parent):
        self.parent_ref = weakref.ref(parent)   # слабая обратная ссылка -> нет цикла


p = Parent()
c = Child(p)
p.children.append(c)

print(c.parent_ref() is p)   # => True
del p                         # сильных ссылок на Parent больше нет
# Parent уничтожается сразу (нет цикла, удерживающего его)
print(c.parent_ref())         # => None
```

**Q8: Чем `weakref.finalize` лучше `__del__`?**

A: `__del__` имеет много проблем: он может не вызваться при наличии циклов (исторически — до Python 3.4), исключения в нём подавляются, объект можно «воскресить», порядок вызова при завершении интерпретатора непредсказуем, и `__del__` может мешать GC. `weakref.finalize` лишён этих недостатков: финализатор гарантированно вызывается ровно один раз, не воскрешает объект (он не получает ссылку на объект), корректно отрабатывает при выходе из интерпретатора, и предоставляет API (`.alive`, `.peek()`, прямой вызов `f()`, флаг `atexit`). Это рекомендуемый способ финализации ресурсов.

**Q9: Что произойдёт с callback слабой ссылки и какой аргумент он получает?**

A: Callback вызывается **в момент уничтожения объекта**, уже после того как объект перестал существовать. Поэтому в callback передаётся **сама слабая ссылка** (которая уже «мертва»), а не объект. Внутри callback нельзя получить доступ к умершему объекту через эту ссылку (`ref()` вернёт `None`). Если нужны данные объекта, их надо «захватить» заранее (как в `finalize`, передавая значения, а не `self`). Нельзя в callback удерживать сильную ссылку на объект — иначе он не сможет умереть.

**Q10: Как weakref взаимодействует с `gc.collect()`?**

A: Если объект освобождается по refcount (нет циклов), слабая ссылка протухает **немедленно**, без участия `gc`. Если же объект участвует в цикле, он освобождается только при проходе циклического сборщика — тогда `gc.collect()` форсирует сборку, и сразу после неё связанные слабые ссылки станут `None`, а их callbacks вызовутся.

```python
import gc
import weakref


class Node:
    def __init__(self):
        self.ref = None


a = Node()
b = Node()
a.ref = b
b.ref = a            # цикл a <-> b

wa = weakref.ref(a)
wb = weakref.ref(b)

del a, b             # внешних ссылок нет, но цикл удерживает объекты
print(wa() is not None)   # => True   (ещё живы из-за цикла)

gc.collect()              # форсируем сборку цикла
print(wa(), wb())         # => None None
```

**Q11: Как реализовать observer/наблюдателя без утечек памяти?**

A: Хранить подписчиков в `WeakSet` (или их методы в списке `WeakMethod`). Тогда подписчик, на который больше нигде нет ссылок, автоматически «отписывается» — субъект не удерживает его живым. Это устраняет классическую утечку, когда забытые слушатели остаются в списке навсегда.

```python
import weakref


class Subject:
    def __init__(self):
        self._observers = weakref.WeakSet()

    def subscribe(self, observer):
        self._observers.add(observer)

    def notify(self, event):
        for obs in self._observers:
            obs.update(event)


class Observer:
    def __init__(self, name):
        self.name = name

    def update(self, event):
        print(f"{self.name} получил: {event}")


s = Subject()
o1 = Observer("A")
o2 = Observer("B")
s.subscribe(o1)
s.subscribe(o2)

s.notify("click")
# => A получил: click
# => B получил: click

del o2                # B забыли -> автоматически отписался
s.notify("scroll")
# => A получил: scroll  (B больше не уведомляется и не течёт)
```

**Q12: Чем weakref-кэш отличается от `functools.lru_cache`?**

A: Это разные стратегии. `functools.lru_cache` держит **сильные** ссылки на результаты и ограничивает кэш по размеру (`maxsize`), вытесняя наименее недавно использованные элементы. Он гарантирует, что закэшированное значение будет доступно, пока не вытеснено, — но удерживает объекты в памяти (риск раздувания, если объекты тяжёлые). `WeakValueDictionary`-кэш, наоборот, не удерживает значения: они живут, только пока используются где-то ещё, и исчезают сами. Выбор: `lru_cache` — когда важна предсказуемая скорость и допустимо держать данные; weak-кэш — когда важна экономия памяти и допустимо, что объект «выпадет» из кэша. Часто их комбинируют (например, identity-map на слабых ссылках + LRU поверх дорогого вычисления).

## Подводные камни (gotchas)

- **Bound-методы умирают мгновенно.** `weakref.ref(obj.method)` почти бесполезен: `obj.method` — временный bound-method, он уничтожается сразу после выражения, и ссылка протухает. Используйте `weakref.WeakMethod`.

  ```python
  import weakref


  class C:
      def m(self):
          return 1


  c = C()
  print(weakref.ref(c.m)())     # => None  (часто)  — ловушка
  print(weakref.WeakMethod(c.m)()())  # => 1   — правильно
  ```

- **Callback не воскрешает объект.** В callback объект уже мёртв; не пытайтесь получить его через переданную ссылку — будет `None`. Захватывайте нужные данные заранее.

- **Нельзя удерживать объект в финализаторе.** Если в `weakref.finalize(self, self.method)` передать связанный метод `self.method`, он держит сильную ссылку на `self`, и объект **никогда не умрёт**. Передавайте `staticmethod`/функцию + данные.

- **Не на всё можно сослаться.** Базовые типы (`int/str/tuple/list/dict`) и `__slots__`-классы без `__weakref__` бросают `TypeError`. Добавляйте `"__weakref__"` в `__slots__`.

- **`proxy` бросает `ReferenceError`.** Любое обращение к прокси мёртвого объекта — исключение, а не `None`. Оборачивайте в `try/except` или проверяйте предварительно через `ref`.

- **Race condition в кэше.** В `WeakValueDictionary` значение может исчезнуть между проверкой `key in cache` и доступом `cache[key]` (особенно в многопоточности). Используйте `cache.get(key)` и обрабатывайте `None`, а не паттерн «проверь-потом-возьми».

- **Слабые ссылки на временные значения бесполезны.** `weakref.ref(SomeClass())` тут же протухнет, т.к. на объект нет сильных ссылок. Объект должен где-то удерживаться сильной ссылкой.

- **Циклы всё равно требуют GC.** Слабые ссылки помогают избежать циклов, но если цикл уже есть (из сильных ссылок), освобождение зависит от `gc`, который запускается не мгновенно.

- **Поведение во время завершения интерпретатора.** При shutdown порядок уничтожения объектов и вызова callbacks/финализаторов может отличаться; `finalize` с `atexit=True` (по умолчанию) гарантированно отработает при выходе.

## Лучшие практики

- Для кэшей объектов, которые «живут, пока используются», берите `WeakValueDictionary`. Для привязки метаданных к чужим объектам — `WeakKeyDictionary`.
- Для реестров живых экземпляров и списков подписчиков используйте `WeakSet`; для callbacks-методов — `WeakMethod`.
- Предпочитайте `weakref.finalize` методу `__del__` для освобождения ресурсов (файлы, соединения, временные каталоги).
- В финализаторах и callbacks **никогда** не удерживайте сильную ссылку на сам объект — передавайте только нужные данные.
- Разрывайте двунаправленные связи слабой обратной ссылкой (`parent_ref = weakref.ref(parent)`), чтобы не плодить циклы.
- В коде с weak-кэшем доступайтесь через `.get()` и проверяйте `None`, чтобы избежать гонок и `KeyError`.
- Для классов с `__slots__`, которым нужны слабые ссылки, явно добавляйте `"__weakref__"` в слоты.
- Когда важна предсказуемость кэша (всегда есть результат) — `functools.lru_cache`; когда важна экономия памяти — weak-кэш. Не путайте их назначение.
- Выбирайте `ref`, когда нужна проверка/контейнер; `proxy` — когда нужно прозрачно использовать объект, и оборачивайте обращения в `try/except ReferenceError`.

## Шпаргалка

```python
import weakref
import gc

# --- Базовая слабая ссылка ---
r = weakref.ref(obj)              # создать
obj_or_none = r()                 # получить объект (или None, если умер)
r = weakref.ref(obj, callback)    # callback(ref) при смерти объекта

# --- Прокси (ведёт себя как объект) ---
p = weakref.proxy(obj)            # p.attr, p.method() — напрямую
# после смерти объекта обращение -> ReferenceError

# --- Слабые контейнеры ---
wvd = weakref.WeakValueDictionary()   # слабые ЗНАЧЕНИЯ -> кэш с автоочисткой
wkd = weakref.WeakKeyDictionary()     # слабые КЛЮЧИ   -> метаданные к объектам
ws  = weakref.WeakSet()               # слабые элементы -> реестры/подписчики

# --- Финализатор (замена __del__) ---
f = weakref.finalize(obj, func, *args)   # func(*args) при смерти obj
f.alive        # жив ли ещё объект
f()            # вызвать вручную (один раз)
f.peek()       # посмотреть (func, args, kwargs) без вызова

# --- Слабая ссылка на метод ---
m = weakref.WeakMethod(obj.method)
bound = m()    # живой bound-method или None
if bound:
    bound()

# --- Утилиты ---
weakref.getweakrefcount(obj)      # сколько слабых ссылок
weakref.getweakrefs(obj)          # список слабых ссылок

# --- Поддержка weakref в классе со __slots__ ---
class C:
    __slots__ = ("x", "__weakref__")   # без "__weakref__" -> TypeError

# --- Циклы и GC ---
gc.collect()   # форсировать сборку циклов -> связанные weakref станут None

# НЕ поддерживают weakref: int, float, str, bytes, tuple, list, dict, bool
# Поддерживают: экземпляры обычных классов, set, classes со __weakref__
```
