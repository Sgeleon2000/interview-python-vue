# multiprocessing — подготовка к собеседованию

## Что это и зачем

Модуль `multiprocessing` запускает задачи в **отдельных процессах операционной системы**, а не в потоках. Каждый процесс имеет **собственный интерпретатор Python, собственный GIL и изолированное адресное пространство (память)**. Именно это позволяет **обойти GIL** и получить **настоящий параллелизм на нескольких ядрах CPU**.

Зачем нужен multiprocessing:

- **CPU-bound задачи** (численные расчёты, обработка изображений, сжатие, шифрование, парсинг больших объёмов, ML-вычисления на чистом Python). Здесь процессы дают линейное (близкое к числу ядер) ускорение, чего потоки из-за GIL дать не могут.
- **Изоляция отказов** — падение одного процесса (например, segfault в C-расширении) не роняет остальные.
- **Обход ограничений одного интерпретатора** — каждый процесс работает независимо.

Цена за это:

- **Дороже создание процесса**, чем потока.
- **Изолированная память** → данные между процессами нужно **сериализовать (pickle)** и передавать через IPC (очереди, пайпы, разделяемую память). Это накладные расходы и ограничения (не всё можно запикапить).
- Сложнее отладка и совместное использование состояния.

```python
from multiprocessing import Process
import os

def worker(name):
    print(f"Процесс {name}, PID={os.getpid()}, родитель={os.getppid()}")

if __name__ == "__main__":            # ОБЯЗАТЕЛЬНО на Windows/spawn
    p = Process(target=worker, args=("A",))
    p.start()
    p.join()
    print("Готово, главный PID:", os.getpid())
```

## Ключевые концепции

### GIL и обход его процессами — главная тема собеседования

Напомним суть GIL: в CPython это глобальный мьютекс, из-за которого **в рамках одного процесса** Python-байткод исполняет только один поток одновременно. Поэтому потоки не ускоряют CPU-bound код.

**Как multiprocessing обходит GIL:**

1. **У каждого процесса свой собственный интерпретатор Python и, соответственно, свой собственный GIL.** Это не «общий замок на все процессы» — каждый процесс блокирует только себя. Восемь процессов на восьмиядерной машине будут реально считать параллельно на восьми ядрах, потому что ни один не конкурирует за чужой GIL.

2. **Изолированная память — причина и следствие.** Процессы не разделяют память, поэтому им не нужен общий замок для защиты счётчиков ссылок. Каждый процесс управляет своими объектами независимо. Отсюда и параллелизм, и необходимость явно передавать данные между процессами.

3. **Поэтому правило простое:** для **CPU-bound** → `multiprocessing` (или `ProcessPoolExecutor`); для **I/O-bound** → `threading`/`asyncio`. На I/O-bound процессы тоже работают, но они тяжелее потоков и требуют сериализации данных, поэтому обычно избыточны.

4. **Граница применимости.** Выигрыш от процессов «съедается», если:
   - задача слишком мелкая (накладные расходы на запуск процесса и pickling больше самой работы);
   - между процессами нужно гонять большие объёмы данных (сериализация дорогая);
   - результат всё равно упирается в один общий ресурс (диск, сеть).

```python
import time
from multiprocessing import Pool

def cpu_bound(n):
    # CPU-нагрузка: суммируем квадраты
    return sum(i * i for i in range(n))

if __name__ == "__main__":
    N = 10_000_000
    tasks = [N] * 8

    # Последовательно (один процесс, упирается в одно ядро)
    start = time.perf_counter()
    [cpu_bound(n) for n in tasks]
    print("Последовательно:", round(time.perf_counter() - start, 2), "сек")

    # Параллельно через пул процессов — обходим GIL, грузим все ядра
    start = time.perf_counter()
    with Pool(processes=8) as pool:
        pool.map(cpu_bound, tasks)
    print("8 процессов:", round(time.perf_counter() - start, 2), "сек")
    # Для CPU-bound процессы дают РЕАЛЬНОЕ ускорение, в отличие от потоков
```

### Методы старта: spawn / fork / forkserver

Способ создания дочернего процесса критично влияет на поведение программы.

- **fork** (по умолчанию на Linux до Python 3.13; на macOS больше не дефолт): дочерний процесс — **копия** родителя (copy-on-write памяти). Быстро, но **наследует всё состояние**: открытые файлы, замки, потоки. Опасно: если в родителе был захвачен замок другим потоком в момент fork, в потомке он останется «вечно захваченным» → deadlock. Также небезопасен с многопоточностью.
- **spawn** (по умолчанию на Windows и macOS, начиная с 3.8 на macOS): запускается **новый чистый интерпретатор**, импортирует модуль заново и передаёт только нужные аргументы (через pickle). Медленнее старт, но безопаснее и предсказуемее. **Именно из-за spawn обязателен `if __name__ == "__main__"`** — иначе при импорте модуля в дочернем процессе код запуска выполнится снова → бесконечное порождение процессов.
- **forkserver** (Linux): отдельный процесс-сервер форкает «чистые» воркеры по запросу. Компромисс между скоростью fork и безопасностью spawn.

```python
import multiprocessing as mp

if __name__ == "__main__":
    # Явно задаём метод старта (делать ОДИН раз, в начале программы)
    mp.set_start_method("spawn")   # или "fork", "forkserver"
    # Альтернатива без глобального изменения:
    ctx = mp.get_context("spawn")
    p = ctx.Process(target=print, args=("hello",))
    p.start(); p.join()
```

### Pickling аргументов

При spawn/forkserver и при передаче данных через `Queue`/`Pipe`/`Pool` объекты **сериализуются через pickle**. Отсюда ограничения:

- Аргументы, возвращаемые значения и сама целевая функция должны быть **picklable**.
- **Нельзя** запикапить: лямбды, локальные (вложенные) функции, замыкания, объекты с открытыми сокетами/файлами, многие объекты БД, генераторы.
- Функции пула должны быть определены на верхнем уровне модуля (чтобы их можно было импортировать по имени).

### IPC: способы обмена данными

- **Queue** — потоко- и процессобезопасная очередь (FIFO), данные сериализуются.
- **Pipe** — двунаправленный канал между ровно двумя процессами, быстрее очереди.
- **Value / Array** — разделяемая память для простых C-типов (числа, массивы).
- **Manager** — сервер-процесс, предоставляющий разделяемые «proxy»-объекты (dict, list и т. д.), но медленнее (всё идёт через прокси и сериализацию).
- **shared_memory** (Python 3.8+) — блок разделяемой памяти для больших данных без копирования (часто с numpy).

## Основные функции/классы/методы

### Process

```python
from multiprocessing import Process
import os, time

def task(name, seconds):
    print(f"[{name}] старт, PID={os.getpid()}")
    time.sleep(seconds)
    print(f"[{name}] конец")

if __name__ == "__main__":
    p = Process(target=task, args=("worker", 1), name="W1", daemon=False)
    p.start()
    print("Жив?", p.is_alive(), "PID:", p.pid)
    p.join(timeout=5)          # ждём (опц. с таймаутом)
    print("Код выхода:", p.exitcode)   # 0 — успех, !=0 / отрицательный — ошибка/сигнал
    # p.terminate()  — послать SIGTERM; p.kill() — SIGKILL
```

### Pool — пул процессов

```python
from multiprocessing import Pool

def square(x):
    return x * x

if __name__ == "__main__":
    with Pool(processes=4) as pool:
        # map — блокирующий, сохраняет порядок
        print(pool.map(square, range(10)))

        # imap — ленивый итератор, сохраняет порядок
        for r in pool.imap(square, range(5)):
            print(r, end=" ")
        print()

        # imap_unordered — отдаёт результаты по мере готовности (быстрее)
        for r in pool.imap_unordered(square, range(5)):
            print(r, end=" ")
        print()

        # apply_async — одна задача асинхронно, возвращает AsyncResult
        res = pool.apply_async(square, (12,))
        print(res.get(timeout=5))

        # starmap — для функций с несколькими аргументами
        print(pool.starmap(pow, [(2, 3), (3, 2)]))   # [8, 9]
```

### Queue — очередь между процессами

```python
from multiprocessing import Process, Queue

def producer(q, n):
    for i in range(n):
        q.put(i)              # данные сериализуются (pickle)
    q.put(None)               # sentinel — сигнал «конец»

def consumer(q):
    while True:
        item = q.get()
        if item is None:
            break
        print("Получено:", item)

if __name__ == "__main__":
    q = Queue()
    p1 = Process(target=producer, args=(q, 5))
    p2 = Process(target=consumer, args=(q,))
    p1.start(); p2.start()
    p1.join(); p2.join()
```

### Pipe — канал между двумя процессами

```python
from multiprocessing import Process, Pipe

def worker(conn):
    conn.send({"msg": "привет", "value": 42})   # отправить
    print("Worker получил:", conn.recv())        # принять
    conn.close()

if __name__ == "__main__":
    parent_conn, child_conn = Pipe()    # два конца двунаправленного канала
    p = Process(target=worker, args=(child_conn,))
    p.start()
    print("Главный получил:", parent_conn.recv())
    parent_conn.send("ответ от родителя")
    p.join()
```

### Value / Array — разделяемая память для C-типов

```python
from multiprocessing import Process, Value, Array, Lock

def increment(counter, lock):
    for _ in range(100_000):
        with lock:                 # синхронизация обязательна!
            counter.value += 1     # value.value += 1 НЕ атомарно

if __name__ == "__main__":
    counter = Value("i", 0)        # 'i' — int (см. коды модуля array)
    arr = Array("d", [0.0, 1.0, 2.0])   # 'd' — double
    lock = Lock()

    procs = [Process(target=increment, args=(counter, lock)) for _ in range(4)]
    for p in procs: p.start()
    for p in procs: p.join()

    print("Счётчик:", counter.value)   # 400000 благодаря замку
    print("Массив:", list(arr))
```

### Manager — разделяемые объекты-прокси

```python
from multiprocessing import Process, Manager

def worker(shared_dict, shared_list, key, value):
    shared_dict[key] = value           # прокси-объекты, изменения видны всем
    shared_list.append(value)

if __name__ == "__main__":
    with Manager() as manager:
        d = manager.dict()
        lst = manager.list()
        procs = [
            Process(target=worker, args=(d, lst, f"k{i}", i))
            for i in range(5)
        ]
        for p in procs: p.start()
        for p in procs: p.join()
        print("dict:", dict(d))
        print("list:", list(lst))
    # Manager медленнее Value/Array: каждое обращение идёт через сервер-процесс
```

### shared_memory — большие данные без копирования (3.8+)

```python
from multiprocessing import shared_memory
import numpy as np

if __name__ == "__main__":
    a = np.array([1, 2, 3, 4], dtype=np.int64)
    shm = shared_memory.SharedMemory(create=True, size=a.nbytes)
    # Создаём numpy-массив поверх разделяемого буфера
    b = np.ndarray(a.shape, dtype=a.dtype, buffer=shm.buf)
    b[:] = a[:]                  # копируем данные в общую память
    print("Имя сегмента:", shm.name)   # его передают в дочерний процесс
    # ... другой процесс открывает SharedMemory(name=shm.name) и читает без копий
    shm.close()
    shm.unlink()                 # освободить сегмент (важно — иначе утечка)
```

## Частые вопросы на собеседовании

**Q1. Чем multiprocessing отличается от threading и когда что использовать?**
Threading — потоки в одном процессе, общая память, упираются в GIL → для I/O-bound. Multiprocessing — отдельные процессы со своей памятью и своим GIL → обходят GIL, дают параллелизм на ядрах → для CPU-bound. Цена процессов: дороже старт и сериализация данных.

**Q2. Как multiprocessing обходит GIL?**
GIL — это замок на уровне процесса/интерпретатора. У каждого процесса свой интерпретатор и свой GIL, поэтому процессы не конкурируют за общий замок и реально исполняют байткод параллельно на разных ядрах.

**Q3. Что такое spawn и fork? В чём разница?**
fork копирует родительский процесс целиком (быстро, но наследует состояние, опасен с потоками/замками). spawn запускает чистый новый интерпретатор и передаёт аргументы через pickle (медленнее, но безопаснее и кроссплатформенно). По умолчанию: fork на Linux (исторически), spawn на Windows и macOS.

**Q4. Зачем нужен `if __name__ == "__main__"` в multiprocessing?**
При spawn дочерний процесс импортирует основной модуль заново. Без защиты `if __name__ == "__main__"` код запуска процессов выполнится при импорте в потомке → рекурсивное бесконечное порождение процессов и краш. На Linux с fork это менее критично, но писать защиту нужно для переносимости.

**Q5. Какие ограничения накладывает pickling?**
Всё, что передаётся между процессами (аргументы, результаты, целевые функции при spawn), должно быть picklable. Нельзя пиклить лямбды, локальные/вложенные функции, замыкания, открытые сокеты/файлы, генераторы. Функции для пула определяют на верхнем уровне модуля.

**Q6. Чем Queue отличается от Pipe?**
Pipe — канал между ровно двумя процессами, двунаправленный, быстрее, но без встроенной синхронизации на много читателей/писателей. Queue — очередь FIFO, безопасна для многих процессов, под капотом использует Pipe + замки + поток-фидер; удобнее, но чуть медленнее.

**Q7. Как разделить состояние между процессами?**
Вариантами: `Value`/`Array` (простые C-типы в разделяемой памяти, нужен Lock), `Manager` (прокси dict/list/etc., медленнее), `shared_memory` (большие буферы без копирования, часто с numpy), либо обмен сообщениями через `Queue`/`Pipe`. Прямого доступа к переменным родителя у дочернего процесса нет.

**Q8. Чем Manager отличается от Value/Array?**
`Value`/`Array` — настоящая разделяемая память для примитивов, быстро. `Manager` поднимает отдельный сервер-процесс, объекты — прокси; поддерживает сложные типы (dict, list, Namespace), но каждое обращение сериализуется и идёт по IPC → значительно медленнее.

**Q9. Является ли `Value.value += 1` атомарным?**
Нет. Это чтение-инкремент-запись, между которыми возможно переключение процессов. Нужен `Lock` (или встроенный `value.get_lock()`), иначе будут потерянные обновления — та же проблема гонки, что и с потоками.

**Q10. Что такое daemon-процесс в multiprocessing?**
Демонический процесс автоматически завершается при завершении родителя и **не может иметь собственных дочерних процессов**. Удобен для фоновых задач, но его нельзя join'ить после завершения родителя.

**Q11. Что вернёт `process.exitcode`?**
0 — успешное завершение; положительное число — код выхода (например, из-за `sys.exit(n)`); отрицательное `-N` — процесс убит сигналом N (например, -9 для SIGKILL); `None` — ещё не завершился.

**Q12. Чем `pool.map` отличается от `imap` и `imap_unordered`?**
`map` — блокирующий, возвращает список, сохраняет порядок, материализует всё сразу. `imap` — ленивый итератор, порядок сохранён, экономит память. `imap_unordered` — отдаёт результаты по мере готовности (порядок не гарантирован), обычно быстрее.

**Q13. Почему мелкие задачи плохо параллелятся процессами?**
Накладные расходы: запуск процесса, сериализация аргументов и результата, IPC. Если работа короче этих расходов, выигрыша нет, а часто потери. Решение — укрупнять задачи (chunking), использовать `chunksize` в `map`.

**Q14. Почему fork опасен в многопоточной программе?**
fork копирует только вызывающий поток, но наследует состояние замков. Если другой поток держал замок в момент fork, в потомке он останется захваченным навсегда → deadlock. Поэтому в многопоточных приложениях предпочитают spawn/forkserver.

**Q15. Как выбрать число процессов?**
Для CPU-bound — обычно `os.cpu_count()` (или чуть меньше, оставив ядро системе). Больше процессов, чем ядер, для чистого CPU смысла не имеет — только переключения контекста. Для смешанной нагрузки настраивают эмпирически.

## Подводные камни (gotchas)

- **Забытый `if __name__ == "__main__"`** при spawn → бесконечное порождение процессов и краш. Всегда оборачивайте точку входа.
- **Непиклящиеся объекты** (лямбды, локальные функции, открытые сокеты) → `PicklingError`. Используйте функции верхнего уровня.
- **`Value.value += 1` без замка** → гонка и потерянные обновления, как и в потоках.
- **`Manager` медленный** — не используйте его для горячих циклов с миллионами обращений; берите `Value`/`Array`/`shared_memory`.
- **Большие данные через Queue/Pipe** дорого сериализуются — не гоняйте гигабайты, используйте `shared_memory`.
- **Deadlock на больших данных в Queue.** Если положить в очередь много данных и не читать их, поток-фидер заблокируется, а `join()` родителя зависнет. Сначала читайте очередь, потом join'ите.
- **fork + потоки** → unsafe, возможен deadlock унаследованных замков. Берите spawn/forkserver.
- **Не вызван `shm.unlink()`** для `shared_memory` → утечка сегмента в ОС (останется после завершения программы).
- **Глобальные переменные не «расшариваются».** При fork это копии (изменения не видны родителю), при spawn их вовсе нет до повторного импорта.
- **`terminate()`/`kill()`** обрывают процесс без финализации — ресурсы, замки, очереди могут остаться в несогласованном состоянии.
- **Запуск из интерактивного интерпретатора** при spawn не работает корректно — код должен быть в импортируемом модуле/скрипте.
- **Исключение в дочернем процессе** не всплывает автоматически в родителя — при `Pool` оно сохраняется и пробрасывается при `get()`/итерации результата; при сыром `Process` — теряется (виден только ненулевой `exitcode`).

## Лучшие практики

- Используйте **`concurrent.futures.ProcessPoolExecutor`** или `Pool` как контекстный менеджер (`with`) — это гарантирует корректное закрытие пула.
- Всегда оборачивайте точку входа в **`if __name__ == "__main__"`**.
- Для **CPU-bound** ставьте число процессов ≈ числу ядер (`os.cpu_count()`), не больше.
- **Укрупняйте задачи** и используйте `chunksize`, чтобы амортизировать накладные расходы.
- **Минимизируйте передаваемые данные** между процессами; для больших массивов — `shared_memory`/numpy.
- Делайте целевые функции **функциями верхнего уровня** (picklable).
- Для общего изменяемого состояния используйте **`Value`/`Array` + Lock**, а `Manager` — только когда нужны сложные структуры и скорость некритична.
- Предпочитайте **`spawn`/`forkserver`** в многопоточных приложениях ради безопасности.
- Завершайте процессы **кооперативно** (sentinel в очереди, флаг), а не `terminate`.
- Для `shared_memory` обязательно вызывайте `close()` и `unlink()`.
- Замеряйте: для мелких задач процессы могут быть **медленнее** последовательного кода.

## Шпаргалка

```python
import multiprocessing as mp
from multiprocessing import Process, Pool, Queue, Pipe, Value, Array, Manager, Lock

# Один процесс
p = Process(target=fn, args=(...), daemon=False)
p.start(); p.join(timeout=None); p.is_alive(); p.exitcode; p.terminate()

# Пул процессов
with Pool(processes=mp.cpu_count()) as pool:
    pool.map(fn, iterable, chunksize=...)      # блокирующий, по порядку
    pool.imap(fn, iterable)                     # ленивый, по порядку
    pool.imap_unordered(fn, iterable)           # по готовности
    pool.starmap(fn, [(a, b), ...])             # несколько аргументов
    r = pool.apply_async(fn, args); r.get(timeout=...)

# IPC
q = Queue(); q.put(x); q.get()                  # очередь (pickle)
a, b = Pipe(); a.send(x); b.recv()              # канал между двумя

# Разделяемая память
v = Value("i", 0)                               # C-тип int
arr = Array("d", [1.0, 2.0])                    # массив double
with v.get_lock(): v.value += 1                 # синхронизация

# Manager (сложные структуры, медленнее)
with Manager() as m:
    d = m.dict(); lst = m.list()

# Метод старта
mp.set_start_method("spawn")                    # один раз в начале
ctx = mp.get_context("forkserver")              # или локальный контекст
```

| Сценарий | Инструмент |
|----------|-----------|
| CPU-bound вычисления | Process / Pool / ProcessPoolExecutor |
| Передать сообщения многим | Queue |
| Канал между двумя процессами | Pipe |
| Общий счётчик/массив примитивов | Value / Array + Lock |
| Общий dict/list | Manager |
| Большой массив без копий | shared_memory (+ numpy) |
| Кроссплатформенность/безопасность | start_method="spawn" |

**Запомнить:** каждый процесс = свой интерпретатор = свой GIL → реальный параллелизм на ядрах для CPU-bound. Платим сериализацией (pickle) и `if __name__ == "__main__"`.
