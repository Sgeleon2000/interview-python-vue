# shelve — подготовка к собеседованию

## Что это и зачем

`shelve` — модуль стандартной библиотеки, реализующий **персистентный словарь**: объект, который ведёт себя как обычный `dict`, но хранит данные на диске. Ключи — строки, значения — **любые picklable-объекты** (сериализуются через `pickle`). По сути shelve = `dbm` (key-value хранилище на диске) + `pickle` (сериализация произвольных Python-значений).

Когда применять:
- простое локальное key-value хранилище без поднятия отдельной БД;
- кэш промежуточных результатов на диск;
- сохранение конфигов/состояния приложения в виде Python-объектов;
- быстрые прототипы.

Когда НЕ применять:
- многопользовательский/конкурентный доступ (shelve не транзакционен и не потокобезопасен);
- обмен данными между языками или с недоверенным вводом (под капотом pickle → небезопасно);
- сложные запросы/индексы (нужна реляционная БД или специализированный KV-store).

## Ключевые концепции

- **Персистентность** — данные переживают перезапуск процесса (лежат в dbm-файле/файлах).
- **`pickle` под капотом** — значения сериализуются pickle, отсюда и плюсы (любые объекты), и минусы (небезопасность, Python-специфичность).
- **`dbm` бэкенд** — конкретная реализация хранилища (`dbm.sqlite3`, `dbm.gnu`, `dbm.ndbm`, `dbm.dumb`); может различаться между системами, отсюда разные имена файлов.
- **`writeback`** — кэширование изменяемых значений в памяти для корректной модификации на месте (с компромиссами по памяти/производительности).
- **Ключи — только `str`** (в отличие от dict, где ключом может быть любой хешируемый объект).

## Основные функции/классы/методы

### shelve.open и базовые операции

```python
import shelve

# Открываем (создаётся, если нет). Это контекстный менеджер -> close() гарантирован.
with shelve.open("mydata") as db:
    db["user"] = {"name": "Андрей", "age": 30}   # значение — любой picklable объект
    db["nums"] = [1, 2, 3]
    db["counter"] = 0

# Чтение в другом запуске процесса — данные сохранились
with shelve.open("mydata") as db:
    print(db["user"])        # {'name': 'Андрей', 'age': 30}
    print("nums" in db)      # True
    print(list(db.keys()))   # ['user', 'nums', 'counter']
    print(db.get("missing", "по умолчанию"))  # по умолчанию
    del db["counter"]        # удаление
```

Интерфейс почти полностью повторяет `dict`: `[]`, `in`, `del`, `keys()`, `values()`, `items()`, `get()`, `len()`, итерация.

### Флаги открытия

```python
import shelve

# flag:
#   'r' — только чтение, БД должна существовать
#   'w' — чтение/запись, БД должна существовать
#   'c' — чтение/запись, создать если нет (ПО УМОЛЧАНИЮ)
#   'n' — всегда создать новую пустую БД (затирает существующую!)
with shelve.open("mydata", flag="c") as db:
    db["x"] = 1

# protocol — версия pickle-протокола для значений
with shelve.open("mydata", protocol=5) as db:
    db["big"] = list(range(1000))
```

### writeback — ключевой нюанс мутаций

Без `writeback=True` значения **не** кэшируются: при `db["key"]` каждый раз возвращается **свежий unpickled-объект**, и его модификация на месте НЕ сохраняется обратно.

```python
import shelve

# --- ПРОБЛЕМА: мутация на месте без writeback ТЕРЯЕТСЯ ---
with shelve.open("data1") as db:
    db["list"] = [1, 2, 3]

with shelve.open("data1") as db:
    db["list"].append(4)        # модифицируем ВРЕМЕННУЮ копию -> не сохранится!
    print(db["list"])           # [1, 2, 3]  — изменение потеряно

# --- РЕШЕНИЕ 1: переприсваивание ---
with shelve.open("data1") as db:
    tmp = db["list"]            # читаем
    tmp.append(4)              # меняем
    db["list"] = tmp           # ОБРАТНО присваиваем -> сохранится
    print(db["list"])          # [1, 2, 3, 4]

# --- РЕШЕНИЕ 2: writeback=True ---
with shelve.open("data1", writeback=True) as db:
    db["list"].append(5)       # теперь мутация на месте работает
    # объект кэшируется в памяти и записывается при close()/sync()
    print(db["list"])          # [1, 2, 3, 4, 5]
```

`writeback=True`:
- плюс: можно менять вложенные изменяемые значения на месте;
- минус: все прочитанные объекты держатся в памяти-кэше до `close()`/`sync()` (рост потребления памяти и медленнее закрытие, т.к. всё переписывается).

### sync и close

```python
import shelve

db = shelve.open("data", writeback=True)
db["a"] = [1, 2]
db["a"].append(3)
db.sync()    # сбросить кэш на диск и очистить writeback-кэш (без закрытия)
db.close()   # обязательно закрыть: сброс на диск + закрытие dbm-файла
```

При `writeback=True` периодический `sync()` помогает контролировать память (сбрасывает кэш). При работе без контекстного менеджера `close()` обязателен, иначе данные могут не записаться.

### Произвольные объекты как значения

```python
import shelve

class Task:
    def __init__(self, title, done=False):
        self.title = title
        self.done = done

with shelve.open("tasks") as db:
    db["t1"] = Task("Подготовиться к собесу")  # экземпляр класса picklable -> ок

with shelve.open("tasks") as db:
    t = db["t1"]
    print(t.title, t.done)  # Подготовиться к собесу False
    # ВАЖНО: класс Task должен быть импортируем при чтении (как и для pickle)
```

## Частые вопросы на собеседовании

**Q1. Что такое shelve и на чём он построен?**
Это персистентный словарь: интерфейс `dict`, но данные на диске. Реализован как `dbm` (key-value хранилище) + `pickle` (сериализация значений). Ключи — строки, значения — любые picklable-объекты.

**Q2. Какие ограничения на ключи и значения?**
Ключи — только строки (`str`). Значения — любой объект, который умеет сериализовать pickle (числа, контейнеры, экземпляры классов верхнего уровня и т.д.); не сериализуются лямбды, сокеты, файлы и т.п.

**Q3. Что такое `writeback` и зачем он нужен?**
Флаг, включающий кэширование прочитанных значений в памяти. Без него мутация изменяемого значения «на месте» (`db["k"].append(...)`) не сохраняется, потому что `db["k"]` возвращает временную копию. С `writeback=True` изменения отслеживаются и пишутся при `sync`/`close`, но растёт потребление памяти.

**Q4. Как сохранить изменение вложенного списка/словаря БЕЗ writeback?**
Прочитать значение в переменную, изменить и присвоить обратно: `tmp = db["k"]; tmp.append(x); db["k"] = tmp`.

**Q5. Чем shelve отличается от обычного dict?**
shelve персистентен (данные на диске), ключи только строки, значения сериализуются (накладные расходы pickle), нет атомарности/транзакций, не потокобезопасен, мутации требуют осторожности (writeback). dict — в памяти, любые хешируемые ключи, мгновенный доступ.

**Q6. Безопасен ли shelve?**
Нет, если файл может прийти из недоверенного источника: значения десериализуются pickle, а это потенциальное выполнение произвольного кода. Открывайте только свои файлы.

**Q7. Потокобезопасен ли shelve / поддерживает ли конкурентный доступ?**
Нет. shelve не предназначен для одновременного доступа из нескольких процессов/потоков на запись. Для конкурентности используйте SQLite или специализированную БД.

**Q8. Зачем `sync()` и `close()`?**
`close()` сбрасывает данные на диск и закрывает dbm-файл — обязателен (или используйте `with`). `sync()` сбрасывает буферы/writeback-кэш на диск без закрытия, помогает ограничить рост памяти при `writeback=True`.

**Q9. Почему имена файлов БД различаются на разных машинах?**
shelve использует доступный `dbm`-бэкенд (`sqlite3`, `gnu`, `ndbm`, `dumb`), который выбирается системой. Разные бэкенды создают разные файлы (один файл или несколько `.dir/.dat/.bak`). Это снижает переносимость файла между системами.

**Q10. Когда выбрать shelve, а когда sqlite3?**
shelve — для простого локального key-value хранилища Python-объектов без запросов и конкурентности. sqlite3 — когда нужны запросы, индексы, транзакции, конкурентный доступ, переносимость и кросс-язычность.

## Подводные камни (gotchas)

- **Мутация на месте без `writeback` теряется.** Главная ловушка: `db["k"].append(x)` ничего не сохранит. Решение: переприсваивание или `writeback=True`.
- **`writeback=True` ест память.** Все прочитанные объекты кэшируются до `close()`/`sync()`; на больших БД это проблема. Периодически вызывайте `sync()`.
- **Безопасность (pickle).** shelve десериализует значения через pickle → не открывайте недоверенные файлы (риск RCE).
- **Ключи только строки.** Передача нестрокового ключа приведёт к ошибке (зависит от бэкенда).
- **Нет транзакций/атомарности.** Сбой посреди записи может повредить БД; нет rollback.
- **Не для конкурентного доступа.** Несколько писателей повредят данные.
- **Переносимость файлов.** Из-за разных dbm-бэкендов файл, созданный на одной системе, может не открыться на другой.
- **Класс значения должен быть импортируем при чтении** (как и для обычного pickle): переименование/перемещение класса сломает загрузку.
- **`flag='n'` затирает БД.** Случайное использование уничтожит данные.
- **Не забывать `close()`** при работе без `with` — иначе изменения могут не записаться.

## Лучшие практики

1. Всегда используйте контекстный менеджер `with shelve.open(...) as db:` — гарантирует `close()`.
2. Для вложенных изменяемых значений: либо переприсваивание, либо `writeback=True` с регулярным `sync()`.
3. Открывайте только доверенные файлы (помните про pickle и безопасность).
4. Для конкурентного/многопользовательского доступа или сложных запросов берите sqlite3.
5. Указывайте явный `flag` (`'r'` для чтения, чтобы случайно не модифицировать/создать).
6. Фиксируйте `protocol` для предсказуемости/совместимости pickle.
7. Не полагайтесь на переносимость файла между ОС/машинами.
8. Используйте осмысленные строковые ключи; для составных ключей делайте префиксы (`user:42`).

## Шпаргалка

```python
import shelve

# --- Открытие (флаги: 'r' / 'w' / 'c'(дефолт) / 'n'(новая, затирает)) ---
with shelve.open("db", flag="c", protocol=5, writeback=False) as db:
    db["key"] = {"any": "picklable"}   # запись (значение -> pickle)
    val = db["key"]                    # чтение (новый объект каждый раз без writeback)
    "key" in db                        # проверка наличия
    del db["key"]                      # удаление
    list(db.keys())                    # ключи (только str!)
    db.get("k", default)               # как у dict

# --- Мутация вложенного значения ---
# БЕЗ writeback: переприсвоить
tmp = db["lst"]; tmp.append(1); db["lst"] = tmp
# С writeback=True: меняем на месте, но память растёт
with shelve.open("db", writeback=True) as db:
    db["lst"].append(1)
    db.sync()   # сбросить кэш и освободить память

# Особенности:
#   ключи только str, значения — picklable
#   НЕ безопасно для недоверенных файлов (pickle!)
#   НЕ потокобезопасно, нет транзакций
#   shelve = dbm + pickle
```
