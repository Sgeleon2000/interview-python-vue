# venv — подготовка к собеседованию

## Что это и зачем

`venv` — встроенный модуль стандартной библиотеки (с Python 3.3) для создания **виртуальных окружений** — изолированных Python-окружений с собственным набором установленных пакетов, отдельным от системного интерпретатора и других проектов.

**Проблема, которую он решает:** разные проекты требуют разных версий библиотек (проект A — Django 3, проект B — Django 5). Если ставить пакеты глобально, версии конфликтуют («dependency hell»). Виртуальное окружение даёт каждому проекту свою изолированную «песочницу» с зависимостями.

**Зачем нужен:**
- Изоляция зависимостей между проектами.
- Воспроизводимость окружения (`requirements.txt` / lock-файлы).
- Установка пакетов без прав администратора и без засорения системного Python.
- Чистота: системный Python (используемый ОС) остаётся нетронутым.

## Ключевые концепции

1. **Виртуальное окружение** — каталог с:
   - символической ссылкой/копией интерпретатора Python,
   - собственным каталогом `site-packages` (куда `pip` ставит пакеты),
   - скриптами активации (`activate`),
   - файлом конфигурации `pyvenv.cfg`.

2. **Изоляция, а не виртуализация.** `venv` не изолирует ОС/процессы (это не контейнер). Он лишь подменяет пути поиска пакетов и интерпретатор. По умолчанию окружение не видит системные `site-packages` (`include-system-site-packages = false`).

3. **Активация** меняет переменную окружения `PATH`, чтобы `python` и `pip` указывали на окружение, и задаёт `VIRTUAL_ENV`. Деактивация (`deactivate`) возвращает прежний `PATH`.

4. **`pip`** — менеджер пакетов, ставит пакеты с PyPI в активное окружение. Часто работают через `requirements.txt` (список зависимостей).

5. **Привязка к версии Python.** Окружение создаётся конкретным интерпретатором и «наследует» его версию. Смена версии Python = новое окружение.

6. **Не переносимо по абсолютным путям.** Окружение содержит абсолютные пути; копировать каталог `.venv` на другую машину/путь нельзя — нужно пересоздать из `requirements.txt`.

## Основные функции/классы/методы

### Создание и активация

```bash
# Создать виртуальное окружение в каталоге .venv
python -m venv .venv

# Активация (Linux / macOS, bash/zsh)
source .venv/bin/activate

# Активация (Windows, PowerShell)
.venv\Scripts\Activate.ps1

# Активация (Windows, cmd.exe)
.venv\Scripts\activate.bat

# После активации приглашение показывает (.venv); проверим интерпретатор:
which python          # .../.venv/bin/python
python -V

# Деактивация
deactivate
```

### Установка зависимостей через pip

```bash
# Внутри активированного окружения
pip install requests "django>=4,<5"

# Зафиксировать текущие версии (снимок окружения)
pip freeze > requirements.txt

# Установить из файла (воспроизведение окружения)
pip install -r requirements.txt

# Обновить pip
python -m pip install --upgrade pip

# Список и информация
pip list
pip show requests
```

Пример `requirements.txt`:

```
requests==2.31.0
Django>=4.2,<5.0
python-dotenv~=1.0
```

### Запуск без активации (надёжнее в скриптах/CI)

```bash
# Можно не активировать, а вызывать интерпретатор окружения напрямую
.venv/bin/python -m pip install requests
.venv/bin/python script.py

# Windows:
.venv\Scripts\python.exe script.py
```

### Полезные флаги venv

```bash
# Дать окружению доступ к системным пакетам
python -m venv --system-site-packages .venv

# Без установки pip (например, для последующего ensurepip)
python -m venv --without-pip .venv

# Пересоздать/обновить окружение поверх существующего
python -m venv --upgrade-deps .venv     # обновить pip/setuptools при создании
python -m venv --clear .venv            # очистить содержимое перед созданием

# Использовать копии вместо симлинков (актуально для переносимости/Windows)
python -m venv --copies .venv

# Своё приглашение
python -m venv --prompt myproject .venv
```

### Создание окружения программно (из Python)

```python
import venv

# Создать окружение с обновлением pip
builder = venv.EnvBuilder(with_pip=True, upgrade_deps=True, prompt="api")
builder.create("/tmp/myenv")

# Можно унаследовать EnvBuilder и переопределить post_setup,
# чтобы автоматически ставить пакеты после создания
class CustomBuilder(venv.EnvBuilder):
    def post_setup(self, context):
        import subprocess
        subprocess.check_call(
            [context.env_exe, "-m", "pip", "install", "requests"]
        )

CustomBuilder(with_pip=True).create("/tmp/env2")
```

### Файл pyvenv.cfg

```ini
# .venv/pyvenv.cfg — конфигурация окружения
home = /usr/local/bin
include-system-site-packages = false
version = 3.12.1
executable = /usr/local/bin/python3.12
command = /usr/local/bin/python3.12 -m venv /path/.venv
```

### Воспроизводимость: requirements vs lock

```bash
# requirements.txt из pip freeze фиксирует версии, но НЕ хеши и не дерево зависимостей.
# Для строгой воспроизводимости используют:
#   - pip-tools: pip-compile -> requirements с пинами и хешами
#   - poetry / pipenv: lock-файлы с полным разрешённым деревом
pip install --require-hashes -r requirements.lock.txt
```

## Сравнение: venv vs virtualenv vs conda vs poetry vs pipenv

| Инструмент   | Что делает | Особенности |
|--------------|-----------|-------------|
| **venv** | Встроен в Python 3.3+, создаёт изолированные окружения. | Стандарт, без зависимостей. Только окружения + `pip`. Не управляет версией самого Python. |
| **virtualenv** | Сторонний предок `venv`. | Быстрее создаёт окружения, поддерживает старые Python (2.x), больше возможностей. Сейчас `venv` достаточно для большинства задач. |
| **conda** | Менеджер окружений И пакетов (не только Python). | Ставит не-Python зависимости (C-библиотеки, CUDA, R). Управляет версиями Python. Свой репозиторий (не PyPI). Популярен в data science. Тяжелее. |
| **poetry** | Управление зависимостями + сборка/публикация. | `pyproject.toml`, lock-файл (`poetry.lock`), разрешение дерева зависимостей, виртуальные окружения «под капотом». Современный стандарт для библиотек/приложений. |
| **pipenv** | `Pipfile` + `Pipfile.lock`, обёртка над venv+pip. | Объединяет создание окружения и lock. Сейчас менее популярен, чем poetry/uv. |
| **uv** | Сверхбыстрый менеджер (на Rust). | Заменяет pip/venv/pip-tools, очень быстрый, lock-файлы, управление версиями Python. Набирает популярность. |

**Кратко:**
- `venv` + `pip` (+ `pip-tools`) — минимализм, стандартная библиотека.
- `poetry` / `uv` — управление зависимостями и сборка с lock-файлами.
- `conda` — когда нужны не-Python зависимости (наука, ML).
- `virtualenv` — legacy / нужна поддержка старых Python или доп. фич.

## Частые вопросы на собеседовании

**Q1. Зачем нужны виртуальные окружения?**
A: Изолировать зависимости проектов друг от друга и от системного Python, избежать конфликтов версий, обеспечить воспроизводимость, ставить пакеты без прав root и не засорять системный интерпретатор.

**Q2. Что физически создаёт `venv`?**
A: Каталог с (символической ссылкой или копией) интерпретатора Python, собственным `site-packages`, скриптами активации и файлом `pyvenv.cfg`. `pip` ставит пакеты в локальный `site-packages` окружения.

**Q3. Что делает активация окружения?**
A: Меняет `PATH` так, чтобы `python`/`pip` указывали на окружение, выставляет `VIRTUAL_ENV` и меняет приглашение. Она не «магия» — можно не активировать, а вызывать `.venv/bin/python` напрямую (надёжнее в CI/скриптах).

**Q4. Чем `venv` отличается от `virtualenv`?**
A: `venv` встроен в стандартную библиотеку (3.3+), не требует установки. `virtualenv` — сторонний, исторически предшественник, быстрее создаёт окружения, поддерживает старые версии Python и имеет больше возможностей. Для большинства современных задач `venv` достаточно.

**Q5. Чем `conda` отличается от `venv`?**
A: `conda` управляет не только Python-пакетами, но и не-Python зависимостями (C/Fortran библиотеки, CUDA) и самими версиями Python; использует собственный репозиторий, а не PyPI. `venv` работает только с Python-пакетами через `pip`. `conda` популярен в data science/ML.

**Q6. Что даёт `poetry` сверх `venv`?**
A: Декларативное управление зависимостями в `pyproject.toml`, lock-файл (`poetry.lock`) с полным разрешённым деревом для воспроизводимости, разрешение конфликтов версий, сборку и публикацию пакетов. Виртуальное окружение создаётся автоматически.

**Q7. Можно ли скопировать каталог `.venv` на другую машину?**
A: Нет, надёжно — нельзя. Окружение содержит абсолютные пути и привязано к конкретному интерпретатору/ОС. Правильно — пересоздать окружение и `pip install -r requirements.txt` (или из lock-файла).

**Q8. В чём разница между `requirements.txt` и lock-файлом?**
A: `requirements.txt` (часто из `pip freeze`) — список версий, но обычно без хешей и без явного дерева транзитивных зависимостей. Lock-файл (`poetry.lock`, `Pipfile.lock`, выход `pip-compile`) фиксирует всё дерево с точными версиями и хешами — строгая воспроизводимость.

**Q9. Где должен лежать `.venv` и нужно ли коммитить его в git?**
A: Обычно в корне проекта (`.venv/`), и его НЕ коммитят — добавляют в `.gitignore`. В репозиторий кладут `requirements.txt`/`pyproject.toml`/lock-файл, из которых окружение воссоздаётся.

**Q10. Что такое `include-system-site-packages` и флаг `--system-site-packages`?**
A: По умолчанию окружение изолировано и не видит системные `site-packages`. Флаг `--system-site-packages` (и соответствующая опция в `pyvenv.cfg`) разрешает окружению видеть системные пакеты — иногда удобно, но снижает изоляцию.

**Q11. Как создать окружение с конкретной версией Python?**
A: Вызвать `venv` нужным интерпретатором: `python3.11 -m venv .venv`. `venv` сам не устанавливает версии Python — нужный интерпретатор должен быть уже установлен (например, через `pyenv`, системный пакетный менеджер; либо использовать `conda`/`uv`, которые умеют ставить Python).

**Q12. Почему лучше `python -m pip` вместо просто `pip`?**
A: `python -m pip` гарантирует, что pip запускается именно для нужного интерпретатора (того, что в окружении). Команда `pip` может указывать на другой Python из `PATH`, что приводит к установке пакетов «не туда».

## Подводные камни (gotchas)

- **Копирование `.venv` не работает** — абсолютные пути и привязка к интерпретатору. Пересоздавайте из requirements/lock.
- **Забыли активировать** — пакеты ставятся в системный Python. Проверяйте `which python` / `pip -V` или используйте `.venv/bin/python -m pip`.
- **`pip freeze` фиксирует всё, включая транзитивные пакеты** — итоговый список «грязный». Для чистоты ведите прямые зависимости отдельно (pip-tools/poetry).
- **`venv` не управляет версией Python** — он лишь использует тот интерпретатор, которым создан. Для нескольких версий нужны `pyenv`/`uv`/`conda`.
- **Глобальная установка пакетов** требует root и ломает системные инструменты ОС — почти всегда нежелательна.
- **Активация только в текущей сессии shell** — в новом терминале нужно активировать заново.
- **`--system-site-packages` ослабляет изоляцию** — системные пакеты «протекают» в окружение.
- **Коммит `.venv` в git** раздувает репозиторий и не переносим — добавляйте в `.gitignore`.
- **Конфликт версий из-за отсутствия lock-файла** — на CI и у коллег могут установиться другие версии транзитивных зависимостей.

## Лучшие практики

- Одно окружение на проект; храните как `.venv/` в корне и добавляйте в `.gitignore`.
- Версионируйте зависимости в репозитории: `requirements.txt` (+ pip-tools для пинов/хешей) или `pyproject.toml` + lock (poetry/uv).
- Для строгой воспроизводимости используйте lock-файлы и `--require-hashes`.
- В CI и скриптах вызывайте интерпретатор окружения напрямую (`.venv/bin/python`), не полагаясь на активацию.
- Используйте `python -m pip` вместо `pip`, чтобы попасть в нужный интерпретатор.
- Обновляйте `pip` в свежем окружении (`--upgrade-deps` или `python -m pip install -U pip`).
- Разделяйте прямые и dev-зависимости (например, `requirements.txt` и `requirements-dev.txt` или группы в poetry).
- Для нескольких версий Python используйте `pyenv`/`uv`/`conda` совместно с окружениями.

## Шпаргалка

```bash
# Создание
python -m venv .venv
python3.11 -m venv .venv            # конкретной версией интерпретатора
python -m venv --system-site-packages .venv   # видеть системные пакеты
python -m venv --prompt api .venv   # своё приглашение
python -m venv --clear .venv        # пересоздать

# Активация / деактивация
source .venv/bin/activate           # Linux/macOS
.venv\Scripts\Activate.ps1          # Windows PowerShell
deactivate

# Без активации (надёжно для CI)
.venv/bin/python -m pip install -r requirements.txt
.venv/bin/python app.py

# pip
python -m pip install -U pip
pip install requests "django>=4,<5"
pip freeze > requirements.txt
pip install -r requirements.txt
pip list ; pip show requests

# Проверки
which python ; pip -V ; echo $VIRTUAL_ENV

# Программно
python -c "import venv; venv.EnvBuilder(with_pip=True).create('.venv')"
```
