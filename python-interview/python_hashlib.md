# hashlib — подготовка к собеседованию

## Что это и зачем

`hashlib` — модуль стандартной библиотеки для криптографических хеш-функций. Хеш-функция отображает данные произвольной длины в строку фиксированной длины (дайджест) так, что:

- **детерминированность** — одинаковый вход → одинаковый хеш;
- **необратимость (preimage resistance)** — по хешу нельзя восстановить вход;
- **устойчивость к коллизиям** — трудно найти два разных входа с одинаковым хешем;
- **лавинный эффект** — изменение одного бита входа меняет ~половину битов хеша.

Зачем нужно:
- проверка целостности файлов (контрольные суммы, чексуммы загрузок);
- дедупликация, кеш-ключи, content-addressable storage (git хранит объекты по SHA-1/SHA-256);
- хеширование паролей (через специальные KDF: `pbkdf2_hmac`, `scrypt`);
- цифровые подписи, HMAC (через модуль `hmac`).

Важно: обычный хеш ≠ хеширование паролей. Для паролей нужны медленные функции с солью (см. ниже).

## Ключевые концепции

- **Алгоритмы:** MD5 (128 бит), SHA-1 (160), семейство SHA-2 (SHA-224/256/384/512), SHA-3, BLAKE2 (blake2b/blake2s). `hashlib.algorithms_guaranteed` — гарантированно доступные.
- **Дайджест:** `digest()` → `bytes`, `hexdigest()` → шестнадцатеричная строка.
- **Инкрементальность:** `update()` можно вызывать многократно — хеш считается по конкатенации. Удобно для больших файлов/потоков.
- **MD5 и SHA-1 криптографически сломаны** — найдены практические коллизии (SHA-1: SHAttered, 2017; MD5 — давно). Нельзя использовать там, где важна устойчивость к коллизиям/подделке.
- **Соль (salt)** — случайные данные, добавляемые к паролю перед хешированием, чтобы одинаковые пароли давали разные хеши и нельзя было использовать радужные таблицы.
- **KDF (Key Derivation Function)** — намеренно медленные функции для паролей: `pbkdf2_hmac`, `scrypt`. Затрудняют перебор.
- **BLAKE2** — быстрый, безопасный, поддерживает ключ (keyed hashing → MAC), соль, персонализацию, настраиваемую длину.
- **`usedforsecurity=False`** (3.9+) — флаг, разрешающий MD5/SHA-1 для несекьюрных целей (контрольные суммы) даже в FIPS-режиме.

## Основные функции/классы/методы

### Базовое использование

```python
import hashlib

# Способ 1: фабрика по имени
h = hashlib.new("sha256")
h.update(b"hello ")
h.update(b"world")           # инкрементально; эквивалентно update(b"hello world")
print(h.hexdigest())         # b94d27b9934d3e08...
print(h.digest())            # сырые байты
print(h.digest_size)         # 32 (байт)
print(h.block_size)          # 64 (внутренний размер блока)
print(h.name)                # 'sha256'

# Способ 2: прямой конструктор (быстрее, читабельнее)
hashlib.sha256(b"data").hexdigest()
hashlib.sha512(b"data").hexdigest()
hashlib.md5(b"data").hexdigest()
hashlib.sha1(b"data").hexdigest()
```

### Хеширование строк (не забыть про кодировку!)

```python
import hashlib

text = "пароль123"
# str НЕЛЬЗЯ хешировать напрямую -> нужно кодировать в bytes
digest = hashlib.sha256(text.encode("utf-8")).hexdigest()
```

### Хеширование большого файла потоково

```python
import hashlib

def file_sha256(path, chunk_size=1024 * 1024):
    h = hashlib.sha256()
    with open(path, "rb") as f:
        for chunk in iter(lambda: f.read(chunk_size), b""):
            h.update(chunk)
    return h.hexdigest()

# Python 3.11+: удобный хелпер, читает файл оптимально
with open("big.iso", "rb") as f:
    digest = hashlib.file_digest(f, "sha256").hexdigest()
```

### BLAKE2 — быстрый, с ключом, солью, кастомной длиной

```python
import hashlib

# Обычный хеш
hashlib.blake2b(b"data").hexdigest()

# Keyed hashing = MAC (альтернатива HMAC, встроено)
mac = hashlib.blake2b(b"message", key=b"secret-key", digest_size=32).hexdigest()

# Настраиваемая длина дайджеста
hashlib.blake2s(b"data", digest_size=16).hexdigest()  # 16 байт

# Соль и персонализация (домен-сепарация)
hashlib.blake2b(b"data", salt=b"01234567", person=b"app-v1").hexdigest()
```

### SHA-3 и shake (расширяемый вывод)

```python
import hashlib

hashlib.sha3_256(b"data").hexdigest()

# SHAKE — eXtendable Output Function: длину задаёт сам вызов
shake = hashlib.shake_128(b"data")
shake.hexdigest(32)    # 32 байта вывода (длину указываем при чтении)
```

### Хеширование паролей: PBKDF2

```python
import hashlib
import os

def hash_password_pbkdf2(password: str, *, iterations: int = 600_000):
    salt = os.urandom(16)                       # уникальная случайная соль на каждый пароль
    dk = hashlib.pbkdf2_hmac(
        "sha256",                               # внутренняя хеш-функция
        password.encode("utf-8"),
        salt,
        iterations,                             # чем больше, тем медленнее перебор
        dklen=32,                               # длина ключа (по умолчанию = размер хеша)
    )
    # Хранить нужно: алгоритм, число итераций, соль и сам ключ
    return f"pbkdf2_sha256${iterations}${salt.hex()}${dk.hex()}"

def verify_pbkdf2(password: str, stored: str) -> bool:
    import hmac
    _, iters, salt_hex, dk_hex = stored.split("$")
    dk = hashlib.pbkdf2_hmac(
        "sha256", password.encode(), bytes.fromhex(salt_hex), int(iters), 32
    )
    # Сравнение в постоянном времени (защита от timing-атак)
    return hmac.compare_digest(dk, bytes.fromhex(dk_hex))
```

### Хеширование паролей: scrypt (memory-hard)

```python
import hashlib
import os

def hash_password_scrypt(password: str):
    salt = os.urandom(16)
    dk = hashlib.scrypt(
        password.encode("utf-8"),
        salt=salt,
        n=2**15,        # CPU/память cost (степень двойки), 16384..32768
        r=8,            # размер блока
        p=1,            # параллелизм
        dklen=32,
    )
    return salt.hex() + "$" + dk.hex()
```

## Частые вопросы на собеседовании

**Q1. Чем `digest()` отличается от `hexdigest()`?**
`digest()` возвращает сырые байты (бинарный дайджест), `hexdigest()` — ту же информацию в виде hex-строки (вдвое длиннее, удобна для логов/хранения текстом).

**Q2. Почему MD5 и SHA-1 небезопасны?**
Для них найдены практические коллизии: можно создать два разных файла с одинаковым хешем (MD5 — тривиально, SHA-1 — атака SHAttered 2017). Это ломает сценарии подписи/проверки подлинности. Для целостности от случайных ошибок MD5 ещё годится, но не для защиты от злоумышленника.

**Q3. Можно ли хешировать пароль через `sha256`?**
Нет. SHA-256 слишком быстрый → миллиарды попыток в секунду на GPU. Пароли нужно хешировать медленными KDF с солью: `pbkdf2_hmac`, `scrypt` (в stdlib) или Argon2 / bcrypt (внешние библиотеки).

**Q4. Зачем нужна соль и почему она должна быть уникальной?**
Соль — случайные данные на каждый пароль. Она делает одинаковые пароли разными хешами, ломает радужные таблицы и не даёт атаковать всех пользователей одним перебором. Соль не секретна — хранится рядом с хешем, но обязана быть уникальной и случайной (`os.urandom`/`secrets`).

**Q5. Чем PBKDF2 отличается от scrypt и Argon2?**
PBKDF2 — итеративный, только CPU-затратный, легко ускоряется на GPU/ASIC. scrypt — memory-hard (требует много RAM), устойчивее к GPU. Argon2 (победитель PHC, не в stdlib) — современный стандарт, настраиваемая память/время/параллелизм. Рекомендация OWASP: Argon2id, иначе scrypt, иначе PBKDF2 с большим числом итераций.

**Q6. Почему пароли надо сравнивать через `hmac.compare_digest`, а не `==`?**
Оператор `==` для строк/байтов завершается на первом несовпавшем байте → время сравнения зависит от данных, что открывает timing-атаку. `hmac.compare_digest` сравнивает за постоянное время.

**Q7. Что такое keyed hashing у BLAKE2 и зачем он?**
BLAKE2 умеет принимать `key` → получается MAC (код аутентификации сообщения) без отдельного HMAC-конструкта. Быстрее, чем HMAC поверх SHA-2, и безопасно. Удобно для проверки целостности с секретом.

**Q8. Что такое лавинный эффект и устойчивость к коллизиям?**
Лавинный эффект: изменение одного бита входа меняет примерно половину битов выхода. Устойчивость к коллизиям: вычислительно невозможно найти два разных входа с одинаковым хешем. Нарушение коллизионной стойкости (как у MD5/SHA-1) делает хеш непригодным для подписей.

**Q9. Как посчитать хеш файла на десятки гигабайт?**
Потоково через `update()` чанками (не читать файл целиком в память), либо `hashlib.file_digest(f, "sha256")` (3.11+). Хеш конкатенации = последовательные `update()`.

**Q10. Что делает флаг `usedforsecurity=False`?**
С Python 3.9 (и в FIPS-сборках) позволяет использовать «слабые» MD5/SHA-1 для несекьюрных задач (например, контрольная сумма кэша), явно декларируя, что это не для безопасности. В FIPS-режиме без флага вызов MD5 может бросить исключение.

**Q11. SHA-512 безопаснее SHA-256? Что выбрать?**
SHA-512 длиннее и на 64-битных системах часто быстрее SHA-256, но обе считаются стойкими. Для большинства задач SHA-256 достаточно. SHA-512/256 (truncated) совмещает скорость SHA-512 с коротким выводом.

**Q12. Чем SHA-3 отличается от SHA-2?**
SHA-2 основан на конструкции Меркла–Дамгора, SHA-3 — на губчатой (sponge) конструкции Keccak. SHA-3 не уязвим к length-extension атаке (в отличие от «голого» SHA-256). BLAKE2 — быстрая современная альтернатива обоим.

## Подводные камни (gotchas)

- **`TypeError: Strings must be encoded`** — нельзя хешировать `str`, только `bytes`. Всегда `.encode("utf-8")`.
- **Использование `sha256` для паролей** — критическая ошибка безопасности. Нужны KDF с солью.
- **Одинаковая или отсутствующая соль** — ломает весь смысл соли; генерируйте уникальную через `os.urandom`/`secrets.token_bytes`.
- **Сравнение хешей через `==`** — timing-атака. Используйте `hmac.compare_digest`.
- **MD5/SHA-1 для подписей и аутентификации** — небезопасно (коллизии).
- **Length-extension атака** на «голый» MD5/SHA-1/SHA-256 — если делаете MAC как `hash(secret + msg)`, злоумышленник может дописать данные. Используйте HMAC или BLAKE2-keyed.
- **Слишком мало итераций PBKDF2** — со временем оборудование ускоряется; держите счётчик высоким (OWASP: 600k+ для SHA-256) и периодически повышайте.
- **Хранение только хеша без параметров** — нужно хранить алгоритм, соль, число итераций/cost, иначе пароль не проверить и не мигрировать.
- **`hexdigest()` нечувствителен к регистру при сравнении вручную** — сравнивайте байты (`digest()`), а не строки разных регистров.
- **scrypt требует много памяти** — большой `n` может уронить процесс при множестве параллельных логинов; балансируйте `n/r/p`.

## Лучшие практики

- Для целостности данных используйте SHA-256 / SHA-512 / BLAKE2, не MD5/SHA-1.
- Для паролей — Argon2id (внешняя библ.) или, из stdlib, `scrypt`; PBKDF2 — приемлемый минимум с большим числом итераций.
- Всегда уникальная случайная соль на каждый пароль (`secrets.token_bytes(16)`).
- Храните «алгоритм$параметры$соль$хеш», чтобы можно было проверять и мигрировать.
- Сравнивайте дайджесты только через `hmac.compare_digest`.
- Для MAC берите `hmac` или keyed-BLAKE2, не самодельное `hash(secret + msg)`.
- Большие файлы — потоково (`file_digest`/чанки `update`).
- MD5/SHA-1 только для неприоритетных контрольных сумм с `usedforsecurity=False`.
- Регулярно пересматривайте cost-параметры KDF по мере роста мощности железа.

## Шпаргалка

```python
import hashlib, hmac, os

# --- Базовые хеши ---
hashlib.sha256(b"data").hexdigest()
hashlib.sha512(b"data").digest()           # сырые байты
hashlib.new("sha3_256", b"data").hexdigest()

# --- Инкрементально / потоково ---
h = hashlib.sha256()
for chunk in stream:
    h.update(chunk)
h.hexdigest()
# Python 3.11+:
hashlib.file_digest(open("f", "rb"), "sha256").hexdigest()

# --- BLAKE2: быстрый, с ключом (MAC), кастомная длина ---
hashlib.blake2b(b"msg", key=b"secret", digest_size=32).hexdigest()
hashlib.blake2s(b"data", digest_size=16).hexdigest()

# --- Пароли: PBKDF2 ---
salt = os.urandom(16)
dk = hashlib.pbkdf2_hmac("sha256", b"password", salt, 600_000, dklen=32)

# --- Пароли: scrypt (memory-hard) ---
dk = hashlib.scrypt(b"password", salt=salt, n=2**15, r=8, p=1, dklen=32)

# --- Безопасное сравнение (анти-timing) ---
hmac.compare_digest(dk_a, dk_b)

# --- Несекьюрный MD5 (контрольная сумма) ---
hashlib.md5(b"data", usedforsecurity=False).hexdigest()

# Память: str -> .encode("utf-8") перед хешированием!
# НЕ для паролей: md5, sha1, sha256 (быстрые) — только KDF с солью.
# Сломаны для подписей: MD5, SHA-1 (коллизии).
```
