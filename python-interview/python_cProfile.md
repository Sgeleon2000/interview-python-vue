# cProfile / profile — подготовка к собеседованию

## Что это и зачем

`cProfile` и `profile` — встроенные детерминированные профилировщики, измеряющие, *сколько времени* и *сколько вызовов* приходится на каждую функцию программы. Они отвечают на вопрос «где код тратит время», помогая найти узкие места (bottlenecks) перед оптимизацией.

- **`cProfile`** — реализован на C, низкие накладные расходы, рекомендуемый профилировщик для большинства задач.
- **`profile`** — чистый Python, медленнее (бóльшие накладные расходы искажают результат), но расширяемый; используется, когда нужно кастомизировать сам профилировщик.

Зачем нужен:
- **Не угадывать.** Интуиция о «медленном месте» часто ошибочна; профиль показывает факты.
- **Приоритизация.** Оптимизировать имеет смысл функции с наибольшим вкладом, а не любые.
- **Контроль регрессий.** Сравнение профилей до/после изменений.

Это *детерминированный* профайлер (инструментирует каждый вызов/возврат), в отличие от *семплирующих* (py-spy, statprof), которые периодически снимают стек с меньшим оверхедом.

## Ключевые концепции

- **ncalls** — число вызовов функции.
- **tottime (total time)** — время, потраченное *внутри самой функции*, БЕЗ времени вложенных вызовов. Главная метрика для поиска «горячего» кода.
- **cumtime (cumulative time)** — *совокупное* время функции ВМЕСТЕ со всеми вызванными ею функциями. Показывает «дорогие» ветки сверху.
- **percall** — время на один вызов (tottime/ncalls и cumtime/ncalls).
- **Stats / pstats** — объект для сортировки, фильтрации и печати собранной статистики.
- **Детерминированность.** Профайлер перехватывает каждый вызов функции, поэтому замедляет программу (особенно `profile`), но точно считает вызовы.

## Основные функции/классы/методы

### Запуск из командной строки

```bash
# Профилировать весь скрипт, вывод в stdout
python -m cProfile script.py arg1

# Отсортировать вывод по совокупному времени
python -m cProfile -s cumtime script.py
python -m cProfile -s tottime script.py     # по собственному времени

# Сохранить бинарный профиль в файл для последующего анализа
python -m cProfile -o profile.out script.py
```

Колонки вывода:
```
   ncalls  tottime  percall  cumtime  percall filename:lineno(function)
```

### Профилирование куска кода: run / runcall

```python
import cProfile


def heavy():
    total = 0
    for i in range(1_000_000):
        total += i * i
    return total


# Профилировать выполнение строки кода:
cProfile.run("heavy()")

# Профилировать и сохранить в файл:
cProfile.run("heavy()", filename="profile.out")

# Профилировать вызов функции с аргументами (без строки):
profiler = cProfile.Profile()
result = profiler.runcall(heavy)     # выполнит heavy() под профилем
profiler.print_stats(sort="tottime")
```

### Контекстное профилирование: enable / disable

Когда нужно профилировать только *часть* программы:

```python
import cProfile

profiler = cProfile.Profile()

setup_data()                 # это НЕ профилируем
profiler.enable()            # начало измерения
process_data()               # профилируем только это
profiler.disable()           # конец измерения

profiler.print_stats(sort="cumtime")
```

С Python 3.8+ `Profile` можно использовать как контекстный менеджер:

```python
import cProfile

with cProfile.Profile() as pr:
    process_data()
pr.print_stats(sort="tottime")
```

### Анализ через pstats.Stats

`pstats` превращает сырой профиль в удобный для анализа объект: сортировка, ограничение вывода, фильтрация по имени, граф вызовов.

```python
import pstats
from pstats import SortKey

# Загрузить сохранённый профиль
stats = pstats.Stats("profile.out")

# Убрать длинные пути к файлам для читаемости
stats.strip_dirs()

# Сортировка и печать топ-10 по собственному времени
stats.sort_stats(SortKey.TIME).print_stats(10)

# По совокупному времени, только функции, чьё имя содержит "process"
stats.sort_stats(SortKey.CUMULATIVE).print_stats("process")

# Топ-5 строк по cumtime
stats.sort_stats(SortKey.CUMULATIVE).print_stats(5)
```

Ключи сортировки (`SortKey` или строки):
```
SortKey.TIME / "tottime"        — собственное время функции
SortKey.CUMULATIVE / "cumtime"  — совокупное время
SortKey.CALLS / "ncalls"        — число вызовов
SortKey.NAME / "name"           — имя функции
SortKey.PCALLS / "pcalls"       — число примитивных (нерекурсивных) вызовов
```

### Кто кого вызывает: callers / callees

```python
import pstats

stats = pstats.Stats("profile.out").strip_dirs()

# Кто ВЫЗЫВАЕТ функции, имя которых содержит "parse"
stats.print_callers("parse")

# Кого ВЫЗЫВАЕТ функция "parse" (её "потомки")
stats.print_callees("parse")
```

Это критично для понимания, *почему* функция дорогая: либо она медленная сама (высокий tottime), либо её часто зовут (смотрим callers).

### Объединение профилей

```python
import pstats

# Сложить статистику из нескольких прогонов
combined = pstats.Stats("run1.out")
combined.add("run2.out")
combined.add("run3.out")
combined.sort_stats("cumtime").print_stats(10)
```

### Визуализация (внешние инструменты)

Сырой `.out` удобно смотреть в графических инструментах:

```bash
# snakeviz — интерактивная веб-визуализация (sunburst / icicle)
pip install snakeviz
snakeviz profile.out

# gprof2dot + graphviz — граф вызовов в виде картинки
pip install gprof2dot
python -m cProfile -o profile.out script.py
gprof2dot -f pstats profile.out | dot -Tpng -o profile.png
```

## Частые вопросы на собеседовании

**Q1. В чём разница между cProfile и profile?**
`cProfile` написан на C, имеет низкие накладные расходы и рекомендуется по умолчанию. `profile` — на чистом Python, заметно медленнее (сильнее искажает измеряемое время), но его можно расширять/кастомизировать. Интерфейс у них совместим.

**Q2. Чем tottime отличается от cumtime?**
`tottime` — время *внутри самой функции*, без вложенных вызовов; показывает, где «горит» собственный код. `cumtime` — совокупное время функции вместе со всем, что она вызвала; показывает дорогие ветки/точки входа. Для микрооптимизации смотрят tottime, для архитектурных узких мест — cumtime.

**Q3. Как понять, оптимизировать саму функцию или уменьшать число её вызовов?**
Смотреть `ncalls` и `tottime/percall`. Если функция вызвана миллион раз с маленьким percall — проблема в *количестве* вызовов (надо звать реже/кешировать). Если вызвана редко, но percall большой — медленна сама реализация. `print_callers` подскажет, откуда идут вызовы.

**Q4. Что такое детерминированное профилирование и каковы его минусы?**
Профайлер инструментирует *каждый* вызов и возврат функции, точно считая вызовы и время. Минус — накладные расходы: программа работает медленнее, и очень частые мелкие функции искажаются (overhead сопоставим с их временем). Альтернатива — семплирующие профайлеры (py-spy), снимающие стек периодически с малым оверхедом.

**Q5. Как профилировать только часть программы?**
Через `Profile().enable()`/`disable()` вокруг нужного участка, или `with cProfile.Profile() as pr:` (3.8+), или `profiler.runcall(func, *args)`. Это исключает из профиля разовую инициализацию и фокусирует измерение.

**Q6. Как анализировать сохранённый профиль?**
Загрузить через `pstats.Stats("file.out")`, вызвать `strip_dirs()`, отсортировать `sort_stats(SortKey.CUMULATIVE)` и напечатать топ `print_stats(N)`. Для понимания связей — `print_callers`/`print_callees`. Для наглядности — snakeviz/gprof2dot.

**Q7. Почему ncalls иногда выглядит как "3/1"?**
Это рекурсия: число вида `total/primitive` означает 3 *всего* вызовов и 1 *примитивный* (нерекурсивный, верхнеуровневый). Так профайлер разделяет общие и непрямые рекурсивные вызовы.

**Q8. cProfile показывает CPU-время или wall-clock?**
По умолчанию используется таймер на основе `perf_counter` (wall-clock). Можно задать свой таймер при создании `Profile(timer=...)`, например, для измерения только CPU-времени. Важно понимать: I/O-ожидание попадёт в cumtime как «время», хотя CPU не работал.

**Q9. cProfile медленный/не подходит для прода — что использовать?**
В продакшене предпочитают семплирующие профайлеры с малым оверхедом: `py-spy` (работает без правки кода, по PID), `scalene`, `austin`. cProfile хорош для локального анализа и тестов, но в нагруженном проде его инструментирование слишком дорого.

**Q10. cProfile не видит, что время уходит в C-расширении или I/O. Почему?**
Детерминированный профайлер измеряет на уровне Python-вызовов. Время внутри одного C-вызова (numpy, БД-драйвер) учитывается как cumtime этого вызова, но не разбивается дальше. Для C-кода нужны нативные профайлеры; для I/O — анализ cumtime и трассировка/логирование запросов.

**Q11. Как профилировать многопоточный/async код?**
cProfile по умолчанию профилирует поток, в котором запущен; другие потоки нужно профилировать отдельно (свой `Profile` на поток). Для asyncio cumtime включает время ожидания корутин, что может вводить в заблуждение. Часто удобнее py-spy/yappi (yappi умеет потоки и async).

**Q12. Какой первый шаг при оптимизации производительности?**
Сначала *измерить*, а не угадывать: запустить cProfile/семплер на реалистичной нагрузке, найти функции с наибольшим cumtime/tottime, и только их оптимизировать (или сократить число вызовов). Затем подтвердить улучшение повторным профилем и timeit на горячем участке.

## Подводные камни (gotchas)

- **Оверхед искажает результат.** Инструментирование замедляет программу; очень частые крошечные функции выглядят дороже, чем в реальности. Относительные сравнения надёжнее абсолютных; для точного времени используйте timeit на конкретном фрагменте.
- **tottime vs cumtime путают.** Сортировка по cumtime поднимет наверх `main`/точки входа (в них «всё» время). Для поиска именно медленного кода сортируйте по tottime.
- **I/O и сон попадают в cumtime.** Ожидание сети/диска/`sleep` учитывается как время, хотя CPU простаивал. Не принимайте I/O-bound за CPU-bound.
- **C-расширения непрозрачны.** Время внутри numpy/драйвера не разбивается по строкам — будет одна строка с большим cumtime. Нужны нативные профайлеры.
- **Профилирование нерепрезентативной нагрузки.** Профиль на игрушечных данных не отражает прод. Профилируйте на реалистичных объёмах и сценариях.
- **profile вместо cProfile.** Использование чистого `profile` без причины сильно замедляет и искажает. По умолчанию берите cProfile.
- **Многопоточность.** cProfile видит один поток; в многопоточном приложении часть работы «исчезнет» из профиля.
- **Преждевременная оптимизация.** Оптимизация функции, не являющейся узким местом, бесполезна. Всегда профилируйте до изменений и проверяйте эффект после.
- **strip_dirs до сортировки.** Порядок операций в pstats важен; вызывайте `strip_dirs()` перед печатью, но он может слить функции с одинаковым именем из разных файлов — учитывайте.

## Лучшие практики

- Сначала профилируйте, потом оптимизируйте — не доверяйте интуиции о «медленном месте».
- Используйте `cProfile` (не `profile`) и сохраняйте профиль в файл (`-o`) для воспроизводимого анализа в pstats.
- Анализируйте по обеим метрикам: cumtime — найти дорогие ветки, tottime — найти горячий код.
- Изучайте `print_callers`/`print_callees`, чтобы понять, оптимизировать функцию или сократить число её вызовов.
- Профилируйте на реалистичных данных и нагрузке; исключайте разовую инициализацию через `enable/disable`.
- Для прод-нагрузки берите семплирующие профайлеры (py-spy, scalene) с малым оверхедом.
- Подтверждайте улучшения повторным профилем; для точного времени горячего фрагмента используйте `timeit`.
- Визуализируйте сложные профили (snakeviz, gprof2dot) — граф вызовов нагляднее таблицы.

## Шпаргалка

```bash
# CLI
python -m cProfile script.py                  # профиль в stdout
python -m cProfile -s cumtime script.py       # сортировка по cumtime
python -m cProfile -s tottime script.py       # по собственному времени
python -m cProfile -o profile.out script.py   # сохранить в файл
```

```python
import cProfile, pstats
from pstats import SortKey

# Профилировать код/функцию
cProfile.run("func()", "profile.out")
cProfile.Profile().runcall(func, *args)

with cProfile.Profile() as pr:    # только часть программы (3.8+)
    work()
pr.print_stats(SortKey.TIME)

# Анализ
st = pstats.Stats("profile.out").strip_dirs()
st.sort_stats(SortKey.CUMULATIVE).print_stats(10)   # топ-10 по cumtime
st.sort_stats(SortKey.TIME).print_stats("regex")    # фильтр по имени
st.print_callers("func")     # кто вызывает
st.print_callees("func")     # кого вызывает
st.add("other.out")          # объединить профили
```

```
Колонки:
  ncalls   число вызовов (total/primitive при рекурсии)
  tottime  время в самой функции (БЕЗ вложенных)   <- горячий код
  cumtime  совокупное время (С вложенными)         <- дорогие ветки
  percall  на один вызов

Метрики выбора:
  много ncalls + малый percall  -> сократить число вызовов / кеш
  малый ncalls + большой percall -> ускорить саму функцию

Прод/малый оверхед: py-spy, scalene (семплирующие)
Точное время фрагмента: timeit
```
