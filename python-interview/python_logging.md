# logging — подготовка к собеседованию

## Что это и зачем

`logging` — это встроенный в стандартную библиотеку Python модуль для гибкого журналирования событий приложения. Он решает задачу, которую часто (и неправильно) пытаются закрыть через `print()`: фиксацию того, что происходит в программе, с возможностью управлять детализацией, форматом и местом назначения вывода без изменения кода.

Зачем нужен `logging`, а не `print()`:

- **Уровни важности.** Можно отделить отладочные сообщения от ошибок и в продакшене показывать только важное, а при отладке — всё.
- **Управление назначением.** Одно и то же сообщение можно отправить в консоль, файл, по почте, в syslog или в систему сбора логов — и менять это конфигурацией, а не правкой кода.
- **Структура и контекст.** Автоматически добавляются время, имя логгера, уровень, имя файла, номер строки, имя потока/процесса.
- **Производительность.** Сообщения ниже текущего уровня не форматируются (ленивое форматирование), а тяжёлые операции можно пропускать через `isEnabledFor`.
- **Централизованная конфигурация.** Поведение всего приложения и сторонних библиотек настраивается в одном месте (`dictConfig`/`fileConfig`).
- **Потокобезопасность.** Запись в один обработчик из разных потоков не приводит к перемешиванию строк.

Главный принцип: **библиотеки только логируют и не настраивают вывод; приложение настраивает вывод**. Это разделение — ключ к пониманию всей архитектуры.

```python
import logging

logger = logging.getLogger(__name__)  # рекомендуемый способ получить логгер

logger.debug("Детальная отладочная информация: x=%s", 42)
logger.info("Обычное событие: пользователь вошёл")
logger.warning("Что-то подозрительное, но не критичное")
logger.error("Ошибка, операция не выполнена")
logger.critical("Критический сбой, приложение может упасть")
```

## Ключевые концепции

### Уровни логирования

Каждое сообщение имеет уровень важности. Уровень — это просто целое число; именованные константы привязаны к конкретным значениям:

| Уровень    | Числовое значение | Когда использовать                                        |
|------------|-------------------|-----------------------------------------------------------|
| `NOTSET`   | 0                 | «не задан» — означает наследование уровня от родителя      |
| `DEBUG`    | 10                | подробная диагностика, интересная при отладке             |
| `INFO`     | 20                | подтверждение, что всё работает как ожидается             |
| `WARNING`  | 30                | произошло что-то неожиданное, но программа работает       |
| `ERROR`    | 40                | из-за проблемы не удалось выполнить функцию               |
| `CRITICAL` | 50                | серьёзная ошибка, программа может не продолжить работу    |

Сообщение проходит, если его уровень **больше или равен** эффективному уровню логгера (а затем — уровню каждого хендлера).

```python
import logging

# Сравнение уровня сообщения с уровнем логгера/хендлера — это просто >=
print(logging.DEBUG, logging.INFO, logging.WARNING, logging.ERROR, logging.CRITICAL)
# 10 20 30 40 50

# Превратить имя в число и наоборот
print(logging.getLevelName("INFO"))   # 20
print(logging.getLevelName(20))       # 'INFO'

# Уровень по умолчанию у root-логгера — WARNING (30).
# Поэтому "из коробки" видны WARNING/ERROR/CRITICAL, но не INFO/DEBUG.
```

Важная деталь: уровень `WARNING` по умолчанию у **root**-логгера — частая ловушка для новичков, которые «не видят» свои `info`-сообщения.

Можно регистрировать собственные уровни через `logging.addLevelName(15, "TRACE")`, но это редко нужно и считается дурным тоном без веской причины.

### Архитектура: Logger / Handler / Formatter / Filter и LogRecord

Это самая важная для собеседования тема. Запоминаем путь сообщения:

1. Вы вызываете `logger.info("msg", arg)`.
2. Логгер проверяет: `info` (20) ≥ эффективного уровня логгера? Если нет — выход немедленно (ничего не форматируется).
3. Создаётся объект **`LogRecord`** — он содержит всё: текст-шаблон, аргументы, уровень, имя логгера, время, имя файла, номер строки, имя потока/процесса, информацию об исключении.
4. К записи применяются **фильтры логгера**. Если хоть один вернул «нет» — запись отбрасывается.
5. Запись передаётся **хендлерам** этого логгера, а затем (если `propagate=True`) — хендлерам всех родительских логгеров вверх по иерархии.
6. Каждый **хендлер** проверяет свой собственный уровень и свои фильтры.
7. Хендлер вызывает свой **форматтер**, который превращает `LogRecord` в строку.
8. Хендлер пишет строку в назначение (консоль, файл, сеть...).

```
logger.info(...) ──► [уровень логгера?] ──► LogRecord ──► [фильтры логгера?]
        │
        ├──► хендлеры логгера ──► [уровень хендлера? фильтры?] ──► Formatter ──► вывод
        │
        └──► (propagate) родительский логгер ──► его хендлеры ──► ... ──► root
```

Роли компонентов:

- **Logger** — точка входа в API. Принимает решение «логировать или нет» по уровню, владеет фильтрами и списком хендлеров, организован в иерархию.
- **Handler** — отправляет запись в конкретное назначение. У него свой уровень и свои фильтры.
- **Formatter** — описывает, как `LogRecord` превращается в строку.
- **Filter** — даёт тонкий контроль: пропустить/отбросить запись или даже изменить её.
- **LogRecord** — структура данных, представляющая одно событие.

Ключевой момент: **уровень проверяется дважды** — на логгере (общий «кран») и на каждом хендлере (можно разные назначения с разной детализацией). Логгер решает, родится ли запись вообще; хендлер решает, попадёт ли она в конкретный пункт назначения.

### getLogger и синглтоны по имени

`logging.getLogger(name)` возвращает **один и тот же** объект логгера для одного и того же имени в пределах процесса. Это менеджер на основе словаря: при первом обращении логгер создаётся, при последующих — возвращается существующий.

```python
import logging

a = logging.getLogger("myapp.db")
b = logging.getLogger("myapp.db")
print(a is b)  # True — это один и тот же объект (синглтон по имени)

# getLogger() без имени или с "root" вернёт корневой логгер
root = logging.getLogger()
```

Почему так сделано: благодаря синглтонам по имени любой модуль в любом месте кода может «достать» свой логгер и настроить его, а конфигурация в одном месте влияет на все обращения. Поэтому стандартный приём — `logger = logging.getLogger(__name__)` на уровне модуля: имя автоматически совпадёт с путём модуля (`myapp.services.user`), а это бесплатно встроит логгер в иерархию пакета.

### basicConfig и root logger

`logging.basicConfig(...)` — быстрый способ настроить **root**-логгер: создаёт `StreamHandler` (по умолчанию в `stderr`) с базовым форматтером и задаёт уровень.

```python
import logging

logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s %(levelname)s %(name)s %(message)s",
    datefmt="%Y-%m-%d %H:%M:%S",
)
logging.getLogger(__name__).info("Теперь INFO виден")
```

Важные особенности:

- `basicConfig` **по умолчанию ничего не делает, если у root уже есть хендлеры**. То есть повторный вызов — no-op. Это сделано специально, чтобы случайные двойные вызовы не плодили хендлеры. (С Python 3.8 есть `force=True`, чтобы сбросить существующие хендлеры и переконфигурировать.)
- Раньше была классическая ловушка: если до `basicConfig` вы успели вызвать, например, `logging.warning(...)`, то root уже получил хендлер по умолчанию, и `basicConfig` молча проигнорируется.
- Параметр `handlers=` позволяет передать готовый список хендлеров (нельзя одновременно с `filename`/`stream`).
- `basicConfig` уместен в маленьких скриптах. В серьёзном приложении используйте `dictConfig`.

### Иерархия логгеров и точечная нотация

Имена логгеров образуют дерево по точкам: `a` — родитель `a.b`, а `a.b` — родитель `a.b.c`. Корень дерева — root-логгер (имя пустое). Иерархия создаётся «лениво»: достаточно запросить `getLogger("a.b.c")`, и промежуточные узлы (placeholder'ы) появятся автоматически.

```python
import logging

parent = logging.getLogger("myapp")
child = logging.getLogger("myapp.db")
print(child.parent is parent)  # True
print(parent.parent is logging.getLogger())  # True — родитель верхнего узла это root
```

Зачем иерархия: можно настроить хендлеры/уровни в одной точке (`myapp`), и это распространится на всех потомков. Это и делает `__name__` удобным — структура пакетов автоматически даёт удобную иерархию логгеров.

### propagate (всплытие к родителю)

По умолчанию `logger.propagate = True`. Это значит, что после обработки записи хендлерами текущего логгера запись «всплывает» к родителю и обрабатывается **его** хендлерами, и так до root. При этом уровень родительского логгера при всплытии **не проверяется** — проверяются только уровни/фильтры самих хендлеров.

```python
import logging

logging.basicConfig(level=logging.INFO)  # хендлер на root

log = logging.getLogger("myapp.payments")
log.addHandler(logging.StreamHandler())  # свой хендлер
log.info("оплата")
# Сообщение будет выведено ДВАЖДЫ: своим хендлером и хендлером root (всплытие)

# Чтобы отключить всплытие к root:
log.propagate = False
log.info("теперь только один раз")
```

Типичные ошибки, связанные с `propagate`:

- **Дублирование строк** — у логгера есть и собственный хендлер, и наследуется хендлер root. Решение: либо `propagate=False`, либо не добавлять хендлер на дочерний логгер, а настраивать всё на одном уровне.
- Отключение `propagate=False` изолирует поддерево: его записи больше не попадут в централизованный root-хендлер. Иногда это именно то, что нужно (отдельный лог-файл для конкретной подсистемы).

### Эффективный уровень (effective level)

У логгера может быть уровень `NOTSET` (значение 0) — это значение по умолчанию для всех логгеров, кроме root (у root по умолчанию `WARNING`). Когда уровень равен `NOTSET`, логгер **не имеет собственного уровня и наследует его** от ближайшего предка, у которого уровень задан явно. Этот вычисленный уровень и называется эффективным.

```python
import logging

logging.getLogger("myapp").setLevel(logging.DEBUG)  # задали на родителе

child = logging.getLogger("myapp.db")  # сам по себе NOTSET
print(child.level)             # 0 (NOTSET) — собственный уровень не задан
print(child.getEffectiveLevel())  # 10 (DEBUG) — унаследован от 'myapp'
print(logging.getLevelName(child.getEffectiveLevel()))  # 'DEBUG'
```

Алгоритм: идём от логгера вверх по родителям, пока не найдём узел с уровнем ≠ `NOTSET`; его уровень и есть эффективный. Если ничего не нашли (теоретически), берётся `WARNING`.

## Основные функции/классы/методы

### Хендлеры

**`StreamHandler`** — пишет в поток (`sys.stderr` по умолчанию, можно `sys.stdout`).

```python
import logging, sys
h = logging.StreamHandler(stream=sys.stdout)
```

**`FileHandler`** — пишет в файл. Открывает файл (по умолчанию в режиме `'a'`, дозапись).

```python
h = logging.FileHandler("app.log", mode="a", encoding="utf-8", delay=True)
# delay=True — файл откроется только при первой записи
```

**`RotatingFileHandler`** — ротация по размеру файла.

```python
from logging.handlers import RotatingFileHandler
h = RotatingFileHandler(
    "app.log",
    maxBytes=10 * 1024 * 1024,  # 10 МБ — порог ротации
    backupCount=5,              # хранить 5 архивных файлов
    encoding="utf-8",
)
# При достижении maxBytes: app.log -> app.log.1, app.log.1 -> app.log.2 ...
# Самый старый (app.log.5) удаляется. backupCount=0 — без ротации, файл усекается.
```

**`TimedRotatingFileHandler`** — ротация по времени.

```python
from logging.handlers import TimedRotatingFileHandler
h = TimedRotatingFileHandler(
    "app.log",
    when="midnight",   # 'S','M','H','D','midnight','W0'..'W6'
    interval=1,        # каждые interval единиц when
    backupCount=7,     # хранить логи за 7 периодов (например 7 дней)
    encoding="utf-8",
    utc=False,
)
# Архивы получают суффикс с датой: app.log.2026-06-19
```

**`NullHandler`** — ничего не делает; нужен библиотекам, чтобы не было предупреждения «No handlers could be found» и чтобы не навязывать вывод приложению.

```python
import logging
logging.getLogger("mylib").addHandler(logging.NullHandler())
```

**`SMTPHandler`** — отправляет лог-записи по email (обычно для ERROR/CRITICAL).

```python
from logging.handlers import SMTPHandler
h = SMTPHandler(
    mailhost=("smtp.example.com", 587),
    fromaddr="alerts@example.com",
    toaddrs=["ops@example.com"],
    subject="Сбой в приложении",
    credentials=("user", "password"),
    secure=(),
)
h.setLevel(logging.ERROR)  # слать только серьёзное
```

Прочие полезные из `logging.handlers`: `SysLogHandler`, `SocketHandler`, `HTTPHandler`, `QueueHandler`/`QueueListener` (асинхронная запись через очередь — важно для высоконагруженных систем, чтобы не блокировать рабочие потоки на медленном I/O), `MemoryHandler`, `WatchedFileHandler` (учитывает внешний logrotate).

### Ротация логов подробно

Ротация нужна, чтобы лог-файлы не росли бесконечно. Два основных подхода:

- **По размеру** (`RotatingFileHandler`): когда текущий файл превышает `maxBytes`, он переименовывается в `.1`, существующие архивы сдвигаются (`.1`→`.2` и т.д.), создаётся новый пустой основной файл. Файлы старше `backupCount` удаляются. Это гарантирует ограничение по диску (примерно `maxBytes * (backupCount + 1)`).
- **По времени** (`TimedRotatingFileHandler`): ротация происходит по расписанию (`when`/`interval`). `backupCount` ограничивает число хранимых периодов. Удобно для «лог за сутки в отдельном файле».

Тонкости:

- При нескольких процессах, пишущих в один файл, встроенная ротация ненадёжна (гонки при переименовании). Варианты: писать каждому процессу в свой файл; использовать внешний `logrotate` ОС вместе с `WatchedFileHandler`; либо отправлять логи в централизованный сборщик (syslog/сетевой хендлер).
- `WatchedFileHandler` отслеживает, что внешний инструмент переименовал/удалил файл (по inode), и переоткрывает его — это правильный спутник системного `logrotate` на Unix.
- `backupCount=0` означает «не хранить архивы»: при ротации файл просто усекается.

### Formatter

Форматтер задаёт, как `LogRecord` превращается в строку. Используются атрибуты записи через `%(name)s`-синтаксис.

```python
import logging

fmt = logging.Formatter(
    fmt="%(asctime)s %(levelname)-8s %(name)s:%(lineno)d %(message)s",
    datefmt="%Y-%m-%d %H:%M:%S",
    style="%",  # стиль плейсхолдеров: '%', '{', '$'
)
```

Часто используемые атрибуты `LogRecord`:

| Плейсхолдер         | Что выводит                                        |
|---------------------|----------------------------------------------------|
| `%(asctime)s`       | время в человекочитаемом виде                       |
| `%(levelname)s`     | имя уровня (`INFO`, `ERROR`...)                      |
| `%(levelno)s`       | числовой уровень                                    |
| `%(name)s`          | имя логгера                                          |
| `%(message)s`       | результат `record.getMessage()` (текст + аргументы) |
| `%(pathname)s` / `%(filename)s` | путь/имя файла                          |
| `%(module)s`        | имя модуля                                           |
| `%(funcName)s`      | имя функции                                          |
| `%(lineno)d`        | номер строки                                         |
| `%(process)d` / `%(processName)s` | PID / имя процесса                    |
| `%(thread)d` / `%(threadName)s`   | id / имя потока                       |
| `%(exc_info)` (через `%(message)s`) | трассировка исключения             |

Три стиля форматирования (`style`):

```python
# '%' (по умолчанию, printf-стиль)
logging.Formatter("%(levelname)s:%(message)s", style="%")
# '{' (str.format)
logging.Formatter("{levelname}:{message}", style="{")
# '$' (string.Template)
logging.Formatter("${levelname}:${message}", style="$")
```

### Filter

Фильтр позволяет тонко решать, пропускать ли запись, и может **обогащать/менять** её. Фильтр — это либо callable, либо объект с методом `filter(record) -> bool` (или интоподобное). Возврат «ложного» значения отбрасывает запись.

```python
import logging

class OnlyMyModule(logging.Filter):
    def filter(self, record: logging.LogRecord) -> bool:
        # Пропускаем только записи логгера 'myapp.db'
        return record.name == "myapp.db"

class AddHostname(logging.Filter):
    def filter(self, record):
        record.hostname = "web-01"  # добавляем поле, которое можно использовать в формате
        return True  # запись не отбрасываем

handler = logging.StreamHandler()
handler.addFilter(OnlyMyModule())
handler.addFilter(AddHostname())
```

Фильтр на основе `logging.Filter("myapp.db")` пропускает записи только этого логгера и его потомков — частый приём.

### Конфигурация через dictConfig и fileConfig

`logging.config.dictConfig(config)` — современный, рекомендуемый способ. Описывает всю систему логирования декларативно.

```python
import logging.config

LOGGING = {
    "version": 1,
    "disable_existing_loggers": False,  # ВАЖНО: не глушить уже созданные логгеры
    "formatters": {
        "default": {
            "format": "%(asctime)s %(levelname)s %(name)s %(message)s",
            "datefmt": "%Y-%m-%d %H:%M:%S",
        },
        "json": {
            "()": "pythonjsonlogger.jsonlogger.JsonFormatter",  # кастомный класс через "()"
            "format": "%(asctime)s %(levelname)s %(name)s %(message)s",
        },
    },
    "filters": {
        "only_errors": {
            "()": "__main__.OnlyErrors",  # путь до класса фильтра
        },
    },
    "handlers": {
        "console": {
            "class": "logging.StreamHandler",
            "level": "DEBUG",
            "formatter": "default",
            "stream": "ext://sys.stdout",
        },
        "file": {
            "class": "logging.handlers.RotatingFileHandler",
            "level": "INFO",
            "formatter": "json",
            "filename": "app.log",
            "maxBytes": 10485760,
            "backupCount": 5,
            "encoding": "utf-8",
        },
    },
    "loggers": {
        "myapp": {                       # логгер приложения
            "level": "DEBUG",
            "handlers": ["console", "file"],
            "propagate": False,          # не дублировать в root
        },
        "sqlalchemy.engine": {           # приглушить шумную библиотеку
            "level": "WARNING",
            "handlers": ["console"],
            "propagate": False,
        },
    },
    "root": {                            # запасной логгер
        "level": "WARNING",
        "handlers": ["console"],
    },
}

logging.config.dictConfig(LOGGING)
logging.getLogger("myapp.db").info("Сконфигурировано через dictConfig")
```

Тот же конфиг в YAML (например для Django/сервиса), который загружают и передают в `dictConfig`:

```yaml
version: 1
disable_existing_loggers: false
formatters:
  default:
    format: "%(asctime)s %(levelname)s %(name)s %(message)s"
    datefmt: "%Y-%m-%d %H:%M:%S"
handlers:
  console:
    class: logging.StreamHandler
    level: DEBUG
    formatter: default
    stream: ext://sys.stdout
  file:
    class: logging.handlers.TimedRotatingFileHandler
    level: INFO
    formatter: default
    filename: app.log
    when: midnight
    backupCount: 7
    encoding: utf-8
loggers:
  myapp:
    level: DEBUG
    handlers: [console, file]
    propagate: false
root:
  level: WARNING
  handlers: [console]
```

```python
import logging.config, yaml

with open("logging.yaml", encoding="utf-8") as f:
    logging.config.dictConfig(yaml.safe_load(f))
```

Особенности `dictConfig`:

- `version` всегда `1` (зарезервировано на будущее).
- `disable_existing_loggers` — критичный параметр. При `True` (значение по умолчанию!) все логгеры, **созданные до** вызова `dictConfig` и не упомянутые в конфиге, отключаются. Это частая причина «логи пропали после настройки». В большинстве случаев ставьте `False`.
- Ключ `"()"` задаёт фабрику — произвольный класс/функцию для создания объекта (форматтера, фильтра, хендлера).
- `ext://` ссылается на внешние объекты (например `ext://sys.stdout`).

`logging.config.fileConfig("logging.ini")` — старый формат на основе INI-файлов (`configparser`). Менее гибкий, чем `dictConfig` (например, сложнее передавать произвольные аргументы), но всё ещё встречается в легаси.

```ini
[loggers]
keys=root,myapp

[handlers]
keys=console

[formatters]
keys=default

[logger_root]
level=WARNING
handlers=console

[logger_myapp]
level=DEBUG
handlers=console
qualname=myapp
propagate=0

[handler_console]
class=StreamHandler
level=DEBUG
formatter=default
args=(sys.stdout,)

[formatter_default]
format=%(asctime)s %(levelname)s %(name)s %(message)s
```

### Ленивое (отложенное) форматирование

Передавайте аргументы **отдельными параметрами**, а не подставляйте их в строку заранее:

```python
import logging
logger = logging.getLogger(__name__)

user_id = 42

# ПРАВИЛЬНО: %-плейсхолдеры + аргументы.
# Форматирование произойдёт ТОЛЬКО если запись реально будет выведена.
logger.debug("Загружаю пользователя %s из БД", user_id)

# НЕПРАВИЛЬНО: f-строка/конкатенация форматируются ВСЕГДА,
# даже если DEBUG отключён и сообщение никуда не пойдёт.
logger.debug(f"Загружаю пользователя {user_id} из БД")
```

Почему это важно:

- Если уровень `DEBUG` выключен, в первом варианте интерполяция строки не выполняется вовсе — экономия CPU. Во втором f-строка вычисляется всегда, до того как `logging` решит выбросить сообщение.
- Особенно заметно, когда аргумент — результат тяжёлого вычисления или `repr` большого объекта.

Если аргумент дорого вычислять, оборачивайте проверку:

```python
if logger.isEnabledFor(logging.DEBUG):
    logger.debug("Дамп: %s", expensive_serialize(obj))
```

### logging.exception и exc_info

Внутри обработчика исключения используйте `logger.exception(...)` — он логирует на уровне `ERROR` и **автоматически добавляет трассировку** текущего исключения.

```python
import logging
logger = logging.getLogger(__name__)

try:
    1 / 0
except ZeroDivisionError:
    logger.exception("Не удалось поделить")  # уровень ERROR + полный traceback

    # Эквивалентно:
    logger.error("Не удалось поделить", exc_info=True)

    # Можно добавить traceback к любому уровню:
    logger.warning("Странно, но продолжаем", exc_info=True)
```

`logger.exception` нужно вызывать **только из блока except**, иначе информации об исключении не будет. `exc_info=True` использует «текущее» исключение; можно передать конкретный кортеж `(type, value, tb)` или сам объект исключения.

### extra и LoggerAdapter

`extra` позволяет добавить в `LogRecord` дополнительные поля, которые затем можно использовать в форматтере.

```python
import logging

logging.basicConfig(format="%(asctime)s %(request_id)s %(message)s")
logger = logging.getLogger(__name__)

logger.info("Обработка запроса", extra={"request_id": "abc-123"})
# Если в формате есть %(request_id)s — поле подставится.
# ВНИМАНИЕ: если поля нет в extra при следующем вызове — будет KeyError в форматтере.
# Имена в extra не должны конфликтовать со стандартными атрибутами LogRecord.
```

Чтобы не передавать `extra` каждый раз, используют `LoggerAdapter` — он «приклеивает» контекст ко всем вызовам:

```python
import logging

base = logging.getLogger(__name__)
logger = logging.LoggerAdapter(base, {"request_id": "abc-123"})

logger.info("Старт")   # автоматически добавит request_id
logger.error("Сбой")   # тоже с request_id
```

Для динамического контекста можно унаследовать `LoggerAdapter` и переопределить `process()`. С Python 3.2 также есть `setLogRecordFactory` для глобального обогащения записей. В вебе под общий контекст (request id, user id) часто берут `contextvars` + фильтр, который вытаскивает значения и кладёт в запись.

### Потокобезопопасность

Модуль `logging` потокобезопасен: внутренние операции защищены блокировками, и одновременная запись из нескольких потоков не приводит к перемешиванию символов одной строки.

Нюансы:

- Гарантируется безопасность в рамках потоков. Для **процессов** (multiprocessing) общий файловый хендлер ненадёжен — используйте `QueueHandler` + `QueueListener`, отдельные файлы или централизованный сборщик.
- Сама конфигурация (`dictConfig`/добавление хендлеров) должна выполняться **до** запуска рабочих потоков, обычно один раз на старте.
- Для асинхронного логирования (не блокировать поток на медленном I/O — сеть, SMTP) применяют `QueueHandler`/`QueueListener`.

### Производительность и isEnabledFor

```python
import logging
logger = logging.getLogger(__name__)

# Быстрая проверка БЕЗ создания LogRecord и форматирования.
if logger.isEnabledFor(logging.DEBUG):
    logger.debug("Тяжёлый дамп: %s", build_huge_report())
```

Что важно для производительности:

- Сам по себе отключённый `logger.debug(...)` дёшев — внутри сразу делается проверка уровня. Но **аргументы вычисляются Python'ом до вызова** функции, поэтому дорогие аргументы оборачивайте в `isEnabledFor`.
- Атрибуты `funcName`, `lineno`, `pathname` требуют интроспекции стека — это небольшая, но не нулевая цена; в горячих циклах учитывайте.
- В критичных по латентности местах выносите I/O в `QueueHandler`.
- Отключайте сбор информации о вызывающем коде (`logging.logProcesses`, `logging.logThreads` и т.п. флаги), если она не нужна.

## Частые вопросы на собеседовании

**Вопрос:** Чем `logging` лучше `print()`?
**Ответ:** `print()` всегда пишет в один поток без структуры и уровней. `logging` даёт уровни важности (фильтрация по детализации), множество назначений (консоль, файл, сеть, почта), автоматический контекст (время, имя логгера, файл/строка, поток), централизованную конфигурацию, потокобезопасность и ленивое форматирование. Главное — назначение и детализацию можно менять конфигурацией, не трогая код, а сторонние библиотеки тоже встраиваются в общую систему. `print` для диагностики в проде — антипаттерн.

**Вопрос:** Назовите уровни логирования и их числовые значения.
**Ответ:** `DEBUG`=10, `INFO`=20, `WARNING`=30, `ERROR`=40, `CRITICAL`=50, плюс `NOTSET`=0. Сообщение проходит, если его уровень ≥ эффективного уровня логгера, а затем ≥ уровня каждого хендлера. По умолчанию root настроен на `WARNING`, поэтому без явной настройки `INFO`/`DEBUG` не видны.

**Вопрос:** Опишите путь, который проходит сообщение от `logger.info()` до вывода.
**Ответ:** Логгер сначала проверяет свой эффективный уровень — если `info` ниже, выходим сразу (ничего не форматируется). Иначе создаётся `LogRecord`, к нему применяются фильтры логгера. Затем запись отдаётся хендлерам логгера и (если `propagate=True`) всплывает к родителям до root, у каждого свои хендлеры. Каждый хендлер проверяет свой уровень и фильтры, вызывает форматтер для превращения `LogRecord` в строку и пишет в назначение. Ключевое: уровень проверяется и на логгере, и на каждом хендлере.

**Вопрос:** Зачем рекомендуют `logging.getLogger(__name__)`?
**Ответ:** `getLogger` возвращает синглтон по имени, а `__name__` равен полному пути модуля (`myapp.services.user`). Это автоматически встраивает логгер в иерархию пакета: можно настроить уровень/хендлеры на `myapp` и они применятся ко всем потомкам. Также в логах сразу видно, из какого модуля пришло сообщение, без ручного указания имён.

**Вопрос:** Что делает `basicConfig` и почему повторный вызов часто игнорируется?
**Ответ:** `basicConfig` быстро настраивает root: добавляет `StreamHandler` (в `stderr`), форматтер и уровень. По умолчанию он **ничего не делает, если у root уже есть хендлеры** — чтобы случайные повторные вызовы не плодили дубликаты. Ловушка: если до `basicConfig` вызвать `logging.warning(...)`, root уже получит хендлер по умолчанию, и `basicConfig` молча проигнорируется. С 3.8 есть `force=True` для пересоздания.

**Вопрос:** Что такое `propagate` и какую проблему он порождает?
**Ответ:** `propagate=True` (по умолчанию) означает, что после хендлеров текущего логгера запись всплывает к родителям до root, обрабатываясь их хендлерами. Если у дочернего логгера есть свой хендлер и при этом запись всплывает к root с его хендлером — получаем **дублирование строк**. Решения: `propagate=False` на дочернем логгере либо настройка хендлеров только на одном уровне иерархии. При всплытии уровень родительского логгера не проверяется — только уровни и фильтры самих хендлеров.

**Вопрос:** Что такое эффективный уровень?
**Ответ:** У большинства логгеров уровень `NOTSET` (0) — собственного уровня нет. Эффективный уровень вычисляется подъёмом по родителям до первого логгера с явно заданным уровнем; его значение и используется. Если такого нет — берётся `WARNING` (уровень root по умолчанию). Метод — `getEffectiveLevel()`. Это объясняет, почему достаточно задать уровень на корне поддерева, чтобы он применился ко всем потомкам.

**Вопрос:** В чём разница между уровнем логгера и уровнем хендлера?
**Ответ:** Уровень логгера — общий «кран»: решает, родится ли `LogRecord` вообще. Уровень хендлера — локальный фильтр конкретного назначения: даже если запись создана, хендлер выведет её, только если её уровень ≥ уровня хендлера. Типичный паттерн: логгер на `DEBUG`, консольный хендлер на `INFO`, файловый — на `DEBUG`. Так в файл попадает всё, а в консоль — только важное.

**Вопрос:** Почему `logger.info("x=%s", x)` лучше, чем `logger.info(f"x={x}")`?
**Ответ:** При первом варианте интерполяция строки выполняется лениво — только если запись реально будет выведена. Если уровень отключён, форматирование вообще не происходит, экономя CPU (особенно при дорогих `repr`/вычислениях). F-строка же вычисляется всегда, ещё до вызова `logging`. Для очень дорогих аргументов дополнительно оборачивают вызов в `if logger.isEnabledFor(logging.DEBUG):`.

**Вопрос:** Как правильно залогировать исключение с трассировкой?
**Ответ:** Внутри `except` вызвать `logger.exception("сообщение")` — он пишет на уровне `ERROR` и автоматически прикрепляет текущий traceback. Эквивалент — `logger.error("...", exc_info=True)`. Параметр `exc_info=True` можно добавить к любому уровню (`warning`, `critical`). Вне блока `except` информации об исключении не будет.

**Вопрос:** Зачем нужен `NullHandler` и кто его использует?
**Ответ:** Его добавляют **библиотеки** к своему корневому логгеру: `logging.getLogger("mylib").addHandler(logging.NullHandler())`. Это предотвращает предупреждение «No handlers could be found» и, главное, не навязывает приложению вывод — библиотека только генерирует записи, а решение, куда их направлять, остаётся за приложением. Это реализация принципа «библиотеки логируют, приложение настраивает».

**Вопрос:** Как настроить ротацию логов и какие есть подводные камни?
**Ответ:** `RotatingFileHandler` (по размеру, `maxBytes`+`backupCount`) или `TimedRotatingFileHandler` (по времени, `when`/`interval`+`backupCount`). При ротации старый файл переименовывается, создаётся новый, лишние архивы удаляются. Подводный камень: при нескольких процессах, пишущих в один файл, встроенная ротация ненадёжна (гонки при переименовании). Решения — отдельные файлы на процесс, внешний `logrotate` + `WatchedFileHandler`, либо `QueueHandler`/централизованный сборщик.

**Вопрос:** Что такое `dictConfig` и зачем `disable_existing_loggers`?
**Ответ:** `dictConfig` — декларативная настройка всей системы логирования из словаря (форматтеры, фильтры, хендлеры, логгеры, root). Это рекомендуемый способ. Параметр `disable_existing_loggers` при значении по умолчанию `True` отключает все логгеры, созданные **до** вызова и не упомянутые в конфиге, — частая причина «логи пропали». В большинстве приложений его ставят в `False`.

**Вопрос:** Как добавить к логам контекст вроде request_id?
**Ответ:** Тремя способами: 1) разово через `extra={"request_id": ...}` в вызове (но если поле есть в формате, его нужно передавать всегда, иначе `KeyError`); 2) через `LoggerAdapter`, который приклеивает контекст ко всем вызовам; 3) через `Filter`, который читает значение из `contextvars`/thread-local и кладёт в `record` — это самый удобный вариант для веба, так как контекст устанавливается в middleware и не нужно его таскать вручную.

**Вопрос:** Потокобезопасен ли `logging`? А для процессов?
**Ответ:** Для потоков — да: внутренние операции защищены блокировками, строки не перемешиваются. Для процессов общий файловый хендлер небезопасен. Решения: отдельные файлы на процесс, `QueueHandler`+`QueueListener` (один процесс-слушатель пишет в файл), `SocketHandler`/`SysLogHandler` или централизованный сборщик логов.

**Вопрос:** Зачем `isEnabledFor` и когда он действительно нужен?
**Ответ:** `logger.isEnabledFor(level)` быстро проверяет, будет ли запись данного уровня обработана, не создавая `LogRecord`. Он нужен, когда **аргументы лога дорого вычислять** (тяжёлая сериализация, большие дампы): аргументы Python вычисляет до вызова `logger.debug`, и ленивое `%`-форматирование тут не спасёт. Для обычных дешёвых аргументов отдельная проверка избыточна — сам `debug()` и так дёшев при отключённом уровне.

**Вопрос:** Что такое структурное логирование и зачем оно?
**Ответ:** Это вывод логов в машиночитаемом виде (обычно JSON), где каждое поле — отдельный ключ (`level`, `message`, `request_id`, `user_id`, `latency_ms`), а не плоская строка. Это удобно для централизованного сбора и анализа (ELK, Loki, Datadog): по полям можно фильтровать и агрегировать. Реализуется кастомным форматтером (`python-json-logger`) или библиотеками вроде `structlog`. В `dictConfig` JSON-форматтер подключают через ключ `"()"`.

**Вопрос:** Что такое `LogRecord` и какие у него полезные атрибуты?
**Ответ:** `LogRecord` — объект, представляющий одно событие логирования. Содержит шаблон сообщения и аргументы, уровень (`levelname`/`levelno`), имя логгера (`name`), время (`created`/`asctime`), местоположение (`pathname`/`filename`/`module`/`funcName`/`lineno`), идентификаторы потока/процесса, информацию об исключении (`exc_info`/`exc_text`). Форматтер обращается к этим атрибутам по именам в строке формата; фильтры могут их читать и дополнять.

## Подводные камни (gotchas)

- **«Не вижу INFO-сообщений».** Уровень root по умолчанию `WARNING`. Без `basicConfig`/настройки `INFO` и `DEBUG` молча отбрасываются.
- **`basicConfig` молча проигнорирован.** Если у root уже есть хендлер (например, после случайного `logging.warning(...)` или повторного вызова), `basicConfig` без `force=True` ничего не делает.
- **Дублирование строк.** Хендлер добавлен и на дочерний логгер, и наследуется через всплытие к root. Лечится `propagate=False` или настройкой на одном уровне.
- **`disable_existing_loggers=True`.** Значение по умолчанию в `dictConfig` глушит ранее созданные логгеры (типично для модулей, импортированных до настройки) — «логи внезапно пропали».
- **f-строки в логах.** `logger.debug(f"...")` вычисляется всегда, теряется ленивость; на горячем пути это лишний CPU.
- **`KeyError` в форматтере из-за `extra`.** Если в строке формата есть `%(request_id)s`, а в каком-то вызове поле не передано — упадёт форматтер. Решение: `LoggerAdapter`/фильтр, гарантирующий наличие поля.
- **`logger.exception` вне `except`.** Без активного исключения traceback не появится, а вызов сам по себе бессмысленен.
- **Добавление хендлеров при каждом вызове функции.** Если в функции вызывать `logger.addHandler(...)`, хендлеры накапливаются и строки множатся. Настраивать логирование нужно один раз на старте.
- **Конфликт имён в `extra`.** Имена не должны совпадать со стандартными атрибутами `LogRecord` (`message`, `args`, `name`...), иначе `KeyError`.
- **Ротация при нескольких процессах.** Встроенные ротационные хендлеры не рассчитаны на несколько писателей в один файл — будут гонки и потеря данных.
- **Шумные сторонние библиотеки.** `urllib3`, `sqlalchemy`, `botocore` могут заваливать логи на `DEBUG`. Их логгеры нужно явно приглушать (`setLevel(WARNING)`).
- **Настройка root-логгера в библиотеке.** Библиотека, вызывающая `basicConfig`/добавляющая хендлеры на root, навязывает поведение приложению. Библиотеки должны только логировать (+ `NullHandler`).
- **`getLevelName` с неизвестным значением.** Для незнакомого числа вернёт строку вида `"Level 25"`, а не исключение.

## Лучшие практики

- **В каждом модуле:** `logger = logging.getLogger(__name__)` на верхнем уровне. Не используйте модульные функции `logging.info(...)` напрямую — они пишут в root.
- **Настраивайте логирование один раз** на старте приложения через `dictConfig` (желательно из конфиг-файла/окружения). В тестах — отдельная конфигурация.
- **В библиотеках:** добавьте `NullHandler` к корневому логгеру библиотеки и никогда не настраивайте root и не добавляйте «реальные» хендлеры — это работа приложения.
- **Используйте ленивое `%`-форматирование**, а для дорогих аргументов — `isEnabledFor`.
- **Логируйте исключения** через `logger.exception(...)` внутри `except`, чтобы получить трассировку.
- **Не используйте `print`** для диагностики в коде, который пойдёт в прод.
- **Выбирайте уровни осмысленно:** `DEBUG` — для разработчика, `INFO` — нормальный ход событий, `WARNING` — подозрительно, `ERROR` — операция не выполнена, `CRITICAL` — угроза работе приложения.
- **Ставьте `propagate=False`** на тех логгерах, для которых добавляете собственные хендлеры, чтобы избежать дублей (или, наоборот, держите хендлеры только на root).
- **Приглушайте шумные библиотеки** заданием уровня их логгерам.
- **Не логируйте секреты** (пароли, токены, персональные данные). Маскируйте через фильтр при необходимости.
- **В продакшене — структурный JSON** и централизованный сбор; ротация или внешний `logrotate` для файлов.
- **Тяжёлый/сетевой I/O** (SMTP, HTTP) выносите в `QueueHandler`/`QueueListener`, чтобы не блокировать рабочие потоки.
- **Добавляйте контекст** (request_id, user_id) через фильтр + `contextvars`, а не вручную в каждый вызов.

## Шпаргалка

```python
# --- Получение логгера (всегда так) ---
import logging
logger = logging.getLogger(__name__)

# --- Уровни и числа ---
# NOTSET=0  DEBUG=10  INFO=20  WARNING=30  ERROR=40  CRITICAL=50
# root по умолчанию = WARNING

# --- Быстрый старт (скрипты) ---
logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s %(levelname)s %(name)s %(message)s",
)

# --- Запись логов ---
logger.debug("отладка %s", x)        # ленивое форматирование!
logger.info("событие")
logger.warning("подозрительно")
logger.error("ошибка")
logger.critical("критично")

# --- Исключения ---
try:
    risky()
except Exception:
    logger.exception("упало")        # ERROR + traceback (только в except)

# --- Дорогой аргумент ---
if logger.isEnabledFor(logging.DEBUG):
    logger.debug("дамп %s", heavy())

# --- Иерархия / уровни ---
logging.getLogger("myapp").setLevel(logging.DEBUG)   # применится к потомкам
logger.getEffectiveLevel()                            # вычисленный уровень
logger.propagate = False                              # отключить всплытие к root

# --- Хендлеры ---
from logging.handlers import RotatingFileHandler, TimedRotatingFileHandler
h1 = logging.StreamHandler()                          # в stderr/stdout
h2 = logging.FileHandler("app.log", encoding="utf-8")
h3 = RotatingFileHandler("app.log", maxBytes=10_485_760, backupCount=5)
h4 = TimedRotatingFileHandler("app.log", when="midnight", backupCount=7)
logging.getLogger("mylib").addHandler(logging.NullHandler())  # в библиотеках

# --- Форматтер ---
fmt = logging.Formatter("%(asctime)s %(levelname)s %(name)s %(message)s")
h1.setFormatter(fmt)
h1.setLevel(logging.INFO)

# --- Фильтр (обогащение записи) ---
class AddCtx(logging.Filter):
    def filter(self, record):
        record.request_id = "abc"
        return True
h1.addFilter(AddCtx())

# --- Контекст ---
logger.info("msg", extra={"request_id": "abc"})       # разово
adapter = logging.LoggerAdapter(logger, {"request_id": "abc"})  # постоянно

# --- Рекомендуемая настройка приложения ---
import logging.config
logging.config.dictConfig({
    "version": 1,
    "disable_existing_loggers": False,   # важно!
    "formatters": {"def": {"format": "%(asctime)s %(levelname)s %(name)s %(message)s"}},
    "handlers": {"console": {"class": "logging.StreamHandler", "formatter": "def"}},
    "loggers": {"myapp": {"level": "DEBUG", "handlers": ["console"], "propagate": False}},
    "root": {"level": "WARNING", "handlers": ["console"]},
})
```

**Памятка-чеклист:**
- логгер = `getLogger(__name__)`; не пиши в root напрямую и не используй `print`
- ленивое `%`-форматирование, не f-строки
- `exception()` только в `except`
- `propagate=False` там, где свои хендлеры (иначе дубли)
- библиотека = только логирует + `NullHandler`
- `dictConfig(disable_existing_loggers=False)` для настройки приложения
- ротация по размеру/времени; для multiprocessing — очередь/внешний logrotate
- приглушай шумные библиотеки; не логируй секреты
