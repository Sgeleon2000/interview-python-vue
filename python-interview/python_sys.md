# sys — подготовка к собеседованию

## Что это и зачем

Модуль `sys` — это встроенный модуль стандартной библиотеки, который даёт доступ к переменным и функциям, тесно связанным с **интерпретатором Python** и его взаимодействием с операционной системой. В отличие от `os` (который отвечает за абстракцию ОС: файлы, процессы, переменные окружения), `sys` отвечает за «внутренности» самого рантайма: как импортируются модули, как устроены потоки ввода-вывода, как обрабатываются исключения, какой лимит рекурсии, какой порядок байтов, какая версия Python и т.д.

Почему это важно на собеседовании:

- `sys` — один из тех модулей, через которые видно глубину понимания того, **как Python устроен внутри**: модель импорта, подсчёт ссылок (reference counting), управление потоками вывода, поведение при завершении программы.
- Вопросы по `sys` часто служат «крючком», чтобы перейти к более фундаментальным темам: GIL, сборка мусора, CPython vs PyPy, буферизация, рекурсия и стек.

```python
import sys

# Базовая «карточка» интерпретатора
print(sys.version)        # человекочитаемая строка версии
print(sys.version_info)   # структурированный кортеж (для сравнений)
print(sys.executable)     # путь к бинарнику python, который сейчас запущен
print(sys.platform)       # 'darwin' / 'linux' / 'win32'
print(sys.prefix)         # корень установки/виртуального окружения
```

`sys` не нужно устанавливать — он часть CPython и встроен в интерпретатор (его нельзя «удалить» из стандартной библиотеки).

## Ключевые концепции

### 1. Интерпретатор как объект, доступный из кода

`sys` позволяет программе **интроспектировать и менять собственный рантайм**: добавить путь поиска модулей, перехватить необработанное исключение, поменять лимит рекурсии, перенаправить вывод. Это «рефлексия уровня процесса».

### 2. Аргументы командной строки: `sys.argv`

`sys.argv` — список строк. `argv[0]` — имя скрипта (или `''` / `'-c'` в особых случаях), далее идут переданные аргументы.

```python
# script.py
import sys
print(sys.argv)
# $ python script.py --name Alice 42
# ['script.py', '--name', 'Alice', '42']
```

Все элементы — **строки**, никакой автоматической типизации нет. Для серьёзного парсинга используют `argparse`, но знать «сырой» `sys.argv` обязательно.

### 3. Поиск модулей: `sys.path` и `sys.modules`

- `sys.path` — список директорий (и zip-архивов), в которых Python ищет модули при `import`. Порядок важен: первый найденный «побеждает».
- `sys.modules` — словарь-кэш уже импортированных модулей. Именно он делает повторный `import` мгновенным: если модуль уже в `sys.modules`, его тело **не выполняется заново**.

### 4. Завершение программы: `sys.exit` и `SystemExit`

`sys.exit()` не «убивает» процесс напрямую — он **возбуждает исключение `SystemExit`**, которое раскручивает стек (выполняя `finally` и `__exit__`), и только потом интерпретатор завершается. Это принципиально отличается от `os._exit()`, который завершает процесс немедленно на уровне ОС.

### 5. Стандартные потоки: `stdin`, `stdout`, `stderr`

Это файловые объекты (текстовые обёртки `io.TextIOWrapper` над буферами). Их можно подменять (перенаправление вывода), но «оригиналы» всегда доступны как `sys.__stdout__`, `sys.__stderr__`, `sys.__stdin__`.

### 6. Подсчёт ссылок и память

CPython использует reference counting. `sys.getrefcount(obj)` показывает счётчик ссылок, `sys.getsizeof(obj)` — размер объекта в байтах. Обе функции — окно во внутреннюю модель памяти.

### 7. Хуки исключений

`sys.excepthook` вызывается для **необработанного** исключения в главном потоке; `sys.unraisablehook` — для исключений, которые нельзя «пробросить» (например, в `__del__`). Их можно переопределять — основа кастомных систем логирования ошибок (Sentry и т.п.).

## Основные функции/классы/методы

### `sys.argv` — аргументы командной строки

```python
import sys

def main():
    # Простейший «ручной» парсинг без argparse
    args = sys.argv[1:]  # отбрасываем имя скрипта
    if not args:
        print("Использование: script.py <имя> [возраст]", file=sys.stderr)
        sys.exit(1)  # ненулевой код = ошибка

    name = args[0]
    age = int(args[1]) if len(args) > 1 else None  # типизация — вручную
    print(f"Привет, {name}! Возраст: {age}")

if __name__ == "__main__":
    main()
```

Нюансы:
- При запуске `python -c "code"` → `sys.argv[0] == '-c'`.
- При запуске из stdin (`python < file`) или интерактивно → `sys.argv[0]` может быть `''`.
- `sys.argv` можно **перезаписать** перед вызовом чужого кода (например, в тестах, имитируя CLI).

### `sys.path` — пути поиска модулей

```python
import sys

# Как формируется sys.path при старте:
# 1. директория скрипта (или '' = текущая для интерактивного режима)
# 2. содержимое переменной окружения PYTHONPATH
# 3. стандартные пути установки + site-packages

for p in sys.path:
    print(p)

# Добавить свой путь во время выполнения
sys.path.insert(0, "/opt/my_libs")   # в начало — высший приоритет
sys.path.append("/opt/extra")        # в конец — низший приоритет

# После этого import найдёт модули из этих директорий
```

Важно: `sys.path[0]` обычно указывает на директорию запускаемого скрипта — отсюда классическая «ловушка теневания»: если назвать свой файл `random.py`, он перекроет стандартный модуль `random`.

### `sys.modules` — кэш импортированных модулей

```python
import sys

print('json' in sys.modules)  # False (ещё не импортирован)
import json
print('json' in sys.modules)  # True
print(sys.modules['json'])    # <module 'json' from '...'>

# Принудительная «перезагрузка» (грубая): удалить из кэша
del sys.modules['json']
import json   # тело модуля выполнится заново

# Правильнее использовать importlib.reload(json)
```

Трюки на собеседовании:
- Можно **подменить** модуль в `sys.modules` ещё до его импорта — приём мокинга в тестах.
- `sys.modules` содержит и пакеты, и подмодули (`os`, `os.path`).
- Циклические импорты «работают» частично именно благодаря тому, что наполовину инициализированный модуль уже лежит в `sys.modules`.

### `sys.exit()` vs `os._exit()` vs `SystemExit`

```python
import sys, os

# 1. sys.exit — «вежливый» выход через исключение
try:
    sys.exit("Сообщение об ошибке")  # строка → печатается в stderr, код = 1
except SystemExit as e:
    print("Перехватили SystemExit:", e.code)  # его МОЖНО поймать!

# Коды:
# sys.exit()      -> код 0 (успех)
# sys.exit(0)     -> код 0
# sys.exit(3)     -> код 3
# sys.exit("msg") -> код 1, "msg" в stderr

# 2. os._exit — немедленное завершение на уровне ОС,
#    без раскрутки стека, без finally, без сброса буферов
def hard_exit():
    print("Это попадёт в буфер...", end="")
    os._exit(0)   # буфер НЕ сбросится — текст может потеряться!
```

Ключевой момент: `SystemExit` наследуется от `BaseException`, а **не** от `Exception`. Поэтому `except Exception:` его НЕ ловит (и это правильно — не мешаем штатному завершению). `os._exit()` обычно нужен только в дочернем процессе после `fork()`.

### `sys.stdin` / `sys.stdout` / `sys.stderr` и `sys.__stdout__`

```python
import sys, io

# Перенаправление stdout в строковый буфер
buffer = io.StringIO()
old_stdout = sys.stdout
sys.stdout = buffer
print("Это уйдёт в буфер, а не на экран")
sys.stdout = old_stdout            # восстановили
print("Захвачено:", buffer.getvalue())

# Оригинальные потоки всегда доступны:
print("Принудительно на настоящий stdout", file=sys.__stdout__)

# Ошибки — в stderr, чтобы не смешивать с полезным выводом
print("Ошибка!", file=sys.stderr)

# Буферизация: форсируем сброс
sys.stdout.write("без перевода строки")
sys.stdout.flush()   # гарантированно вытолкнуть буфер
```

Современный способ перенаправления — `contextlib.redirect_stdout`:

```python
import contextlib, io
buf = io.StringIO()
with contextlib.redirect_stdout(buf):
    print("захвачено контекстным менеджером")
print(buf.getvalue())
```

Буферизация:
- stdout по умолчанию **построчно** буферизован, если подключён к терминалу, и **блочно** — если перенаправлен в файл/пайп.
- stderr исторически небуферизован (или строчно) — чтобы ошибки появлялись сразу.
- Флаг `python -u` или `PYTHONUNBUFFERED=1` отключает буферизацию (важно для логов в Docker).

### `sys.version_info` и `sys.version`

```python
import sys

print(sys.version)        # '3.12.1 (main, ...) [Clang ...]' — для людей
print(sys.version_info)   # sys.version_info(major=3, minor=12, micro=1, ...)

# Правильное сравнение версий — через version_info, а НЕ парсинг строки!
if sys.version_info >= (3, 10):
    print("Доступен match-statement")

if sys.version_info < (3, 8):
    raise RuntimeError("Требуется Python 3.8+")
```

### `sys.getsizeof()` — размер объекта

```python
import sys

print(sys.getsizeof(0))          # ~28 байт (int — не «4 байта»!)
print(sys.getsizeof(""))         # ~49 (пустая str)
print(sys.getsizeof("a"))        # больше — Python хранит длину/кодировку
print(sys.getsizeof([]))         # пустой список
print(sys.getsizeof([1, 2, 3]))  # список + ссылки на элементы

# ВАЖНО: getsizeof НЕ рекурсивен!
data = [list(range(1000))]
print(sys.getsizeof(data))  # покажет размер ВНЕШНЕГО списка (одна ссылка),
                            # а не суммарный размер вложенных данных
```

Главный подвох: для контейнеров `getsizeof` считает только **сам контейнер** (массив указателей), но не объекты, на которые он ссылается. Для «настоящего» размера нужен рекурсивный обход (см. документацию или `pympler.asizeof`).

### `sys.setrecursionlimit()` / `sys.getrecursionlimit()`

```python
import sys

print(sys.getrecursionlimit())   # обычно 1000

def deep(n):
    if n == 0:
        return 0
    return deep(n - 1)

# deep(2000)  # RecursionError при лимите 1000

sys.setrecursionlimit(3000)      # подняли лимит
print(deep(2000))                # теперь ок

# ОПАСНОСТЬ: слишком большой лимит может привести к реальному
# переполнению стека C и краху интерпретатора (segfault), а не к
# аккуратному RecursionError.
```

Лимит — это защита от бесконечной рекурсии: ограничение на глубину Python-стека, не на стек C напрямую.

### `sys.getrefcount()` — счётчик ссылок

```python
import sys

x = []
print(sys.getrefcount(x))   # покажет на 1 БОЛЬШЕ ожидаемого

# Почему +1?
# В момент вызова getrefcount(x) аргумент передаётся в функцию,
# создавая ВРЕМЕННУЮ дополнительную ссылку на объект (параметр функции).
# Поэтому результат всегда на единицу больше «реального» числа ссылок.

a = object()
b = a                       # 2 ссылки: a и b
print(sys.getrefcount(a))   # 3 = a + b + временная в getrefcount
```

Нюансы: маленькие int (от -5 до 256) и интернированные строки имеют огромный refcount, потому что переиспользуются по всему интерпретатору. В Python 3.12+ есть «бессмертные» (immortal) объекты (например, `None`, `True`, `False`), у которых refcount искусственно зафиксирован.

### `sys.platform` и определение ОС

```python
import sys

# Типичные значения:
#   'linux'   — Linux
#   'darwin'  — macOS
#   'win32'   — Windows (даже на 64-битной!)
#   'cygwin'  — Cygwin

if sys.platform.startswith("linux"):
    ...
elif sys.platform == "darwin":
    ...
elif sys.platform == "win32":
    ...

# Для более детальной инфы — модуль platform (другой!):

```

### `sys.excepthook` и `sys.unraisablehook`

```python
import sys

# Перехват ЛЮБОГО необработанного исключения в главном потоке
def my_excepthook(exc_type, exc_value, exc_traceback):
    print("!!! Поймано необработанное исключение:", exc_type.__name__)
    # здесь можно залогировать в файл, отправить в Sentry и т.п.
    sys.__excepthook__(exc_type, exc_value, exc_traceback)  # стандартное поведение

sys.excepthook = my_excepthook

# raise ValueError("бум")  # вызовет my_excepthook вместо стандартного трейсбэка
```

```python
# unraisablehook — для исключений, которые нельзя пробросить наружу.
# Классический случай: исключение в __del__ или в callback'е GC.
import sys

def my_unraisablehook(unraisable):
    print("Неподнимаемое исключение в:", unraisable.object)
    print("Исключение:", unraisable.exc_value)

sys.unraisablehook = my_unraisablehook

class Boom:
    def __del__(self):
        raise RuntimeError("ошибка в деструкторе")

Boom()   # исключение в __del__ уйдёт в unraisablehook, не сломав поток
```

Важно: `sys.excepthook` **не** вызывается для исключений в дочерних потоках (`threading.Thread`) — там работает `threading.excepthook` (с Python 3.8).

### `sys.maxsize`, `sys.byteorder`, `sys.flags`, `sys.intern`

```python
import sys

# maxsize — максимальный размер контейнера / индекса.
# На 64-битной системе это 2**63 - 1. НЕ путать с максимальным int
# (в Python 3 int не ограничен). Хороший индикатор разрядности:
print(sys.maxsize)                      # 9223372036854775807 на 64 бит
print(sys.maxsize > 2**32)              # True => 64-битный интерпретатор

# byteorder — порядок байтов платформы
print(sys.byteorder)                    # 'little' или 'big'

# flags — флаги интерпретатора (read-only)
print(sys.flags.optimize)               # уровень -O
print(sys.flags.debug)
print(sys.flags.dont_write_bytecode)    # соответствует -B / PYTHONDONTWRITEBYTECODE

# intern — принудительное интернирование строки (хранение в общем пуле).
# Ускоряет сравнение по identity (is) и экономит память при множестве
# одинаковых строк (например, ключей словарей в парсинге).
a = sys.intern("очень_длинный_повторяющийся_ключ")
b = sys.intern("очень_длинный_повторяющийся_ключ")
print(a is b)   # True — это один и тот же объект в памяти
```

### `sys.settrace` (кратко)

```python
import sys

# settrace задаёт глобальную трейс-функцию, вызываемую интерпретатором
# на каждое событие (вызов функции, строка, возврат, исключение).
# Это основа отладчиков (pdb) и инструментов покрытия кода (coverage.py).

def tracer(frame, event, arg):
    if event == "call":
        print(f"Вызов функции: {frame.f_code.co_name}")
    return tracer   # вернуть локальную трейс-функцию для построчной трассировки

sys.settrace(tracer)
# ... код, который хотим трассировать ...
sys.settrace(None)  # отключить

# На практике вручную почти не используют — знают как механизм
# под капотом pdb/coverage/профайлеров. Парный метод: sys.setprofile.
```

## Частые вопросы на собеседовании

**Вопрос:** В чём разница между `sys.exit()` и `os._exit()`?

**Ответ:** `sys.exit()` возбуждает исключение `SystemExit`, которое раскручивает стек: выполняются блоки `finally`, методы `__exit__` контекстных менеджеров, обработчики `atexit`, сбрасываются буферы ввода-вывода. Его можно перехватить через `except SystemExit`. `os._exit()` завершает процесс **немедленно** через системный вызов, минуя любую очистку Python — буферы не сбрасываются, `finally` не выполняется. `os._exit()` используют почти исключительно в дочернем процессе после `os.fork()`, чтобы не запускать повторно очистку родителя.

---

**Вопрос:** Почему `sys.getrefcount(x)` возвращает значение на 1 больше ожидаемого?

**Ответ:** Когда вы вызываете `getrefcount(x)`, объект передаётся в функцию как аргумент. Параметр функции создаёт временную дополнительную ссылку на объект на время вызова. Поэтому возвращаемое значение всегда на единицу больше числа «настоящих» ссылок в вашем коде. Дополнительно: refcount специфичен для CPython (в PyPy/Jython подсчёта ссылок нет), а у мелких целых, интернированных строк и «бессмертных» объектов вроде `None` счётчик не отражает реальность.

---

**Вопрос:** Как Python находит модуль при `import`? Какова роль `sys.path` и `sys.modules`?

**Ответ:** Сначала Python проверяет `sys.modules` — кэш уже импортированных модулей. Если модуль там есть, он возвращается мгновенно, тело **не выполняется повторно**. Если нет — запускается механизм импорта: finders и loaders из `sys.meta_path` ищут модуль, в том числе по директориям из `sys.path` (директория скрипта, `PYTHONPATH`, site-packages). Первое совпадение в `sys.path` побеждает. После успешного импорта модуль кладётся в `sys.modules`. Именно `sys.modules` обеспечивает работу циклических импортов (наполовину инициализированный модуль уже доступен) и используется для мокинга в тестах.

---

**Вопрос:** Почему `sys.getsizeof([1, 2, 3])` может ввести в заблуждение?

**Ответ:** `getsizeof` не рекурсивен: он возвращает размер только самого объекта-контейнера, то есть структуры списка и массива указателей на элементы, но **не** размер объектов, на которые эти указатели ссылаются. Для списка из вложенных больших структур результат покажет лишь несколько десятков байт «обёртки». Чтобы посчитать полный размер, нужно рекурсивно обойти все ссылки (с защитой от циклов и повторно посещённых объектов) или использовать `pympler.asizeof`.

---

**Вопрос:** Что произойдёт, если выставить `sys.setrecursionlimit(10**8)`?

**Ответ:** Лимит рекурсии Python — это защита, ограничивающая глубину Python-стека, чтобы выдать аккуратный `RecursionError` до того, как закончится реальный стек C. Если поднять лимит слишком высоко, глубокая рекурсия исчерпает стек уровня C/ОС раньше, чем сработает проверка лимита, и интерпретатор **упадёт с segfault** (или stack overflow) вместо исключения. Поэтому лимит поднимают аккуратно и небольшими значениями, а лучше — переписывают алгоритм итеративно.

---

**Вопрос:** Чем `sys.stdout` отличается от `sys.__stdout__`? Зачем нужен второй?

**Ответ:** `sys.stdout` — текущий объект стандартного вывода, который можно переназначить (например, перенаправить в `StringIO` для захвата вывода). `sys.__stdout__` хранит **оригинальный** объект, каким он был на старте интерпретатора. Это нужно, чтобы можно было гарантированно восстановить вывод после перенаправления или принудительно написать в «настоящую» консоль, даже если `sys.stdout` подменён. Аналогично есть `sys.__stderr__` и `sys.__stdin__`.

---

**Вопрос:** Ловит ли `except Exception` исключение от `sys.exit()`?

**Ответ:** Нет. `sys.exit()` возбуждает `SystemExit`, которое наследуется от `BaseException`, а не от `Exception`. Поэтому `except Exception:` его не перехватит, и программа корректно завершится. Это сделано намеренно: широкий `except Exception` для обработки прикладных ошибок не должен случайно блокировать штатное завершение программы (как и `KeyboardInterrupt`, который тоже от `BaseException`). Перехватить `SystemExit` можно только явным `except SystemExit:` или `except BaseException:`.

---

**Вопрос:** Как глобально логировать все необработанные исключения?

**Ответ:** Переопределить `sys.excepthook` — функцию, которую интерпретатор вызывает для любого необработанного исключения в главном потоке (с аргументами `(type, value, traceback)`). Внутри можно залогировать ошибку, отправить её в систему мониторинга и при желании вызвать `sys.__excepthook__` для стандартного трейсбэка. Важно помнить: для дочерних потоков работает отдельный `threading.excepthook`, а для исключений, которые нельзя пробросить (например, в `__del__`), — `sys.unraisablehook`.

---

**Вопрос:** Как по `sys` определить разрядность интерпретатора (32 vs 64 бита)?

**Ответ:** Через `sys.maxsize`: на 64-битной сборке он равен `2**63 - 1`, на 32-битной — `2**31 - 1`. Проверка `sys.maxsize > 2**32` истинна для 64-битного интерпретатора. Это надёжнее, чем смотреть на разрядность ОС, потому что важна именно сборка Python. Важно не путать `sys.maxsize` (предел размера контейнеров/индексов) с максимальным значением `int` — в Python 3 целые имеют произвольную точность и не ограничены.

---

**Вопрос:** Что делает `sys.intern` и когда это полезно?

**Ответ:** `sys.intern(s)` помещает строку в общий внутренний пул и возвращает каноничный объект: для двух одинаковых интернированных строк `a is b` будет `True`. Это даёт два эффекта: экономия памяти при большом числе повторяющихся строк (например, имена ключей при парсинге большого JSON/CSV) и ускорение сравнений, поскольку сравнение по identity (`is`) дешевле посимвольного. Python автоматически интернирует строковые литералы, похожие на идентификаторы, но динамически собранные строки нужно интернировать вручную.

---

**Вопрос:** Почему вывод программы «теряется» при использовании `os._exit()` или при падении в Docker?

**Ответ:** `stdout` буферизуется. При подключении к терминалу буферизация построчная, но при перенаправлении в файл/пайп (как в Docker) — блочная: данные накапливаются и сбрасываются крупными порциями. `os._exit()` завершает процесс, не сбрасывая буферы, поэтому накопленное теряется. То же может происходить при аварийном завершении. Решения: явный `sys.stdout.flush()`, запуск с `python -u`, переменная `PYTHONUNBUFFERED=1`, либо `print(..., flush=True)`.

---

**Вопрос:** Можно ли мокать модуль через `sys.modules`?

**Ответ:** Да. Если положить в `sys.modules['имя_модуля']` подставной объект (Mock) **до** того, как тестируемый код выполнит `import имя_модуля`, импорт вернёт ваш объект из кэша, не выполняя реальный модуль. Это используют для подмены тяжёлых или недоступных зависимостей. Однако это грубый приём; чаще применяют `unittest.mock.patch`, который как раз и манипулирует ссылками/`sys.modules` под капотом, но с автоматическим откатом после теста.

## Подводные камни (gotchas)

1. **Теневание стандартных модулей.** Файл `random.py`, `json.py`, `email.py` в директории скрипта попадёт в `sys.path[0]` и перекроет одноимённый стандартный модуль. Симптом — странные `ImportError`/`AttributeError`. Не называйте файлы именами стандартных модулей.

2. **`getsizeof` не рекурсивен.** Для вложенных контейнеров возвращает размер только верхнего уровня. Нельзя использовать «в лоб» для оценки потребления памяти структурой данных.

3. **`getrefcount` всегда +1** из-за временной ссылки-аргумента. И он бессмыслен для интернированных мелких int, строк и «бессмертных» объектов.

4. **`sys.exit()` ловится случайным `except`.** Голый `except:` (без типа) перехватывает даже `SystemExit` и `KeyboardInterrupt`, ломая нормальное завершение. Всегда уточняйте тип: `except Exception:`.

5. **`os._exit()` без сброса буферов.** Вывод и данные в буферах теряются, `finally` и `atexit` не выполняются. Использовать только осознанно (после `fork`).

6. **`sys.excepthook` не работает для потоков.** В дочерних потоках исключения идут через `threading.excepthook`, а не `sys.excepthook`.

7. **Высокий `setrecursionlimit` = риск segfault.** Лимит не защищает от переполнения C-стека, если задрать его слишком высоко.

8. **Изменение `sys.path` глобально.** Вставка путей затрагивает весь процесс. В библиотеках так делать не стоит — это побочный эффект на чужой код. Лучше пакеты и виртуальные окружения.

9. **`sys.platform == 'win32'` даже на 64-битной Windows.** Имя историческое; не полагайтесь на «32» в нём.

10. **Перенаправил `sys.stdout` и забыл восстановить.** Если не вернуть оригинал (или не использовать `redirect_stdout` как контекстный менеджер), весь последующий вывод уйдёт «в никуда» или в чужой буфер.

11. **`del sys.modules['x']` ≠ полная перезагрузка.** Уже импортированные где-то ссылки на старый модуль/его объекты останутся жить. Для аккуратной перезагрузки — `importlib.reload`, и то с оговорками.

## Лучшие практики

- **Для CLI используйте `argparse`/`click`/`typer`, а не голый `sys.argv`.** `sys.argv` — для самых простых случаев и для подмены аргументов в тестах.
- **Версию проверяйте через `sys.version_info` кортежем** (`>= (3, 10)`), а не парсингом `sys.version`.
- **Ошибки и диагностику пишите в `sys.stderr`**, полезные данные — в `sys.stdout`. Это позволяет раздельно перенаправлять потоки в пайплайнах.
- **Для перенаправления вывода используйте `contextlib.redirect_stdout`/`redirect_stderr`** — они корректно восстанавливают потоки даже при исключении.
- **Не поднимайте лимит рекурсии бездумно** — предпочитайте итеративные алгоритмы или явный стек.
- **Не модифицируйте `sys.path` из библиотечного кода** — это глобальный побочный эффект. Используйте упаковку и зависимости.
- **`os._exit()` — только после `fork()`** в дочернем процессе.
- **Кастомный `sys.excepthook`** — хороший централизованный пункт логирования необработанных ошибок; не забудьте про `threading.excepthook` для потоков.
- **`sys.intern`** применяйте точечно — там, где реально много повторяющихся строк (парсинг больших данных), а не «на всякий случай».
- **При работе с буферизацией в контейнерах** включайте `PYTHONUNBUFFERED=1`/`python -u` или используйте `flush=True`.

## Шпаргалка

```python
import sys

# --- Аргументы ---
sys.argv                 # ['script.py', 'arg1', ...] — всё строки
sys.argv[1:]             # переданные аргументы (без имени скрипта)

# --- Пути и модули ---
sys.path                 # список директорий поиска модулей
sys.path.insert(0, p)    # добавить путь с высшим приоритетом
sys.modules              # кэш {имя: модуль} импортированных модулей
'json' in sys.modules    # проверка, импортирован ли модуль

# --- Завершение ---
sys.exit()               # код 0
sys.exit(1)              # код 1 (ошибка)
sys.exit("msg")          # код 1, msg в stderr; возбуждает SystemExit
os._exit(0)              # немедленный выход без очистки (после fork)
# SystemExit -> BaseException (НЕ ловится except Exception)

# --- Потоки ввода-вывода ---
sys.stdin, sys.stdout, sys.stderr        # текущие потоки
sys.__stdout__, sys.__stderr__           # оригиналы (для восстановления)
print("err", file=sys.stderr)            # вывод в stderr
sys.stdout.flush()                       # сброс буфера
# contextlib.redirect_stdout(buf) — безопасное перенаправление

# --- Версия ---
sys.version              # человекочитаемая строка
sys.version_info         # кортеж для сравнений: >= (3, 10)

# --- Память и ссылки ---
sys.getsizeof(obj)       # размер в байтах (НЕ рекурсивно!)
sys.getrefcount(obj)     # счётчик ссылок (+1 от аргумента)

# --- Рекурсия ---
sys.getrecursionlimit()  # текущий лимит (обычно 1000)
sys.setrecursionlimit(n) # поднять (осторожно: риск segfault)

# --- Платформа и интерпретатор ---
sys.platform             # 'linux' / 'darwin' / 'win32'
sys.executable           # путь к бинарнику python
sys.maxsize              # 2**63-1 на 64 бит; > 2**32 => 64-битный
sys.byteorder            # 'little' / 'big'
sys.flags                # флаги интерпретатора (-O, -B, ...)
sys.prefix               # корень установки/venv

# --- Хуки ---
sys.excepthook           # обработчик необработанных исключений (main thread)
sys.unraisablehook       # для исключений в __del__ / GC
sys.__excepthook__       # стандартный обработчик
# threading.excepthook   # для дочерних потоков (отдельно!)

# --- Прочее ---
sys.intern("s")          # интернирование строки (общий пул, is == True)
sys.settrace(func)       # глобальная трейс-функция (основа pdb/coverage)
sys.setprofile(func)     # профилирующая трейс-функция
```

**Мнемоника различий:**

| Действие | Инструмент | Что делает |
|---|---|---|
| Аргументы CLI | `sys.argv` | список строк |
| Пути модулей | `sys.path` | поиск import |
| Кэш модулей | `sys.modules` | повторный import без выполнения |
| Вежливый выход | `sys.exit()` | `SystemExit`, очистка |
| Жёсткий выход | `os._exit()` | мгновенно, без очистки |
| Версия | `sys.version_info` | сравнение кортежем |
| Размер | `sys.getsizeof()` | байты, не рекурсивно |
| Ссылки | `sys.getrefcount()` | +1 за аргумент |
| Лимит рекурсии | `sys.setrecursionlimit()` | риск segfault |
| Перехват ошибок | `sys.excepthook` | только main thread |
