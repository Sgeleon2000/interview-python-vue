# os — подготовка к собеседованию

## Что это и зачем

Модуль `os` — это стандартная библиотека Python, предоставляющая переносимый (portable) интерфейс к функциям операционной системы. Через него Python-программа общается с ОС: читает переменные окружения, работает с файловой системой, управляет процессами, правами доступа и т. д.

Ключевая идея — **абстракция над платформой**. Один и тот же код с `os.path.join`, `os.sep`, `os.listdir` работает на Linux, macOS и Windows. Там, где переносимость невозможна (например, `os.fork`), функции доступны только на соответствующих платформах.

Зачем он нужен на практике:

- Чтение конфигурации из переменных окружения (12-factor приложения, секреты, настройки).
- Работа с файлами и директориями: обход, создание, удаление, переименование.
- Управление текущей рабочей директорией.
- Запуск внешних процессов (хотя для этого предпочтительнее `subprocess`).
- Получение информации о системе: число CPU, PID процесса, разделители путей.
- Генерация криптографически стойких случайных байтов (`os.urandom`).

Важно понимать соотношение `os` с соседними модулями:

- `os.path` — работа с путями как со строками (склейка, проверка существования). В современном коде часто заменяется на `pathlib`.
- `subprocess` — запуск внешних процессов (заменяет устаревшие `os.system`/`os.popen`).
- `shutil` — высокоуровневые операции (копирование деревьев, `rmtree`, `which`).
- `pathlib` — объектно-ориентированный API для путей.

```python
import os

# os — это, по сути, тонкая обёртка над системными вызовами POSIX/WinAPI.
# Многие функции напрямую соответствуют системным вызовам:
# os.open -> open(2), os.fork -> fork(2), os.stat -> stat(2).
print(os.name)  # 'posix' на Unix/macOS, 'nt' на Windows
```

## Ключевые концепции

**1. Переносимость и платформозависимость.** Часть `os` работает везде, часть — только на Unix (`os.fork`, `os.getppid`, расширенные права). Перед использованием платформозависимых вещей стоит проверять `os.name` или ловить `AttributeError`/`OSError`.

**2. `os.environ` — это снимок окружения процесса.** Это специальный mapping-объект (`os._Environ`), который при изменении синхронизируется с реальным окружением процесса через `putenv`/`unsetenv`. Дочерние процессы наследуют его копию.

**3. Файловые операции работают на «сыром» уровне.** `os.remove`, `os.mkdir`, `os.rmdir` оперируют отдельными файлами/директориями и кидают исключения при нарушении условий (директория не пуста, файл не существует). Высокоуровневые операции — в `shutil`.

**4. Дескрипторы и пути.** Многие функции `os` могут принимать как строковый путь, так и файловый дескриптор (целое число). Есть низкоуровневые `os.open`, `os.read`, `os.write`, `os.close` — это не то же самое, что встроенный `open()`.

**5. Исключения — это `OSError` и его подклассы.** В Python 3 множество мелких исключений объединены в иерархию `OSError`: `FileNotFoundError`, `FileExistsError`, `PermissionError`, `NotADirectoryError`, `IsADirectoryError`. Все они имеют атрибут `errno`.

**6. Права доступа в Unix-стиле.** Режим файла (`st_mode`) кодирует тип файла и права (rwx для owner/group/others) в виде битовой маски. `umask` влияет на права создаваемых файлов.

```python
import os

# Иерархия исключений: всё наследуется от OSError
try:
    os.remove("несуществующий.txt")
except FileNotFoundError as e:        # подкласс OSError
    print(e.errno, e.strerror)        # 2 No such file or directory
except OSError as e:
    print("Другая ошибка ОС:", e)
```

## Основные функции/классы/методы

### Переменные окружения: environ, getenv, setdefault

```python
import os

# os.environ — mapping-объект (тип os._Environ), ведёт себя как dict,
# но синхронизирован с окружением процесса.

# Чтение (KeyError, если ключа нет)
path = os.environ["PATH"]

# Безопасное чтение со значением по умолчанию
home = os.environ.get("HOME", "/tmp")

# os.getenv — то же самое, что environ.get, но более идиоматично
debug = os.getenv("DEBUG", "0")           # вернёт строку "0", если нет
db_url = os.getenv("DATABASE_URL")        # вернёт None, если нет

# Запись: попадает в окружение текущего процесса и наследуется потомками
os.environ["MY_VAR"] = "значение"         # под капотом вызывает putenv

# setdefault: установить, только если ключа ещё нет (атомарно для словаря)
os.environ.setdefault("LANG", "en_US.UTF-8")

# Удаление переменной
del os.environ["MY_VAR"]                   # под капотом вызывает unsetenv

# ВАЖНО: значения окружения — всегда строки. Числа/булевы нужно парсить вручную.
port = int(os.environ.get("PORT", "8000"))
is_prod = os.environ.get("ENV") == "production"
```

Особенность: `os.environ` читается один раз при старте интерпретатора. Если внешняя программа изменит окружение после запуска Python, это не отразится в `os.environ` (но `os.getenv` тоже читает из `os.environ`, не из ОС напрямую).

### Рабочая директория: getcwd, chdir

```python
import os

cwd = os.getcwd()           # текущая рабочая директория (абсолютный путь)
os.chdir("/tmp")            # сменить рабочую директорию (OSError, если нет)

# chdir влияет на ВЕСЬ процесс, а не на текущую функцию — это глобальное
# состояние. В многопоточном коде это источник трудноуловимых багов.

# Безопасный паттерн: вернуться в исходную директорию через try/finally
prev = os.getcwd()
try:
    os.chdir("/some/path")
    # ... работа ...
finally:
    os.chdir(prev)

# В современном коде лучше использовать contextlib.chdir (Python 3.11+)
import contextlib
with contextlib.chdir("/tmp"):
    print(os.getcwd())      # /tmp
# здесь cwd автоматически восстановлена
```

### Просмотр содержимого: listdir, scandir, walk

```python
import os

# listdir — список имён (str) в директории, без '.' и '..'
names = os.listdir(".")     # ['file.py', 'subdir', ...]

# scandir — БОЛЕЕ БЫСТРЫЙ способ: возвращает итератор DirEntry,
# которые кэшируют метаданные (тип, stat) из одного системного вызова.
# Это критично для больших директорий: не нужен отдельный stat на каждый файл.
with os.scandir(".") as it:               # контекстный менеджер закрывает дескриптор
    for entry in it:
        # entry.is_file()/is_dir() используют кэш — без лишних syscall
        if entry.is_file():
            print(entry.name, entry.stat().st_size)
        elif entry.is_dir(follow_symlinks=False):
            print("[dir]", entry.path)    # entry.path = полный путь

# walk — рекурсивный обход дерева директорий (сверху вниз по умолчанию)
for dirpath, dirnames, filenames in os.walk("."):
    # dirnames можно МУТИРОВАТЬ на месте, чтобы пропустить ветки обхода
    dirnames[:] = [d for d in dirnames if not d.startswith(".")]  # пропустить .git и т.п.
    for f in filenames:
        full = os.path.join(dirpath, f)
        print(full)

# topdown=False даёт обход снизу вверх — удобно для удаления дерева
for dirpath, dirnames, filenames in os.walk("./trash", topdown=False):
    for f in filenames:
        os.remove(os.path.join(dirpath, f))
    os.rmdir(dirpath)        # директория уже пуста к этому моменту
```

Внутри `os.walk` начиная с Python 3.5 реализован через `os.scandir`, поэтому он тоже быстрый.

### Создание директорий: mkdir, makedirs

```python
import os

# mkdir — создаёт ОДНУ директорию. Кидает FileExistsError, если есть,
# и FileNotFoundError, если родителя не существует.
os.mkdir("newdir")
os.mkdir("newdir", mode=0o755)            # с указанием прав

# makedirs — создаёт всю цепочку родительских директорий (mkdir -p)
os.makedirs("a/b/c/d")

# exist_ok=True — НЕ кидать ошибку, если конечная директория уже существует.
# Это идиоматичная замена проверке os.path.exists перед созданием
# (избегает гонки TOCTOU — time-of-check-to-time-of-use).
os.makedirs("logs/app", exist_ok=True)

# Подвох: exist_ok влияет ТОЛЬКО на конечный путь. Если по пути встретится
# файл (не директория) — всё равно будет ошибка.
```

### Удаление: remove, unlink, rmdir, removedirs

```python
import os

# remove и unlink — ПОЛНЫЕ синонимы, удаляют файл (не директорию).
# Имя unlink ближе к POSIX. Для директории кидают IsADirectoryError/PermissionError.
os.remove("file.txt")
os.unlink("file.txt")

# rmdir — удаляет ПУСТУЮ директорию. Если не пуста — OSError (ENOTEMPTY).
os.rmdir("emptydir")

# removedirs — удаляет директорию И пытается удалить пустых родителей вверх.
# Например removedirs("a/b/c") удалит c, потом b (если пуста), потом a (если пуста).
os.removedirs("a/b/c")

# Для рекурсивного удаления НЕпустого дерева — shutil.rmtree (НЕ в os):
import shutil
shutil.rmtree("nonempty_dir")
```

### Переименование и перемещение: rename, replace

```python
import os

# rename — переименовать/переместить файл или директорию.
# Поведение при существующем dst различается по платформам:
#   - Unix: молча перезаписывает (если dst — файл)
#   - Windows: кидает FileExistsError
os.rename("old.txt", "new.txt")

# replace — то же, но с ГАРАНТИРОВАННОЙ атомарной перезаписью dst
# на ВСЕХ платформах. Предпочтительнее для безопасной замены файлов.
os.replace("temp.txt", "config.txt")

# Классический паттерн атомарной записи: пишем во временный файл,
# затем атомарно подменяем. Читатели никогда не увидят «недописанный» файл.
import tempfile
fd, tmp = tempfile.mkstemp(dir=".")
with os.fdopen(fd, "w") as f:
    f.write("новые данные")
os.replace(tmp, "data.txt")   # атомарная подмена

# ВАЖНО: rename/replace работают атомарно только в пределах одной файловой
# системы. Между разными ФС они упадут (EXDEV) — там нужен shutil.move.
```

### Запуск процессов: system, popen и почему subprocess лучше

```python
import os

# os.system — запускает команду через системный shell, возвращает код выхода
# (на Unix это «raw» статус из wait, см. os.waitstatus_to_exitcode).
code = os.system("ls -la")   # вывод идёт напрямую в stdout процесса

# os.popen — открывает pipe к команде, возвращает файловый объект
output = os.popen("ls -la").read()

# ПОЧЕМУ subprocess ПРЕДПОЧТИТЕЛЬНЕЕ:
# 1) Безопасность: os.system/os.popen передают строку в shell -> shell injection.
#    Если в команду попадёт пользовательский ввод, это дыра в безопасности.
# 2) Контроль: subprocess даёт раздельный доступ к stdin/stdout/stderr,
#    кодам возврата, таймаутам, окружению, рабочей директории.
# 3) Без shell: subprocess по умолчанию запускает без shell (список аргументов),
#    что устраняет проблемы экранирования и инъекций.
# 4) os.system возвращает закодированный статус, а не чистый код выхода.

# Современная замена:
import subprocess
result = subprocess.run(
    ["ls", "-la"],            # список аргументов — НЕ строка, нет shell
    capture_output=True,
    text=True,
    check=True,              # кинуть CalledProcessError при ненулевом коде
    timeout=10,
)
print(result.stdout)

# Документация прямо рекомендует subprocess вместо os.system/os.popen.
```

### Идентификаторы процессов: getpid, getppid, fork

```python
import os

pid = os.getpid()            # PID текущего процесса
ppid = os.getppid()          # PID родительского процесса (Unix; на Windows тоже есть)

# os.fork() — ТОЛЬКО Unix. Создаёт копию процесса (системный вызов fork(2)).
# Возвращает 0 в дочернем процессе и PID ребёнка в родительском.
if hasattr(os, "fork"):      # проверка доступности (нет на Windows)
    pid = os.fork()
    if pid == 0:
        print("Я дочерний процесс, мой PID:", os.getpid())
        os._exit(0)          # в дочернем процессе выходить через os._exit
    else:
        print("Я родитель, PID ребёнка:", pid)
        os.waitpid(pid, 0)   # дождаться завершения ребёнка (иначе зомби)

# На практике вместо «голого» fork используют multiprocessing/subprocess.
# fork опасен в многопоточных программах и с открытыми ресурсами (БД, локи).
```

### Права доступа: chmod, stat, st_mode, access, umask

```python
import os
import stat

# os.chmod — изменить права (режим) файла. Режим — восьмеричное число.
os.chmod("script.sh", 0o755)   # rwxr-xr-x

# os.stat — получить метаданные файла (структура stat_result)
st = os.stat("file.txt")
print(st.st_size)              # размер в байтах
print(st.st_mode)              # тип файла + права (битовая маска)
print(st.st_mtime)             # время изменения (float, секунды эпохи)
print(st.st_uid, st.st_gid)    # владелец и группа (Unix)

# Модуль stat помогает разбирать st_mode:
mode = st.st_mode
print(stat.S_ISDIR(mode))      # это директория?
print(stat.S_ISREG(mode))      # это обычный файл?
print(oct(stat.S_IMODE(mode))) # только биты прав, например 0o644
print(stat.filemode(mode))     # человекочитаемо: '-rw-r--r--'

# os.access — проверить доступность пути для текущего процесса.
# ВНИМАНИЕ: использует реальные (а не эффективные) UID/GID и подвержен
# гонке TOCTOU. Часто лучше просто попытаться открыть файл и поймать ошибку.
print(os.access("file.txt", os.R_OK))            # читаем?
print(os.access("file.txt", os.W_OK | os.X_OK)) # пишем И исполняем?

# os.umask — маска, которая ВЫЧИТАЕТСЯ из прав при создании файлов/директорий.
# Например, при umask 0o022 файл, создаваемый с 0o666, получит 0o644.
old = os.umask(0o077)          # установить новую маску, вернуть старую
# ... создаём файлы только с правами для владельца ...
os.umask(old)                  # восстановить (umask — глобальное состояние процесса)
```

### Работа с путями: os.path (обзор)

`os.path` — подмодуль для работы с путями как со строками. В современном коде его часто заменяет объектный `pathlib`, но на собеседовании знать базовые функции обязательно.

```python
import os

# join — переносимая склейка путей (использует правильный разделитель)
p = os.path.join("dir", "sub", "file.txt")   # 'dir/sub/file.txt' (или с \ на Windows)

# Подвох join: абсолютный сегмент «сбрасывает» предыдущие
os.path.join("/a", "/b")     # '/b', а не '/a/b'

# Проверки существования и типа
os.path.exists(p)            # путь существует (файл/директория/симлинк)?
os.path.isfile(p)            # это обычный файл?
os.path.isdir(p)             # это директория?
os.path.islink(p)            # это символическая ссылка?

# Разбор путей
os.path.basename("/a/b/c.txt")   # 'c.txt'
os.path.dirname("/a/b/c.txt")    # '/a/b'
os.path.splitext("c.txt")        # ('c', '.txt')
os.path.split("/a/b/c.txt")      # ('/a/b', 'c.txt')
os.path.abspath("rel")           # абсолютный путь от cwd
os.path.expanduser("~/cfg")      # подставить домашнюю директорию
os.path.normpath("a/./b/../c")   # 'a/c'

# Эквивалент через pathlib (современный стиль):
from pathlib import Path
p = Path("dir") / "sub" / "file.txt"
p.exists(), p.is_file(), p.name, p.suffix, p.parent
```

### Прочие полезные функции: urandom, cpu_count, sep, linesep

```python
import os

# os.urandom — криптографически стойкие случайные байты от ОС
# (/dev/urandom на Unix, CryptGenRandom на Windows). Используется для токенов,
# соли, ключей. Для удобства есть модуль secrets (поверх os.urandom).
token = os.urandom(16)             # 16 случайных байт (bytes)
import secrets
hex_token = secrets.token_hex(16)  # рекомендуемый способ для токенов

# os.cpu_count — число логических CPU (или None, если не определить).
# Полезно для выбора размера пула воркеров.
n = os.cpu_count() or 1
# Для числа РЕАЛЬНО доступных CPU (с учётом affinity) — len(os.sched_getaffinity(0))
# или os.process_cpu_count() (Python 3.13+).

# os.sep — разделитель компонентов пути ('/' на Unix, '\\' на Windows)
print(os.sep)        # '/'
print(os.altsep)     # альтернативный разделитель ('/' на Windows, None на Unix)
print(os.extsep)     # '.' — разделитель расширения
print(os.pathsep)    # разделитель в PATH (':' на Unix, ';' на Windows)

# os.linesep — разделитель строк ОС ('\n' на Unix, '\r\n' на Windows).
# ВАЖНО: НЕ использовать при записи в текстовый режим — там Python сам
# делает перевод. linesep нужен только для бинарного режима/низкого уровня.
print(repr(os.linesep))   # '\n'
```

## Частые вопросы на собеседовании

**Вопрос:** В чём разница между `os.environ` и `os.getenv`?

**Ответ:** `os.environ` — это mapping-объект (словарь), синхронизированный с окружением процесса. Доступ через `os.environ["KEY"]` кидает `KeyError`, если ключа нет. `os.getenv("KEY", default)` — это эквивалент `os.environ.get("KEY", default)`, возвращает `None` (или указанный default) вместо исключения. Для записи используется только `os.environ` (`os.environ["KEY"] = ...`), у `getenv` записи нет. Все значения — всегда строки.

---

**Вопрос:** Чем `os.scandir` лучше `os.listdir`?

**Ответ:** `listdir` возвращает только имена, и если нужны метаданные (размер, тип), на каждый файл придётся делать отдельный `os.stat` — это лишние системные вызовы. `scandir` возвращает итератор объектов `DirEntry`, которые кэшируют тип и stat-информацию, полученную за один проход директории. На больших директориях это значительно быстрее. Поэтому `os.walk` внутри тоже реализован через `scandir`. Минус: `scandir` возвращает контекстный менеджер, который нужно закрывать (держит дескриптор директории).

---

**Вопрос:** Почему `subprocess` предпочтительнее `os.system` и `os.popen`?

**Ответ:** Три главные причины. (1) Безопасность: `os.system`/`os.popen` исполняют строку через shell, что открывает shell injection при попадании пользовательского ввода. `subprocess.run(["cmd", arg])` принимает список аргументов и не использует shell по умолчанию. (2) Контроль: `subprocess` даёт раздельный доступ к stdin/stdout/stderr, таймауты, коды возврата, кастомное окружение и cwd. (3) Корректность: `os.system` возвращает закодированный статус (на Unix нужно ещё разбирать через `waitstatus_to_exitcode`), а `subprocess` даёт чистый returncode. Документация прямо рекомендует `subprocess`.

---

**Вопрос:** В чём разница между `os.rename` и `os.replace`?

**Ответ:** Обе переименовывают/перемещают файл. Разница в поведении, когда целевой путь уже существует: `os.rename` на Unix молча перезаписывает, а на Windows кидает `FileExistsError` — поведение непереносимо. `os.replace` гарантирует атомарную перезапись на всех платформах. Поэтому для безопасной атомарной подмены файла (паттерн «запиши во временный, потом подмени») всегда используют `os.replace`. Важно: атомарность работает только в пределах одной файловой системы; между разными ФС нужен `shutil.move`.

---

**Вопрос:** Что делает параметр `exist_ok` в `os.makedirs`?

**Ответ:** Если `exist_ok=True`, функция не кидает `FileExistsError`, когда конечная директория уже существует. Это идиоматичная замена антипаттерну «проверить `os.path.exists`, потом создать», который подвержен гонке TOCTOU (между проверкой и созданием директорию может создать другой процесс). Подвох: `exist_ok` влияет только на конечный путь; если по пути встретится обычный файл вместо директории, всё равно будет ошибка.

---

**Вопрос:** Что такое `st_mode` и как извлечь из него права доступа?

**Ответ:** `st_mode` — это поле структуры `stat_result`, битовая маска, кодирующая одновременно тип файла (обычный/директория/симлинк/устройство) и права доступа (rwx для владельца/группы/остальных) плюс спецбиты (setuid, setgid, sticky). Разбирают его через модуль `stat`: `stat.S_ISDIR(mode)`/`stat.S_ISREG(mode)` определяют тип, `stat.S_IMODE(mode)` извлекает только биты прав (например `0o644`), `stat.filemode(mode)` даёт человекочитаемую строку вроде `-rw-r--r--`.

---

**Вопрос:** Что такое `umask` и как он влияет на права создаваемых файлов?

**Ответ:** `umask` — маска процесса, биты которой *снимаются* с прав при создании новых файлов и директорий. Итоговые права = запрошенные `& ~umask`. Например, при `umask 0o022` файл, создаваемый с режимом `0o666`, получит `0o644` (убраны write-биты для группы и остальных). `os.umask(new)` устанавливает новую маску и возвращает старую. Это глобальное состояние процесса, поэтому менять его нужно осторожно и восстанавливать. Именно из-за umask переданный в `os.mkdir(path, mode)` режим может не совпасть с фактическим.

---

**Вопрос:** Почему `os.access` считается небезопасным для проверки прав?

**Ответ:** Две причины. (1) Гонка TOCTOU: между проверкой `os.access` и реальным открытием файла права или сам файл могут измениться, поэтому проверка ничего не гарантирует. (2) `os.access` использует реальные UID/GID, а не эффективные, что в setuid-программах даёт неверный результат. Идиоматичный подход в Python — «проще попросить прощения, чем разрешения» (EAFP): просто попытаться открыть файл и поймать `PermissionError`/`FileNotFoundError`.

---

**Вопрос:** Чем опасен `os.chdir` в многопоточном или библиотечном коде?

**Ответ:** Рабочая директория — это глобальное состояние всего процесса, а не отдельного потока. Если один поток вызовет `os.chdir`, это повлияет на все остальные потоки, которые работают с относительными путями, — источник недетерминированных багов. В библиотеках вообще не стоит менять cwd. Безопасные подходы: работать с абсолютными путями, или использовать `contextlib.chdir` (Python 3.11+) / `try/finally` для гарантированного восстановления, или передавать `cwd=` в `subprocess.run`.

---

**Вопрос:** Как `os.walk` позволяет пропускать определённые директории?

**Ответ:** `os.walk` на каждой итерации отдаёт кортеж `(dirpath, dirnames, filenames)`, где `dirnames` — изменяемый список. При `topdown=True` (по умолчанию) можно мутировать `dirnames` *на месте* (через срез `dirnames[:] = ...`), и `walk` не будет заходить в удалённые из списка поддиректории. Это стандартный приём, чтобы пропустить, например, `.git` или `node_modules`. Важно менять список именно на месте, а не переприсваивать переменную — иначе walk не увидит изменений.

---

**Вопрос:** Что такое `os.urandom` и когда его использовать?

**Ответ:** `os.urandom(n)` возвращает `n` криптографически стойких случайных байтов, полученных от источника энтропии ОС (`/dev/urandom` на Unix, CryptGenRandom на Windows). Используется для генерации токенов, солей, ключей, CSRF-токенов — там, где `random` (псевдослучайный, предсказуемый) непригоден. На практике для токенов рекомендуют модуль `secrets` (`secrets.token_hex`, `secrets.token_urlsafe`), который построен поверх `os.urandom` и даёт удобный API.

---

**Вопрос:** В чём разница между `os.remove`, `os.unlink`, `os.rmdir` и `shutil.rmtree`?

**Ответ:** `os.remove` и `os.unlink` — полные синонимы, удаляют один файл (не директорию). `os.rmdir` удаляет только *пустую* директорию (иначе `OSError`). `os.removedirs` удаляет директорию и пытается удалить ставших пустыми родителей. Для рекурсивного удаления непустого дерева в `os` функции нет — нужен `shutil.rmtree`, который удаляет всё содержимое рекурсивно.

## Подводные камни (gotchas)

```python
import os

# 1) Значения окружения — ВСЕГДА строки, даже числа и булевы
os.environ["PORT"] = "8080"
# port = os.environ["PORT"] + 1     # TypeError: str + int
port = int(os.environ["PORT"]) + 1  # правильно

# 2) os.path.join и абсолютный сегмент: предыдущие части СБРАСЫВАЮТСЯ
print(os.path.join("/base", "/etc/passwd"))   # '/etc/passwd' — НЕ '/base/etc/passwd'!
# Опасно при склейке с пользовательским вводом — потенциальный path traversal.

# 3) os.listdir даёт ОТНОСИТЕЛЬНЫЕ имена, а не полные пути
for name in os.listdir("/some/dir"):
    # os.stat(name)                 # упадёт: name — это просто имя файла
    os.stat(os.path.join("/some/dir", name))   # правильно

# 4) os.rename перезаписывает по-разному на Unix и Windows — используйте os.replace

# 5) os.chdir — глобальное состояние процесса, влияет на все потоки

# 6) os.umask «съедает» биты у mode в mkdir/chmod-при-создании:
os.mkdir("d", 0o777)
# Реальные права могут оказаться 0o755 из-за umask 0o022 — это не баг.

# 7) os.system возвращает НЕ код выхода напрямую (на Unix):
status = os.system("exit 3")
code = os.waitstatus_to_exitcode(status)   # вот так получают реальный код

# 8) scandir нужно закрывать — используйте with, иначе утечёт дескриптор
with os.scandir(".") as it:
    files = [e.name for e in it]

# 9) os.environ изменяется в текущем процессе, но menять окружение УЖЕ
# запущенного процесса извне нельзя; потомки получают КОПИЮ на момент запуска.

# 10) os.linesep НЕ нужен при записи в текстовом режиме — будет двойной \r\r\n
# на Windows. В текстовом режиме всегда пишите '\n', Python переведёт сам.
```

## Лучшие практики

- **Читайте окружение через `os.getenv` с дефолтами и явным парсингом типов.** Все значения — строки; оборачивайте в `int()`/булевую логику. Для конфигурации рассматривайте `pydantic-settings` или `python-dotenv`.
- **Не используйте `os.system`/`os.popen`.** Всегда `subprocess.run` со списком аргументов, `check=True`, `timeout=...`. Избегайте `shell=True`.
- **Используйте `os.replace` для атомарной подмены файлов** (паттерн temp-file + replace), а не `os.rename`.
- **Создавайте директории через `os.makedirs(path, exist_ok=True)`** вместо проверки `exists` + create — это избавляет от гонки TOCTOU.
- **Не меняйте cwd через `os.chdir`** в библиотеках и многопоточном коде. Работайте с абсолютными путями или `contextlib.chdir` / передавайте `cwd=` в subprocess.
- **Не проверяйте права через `os.access`** — следуйте EAFP: пробуйте операцию и ловите `OSError`/`PermissionError`.
- **Для больших директорий используйте `os.scandir`** (с `with`) или `os.walk`, а не `listdir` + множественные `stat`.
- **Для путей предпочитайте `pathlib`** в новом коде; `os.path.join` помните для совместимости и интервью.
- **Для случайности в безопасности** используйте `secrets` (поверх `os.urandom`), а не `random`.
- **Ловите конкретные подклассы `OSError`** (`FileNotFoundError`, `PermissionError`, `FileExistsError`), а не голый `except Exception`.
- **Проверяйте платформозависимые функции** (`os.fork`, affinity) через `hasattr(os, "fork")`.

os.path.exists/isfile/isdir/islink(p)
