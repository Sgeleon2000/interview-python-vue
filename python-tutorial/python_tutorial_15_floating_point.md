# Глава 15. Floating Point Arithmetic: Issues and Limitations — конспект и вопросы

> **ВАЖНАЯ тема для собеседований.** Вопрос «почему `0.1 + 0.2 != 0.3`?» — классика. Здесь разбираемся глубоко: как устроен IEEE 754, почему дроби не представимы точно, как сравнивать float'ы, чем спасает `Decimal`/`Fraction`, как ведёт себя `round`, что такое `nan`/`inf`.

## О чём глава

Числа с плавающей точкой (`float`) в Python — это двоичные дроби двойной точности (IEEE 754 binary64, он же C `double`). Из-за двоичного представления **большинство десятичных дробей нельзя записать точно**, отсюда «странности» арифметики. Это не баг Python, а фундаментальное свойство всех языков с IEEE 754 (C, Java, JavaScript и т. д.). Глава объясняет причины и даёт инструменты, как с этим жить.

---

## Ключевые концепции (с примерами и выводом)

### 1. Канонический пример: 0.1 + 0.2

```python
print(0.1 + 0.2)            # => 0.30000000000000004
print(0.1 + 0.2 == 0.3)    # => False
```

Это пугает новичков, но абсолютно нормально. Причина — ниже.

### 2. repr показывает «настоящее» значение

```python
print(0.1)                 # => 0.1   (короткое, удобное представление)
print(f"{0.1:.17f}")       # => 0.10000000000000000555  (что хранится на самом деле)
from decimal import Decimal
print(Decimal(0.1))        # => 0.1000000000000000055511151231257827021181583404541015625
```

`0.1` не равно ровно 1/10 — это ближайшее представимое двоичное число. `repr()` печатает **кратчайшую** строку, которая при обратном чтении даёт то же самое число.

### 3. Сравнение через math.isclose

```python
import math
print(math.isclose(0.1 + 0.2, 0.3))   # => True
```

### 4. Точные альтернативы — Decimal и Fraction

```python
from decimal import Decimal
from fractions import Fraction

print(Decimal("0.1") + Decimal("0.2") == Decimal("0.3"))   # => True
print(Fraction(1, 10) + Fraction(2, 10) == Fraction(3, 10)) # => True
```

## Подробный разбор по темам

### Почему дробь нельзя представить точно

Компьютер хранит числа в **двоичной** системе. Десятичная дробь представима точно, только если её знаменатель — степень двойки (1/2, 1/4, 1/8, ...). Число `0.1 = 1/10` имеет в знаменателе множитель 5, поэтому в двоичной системе это **бесконечная периодическая дробь**:

```text
0.1 (десятичное) = 0.0001100110011001100110011... (двоичное, период повторяется)
```

Аналогично 1/3 = 0.333... бесконечна в десятичной системе. Чтобы поместиться в 64 бита, дробь **обрезается** до ближайшего представимого значения — отсюда крошечная погрешность.

```python
print(1/3)                 # => 0.3333333333333333
print(0.1 == 1/10)         # => False  (0.1 хранится приближённо)
print(0.5 == 1/2)          # => True   (1/2 = степень двойки, точно)
print(0.25 + 0.75 == 1.0)  # => True   (обе — суммы степеней двойки)
```

### Представление IEEE 754 (binary64 / double)

`float` в CPython — это 64-битное число двойной точности:

| Поле | Бит | Назначение |
|------|-----|------------|
| Знак (sign) | 1 | + или − |
| Порядок (exponent) | 11 | масштаб (степень двойки), смещение 1023 |
| Мантисса (significand/fraction) | 52 | значащие цифры (плюс неявный ведущий бит) |

Формула: `(-1)^sign × 1.mantissa × 2^(exponent - 1023)`.

Следствия:
- Около **15–17 значащих десятичных цифр** точности.
- Чем больше число, тем больше «шаг» между соседними представимыми значениями (точность относительная, не абсолютная).

```python
import sys
print(sys.float_info.dig)        # => 15   (гарантированные значащие цифры)
print(sys.float_info.epsilon)    # => 2.220446049250313e-16
print(sys.float_info.max)        # => 1.7976931348623157e+308
print(sys.float_info.min)        # => 2.2250738585072014e-308
```

### repr vs реальное значение и round-trip

С Python 3.1 `repr(float)` выводит кратчайшую строку, гарантирующую корректный round-trip: `float(repr(x)) == x`. Поэтому `repr(0.1)` это `'0.1'`, хотя хранится более длинное значение. Чтобы увидеть «истину» — `Decimal(x)` или формат `:.17g`.

```python
x = 0.1 + 0.2
print(repr(x))                 # => '0.30000000000000004'
print(float(repr(x)) == x)     # => True  (round-trip сохранён)
print(f"{x:.17g}")             # => 0.30000000000000004
```

### Поведение round() — банковское округление (round half to even)

`round()` в Python 3 использует **«округление половины к чётному»** (banker's rounding, IEEE 754). Это уменьшает систематический сдвиг при массовом округлении.

```python
print(round(0.5))    # => 0   (к чётному 0, а не 1!)
print(round(1.5))    # => 2   (к чётному 2)
print(round(2.5))    # => 2   (к чётному 2, а не 3!)
print(round(3.5))    # => 4
print(round(-0.5))   # => 0
```

Дополнительная тонкость: из-за неточного представления «ровно .5» иногда им не является:

```python
print(round(2.675, 2))   # => 2.67  (а ждали 2.68!)
# Потому что 2.675 хранится как 2.67499999999999982...
from decimal import Decimal
print(Decimal(2.675))    # => 2.67499999999999982236431605997495353221893310546875
```

Для предсказуемого округления денег используйте `Decimal` с явным режимом:

```python
from decimal import Decimal, ROUND_HALF_UP
print(Decimal("2.675").quantize(Decimal("0.01"), rounding=ROUND_HALF_UP))  # => 2.68
```

### math.isclose для сравнения

**Никогда не сравнивайте float через `==`.** Используйте `math.isclose(a, b, rel_tol=..., abs_tol=...)`:

```python
import math
print(math.isclose(0.1 + 0.2, 0.3))                # => True
print(math.isclose(1e-10, 0.0))                    # => False  (rel_tol не ловит около нуля)
print(math.isclose(1e-10, 0.0, abs_tol=1e-9))      # => True   (нужен abs_tol около нуля)
```

- `rel_tol` (по умолчанию `1e-09`) — относительный допуск, хорош для крупных чисел.
- `abs_tol` (по умолчанию `0.0`) — абсолютный допуск, **обязателен при сравнении с нулём**.

### Потеря точности при суммировании

Сложение многих float'ов накапливает погрешность. `math.fsum` ведёт точный учёт:

```python
import math
data = [0.1] * 10
print(sum(data))        # => 0.9999999999999999  (накопленная ошибка)
print(math.fsum(data))  # => 1.0                  (точное суммирование)
```

Также теряется точность при сложении сильно разных по величине чисел (поглощение малого большим):

```python
print(1e16 + 1)         # => 1e+16   (единица «потерялась»)
print(1e16 + 1 == 1e16) # => True
```

### Decimal и Fraction как решения

```python
from decimal import Decimal, getcontext
from fractions import Fraction

# Decimal — десятичная арифметика с заданной точностью (деньги, финансы)
print(Decimal("0.1") + Decimal("0.2"))   # => 0.3
getcontext().prec = 6
print(Decimal(1) / Decimal(7))           # => 0.142857  (контролируемая точность)

# ВАЖНО: создавайте Decimal из СТРОКИ, не из float!
print(Decimal(0.1))      # => 0.1000000000000000055511151231257827021181583404541015625
print(Decimal("0.1"))    # => 0.1  (правильно)

# Fraction — точные рациональные дроби
print(Fraction(1, 3) + Fraction(1, 6))   # => 1/2  (точно)
print(Fraction("0.1") + Fraction("0.2")) # => 3/10
print(float(Fraction(3, 10)))            # => 0.3
```

Сравнение подходов:

| Тип | Точность | Когда применять | Минус |
|-----|----------|------------------|-------|
| `float` | приближённая (IEEE 754) | наука, графика, скорость | погрешности |
| `Decimal` | точная десятичная | деньги, финансы, бухгалтерия | медленнее |
| `Fraction` | точная рациональная | символьные/точные дроби | не для иррациональных |

### format для отображения

Округление при **показе** не меняет хранимое значение — это часто всё, что нужно:

```python
x = 0.1 + 0.2
print(f"{x:.2f}")     # => 0.30
print(f"{x:.10f}")    # => 0.3000000000000
print("{:.2%}".format(0.1234))  # => 12.34%
print(format(1234567.891, ",.2f"))  # => 1,234,567.89  (разделители тысяч)
```

### nan и inf

```python
inf = float("inf")
ninf = float("-inf")
nan = float("nan")

print(inf > 1e308)        # => True
print(inf + 1)            # => inf
print(inf - inf)          # => nan  (неопределённость)
print(1.0 / inf)          # => 0.0

import math
print(math.isinf(inf))    # => True
print(math.isnan(nan))    # => True
print(math.inf, math.nan) # => inf nan
```

### Почему nan != nan

`nan` (Not a Number) по стандарту IEEE 754 **не равен ничему, включая себя**:

```python
nan = float("nan")
print(nan == nan)         # => False  (!)
print(nan != nan)         # => True
print(nan < 1, nan > 1)   # => False False  (любые сравнения дают False)

import math
print(math.isnan(nan))    # => True   единственно верный способ проверки
```

Практическое следствие: проверяйте `nan` **только** через `math.isnan()`, а не `x == float('nan')`. Любопытная деталь: `[nan].count(nan)` вернёт 1 и `nan in [nan]` вернёт `True`, потому что списки/словари сначала проверяют **идентичность** (`is`), а уже потом равенство.

```python
nan = float("nan")
print(nan in [nan])       # => True  (сравнение по is срабатывает раньше ==)
```

## Частые вопросы на собеседовании (Q/A)

**Q: Почему `0.1 + 0.2 == 0.3` возвращает False?**
A: `0.1`, `0.2`, `0.3` нельзя точно представить в двоичном IEEE 754 (их знаменатели не степени двойки). Они хранятся как ближайшие приближения; сумма приближений `0.1` и `0.2` не совпадает бит-в-бит с приближением `0.3`. Результат `0.30000000000000004`. Это поведение всех языков с IEEE 754.

**Q: Как правильно сравнивать два float?**
A: Через `math.isclose(a, b, rel_tol=1e-9, abs_tol=...)`, а не `==`. При сравнении с нулём обязательно задавать `abs_tol`.

**Q: Сколько байт занимает Python float и какая точность?**
A: 64 бита (IEEE 754 double): 1 знак, 11 порядок, 52 мантисса. Около 15–17 значащих десятичных цифр.

**Q: Чему равен `round(2.5)` и почему?**
A: `2`. Python использует банковское округление (round half to even) — половина округляется к ближайшему чётному. `round(0.5)=0`, `round(1.5)=2`, `round(2.5)=2`.

**Q: Почему `round(2.675, 2)` даёт 2.67, а не 2.68?**
A: `2.675` хранится как `2.67499999...`, то есть фактически меньше «ровно .675», поэтому округляется вниз. Для денег — `Decimal("2.675").quantize(Decimal("0.01"), ROUND_HALF_UP)`.

**Q: Чем `Decimal` отличается от `float` и когда его брать?**
A: `Decimal` — точная десятичная арифметика с контролируемой точностью и режимами округления. Берут для денег/финансов. `float` быстрее, для науки/графики. Создавать `Decimal` нужно из строки: `Decimal("0.1")`, иначе унаследует неточность float.

**Q: Что вернёт `float('nan') == float('nan')`?**
A: `False`. По IEEE 754 `nan` не равен ничему, включая себя. Проверять — `math.isnan(x)`.

**Q: Как точно просуммировать список float'ов?**
A: `math.fsum(iterable)` — ведёт компенсированное суммирование без накопления ошибки, в отличие от обычного `sum`.

**Q: Является ли это багом Python?**
A: Нет. Это свойство стандарта IEEE 754, общее для C, Java, JS, C# и других. Python лишь честно показывает значения.

## Подводные камни (gotchas)

- **`==` на float** — почти всегда ошибка. Используйте `math.isclose`.
- **`Decimal(0.1)`** наследует неточность float. Правильно — `Decimal("0.1")` из строки.
- **`round()` ≠ школьное округление.** Банковское (half-to-even) + неточность хранения дают неожиданные `round(2.5)==2`, `round(2.675,2)==2.67`.
- **Накопление ошибки** при `sum` многих/разномасштабных чисел. Спасает `math.fsum`.
- **Поглощение малого большим:** `1e16 + 1 == 1e16` это `True`.
- **`nan` ломает сортировку и сравнения:** любые `<`, `>`, `==` с `nan` дают `False`; сортировка списка с `nan` может дать неверный порядок.
- **`nan in [nan]` → True** из-за проверки `is` перед `==` в контейнерах — контринтуитивно.
- **`inf - inf` и `0 * inf` → nan** (неопределённости).

## Шпаргалка

```python
import math
from decimal import Decimal, ROUND_HALF_UP
from fractions import Fraction

# Сравнение float
math.isclose(a, b, rel_tol=1e-9, abs_tol=1e-12)

# Точное суммирование
math.fsum([0.1] * 10)                 # => 1.0

# Проверки спецзначений
math.isnan(x); math.isinf(x); math.isfinite(x)

# Деньги: Decimal из строки + явное округление
Decimal("19.99") + Decimal("0.01")    # => 20.00
Decimal("2.675").quantize(Decimal("0.01"), ROUND_HALF_UP)  # => 2.68

# Точные дроби
Fraction(1, 3) + Fraction(1, 6)       # => Fraction(1, 2)

# Показать «истинное» значение float
Decimal(0.1)          # полное двоичное приближение
f"{0.1:.17g}"         # 0.10000000000000001

# Форматирование для вывода
f"{0.1 + 0.2:.2f}"    # => '0.30'
```

**Запомнить главное:**
1. `float` = IEEE 754 double, многие десятичные дроби неточны → `0.1+0.2 != 0.3`.
2. Сравнивай через `math.isclose`, не `==`.
3. `round` — банковское (half-to-even); деньги — через `Decimal`.
4. `nan != nan`; проверка только `math.isnan`.
5. Точность нужна → `Decimal` (десятичная) или `Fraction` (рациональная).
