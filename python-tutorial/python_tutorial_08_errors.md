# Глава 8. Ошибки и исключения — конспект и вопросы

## О чём глава

Глава официального туториала Python посвящена двум видам ошибок и механизму их обработки:

- **Синтаксические ошибки** (`SyntaxError`) — обнаруживаются на этапе разбора (парсинга) кода, до выполнения.
- **Исключения** (exceptions) — возникают во время выполнения уже синтаксически корректного кода.

Python использует исключения как основной механизм сигнализации об ошибках и нештатных ситуациях. Понимание модели «попробуй — поймай — подчисти» (`try/except/else/finally`), иерархии исключений, цепочек (`raise from`), пользовательских классов и групп исключений (`ExceptionGroup`, `except*`) — обязательная база для уровня middle/senior.

Ключевая идиома Python — **EAFP** («проще попросить прощения, чем разрешения»): пытаемся выполнить операцию и ловим исключение, вместо того чтобы заранее проверять все условия (**LBYL**).

## Ключевые концепции (с примерами и выводом)

### Синтаксическая ошибка vs исключение

```python
# Синтаксическая ошибка — код даже не запустится.
# while True print("hi")   # SyntaxError: invalid syntax (нет двоеточия)

# Исключение — код синтаксически верный, но падает во время выполнения.
print(10 / 0)
# => Traceback (most recent call last):
# =>   File "<stdin>", line 1, in <module>
# => ZeroDivisionError: division by zero
```

### Базовый try/except

```python
while True:
    try:
        x = int(input("Введите число: "))
        break
    except ValueError:
        print("Это не число, попробуйте снова")
# Ввод "abc" => "Это не число, попробуйте снова"
# Ввод "42"  => цикл прерывается
```

### Перехват нескольких исключений

```python
try:
    risky()
except (TypeError, ValueError) as exc:   # один блок на несколько типов
    print(f"Ошибка данных: {exc}")
except KeyError as exc:                   # отдельный блок
    print(f"Нет ключа: {exc}")
```

### else и finally

```python
def read_value(d, key):
    try:
        value = d[key]
    except KeyError:
        print("ключа нет")
        return None
    else:
        # выполняется ТОЛЬКО если try прошёл без исключения
        print("успешно прочитали")
        return value
    finally:
        # выполняется ВСЕГДА: и при успехе, и при исключении, и при return
        print("finally: подчистка")

read_value({"a": 1}, "a")
# => успешно прочитали
# => finally: подчистка   (выполнился перед возвратом значения)
```

### raise и повторный проброс

```python
def validate(age):
    if age < 0:
        raise ValueError(f"возраст не может быть отрицательным: {age}")
    return age

try:
    validate(-5)
except ValueError as exc:
    print("поймали:", exc)
    # raise          # «голый» raise пробросит то же исключение дальше
# => поймали: возраст не может быть отрицательным: -5
```

### Цепочки исключений (raise from)

```python
def load_config(path):
    try:
        open(path)
    except FileNotFoundError as exc:
        raise RuntimeError("конфиг не загружен") from exc

try:
    load_config("/no/such/file")
except RuntimeError as e:
    print(type(e).__name__, "->", type(e.__cause__).__name__)
# => RuntimeError -> FileNotFoundError
```

### Группы исключений и except* (Python 3.11+)

```python
try:
    raise ExceptionGroup("несколько проблем", [
        ValueError("плохое значение"),
        TypeError("плохой тип"),
    ])
except* ValueError as eg:
    print("поймали ValueError:", eg.exceptions)
except* TypeError as eg:
    print("поймали TypeError:", eg.exceptions)
# => поймали ValueError: (ValueError('плохое значение'),)
# => поймали TypeError: (TypeError('плохой тип'),)
```

### Заметки add_note (Python 3.11+)

```python
try:
    raise ValueError("основная ошибка")
except ValueError as exc:
    exc.add_note("контекст: при обработке записи #42")
    # raise  # в трейсбэке заметка покажется отдельной строкой
```

## Подробный разбор по темам

### 1. Синтаксические ошибки vs исключения

**Синтаксическая ошибка** (`SyntaxError`) возникает на этапе компиляции исходника в байт-код. Парсер указывает на проблемное место «стрелочкой». Пока есть хоть одна синтаксическая ошибка, **ни одна строка модуля не выполнится**.

```python
# def f(:   # SyntaxError — модуль не запустится целиком
```

**Исключение** — это объект, который возникает во время выполнения корректного кода: деление на ноль, обращение к несуществующему ключу, неверный тип и т.д. Исключения можно перехватывать и обрабатывать; синтаксические ошибки в импортируемом модуле — нет (они валят импорт).

Все исключения — экземпляры классов, наследующихся от `BaseException`. Большинство «обычных» — от `Exception`.

### 2. Иерархия исключений и порядок except

Корень иерархии — `BaseException`. От него:

```
BaseException
 ├── SystemExit          # sys.exit()
 ├── KeyboardInterrupt   # Ctrl+C
 ├── GeneratorExit
 └── Exception           # ← почти всё «прикладное» здесь
      ├── ArithmeticError
      │    └── ZeroDivisionError
      ├── LookupError
      │    ├── KeyError
      │    └── IndexError
      ├── ValueError
      ├── TypeError
      ├── OSError
      │    ├── FileNotFoundError
      │    └── PermissionError
      └── ...
```

`SystemExit`, `KeyboardInterrupt`, `GeneratorExit` наследуются от `BaseException`, **а не от `Exception`**. Поэтому `except Exception` их НЕ ловит — и это правильно: вы не хотите случайно проглотить Ctrl+C или `sys.exit()`.

**Порядок блоков `except` важен.** `except` ловит указанный класс И все его подклассы. Поэтому более общие исключения нужно ставить **после** более специфичных, иначе специфичный блок станет недостижимым:

```python
try:
    d = {}
    d["x"]
except LookupError:          # ловит и KeyError, и IndexError
    print("это сработает первым")
except KeyError:             # НЕДОСТИЖИМО — KeyError уже пойман выше
    print("сюда не попадём")
# => это сработает первым
```

Правильно — сначала узкий `KeyError`, потом широкий `LookupError`.

Срабатывает **первый подходящий** блок, остальные пропускаются.

### 3. else у try — когда выполняется

Блок `else` выполняется, **только если в `try` НЕ возникло исключения**. Смысл — отделить «код, который может бросить исключение» (в `try`) от «кода, который выполняется при успехе» (в `else`).

Зачем не положить код успеха прямо в конец `try`? Чтобы случайно **не перехватить** исключение, возникшее уже в логике успеха:

```python
def get_and_use(d, key):
    try:
        value = d[key]          # ожидаемо может бросить KeyError
    except KeyError:
        print("нет ключа")
    else:
        # если положить это в try, то KeyError из process() ошибочно
        # попадёт в except выше и будет неверно интерпретирован
        process(value)
```

Порядок: `try` (без исключения) → `else` → `finally`.

### 4. finally — гарантированное выполнение (в т.ч. при return)

`finally` выполняется **всегда**, независимо от того, что произошло в `try`/`except`/`else`:

- штатное завершение `try`;
- возникло и поймано исключение;
- возникло и НЕ поймано исключение (тогда `finally` выполнится, а затем исключение полетит дальше);
- из `try`/`except`/`else` сделан `return`, `break` или `continue`.

**`finally` + `return` — важный подводный камень.** Если `finally` содержит `return`, он **перезаписывает** значение, возвращаемое из `try`:

```python
def f():
    try:
        return 1        # это значение «запомнится»...
    finally:
        return 2        # ...но finally его перезатрёт

print(f())
# => 2
```

Аналогично `finally` может **проглотить исключение**, если в нём есть `return`/`break`/`continue`:

```python
def g():
    try:
        raise ValueError("ошибка")
    finally:
        return "проглотили"   # исключение НЕ полетит дальше!

print(g())
# => проглотили   (ValueError потерян — это антипаттерн)
```

Без `return` в `finally` исключение, не пойманное в `except`, корректно пробрасывается после выполнения `finally`:

```python
def h():
    try:
        return 1
    finally:
        print("finally до возврата")

print(h())
# => finally до возврата
# => 1
```

Вывод: не ставьте `return`/`break`/`continue` в `finally`, если не хотите намеренно подавить результат/исключение из `try`.

### 5. as — доступ к объекту исключения

`except SomeError as exc:` связывает пойманное исключение с именем `exc`. У объекта есть полезные атрибуты:

```python
try:
    int("abc")
except ValueError as exc:
    print(exc.args)        # => ('invalid literal for int() with base 10: ...',)
    print(str(exc))        # => текст сообщения
    print(type(exc))       # => <class 'ValueError'>
```

**Важно:** имя `exc` после выхода из блока `except` **удаляется** (для разрыва циклических ссылок и освобождения трейсбэка). Если значение нужно дальше — сохраните его в другую переменную:

```python
err = None
try:
    int("abc")
except ValueError as exc:
    err = exc            # сохранили наружу
# print(exc)            # NameError: exc уже удалён
print(repr(err))         # => ValueError(...)
```

### 6. raise и повторный проброс

`raise Exc("msg")` бросает исключение. `raise Exc` (без аргументов) — то же, что `raise Exc()` (создаст экземпляр). «Голый» `raise` внутри `except` пробрасывает **текущее** пойманное исключение дальше, сохраняя оригинальный трейсбэк:

```python
def with_logging():
    try:
        risky()
    except Exception:
        log("упало, пробрасываю дальше")
        raise          # пробросить тот же объект с тем же трейсбэком
```

Это лучше, чем `raise exc`, потому что не «обрезает» и не меняет трейсбэк.

### 7. Цепочки исключений: raise from, __cause__, __context__

Когда внутри `except` бросается новое исключение, Python **автоматически** связывает их через атрибут `__context__` — это «неявное сцепление». В трейсбэке появляется строка:
`During handling of the above exception, another exception occurred:`

```python
try:
    1 / 0
except ZeroDivisionError:
    raise ValueError("обёртка")   # __context__ = ZeroDivisionError (неявно)
```

**`raise ... from ...`** задаёт причину **явно** через `__cause__`. В трейсбэке:
`The above exception was the direct cause of the following exception:`

```python
def parse(s):
    try:
        return int(s)
    except ValueError as exc:
        raise RuntimeError("не удалось распарсить") from exc

try:
    parse("x")
except RuntimeError as e:
    print("cause:", repr(e.__cause__))      # => ValueError(...)
    print("context:", repr(e.__context__))  # => ValueError(...)
```

`raise ... from None` **подавляет** цепочку (скрывает исходное исключение из трейсбэка) — полезно, когда внутреннее исключение деталь реализации:

```python
def lookup(d, k):
    try:
        return d[k]
    except KeyError:
        raise ValueError(f"неизвестный ключ {k!r}") from None  # без KeyError в трейсбэке
```

Разница: `__cause__` (явная причина, через `from`) подразумевает «это прямо вызвало ошибку»; `__context__` (неявный) — «это случилось во время обработки».

### 8. Пользовательские классы исключений

Наследуйтесь от `Exception` (не от `BaseException`!). Можно строить свою иерархию для домена:

```python
class AppError(Exception):
    """Базовое исключение приложения."""

class ConfigError(AppError):
    """Проблемы с конфигурацией."""

class NetworkError(AppError):
    """Проблемы с сетью."""
    def __init__(self, message, *, status_code=None):
        super().__init__(message)
        self.status_code = status_code

try:
    raise NetworkError("таймаут", status_code=504)
except AppError as exc:          # ловим всё семейство одним блоком
    print(type(exc).__name__, getattr(exc, "status_code", None))
# => NetworkError 504
```

Преимущество базового класса `AppError`: вызывающий код может ловить либо узко (`ConfigError`), либо всё приложение целиком (`AppError`), не перехватывая чужие исключения.

Конвенция: имена классов исключений обычно заканчиваются на `Error`.

### 9. Очистка ресурсов: with как альтернатива try/finally

Классическая подчистка через `try/finally`:

```python
f = open("data.txt")
try:
    data = f.read()
finally:
    f.close()   # гарантированно закроем, даже при исключении
```

Идиоматичнее — **менеджер контекста** `with`, который сам вызывает подчистку (`__exit__`) при выходе из блока, в том числе при исключении:

```python
with open("data.txt") as f:
    data = f.read()
# файл закрыт автоматически, даже если read() бросил исключение
```

Свой менеджер контекста:

```python
class Resource:
    def __enter__(self):
        print("захват ресурса")
        return self
    def __exit__(self, exc_type, exc_value, tb):
        print("освобождение ресурса")
        # вернуть True, чтобы ПОДАВИТЬ исключение; False/None — пробросить
        return False

with Resource():
    raise ValueError("бум")
# => захват ресурса
# => освобождение ресурса
# => затем ValueError пробрасывается дальше
```

Или через `contextlib`:

```python
from contextlib import contextmanager

@contextmanager
def managed():
    print("setup")
    try:
        yield "ресурс"
    finally:
        print("teardown")   # аналог finally
```

`with` короче, безопаснее (нельзя забыть `close`) и поддерживает несколько ресурсов сразу: `with open(a) as fa, open(b) as fb:`.

### 10. Группы исключений ExceptionGroup и except* (Python 3.11+)

`ExceptionGroup` позволяет бросить **несколько исключений одновременно** (актуально для конкурентного кода: `asyncio.TaskGroup`, параллельные задачи). Перехват — через `except*`, который обрабатывает **подмножество** группы по типу:

```python
def process_all():
    errors = []
    for item in ["1", "x", "2", "y"]:
        try:
            int(item)
        except ValueError as e:
            errors.append(e)
    if errors:
        raise ExceptionGroup("ошибки парсинга", errors)

try:
    process_all()
except* ValueError as eg:
    # eg — это ExceptionGroup с подмножеством ValueError'ов
    print(f"поймали {len(eg.exceptions)} ValueError")
# => поймали 2 ValueError
```

Особенности `except*`:
- Несколько `except*` блоков могут сработать в рамках одной группы (в отличие от обычного `except`, где срабатывает один).
- Нельзя смешивать `except` и `except*` в одном `try`.
- Непойманные части группы пробрасываются дальше как остаточная группа.

### 11. Заметки add_note (Python 3.11+)

Метод `add_note(str)` добавляет к исключению контекстную строку, которая отображается в трейсбэке. Удобно обогащать ошибку, не меняя её тип и не теряя оригинал:

```python
def process_records(records):
    for i, r in enumerate(records):
        try:
            handle(r)
        except Exception as exc:
            exc.add_note(f"при обработке записи #{i}")
            raise   # тип и трейсбэк сохранены, добавлен контекст

# В трейсбэке после строки исключения появится:
#   при обработке записи #3
```

Заметки накапливаются в `exc.__notes__` (список строк).

### 12. EAFP vs LBYL

**LBYL** (Look Before You Leap) — «семь раз проверь»: предварительные проверки условий.
**EAFP** (Easier to Ask Forgiveness than Permission) — «проще извиниться»: пробуем и ловим исключение. Это **питоничный** стиль.

```python
# LBYL — гонки состояний, многословно, дублирование проверок
import os
if os.path.exists(path):      # между проверкой и open файл могут удалить!
    with open(path) as f:
        data = f.read()
else:
    data = None

# EAFP — короче и без гонки
try:
    with open(path) as f:
        data = f.read()
except FileNotFoundError:
    data = None
```

```python
# LBYL для словаря
if "key" in d and isinstance(d["key"], int):
    x = d["key"]

# EAFP — короче, без двойного обращения
try:
    x = int(d["key"])
except (KeyError, ValueError, TypeError):
    x = default
```

EAFP лучше в многопоточной/конкурентной среде (нет окна гонки между проверкой и действием) и часто быстрее на «счастливом пути». LBYL уместен, когда проверка дешёвая, а исключение — дорогое или маловероятное событие, или когда нужно избежать частичного побочного эффекта.

## Частые вопросы на собеседовании (Q/A)

**Q1. В чём разница между синтаксической ошибкой и исключением?**
Синтаксическая ошибка обнаруживается парсером до выполнения — модуль не запустится вообще. Исключение возникает во время выполнения синтаксически корректного кода и может быть перехвачено через `try/except`.

**Q2. Когда выполняется блок `else` в `try/except/else`?**
Только если в `try` не возникло исключения. Идёт после `try` и `except`, но до `finally`. Нужен, чтобы отделить «опасный» код от «кода успеха» и не ловить случайно исключения из логики успеха.

**Q3. Гарантированно ли выполняется `finally`? Что будет с `return` в нём?**
Да, `finally` выполняется всегда — при успехе, при исключении (даже непойманном), при `return`/`break`/`continue`. Если в `finally` есть `return`, он перезаписывает значение из `try` и **подавляет** непойманное исключение. Это антипаттерн — избегайте управляющих переходов в `finally`.

**Q4. Почему `except Exception` не ловит `KeyboardInterrupt`?**
Потому что `KeyboardInterrupt`, `SystemExit`, `GeneratorExit` наследуются от `BaseException`, а не от `Exception`. Это сделано специально, чтобы прикладной `except Exception` не проглатывал Ctrl+C и выход из программы.

**Q5. Чем `except Exception` лучше «голого» `except:`?**
Голый `except:` ловит `BaseException`, включая `KeyboardInterrupt` и `SystemExit` — программу станет невозможно прервать или корректно завершить. Голый `except` также скрывает любые баги. Всегда указывайте конкретный тип или хотя бы `except Exception`.

**Q6. Почему важен порядок блоков `except`?**
`except` ловит указанный класс и все подклассы. Если поставить общий класс раньше специфичного, специфичный станет недостижимым. Сначала узкие исключения, потом широкие.

**Q7. В чём разница между `raise` и `raise exc`?**
Голый `raise` внутри `except` пробрасывает текущее исключение, **сохраняя оригинальный трейсбэк**. `raise exc` бросает заново и может изменить/обрезать контекст. Для повторного проброса предпочтительнее голый `raise`.

**Q8. Что делает `raise ... from ...` и чем `__cause__` отличается от `__context__`?**
`raise X from Y` явно задаёт причину `X.__cause__ = Y` («прямая причина»). `__context__` устанавливается автоматически, когда новое исключение бросается внутри обработки другого («во время обработки»). `from None` подавляет отображение цепочки.

**Q9. EAFP vs LBYL — что и когда?**
EAFP (try/except) — питоничный стиль, короче, без гонки состояний между проверкой и действием, быстрее на счастливом пути. LBYL (предварительные проверки) уместен при дешёвых проверках, редких/дорогих исключениях или чтобы избежать частичных побочных эффектов.

**Q10. Как создать своё исключение и зачем нужна базовая иерархия?**
Наследоваться от `Exception` (не `BaseException`). Базовый класс приложения (например, `AppError`) позволяет ловить либо всё семейство ошибок приложения одним `except AppError`, либо отдельные подтипы, не перехватывая чужие исключения.

**Q11. Почему `with` предпочтительнее `try/finally` для ресурсов?**
`with` гарантирует вызов `__exit__` (подчистку) при любом выходе из блока, включая исключения, при этом короче и не даёт забыть `close()`. Под капотом это тот же `try/finally`, но безопаснее и читабельнее.

**Q12. Что такое `ExceptionGroup` и `except*`?**
`ExceptionGroup` (3.11+) — контейнер для нескольких одновременных исключений (конкурентные задачи, `asyncio.TaskGroup`). `except*` ловит подмножество группы по типу; несколько `except*` могут сработать за один `try`. Нельзя смешивать `except` и `except*`.

**Q13. Что делает `add_note` и зачем?**
`exc.add_note("...")` (3.11+) добавляет контекстную строку к исключению, видимую в трейсбэке, не меняя его тип и не теряя оригинал. Удобно обогащать ошибку информацией о том, где/при каких данных она произошла.

**Q14. Что произойдёт, если в `except` бросить новое исключение?**
Старое исключение автоматически попадёт в `__context__` нового, и в трейсбэке будет «During handling of the above exception...». Чтобы задать причину явно — `raise New from old`; чтобы скрыть цепочку — `raise New from None`.

## Подводные камни (gotchas)

1. **`finally` с `return`/`break`/`continue`** перезаписывает результат и глотает непойманное исключение из `try`. Почти всегда баг.

   ```python
   def f():
       try:
           return 1
       finally:
           return 2   # вернётся 2, а не 1
   ```

2. **Голый `except:`** ловит `BaseException` → перехватывает `KeyboardInterrupt`/`SystemExit`, делает программу непрерываемой и скрывает баги. Используйте `except Exception` или конкретный тип.

3. **Проглатывание исключений** (`except Exception: pass`) — ошибка «исчезает» без следа, отладка превращается в ад. Минимум — логируйте.

4. **Имя из `as exc` удаляется** после блока `except`. Снаружи `exc` недоступно (`NameError`) — сохраняйте в отдельную переменную при необходимости.

5. **Неправильный порядок `except`**: общий класс перед специфичным делает специфичный недостижимым.

6. **Мутабельный/дорогой код в `try`**: чем больше строк в `try`, тем выше риск поймать «не то» исключение. Держите в `try` минимум, остальное — в `else`.

7. **`raise exc` вместо голого `raise`** при перепробросе может обрезать/переписать трейсбэк. Для проброса того же исключения — просто `raise`.

8. **LBYL и гонки состояний**: `if os.path.exists(p): open(p)` — между проверкой и открытием файл может исчезнуть. EAFP надёжнее.

9. **Наследование своих исключений от `BaseException`** вместо `Exception` — их не поймает `except Exception`, что обычно нежелательно.

10. **Смешивание `except` и `except*`** в одном `try` запрещено и вызовет ошибку.

11. **Сравнение исключений по тексту сообщения** вместо типа/атрибутов — хрупко. Ловите по классу, передавайте данные через атрибуты/`args`.

## Шпаргалка

```python
try:
    # код, который может бросить исключение
    ...
except (TypeError, ValueError) as exc:   # несколько типов + доступ к объекту
    ...
except SpecificError:                     # узкие — выше
    ...
except Exception:                         # широкие — ниже (НЕ голый except!)
    ...
else:
    # выполняется ТОЛЬКО при отсутствии исключения в try
    ...
finally:
    # выполняется ВСЕГДА (не ставьте сюда return/break/continue!)
    ...
```

| Конструкция | Когда выполняется / что делает |
|---|---|
| `try` | основной блок, который может бросить исключение |
| `except T` | если в `try` возникло `T` или подкласс `T` |
| `except (A, B)` | несколько типов одним блоком |
| `except T as e` | связывает исключение с `e` (удаляется после блока) |
| `else` | только если `try` прошёл БЕЗ исключения |
| `finally` | ВСЕГДА (успех, исключение, return/break/continue) |
| `raise` | бросить / голый `raise` — пробросить текущее (трейсбэк сохранён) |
| `raise X from Y` | явная причина: `X.__cause__ = Y` |
| `raise X from None` | подавить цепочку исключений |
| `__context__` | неявная цепочка (исключение во время обработки) |
| `__cause__` | явная причина (через `from`) |
| `with` | менеджер контекста = безопасная замена `try/finally` |
| `ExceptionGroup` | несколько исключений сразу (3.11+) |
| `except*` | перехват подмножества группы по типу (3.11+) |
| `exc.add_note(s)` | добавить контекст в трейсбэк (3.11+) |

**Иерархия (ключевое):**
`BaseException` → `Exception` (прикладное), а также `SystemExit` / `KeyboardInterrupt` / `GeneratorExit` (НЕ под `Exception`).

**Правила хорошего тона:**
- Ловите узко; `except Exception` — максимум, голый `except:` — нет.
- Не глотайте исключения: логируйте или пробрасывайте.
- Минимум кода в `try`, остальное — в `else`.
- Ресурсы — через `with`.
- Перепроброс — голым `raise`.
- Предпочитайте EAFP; LBYL — только при дешёвых проверках и редких исключениях.
- Свои исключения — от `Exception`, стройте доменную иерархию.
