# Как объекты Python устроены в памяти (CPython)

Конспект по внутреннему устройству встроенных типов в основной реализации Python — **CPython**. Везде, где приводится C-структура, она упрощена: убраны второстепенные поля, оставлено главное для понимания раскладки в памяти.

Общая идея, которая повторяется во всех типах:

- **Неизменяемые** типы (`tuple`, `str`, `bytes`, `int`) хранят данные **встроенно, одним блоком памяти** и часто кэшируют свой хеш рядом с данными.
- **Изменяемые** типы (`list`, `bytearray`, `dict`, `set`) держат **отдельный буфер** с запасом ёмкости (overallocation), что даёт быстрые вставки ценой лишней памяти.
- Почти везде есть оптимизация — **кэширование часто используемых мелких объектов** (малые целые, односимвольные строки/байты, `True`/`False`/`None`).

## list

`list` — это **динамический массив указателей**, а не массив самих объектов.

```c
typedef struct {
    PyObject_VAR_HEAD          // ob_refcnt, ob_type, ob_size (= len)
    PyObject **ob_item;        // указатель на ОТДЕЛЬНЫЙ массив указателей
    Py_ssize_t allocated;      // сколько памяти реально выделено (ёмкость)
} PyListObject;
```

```
PyListObject                 ob_item → [ ptr ][ ptr ][ ptr ][ free ]
  ob_size   = 3                          │      │      │
  allocated = 4                          ▼      ▼      ▼
  ob_item ●───────────────────────►   объект объект объект   (разбросаны по куче)
```

- Список хранит **указатели**, поэтому в нём могут лежать элементы разных типов, а размер ячейки всегда одинаков (один указатель — 8 байт на 64-битной системе).
- Доступ по индексу — **O(1)**: `ob_item[i]` — простая адресная арифметика.
- **Overallocation**: при росте память выделяется с запасом (`allocated >= ob_size`). Рост идёт примерно по схеме `0, 4, 8, 16, 25, 35, 46, 58, 72, 88, …`, поэтому `append` амортизированно **O(1)**.
- При переполнении выделяется новый блок побольше, и **указатели копируются** туда (сами объекты не двигаются). Поэтому `insert(0, x)` — **O(n)**.
- По памяти список «тяжелее», чем `array.array` или `numpy`-массив, где значения упакованы подряд без обёрток-объектов.

```python
import sys
lst = []
prev = sys.getsizeof(lst)
for i in range(20):
    lst.append(i)
    size = sys.getsizeof(lst)
    if size != prev:
        print(f"len={len(lst):2d}  {size} байт")   # размер растёт «ступеньками»
        prev = size
```

## tuple

`tuple` устроен как список, но массив указателей **встроен прямо в объект** — это следствие неизменяемости.

```c
typedef struct {
    PyObject_VAR_HEAD
    PyObject *ob_item[1];   // встроенный массив, реальная длина = ob_size
} PyTupleObject;
```

Сравните: у списка `PyObject **ob_item` (указатель на отдельный массив), у кортежа `PyObject *ob_item[]` (массив — часть самого объекта).

```
PyTupleObject (один блок памяти)
  ob_size = 3
  ob_item[0] ●──► объект
  ob_item[1] ●──► объект
  ob_item[2] ●──► объект
```

- **Одно выделение памяти** вместо двух → лучшая локальность кэша, меньше накладных расходов.
- Нет поля `allocated` и нет overallocation: длина известна при создании, выделяется ровно `ob_size` указателей. Кортеж легче списка из тех же элементов.
- Нет перевыделения и копирования; нет `append`/`insert`.
- CPython кэширует маленькие кортежи (free list) и переиспользует их.

```python
import sys
sys.getsizeof((1, 2, 3))   # обычно 64 байта
sys.getsizeof([1, 2, 3])   # обычно 88 байт
```

**Неизменяемость поверхностна.** «Заморожен» массив указателей, но не объекты, на которые они указывают:

```python
t = ([1, 2], 'hi')
t[0].append(3)     # работает — меняется список, а не кортеж
print(t)           # ([1, 2, 3], 'hi')
t[0] = [9]         # TypeError — нельзя заменить сам указатель
```

## Срезы

Нужно различать **объект-срез** `slice(start, stop, step)` и **результат** среза `lst[1:3]`.

`a[1:5:2]` сначала создаёт объект `slice(1, 5, 2)` и передаёт его в `a.__getitem__`:

```c
typedef struct {
    PyObject_HEAD
    PyObject *start;   // указатель на объект (int или None)
    PyObject *stop;
    PyObject *step;
} PySliceObject;
```

Сам объект-срез ничего не копирует — лишь описывает «правило выборки».

**Применение** среза к встроенной последовательности (`list`, `tuple`, `str`) создаёт **новый объект и копирует указатели** — это поверхностная копия, а не «окно» в старую память:

```
a = [10, 'hi', 3.14, True]      b = a[1:3]
  a.ob_item → [→10][→'hi'][→3.14][→True]
                     │       │
                     ▼       ▼          ← общие объекты
                   'hi'    3.14
                     ▲       ▲
  b.ob_item → [─────┘       └─────]      (новый массив, те же объекты)
```

- Срез — **поверхностная копия**: новый массив `ob_item`, но указатели ведут на те же объекты.
- Контейнер независим (можно менять `b`, не трогая `a`), но **общие изменяемые элементы видны через оба**:

```python
a = [[1, 2], 'hi', 3.14]
b = a[0:2]
b.append('new')      # a не меняется — у b свой массив
b[0].append(99)      # меняем общий вложенный список
print(a[0])          # [1, 2, 99] — видно и через a
```

- Стоимость — **O(k)**, где `k` — длина среза (копируются `k` указателей). `a[:]` — полная поверхностная копия.
- Для `tuple` и `str` — то же самое: копия. (Мелкая оптимизация: `t[:]`/`s[:]` для неизменяемого часто возвращают тот же объект.)
- **Исключение — другие типы.** Срез `numpy`-массива возвращает **view** (общий буфер + смещение и strides), запись в срез меняет исходный массив. `memoryview` поверх `bytes`/`bytearray` — тоже view. Но это свойство конкретных типов, а не среза.

```python
import numpy as np
arr = np.array([0, 1, 2, 3, 4])
v = arr[1:4]         # view, общий буфер
v[0] = 99
print(arr)           # [ 0 99  2  3  4] — исходный массив изменился
```

## str

Строка **неизменяема** и хранит символы **встроенно** (compact-представление), одним блоком. Главная особенность — **гибкое представление** (PEP 393): ширина символа выбирается по самому «широкому» символу: **1, 2 или 4 байта**.

```c
typedef struct {
    PyObject_HEAD
    Py_ssize_t length;     // длина в кодовых точках
    Py_hash_t  hash;       // кэш хеша, -1 пока не вычислен
    struct {
        unsigned int interned : 2;
        unsigned int kind : 3;      // 1, 2 или 4 байта на символ
        unsigned int compact : 1;
        unsigned int ascii : 1;
    } state;
} PyASCIIObject;
// СРАЗУ за структурой в памяти идут байты символов
```

Гибкая ширина (определяется максимальной кодовой точкой):

```
'cat'   kind=1  1 Б/символ   [63][61][74]
'мир'   kind=2  2 Б/символ   [043C][0438][0440]
'a' + символ > U+FFFF
        kind=4  4 Б/символ   [00000061][0001F389]
                              ▲ даже обычная 'a' занимает 4 байта —
                                ширину задаёт самый широкий символ
```

- ASCII/Latin-1 (≤255) → 1 байт; до U+FFFF → 2 байта; выше → 4 байта. **Один «широкий» символ переводит всю строку** в 4 байта/символ.
- **Индексация и `len` — O(1)**: длина в поле `length`, символы фиксированной ширины → `s[i]` = `data + i*ширина`. (Поэтому хранят фиксированную ширину, а не UTF-8, где индексация была бы O(n).)
- **Хеш кэшируется** при первом обращении (например, как ключ словаря).
- **UTF-8 тоже кэшируется** при первом кодировании; для ASCII отдельный буфер не нужен.
- **Интернирование**: короткие строки-идентификаторы и литералы дедуплицируются.

```python
import sys
sys.getsizeof("hello")    # ASCII   → ~1 байт/символ
sys.getsizeof("привет")   # UCS-2   → 2 байта/символ
sys.getsizeof("hi🎉")      # UCS-4   → 4 байта/символ (из-за эмодзи)

a = "hello"; b = "hello"
a is b                    # часто True — литералы интернируются
```

> Для сравнения строк всегда `==`, не `is` (интернирование — деталь реализации).

## dict

Современный словарь («компактный», с Python 3.6) использует **две части**: разреженную хеш-таблицу индексов и плотный массив записей в порядке вставки.

```c
typedef struct {
    PyObject_HEAD
    Py_ssize_t ma_used;        // количество элементов
    PyDictKeysObject *ma_keys; // таблица ключей (две части)
    PyObject **ma_values;      // NULL для обычных (combined) словарей
} PyDictObject;
```

Запись `{'a':1, 'b':2, 'c':3}`:

```
dk_indices (разрежено)        dk_entries (порядок вставки)
 [ — ]                        [0]  h('a') | 'a' | 1
 [ 0 ] ───────────────────►   [1]  h('b') | 'b' | 2
 [ — ]                        [2]  h('c') | 'c' | 3
 [ 2 ] ──────────────┐         ▲          ▲     ▲
 [ — ]               │     me_hash    me_key  me_value
 [ — ]               │
 [ 1 ] ──────┐       └──────►  [2]
 [ — ]       └──────────────►  [1]
```

- **Поиск**: хеш ключа → младшие биты как индекс в `dk_indices` → номер записи → `dk_entries[номер]`. Сначала сравнивается **кэш хеша** (быстрая целочисленная проверка), потом `==` ключей. Для строковых ключей переиспользуется их кэшированный хеш.
- **Открытая адресация** (коллизии — поиском другого слота, не цепочками). Зондирование с возмущением: `perturb >>= 5; i = (i*5 + perturb + 1) & mask` — задействует все биты хеша.
- **Упорядоченность** (гарантирована с 3.7) — побочный эффект плотного `dk_entries` в порядке вставки.
- **Экономия памяти**: разрежена только таблица маленьких индексов, записи лежат плотно. `dk_indices` использует целые переменной ширины (1 байт/слот при размере ≤ 128).
- **Перехеширование** при заполнении ~2/3 в большую таблицу (степень двойки).
- **Раздельные таблицы** (`ma_values != NULL`) — для `__dict__` экземпляров: ключи-атрибуты общие, у каждого объекта свой массив значений.

## set / frozenset

Множество — тоже хеш-таблица с открытой адресацией, но **проще** словаря: одна таблица слотов `(ключ, хеш)`, без значений и без отдельного плотного массива.

```c
typedef struct {
    PyObject_HEAD
    Py_ssize_t fill;          // занятые + удалённые слоты
    Py_ssize_t used;          // количество элементов
    Py_ssize_t mask;          // размер таблицы минус 1
    setentry *table;          // указатель на таблицу
    Py_hash_t hash;           // кэш хеша (только для frozenset)
    setentry smalltable[8];   // встроенная таблица на 8 слотов
} PySetObject;

typedef struct {
    PyObject *key;
    Py_hash_t hash;           // кэш хеша ключа
} setentry;
```

```
smalltable[8]   me_hash | key
 [ — ]
 [ — ]
 [ h('c') | 'c' ]      ← позиции заданы хешем → порядок не хранится
 [ h('a') | 'a' ]
 [ — ]
 [ — ]
 [ h('b') | 'b' ]
 [ — ]
```

Отличия от словаря:

- **Нет плотного массива записей → множество неупорядочено.** Элементы лежат прямо в слотах по позициям хеша.
- Запись — `(ключ, хеш)`, **без значения**.
- **Встроенная `smalltable[8]`**: маленькое множество (до 5 элементов) не требует отдельного выделения под таблицу.
- При коллизии сначала пробуются несколько соседних слотов (линейное зондирование — выгодно для кэша CPU), затем — та же возмущающая последовательность. Перехеширование при заполнении ~3/5.
- **`frozenset` кэширует свой хеш** (поле `hash`) → его можно класть ключом в словарь или элементом в другое множество. У обычного `set` это поле не используется (он не хешируется).

## int

`int` — **число произвольной точности** (bignum). Никогда не переполняется, в отличие от C-`int` или `numpy.int64`. Хранится как массив «цифр» по основанию **2³⁰** (на 64-битных платформах; 2¹⁵ на 32-битных), встроенных в объект.

```c
typedef struct {
    PyObject_VAR_HEAD     // в т.ч. ob_size: знак И число цифр
    digit ob_digit[1];    // массив цифр по основанию 2^30
} PyLongObject;
```

Число `2³⁰ + 5 = 1 073 741 829` (две цифры, младшая первая):

```
PyLongObject (один блок)
  ob_size = 2  → знак «+», 2 цифры
  ob_digit[0] = 5    (30 бит)
  ob_digit[1] = 1    (30 бит)

  значение = 5 + 1·2³⁰ = 1 073 741 829
```

- **Произвольная точность**: значение = `Σ ob_digit[i]·(2³⁰)ⁱ`. Растёт число — добавляются цифры; предел только память.
- **Знак — в `ob_size`** (классическая схема): «+»/«−»/0. В Python 3.12 представление переписали на отдельный тег (`lv_tag`), но идея та же: знак + массив 30-битных цифр.
- **Один блок**: размер растёт ступеньками — ~28 байт для маленького числа и примерно +4 байта на каждую 30-битную цифру.
- Даже `5` — полноценный объект в куче, поэтому массив питоновских `int` намного «тяжелее» C/numpy-массива.

```python
import sys
sys.getsizeof(0)        # ~28 байт — цифр нет
sys.getsizeof(2**30)    # ~32 — нужны две цифры
sys.getsizeof(2**1000)  # заметно больше — ~34 цифры
```

**Кэш малых целых (−5 … 256).** Эти объекты предсозданы и общие:

```python
a = 256; b = 256
a is b      # True — кэшированный объект

x = 257; y = 257
x is y      # обычно False — два разных объекта 257
x == y      # True — по значению равны
```

> Для сравнения чисел всегда `==`, не `is`. `bool` — подкласс `int`: `True`/`False` физически равны 1/0 и тоже синглтоны. `n += 1` не меняет объект на месте, а создаёт новый.

## bytes / bytearray

### bytes

**Неизменяемая** последовательность сырых байтов; каждый элемент — число 0–255 (никакой гибкой ширины, всегда 1 байт). Данные встроены, как у строки, плюс кэш хеша.

```c
typedef struct {
    PyObject_VAR_HEAD     // ob_size = длина в байтах
    Py_hash_t ob_shash;   // кэш хеша, -1 пока не вычислен
    char ob_sval[1];      // сами байты, встроены, + завершающий '\0'
} PyBytesObject;
```

```
b'cat'  →  ob_size=3, ob_shash=кэш
           [0x63][0x61][0x74][\0]
            99    97    116  ← завершающий ноль (для C)
```

- **Сырые байты, не символы**: индексация возвращает `int` (`b'cat'[0] == 99`), срез возвращает `bytes` (копию).
- **Один блок и неизменяемость** — как у `tuple`/`str`/`int`.
- **Завершающий `\0`** для совместимости с C; реальная длина — из `ob_size` (внутри могут быть нули).
- **Кэш хеша** → хешируется, можно ключом в словарь. `sys.getsizeof(b'') ≈ 33`, +1 байт на байт данных.
- Пустой `bytes` и все односимвольные `b'\x00'…b'\xff'` предсозданы (синглтоны).

### bytearray

**Изменяемый** брат — устроен как `list` относительно `tuple`: отдельный буфер с запасом.

```c
typedef struct {
    PyObject_VAR_HEAD
    Py_ssize_t ob_alloc;  // выделенная ёмкость (с запасом)
    char *ob_bytes;       // указатель на ОТДЕЛЬНЫЙ буфер
    char *ob_start;       // начало данных (дешёвое удаление слева)
} PyByteArrayObject;
```

```
bytearray b'cat'                ob_bytes → [0x63][0x61][0x74][ — ][ — ][ — ]
  ob_size  = 3                              └── заняты ──┘ └─ свободны ─┘
  ob_alloc = 6
  ob_bytes ●──────────────────────────────►  (отдельный буфер с запасом)
```

- **Буфер отдельно** (не встроен), как у списка.
- **Запас под рост** (`ob_alloc`) и overallocation → `extend`/`+=` амортизированно **O(1)**.
- **Изменяемый → не хешируется**: нельзя ключом словаря / элементом множества. Поля `ob_shash` нет.
- **`ob_start`** позволяет дёшево удалять слева: вместо сдвига всего содержимого двигается указатель начала.

## Ленивые итераторы

Устроены **противоположно** контейнерам: хранят не данные, а **состояние** — где мы сейчас и как вычислить следующий элемент. Память — **O(1)** независимо от числа значений. Протокол: `__next__` отдаёт значение или возбуждает `StopIteration`.

### Итератор списка

Не копирует список — хранит указатель и индекс:

```c
typedef struct {
    PyObject_HEAD
    Py_ssize_t it_index;   // текущая позиция
    PyListObject *it_seq;  // указатель на список (NULL когда исчерпан)
} listiterobject;
```

```
список a (ob_item):  [10][20][30][40][50]…
                          ▲
итератор:  it_seq ●──► a   it_index = 2  → next() даст 30
```

- Итератор — это «позиция», а не «данные»; перебор миллиона элементов создаёт тот же крошечный объект.
- Смотрит на **живой** список → менять коллекцию во время итерации опасно.
- Одноразовый: дойдя до конца, обнуляет `it_seq`.

### range и ленивые конвейеры

`range` сам ленив — хранит только границы, не числа (`range(10**18)` — десятки байт):

```c
typedef struct {
    PyObject_HEAD
    PyObject *start, *stop, *step, *length;  // только границы, не сами числа
} rangeobject;
```

`map`, `filter`, `zip`, `enumerate`, `itertools` — обёртки, хранящие указатель на источник + минимум состояния; тянут по одному элементу сверху по требованию (**ленивый конвейер вытягивания**). Память O(1) при любом объёме:

```python
# проходит миллиард чисел, но в памяти — лишь несколько маленьких итераторов
total = sum(x * x for x in range(10**9) if x % 2)
```

(Сравните со списковым включением `[x*x for x in range(10**9)]`, которое создаёт миллиард объектов сразу.)

### Генераторы

Вызов генераторной функции не выполняет тело — возвращает объект-генератор с **замороженным кадром выполнения**: локальные переменные + указатель на инструкцию, где остановились.

```c
typedef struct {
    PyObject_HEAD
    char gi_frame_state;            // приостановлен / выполняется / завершён
    PyObject *gi_code;              // байткод функции
    _PyInterpreterFrame gi_iframe;  // встроенный кадр: локальные, стек, указатель инструкции
} PyGenObject;
```

```
генератор g                         код:
  gi_frame_state = приостановлен       def gen(n):
  ┌─ замороженный кадр ─────────┐          i = 0
  │ локальные: i = 2, n = 5     │          while i < n:
  │ стек значений               │  ◄──────────  yield i   ← точка возобновления
  │ f_lasti → байткод «yield i» │              i += 1
  └─────────────────────────────┘

next(g): возобновить кадр → выполнять до следующего yield → снова заморозить
```

- **`next()` размораживает и снова замораживает**: восстанавливает локальные и стек, прыгает на `f_lasti`, выполняет до `yield`, сохраняет новое состояние и замирает. В памяти — ровно один кадр, а не вся последовательность.
- **Память O(1) даже для «бесконечности»** (`while True: yield ...`). С Python 3.11 кадр встроен в объект генератора (`gi_iframe`).
- Одноразовость и защита от повторного входа через `gi_frame_state`. Есть `.send()`, `.throw()`, `.close()`. На том же механизме приостановки кадра построены `async`/`await` и асинхронные генераторы.
