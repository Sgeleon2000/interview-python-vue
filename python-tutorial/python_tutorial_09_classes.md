# Глава 9. Классы — конспект и вопросы

## О чём глава

Девятая глава официального туториала Python — это введение в объектно-ориентированное программирование (ООП). Это **топовая тема собеседований**: про классы, наследование, MRO, `super()`, итераторы и генераторы спрашивают практически всегда на позициях middle и senior.

Классы — это механизм связывания **данных** (атрибутов) и **поведения** (методов) в единую сущность. Класс — это «чертёж» (blueprint), а экземпляр (instance) — конкретный объект, построенный по этому чертежу.

Ключевые особенности классов в Python:
- Множественное наследование (а значит — MRO и C3-линеаризация).
- Наследование позволяет переопределять методы базового класса с тем же именем.
- Все члены класса (включая данные) по умолчанию **публичные**; «приватность» эмулируется соглашениями и name mangling.
- Все методы **виртуальные** (позднее связывание): вызов метода всегда ищется в реальном типе объекта.
- Классы сами являются объектами (метаклассы — это уже за рамками главы, но важно понимать «классы — объекты первого класса»).

В этой главе также разбираются **области видимости и пространства имён** (scopes/namespaces), без понимания которых ООП в Python осмыслить невозможно, а также **итераторы** и **генераторы**.

## Ключевые концепции (с примерами и выводом)

### 1. Пространства имён и области видимости (namespaces & scopes)

**Пространство имён (namespace)** — это отображение имён в объекты (по сути — словарь). Примеры: встроенные имена (`print`, `len`), глобальные имена модуля, локальные имена внутри функции, атрибуты объекта.

**Область видимости (scope)** — это текстовая (лексическая) область программы, где пространство имён доступно напрямую (без точечной нотации).

Правило поиска имён — **LEGB**:
- **L**ocal — локальная область функции.
- **E**nclosing — области объемлющих функций (для вложенных функций).
- **G**lobal — глобальная область модуля.
- **B**uilt-in — встроенные имена.

```python
x = "глобальный"  # global

def outer():
    x = "enclosing"  # переменная объемлющей функции

    def inner():
        x = "локальный"  # local
        print(x)

    inner()
    print(x)

outer()
print(x)
# => локальный
# => enclosing
# => глобальный
```

### 2. `global` и `nonlocal`

По умолчанию присваивание создаёт/перезаписывает имя в **локальной** области. Чтобы изменить имя из внешней области:

```python
counter = 0

def increment():
    global counter      # говорим: работаем с глобальной переменной
    counter += 1

increment()
increment()
print(counter)  # => 2


def outer():
    n = 10

    def inner():
        nonlocal n      # ближайшая объемлющая (НЕ глобальная) область
        n += 5

    inner()
    print(n)            # => 15

outer()
```

Важно: **чтение** внешней переменной не требует `global`/`nonlocal`, а **присваивание** — требует, иначе Python создаст новую локальную переменную (и при чтении до присваивания вы получите `UnboundLocalError`).

```python
val = 100

def buggy():
    print(val)   # UnboundLocalError!
    val = 5      # из-за этого присваивания val считается локальной во ВСЕЙ функции

# buggy()  # UnboundLocalError: local variable 'val' referenced before assignment
```

### 3. Определение класса и создание экземпляра

```python
class Dog:
    """Простой класс собаки."""
    kind = "собака"          # атрибут КЛАССА (общий для всех экземпляров)

    def __init__(self, name):
        self.name = name     # атрибут ЭКЗЕМПЛЯРА (свой у каждого)

    def bark(self):
        return f"{self.name} говорит: Гав!"


d = Dog("Рекс")             # вызов класса создаёт экземпляр (вызывается __init__)
print(d.name)               # => Рекс
print(d.kind)               # => собака
print(d.bark())             # => Рекс говорит: Гав!
print(type(d))              # => <class '__main__.Dog'>
print(isinstance(d, Dog))   # => True
```

- `Dog` — **объект класса** (class object), поддерживает атрибутные ссылки (`Dog.kind`) и инстанцирование (`Dog(...)`).
- `d` — **объект экземпляра** (instance object).
- `__init__` — конструктор-инициализатор: вызывается автоматически после создания объекта.

### 4. Атрибут класса vs атрибут экземпляра — и ЛОВУШКА изменяемого атрибута

```python
class Counter:
    count = 0            # атрибут класса

    def __init__(self):
        self.value = 0   # атрибут экземпляра


a = Counter()
b = Counter()

a.value = 5              # меняем ТОЛЬКО атрибут экземпляра a
print(a.value, b.value)  # => 5 0

Counter.count = 10       # меняем атрибут КЛАССА — видят все
print(a.count, b.count)  # => 10 10
```

**Главная ловушка собеседований** — изменяемый (mutable) атрибут класса, общий для всех экземпляров:

```python
class Dog:
    tricks = []          # ОШИБКА: один список на ВСЕ экземпляры!

    def __init__(self, name):
        self.name = name

    def add_trick(self, trick):
        self.tricks.append(trick)   # мутируем общий список класса


fido = Dog("Фидо")
buddy = Dog("Бадди")
fido.add_trick("сидеть")
buddy.add_trick("кататься")

print(fido.tricks)   # => ['сидеть', 'кататься']  (!) общий список
print(buddy.tricks)  # => ['сидеть', 'кататься']  (!) тот же объект
```

Правильное решение — заводить изменяемые данные **в `__init__`** как атрибуты экземпляра:

```python
class Dog:
    def __init__(self, name):
        self.name = name
        self.tricks = []     # свой список у каждого экземпляра

    def add_trick(self, trick):
        self.tricks.append(trick)


fido = Dog("Фидо")
buddy = Dog("Бадди")
fido.add_trick("сидеть")
buddy.add_trick("кататься")
print(fido.tricks)   # => ['сидеть']
print(buddy.tricks)  # => ['кататься']
```

### 5. `self`, поиск атрибута и затенение

`self` — это просто соглашение об имени первого параметра метода (можно назвать иначе, но не стоит). При вызове `d.bark()` Python неявно передаёт `d` как `self`: `Dog.bark(d)`.

Поиск атрибута идёт: сначала в `__dict__` экземпляра, затем в классе и его базовых классах (по MRO).

```python
class A:
    x = "класс"

a = A()
print(a.x)        # => класс  (нашли в классе)
a.x = "экземпляр"
print(a.x)        # => экземпляр  (атрибут экземпляра затенил атрибут класса)
print(A.x)        # => класс  (в классе ничего не изменилось)
del a.x
print(a.x)        # => класс  (снова видим атрибут класса)
```

### 6. Методы экземпляра / класса / статические

```python
class Pizza:
    base_price = 200

    def __init__(self, toppings):
        self.toppings = toppings

    # метод ЭКЗЕМПЛЯРА: получает self
    def price(self):
        return self.base_price + 50 * len(self.toppings)

    # метод КЛАССА: получает cls, удобен как альтернативный конструктор
    @classmethod
    def margherita(cls):
        return cls(["сыр", "томаты"])

    # СТАТИЧЕСКИЙ метод: не получает ни self, ни cls — просто функция в namespace класса
    @staticmethod
    def is_vegetarian(topping):
        return topping not in ("пепперони", "ветчина")


p = Pizza.margherita()              # classmethod как фабрика
print(p.price())                    # => 300
print(Pizza.is_vegetarian("сыр"))   # => True
```

- **classmethod** получает класс (`cls`) — корректно работает с наследованием (вернёт подкласс, если вызвать у подкласса).
- **staticmethod** — логически связанная с классом функция без доступа к состоянию.

### 7. Наследование, `super()`, переопределение

```python
class Animal:
    def __init__(self, name):
        self.name = name

    def speak(self):
        return "..."

    def describe(self):
        return f"{self.name} говорит: {self.speak()}"


class Cat(Animal):
    def __init__(self, name, color):
        super().__init__(name)   # вызываем конструктор родителя
        self.color = color

    def speak(self):             # переопределение метода
        return "Мяу"


c = Cat("Барсик", "рыжий")
print(c.describe())               # => Барсик говорит: Мяу  (позднее связывание!)
print(isinstance(c, Animal))      # => True
print(issubclass(Cat, Animal))    # => True
```

Обратите внимание: `describe()` определён в `Animal`, но вызывает `self.speak()`, который разрешается в `Cat.speak` — это **позднее связывание** (все методы виртуальные).

### 8. Множественное наследование и MRO

```python
class A:
    def who(self):
        return "A"

class B(A):
    def who(self):
        return "B"

class C(A):
    def who(self):
        return "C"

class D(B, C):   # ромбовидное наследование
    pass


d = D()
print(d.who())          # => B
print([cls.__name__ for cls in D.__mro__])
# => ['D', 'B', 'C', 'A', 'object']
```

MRO (Method Resolution Order) — порядок, в котором Python ищет методы. Вычисляется по алгоритму **C3-линеаризации** (подробности ниже).

### 9. Name mangling («приватные» атрибуты)

```python
class Account:
    def __init__(self, balance):
        self.__balance = balance     # станет _Account__balance

    def get_balance(self):
        return self.__balance


acc = Account(1000)
# print(acc.__balance)              # AttributeError
print(acc.get_balance())            # => 1000
print(acc._Account__balance)        # => 1000  (доступ есть, но через искажённое имя)
print([k for k in acc.__dict__])    # => ['_Account__balance']
```

### 10. Итераторы (протокол)

```python
class Countdown:
    def __init__(self, start):
        self.current = start

    def __iter__(self):
        return self                  # сам объект — итератор

    def __next__(self):
        if self.current <= 0:
            raise StopIteration      # сигнал конца итерации
        self.current -= 1
        return self.current + 1


for n in Countdown(3):
    print(n, end=" ")                # => 3 2 1
print()
```

### 11. Генераторы

```python
def countdown(start):
    while start > 0:
        yield start                  # yield "замораживает" функцию, сохраняя состояние
        start -= 1


gen = countdown(3)
print(type(gen))                     # => <class 'generator'>
print(list(gen))                     # => [3, 2, 1]

# Генераторное выражение (ленивое, не создаёт список в памяти):
squares = (x * x for x in range(5))
print(sum(squares))                  # => 30
```

## Подробный разбор по темам

### Пространства имён: время жизни и доступ

- **Встроенное (built-in)** пространство имён создаётся при старте интерпретатора и живёт до конца.
- **Глобальное** пространство имён модуля создаётся при его импорте/запуске и живёт до завершения интерпретатора.
- **Локальное** пространство имён функции создаётся при её вызове и удаляется при возврате (или исключении).

Важная мысль из туториала: **атрибуты — это тоже пространство имён**. `module.func`, `obj.attr`, `MyClass.method` — всё это обращения к разным пространствам имён через точечную нотацию. При этом `obj.attr` ищется динамически по правилам атрибутного доступа (экземпляр → класс → базовые классы по MRO), а не по LEGB.

Тонкость, которую любят на собеседованиях: **тело класса — это отдельная область видимости, но она НЕ участвует как enclosing-scope для методов**. Поэтому внутри метода нельзя обратиться к атрибуту класса просто по имени — нужно `self.attr` или `ClassName.attr`.

```python
class C:
    z = 10

    def method(self):
        # return z          # NameError: name 'z' is not defined
        return self.z       # правильно: => 10
```

Ещё одна классическая ловушка — list/generator comprehension внутри тела класса не видит атрибуты класса:

```python
class Demo:
    nums = [1, 2, 3]
    # squared = [n for n in nums]            # OK: первая переменная (источник) видна
    # bad = [n * factor for n in nums]       # NameError, если factor — атрибут класса
```

### Объект класса vs объект экземпляра

Когда выполняется `class Dog: ...`, Python:
1. Создаёт новое пространство имён (locals тела класса).
2. Выполняет тело класса в нём (определения методов и атрибутов попадают в это пространство).
3. Создаёт **объект класса** `Dog`, чьим `__dict__` становится это пространство имён.

Объект класса поддерживает две операции:
- **Атрибутная ссылка**: `Dog.kind`, `Dog.bark`.
- **Инстанцирование**: `Dog(...)` создаёт пустой объект-экземпляр и вызывает `Dog.__init__(instance, ...)`.

Экземпляр имеет собственный `__dict__` для атрибутов экземпляра. Атрибуты класса хранятся в `__dict__` класса и **разделяются** всеми экземплярами.

### Атрибут класса vs атрибут экземпляра — механика

Поиск **чтения** `obj.x`:
1. Дескрипторы данных класса (data descriptors) — высший приоритет (для простоты опустим, это уровень property/слотов).
2. `obj.__dict__['x']` — атрибут экземпляра.
3. `type(obj).__dict__['x']` и далее по MRO — атрибут класса.
4. `__getattr__`, если определён.

**Запись** `obj.x = ...` (без property/`__set__`) **всегда** пишет в `obj.__dict__`, создавая/перезаписывая атрибут экземпляра. Именно поэтому `a.value = 5` не трогает другие экземпляры, а `Counter.count = 10` (запись по классу) видна всем.

Ключевой нюанс: `obj.shared.append(...)` — это **чтение** `obj.shared` (находим список класса) + **мутация найденного объекта**. Присваивания нет, поэтому мутируется общий объект класса. А `obj.shared = [...]` (с присваиванием) — создаёт атрибут экземпляра. Понимание этой разницы критично для ловушки из раздела 4.

### Методы: bound vs unbound, дескрипторы

Когда вы пишете `d.bark`, происходит магия дескрипторов: функция `Dog.bark` — это объект-дескриптор, и доступ через экземпляр возвращает **связанный метод** (bound method), в котором `self` уже зафиксирован.

```python
d = Dog("Рекс")
m = d.bark              # связанный метод
print(m())              # self уже = d
print(Dog.bark)         # обычная функция
print(d.bark.__self__)  # => экземпляр d (на что привязан метод)
```

`classmethod` и `staticmethod` — тоже дескрипторы, которые меняют поведение привязки:
- `classmethod` привязывает первым аргументом **класс** (`cls`).
- `staticmethod` ничего не привязывает — возвращает исходную функцию.

Почему `classmethod` лучше для альтернативных конструкторов:

```python
class Base:
    @classmethod
    def create(cls):
        return cls()        # cls — реальный класс вызова

class Child(Base):
    pass

print(type(Base.create()))   # => <class '__main__.Base'>
print(type(Child.create()))  # => <class '__main__.Child'>  (правильно!)
```

Если бы `create` хардкодил `Base()`, то `Child.create()` вернул бы `Base`, а не `Child`.

### Наследование и `super()` подробно

`super()` (без аргументов в Python 3) возвращает прокси-объект, который делегирует вызовы **следующему классу в MRO** текущего объекта — а не «родителю» в наивном смысле. Это критически важно при множественном наследовании.

```python
class A:
    def __init__(self):
        print("A.__init__")

class B(A):
    def __init__(self):
        print("B.__init__")
        super().__init__()

class C(A):
    def __init__(self):
        print("C.__init__")
        super().__init__()

class D(B, C):
    def __init__(self):
        print("D.__init__")
        super().__init__()


D()
# => D.__init__
# => B.__init__
# => C.__init__   <-- super() в B пошёл в C, а не сразу в A!
# => A.__init__
```

Здесь видно, что `super()` следует по MRO `[D, B, C, A, object]`. Это позволяет реализовать **кооперативное множественное наследование**: каждый класс вызывает `super().__init__()`, и все инициализаторы выполняются ровно по одному разу, в правильном порядке. Если бы в `B` стояло `A.__init__(self)` напрямую, `C.__init__` был бы пропущен, а `A.__init__` вызвался бы дважды.

Старый синтаксис `super(B, self)` эквивалентен `super()` внутри методов класса `B` — иногда встречается в legacy-коде или при вызове вне обычного контекста.

### MRO и C3-линеаризация — глубоко

MRO — это линейный (упорядоченный, без повторов) список классов, по которому ищутся атрибуты и методы. Python (начиная с new-style классов) использует алгоритм **C3-линеаризации**, который гарантирует:

1. **Сохранение локального порядка предков** (если в `class D(B, C)` указано `B` раньше `C`, то в MRO `B` встретится раньше `C`).
2. **Монотонность**: MRO подкласса согласован с MRO каждого родителя.
3. Каждый класс встречается **ровно один раз**.

Формула линеаризации `L`:

```
L[C] = C + merge(L[B1], L[B2], ..., L[Bn], [B1, B2, ..., Bn])
```

где `merge` берёт «голову» (первый элемент) первого списка, которая **не входит в "хвост"** (не в начало) ни одного из остальных списков, добавляет её в результат и удаляет из всех списков; повторяет, пока списки не опустеют.

Разберём ромб `D(B, C)`, где `B(A)`, `C(A)`:

```
L[A] = [A, object]
L[B] = [B, A, object]
L[C] = [C, A, object]

L[D] = D + merge(L[B], L[C], [B, C])
     = D + merge([B,A,object], [C,A,object], [B,C])

шаг 1: голова B — не в хвостах остальных? да -> берём B
       merge([A,object], [C,A,object], [C])
шаг 2: голова A — но A в хвосте [C,A,object]! пропускаем.
       следующий список, голова C — не в хвостах? да -> берём C
       merge([A,object], [A,object], [])
шаг 3: голова A — теперь не в хвостах -> берём A
       merge([object], [object], [])
шаг 4: object -> берём object

L[D] = [D, B, C, A, object]
```

Именно поэтому `A` встречается в конце (после `B` и `C`) — общий предок «откладывается» до тех пор, пока не пройдены все его потомки. Это и решает **ромбовидную проблему** (diamond problem): `A.__init__` вызовется один раз.

Если C3 не может построить корректный порядок (противоречивые требования), Python выбрасывает ошибку **уже при определении класса**:

```python
class X: pass
class Y: pass
class A1(X, Y): pass
class B1(Y, X): pass
# class C1(A1, B1): pass
# TypeError: Cannot create a consistent method resolution order (MRO)
# для базовых классов A1, B1 — порядок X,Y и Y,X несовместим
```

Проверить MRO можно так:

```python
print(D.__mro__)          # кортеж классов
print(D.mro())            # метод, возвращает список
import inspect
print(inspect.getmro(D))  # то же через inspect
```

### Name mangling детально

Любой идентификатор вида `__spam` (минимум два ведущих подчёркивания, **максимум одно** замыкающее) внутри тела класса **текстуально** заменяется на `_ClassName__spam`. Это происходит на этапе компиляции, независимо от того, атрибут это, метод или просто локальная переменная-обращение.

Цель — **не безопасность**, а защита от случайного **переопределения** атрибута в подклассах:

```python
class Base:
    def __init__(self):
        self.__id = 1        # _Base__id

    def get_id(self):
        return self.__id     # обращается к _Base__id


class Sub(Base):
    def __init__(self):
        super().__init__()
        self.__id = 2        # _Sub__id — НЕ конфликтует с _Base__id


s = Sub()
print(s.get_id())            # => 1  (метод Base видит свой _Base__id)
print(s.__dict__)            # => {'_Base__id': 1, '_Sub__id': 2}
```

Нюансы:
- `__dunder__` (с двумя подчёркиваниями с обеих сторон) **не** искажаются — это специальные имена.
- Одно подчёркивание `_name` — просто **соглашение** «это внутреннее/защищённое», интерпретатор его не трогает (кроме `from module import *`, который игнорирует `_name` на уровне модуля).
- Mangling — текстовая замена, поэтому через искажённое имя `_Class__attr` доступ остаётся.

### Итераторы: протокол целиком

Цикл `for x in obj` под капотом делает:
1. `it = iter(obj)` → вызывает `obj.__iter__()`, получает итератор.
2. В цикле повторно `next(it)` → вызывает `it.__next__()`.
3. Когда `__next__` бросает `StopIteration` — цикл завершается.

```python
nums = [10, 20, 30]
it = iter(nums)
print(next(it))   # => 10
print(next(it))   # => 20
print(next(it))   # => 30
# print(next(it)) # StopIteration
```

Различайте:
- **Iterable (итерируемое)** — объект с `__iter__` (можно получить итератор): список, строка, словарь, файл.
- **Iterator (итератор)** — объект с `__next__` и `__iter__` (возвращающим себя). Итератор **одноразовый**: после исчерпания он пуст.

```python
class Reverse:
    """Итератор для обхода последовательности в обратном порядке."""
    def __init__(self, data):
        self.data = data
        self.index = len(data)

    def __iter__(self):
        return self

    def __next__(self):
        if self.index == 0:
            raise StopIteration
        self.index -= 1
        return self.data[self.index]


for char in Reverse("спам"):
    print(char, end="")     # => мапс
print()
```

Хорошая практика: разделять iterable и iterator, чтобы по одной коллекции можно было итерироваться многократно. Например, у списка `__iter__` каждый раз возвращает **новый** итератор — поэтому два вложенных `for` по одному списку работают независимо.

### Генераторы: глубокое погружение

**Генератор** — это компактный способ создать итератор. Любая функция, содержащая `yield`, при вызове **не выполняет тело**, а возвращает объект-генератор. Тело выполняется по кусочкам при каждом `next()`.

Чем генератор отличается от обычной функции:

| Обычная функция | Генератор |
|---|---|
| `return` возвращает значение и завершает функцию | `yield` отдаёт значение и **приостанавливает** функцию, сохраняя состояние |
| При повторном вызове начинается заново | Возобновляется с места после `yield` |
| Возвращает результат сразу | Возвращает объект-генератор (ленивый) |
| Локальные переменные теряются после возврата | Локальные переменные и счётчик команд сохраняются между `yield` |

```python
def fib(n):
    a, b = 0, 1
    for _ in range(n):
        yield a
        a, b = b, a + b


print(list(fib(8)))   # => [0, 1, 1, 2, 3, 5, 8, 13]

# Ленивость: значения вычисляются по требованию, память не растёт
g = fib(1_000_000)
print(next(g))        # => 0  (вычислили только первое, миллион чисел НЕ в памяти)
print(next(g))        # => 1
```

Генератор автоматически реализует `__iter__` и `__next__`, а `StopIteration` бросается **автоматически** при достижении конца тела функции (или `return`). Состояние (значения локальных переменных, точка остановки) сохраняется в самом объекте генератора.

**Генераторные выражения** — синтаксис как у list comprehension, но в круглых скобках; они ленивые и экономят память:

```python
# list comprehension — создаёт весь список сразу
total_list = sum([x * x for x in range(1000)])
# generator expression — вычисляет по одному, без промежуточного списка
total_gen = sum(x * x for x in range(1000))
print(total_list == total_gen)   # => True

# Удобно передавать в функции-агрегаторы:
print(max(len(word) for word in ["раз", "два", "четыре"]))  # => 6
```

**`yield from`** (кратко) — делегирует итерацию вложенному итерируемому/генератору, прокидывая значения (и поддерживая отправку значений/исключений в подгенератор):

```python
def chain(*iterables):
    for it in iterables:
        yield from it        # эквивалент: for x in it: yield x


print(list(chain([1, 2], (3, 4), "ab")))  # => [1, 2, 3, 4, 'a', 'b']
```

### Произвольные атрибуты, `__dict__`, `__slots__`

По умолчанию экземплярам можно присваивать **произвольные** атрибуты — они хранятся в `__dict__` (словаре экземпляра):

```python
class Empty:
    pass

e = Empty()
e.x = 1
e.label = "привет"
print(e.__dict__)        # => {'x': 1, 'label': 'привет'}
```

`__slots__` фиксирует допустимый набор атрибутов, убирает `__dict__` (экономит память при большом числе объектов) и **запрещает** добавление новых атрибутов:

```python
class Point:
    __slots__ = ("x", "y")    # только эти атрибуты, без __dict__

    def __init__(self, x, y):
        self.x = x
        self.y = y


p = Point(1, 2)
print(p.x, p.y)          # => 1 2
# p.z = 3                # AttributeError: 'Point' object has no attribute 'z'
# print(p.__dict__)     # AttributeError: нет __dict__
```

Когда применять `__slots__`: миллионы мелких объектов, где важна память; либо чтобы явно ограничить атрибуты. Минусы: нельзя добавлять атрибуты «на лету», тонкости с наследованием (`__dict__` появляется, если у родителя его нет в slots).

### `dataclasses` — удобная альтернатива для классов-данных

Если класс — это в основном **контейнер данных**, модуль `dataclasses` (Python 3.7+) автогенерирует `__init__`, `__repr__`, `__eq__` и др.:

```python
from dataclasses import dataclass, field


@dataclass
class Product:
    name: str
    price: float
    tags: list = field(default_factory=list)  # ИЗБЕГАЕМ ловушки изменяемого default!


p1 = Product("Книга", 500)
p2 = Product("Книга", 500)
print(p1)             # => Product(name='Книга', price=500, tags=[])
print(p1 == p2)       # => True  (__eq__ сгенерирован автоматически)
```

Важно: для изменяемых значений по умолчанию (`list`, `dict`) **обязателен** `field(default_factory=...)` — это тот же урок про общий изменяемый объект, что и с атрибутами класса. Прямое `tags: list = []` вызовет ошибку на этапе определения класса (dataclass это специально проверяет).

Альтернативы для классов-данных: `collections.namedtuple` (неизменяемый), `typing.NamedTuple`, `attrs`, `pydantic` (с валидацией).

## Частые вопросы на собеседовании (Q/A)

**Q1. В чём разница между атрибутом класса и атрибутом экземпляра?**
Атрибут класса определяется в теле класса и хранится в `__dict__` класса; он **один на все экземпляры**. Атрибут экземпляра обычно создаётся в `__init__` через `self.x = ...` и хранится в `__dict__` экземпляра — у каждого свой. При чтении `obj.x` сначала ищется в экземпляре, затем в классе (по MRO). При записи `obj.x = ...` всегда создаётся атрибут экземпляра.

**Q2. Объясните ловушку изменяемого атрибута класса.**
Если объявить `tricks = []` на уровне класса и мутировать его через `self.tricks.append(...)`, то все экземпляры разделяют **один и тот же список**, потому что `self.tricks` находит атрибут класса (присваивания нет — только мутация). Решение: создавать список в `__init__` как `self.tricks = []`. Тот же урок касается mutable default-аргументов функций и dataclass-полей (`default_factory`).

**Q3. Что такое `self` и обязательно ли так называть первый параметр?**
`self` — ссылка на текущий экземпляр, передаётся неявно при вызове `obj.method()`. Имя `self` — соглашение, технически можно назвать иначе, но не нужно — это нарушит читаемость и ожидания инструментов.

**Q4. Чем отличаются `@classmethod`, `@staticmethod` и обычный метод?**
Обычный метод получает `self` (экземпляр). `@classmethod` получает `cls` (класс) — полезен для альтернативных конструкторов и корректно работает с наследованием. `@staticmethod` не получает ничего особого — это просто функция в пространстве имён класса, логически с ним связанная.

**Q5. Что делает `super()` и почему он лучше прямого вызова `Parent.method(self)`?**
`super()` делегирует **следующему классу в MRO** текущего объекта, а не жёстко родителю. Это обеспечивает кооперативное множественное наследование: каждый метод в цепочке вызывается ровно один раз и в правильном порядке. Прямой вызов `Parent.method(self)` ломает MRO: может пропустить классы или вызвать общего предка несколько раз.

**Q6. Что такое MRO и как он вычисляется?**
MRO (Method Resolution Order) — порядок поиска атрибутов/методов. Вычисляется алгоритмом **C3-линеаризации**: `L[C] = C + merge(L[base1], ..., L[basen], [base1..basen])`. Он сохраняет локальный порядок наследования, обеспечивает монотонность и единократное вхождение каждого класса. Посмотреть: `Cls.__mro__` или `Cls.mro()`.

**Q7. Что такое ромбовидная проблема (diamond problem) и как Python её решает?**
Когда `D(B, C)`, а `B` и `C` оба наследуют `A`. Вопрос — сколько раз вызывать код `A` и в каком порядке. C3-линеаризация ставит `A` в конец MRO (`[D, B, C, A, object]`), а `super()` гарантирует единократный вызов `A.__init__`. Это устраняет дублирование и неоднозначность.

**Q8. Объясните разницу между `is`-проверкой типа, `type(x) == T`, `isinstance` и `issubclass`.**
`isinstance(obj, T)` — `True`, если объект экземпляр `T` **или его подкласса** (учитывает наследование). `issubclass(A, B)` — `True`, если `A` подкласс `B`. `type(x) == T` проверяет точный тип без учёта наследования. На практике почти всегда нужен `isinstance` (полиморфизм).

**Q9. Что такое name mangling? `__attr` — это приватность?**
Имя `__attr` внутри класса текстуально превращается в `_ClassName__attr`. Это **не** настоящая приватность (доступ остаётся через искажённое имя), а защита от случайного переопределения атрибута в подклассах. Соглашение `_attr` (одно подчёркивание) — просто «внутреннее», интерпретатор его не трогает.

**Q10. В чём разница между iterable и iterator?**
Iterable — объект с `__iter__()`, из которого можно получить итератор (список, строка). Iterator — объект с `__next__()` и `__iter__()` (возвращающим себя); он хранит состояние обхода и **одноразовый**. `for` вызывает `iter()` для получения итератора, затем `next()` до `StopIteration`.

**Q11. Опишите протокол итератора.**
`iter(obj)` → `obj.__iter__()` возвращает итератор. `next(it)` → `it.__next__()` возвращает следующий элемент или бросает `StopIteration`. Цикл `for` инкапсулирует это и корректно ловит `StopIteration`.

**Q12. Чем генератор отличается от обычной функции?**
Функция с `yield` при вызове возвращает объект-генератор и **не выполняет тело сразу**. На каждом `next()` тело выполняется до следующего `yield`, который приостанавливает функцию, **сохраняя локальное состояние и точку остановки**. По завершении тела автоматически бросается `StopIteration`. Генератор ленив и экономит память.

**Q13. Когда использовать генератор вместо списка?**
Когда данные большие/бесконечные или нужны по одному (потоковая обработка, конвейеры), чтобы не держать всё в памяти. Генераторные выражения `(x for x in ...)` лучше list comprehension в `sum()`, `any()`, `max()` и т.п. Список нужен, когда требуется многократный обход, индексирование или `len()`.

**Q14. Что такое `__dict__` и `__slots__`?**
`__dict__` — словарь атрибутов экземпляра; позволяет добавлять произвольные атрибуты. `__slots__` фиксирует набор атрибутов, убирает `__dict__`, экономит память и запрещает новые атрибуты. Используется при большом числе объектов; усложняет наследование и динамику.

**Q15. Объясните LEGB и разницу `global` / `nonlocal`.**
LEGB — порядок поиска имён: Local → Enclosing → Global → Built-in. `global x` позволяет присваивать глобальной переменной модуля внутри функции. `nonlocal x` — переменной ближайшей объемлющей функции (не глобальной). Без них присваивание создаёт локальную переменную.

**Q16. Почему обращение к переменной до присваивания внутри функции даёт `UnboundLocalError`?**
Если в теле функции есть присваивание имени, Python на этапе компиляции считает это имя **локальным во всей функции**. Чтение до присваивания обращается к ещё не инициализированной локальной переменной → `UnboundLocalError`. Лечится `global`/`nonlocal` или переименованием.

**Q17. Все ли методы в Python виртуальные? Что такое позднее связывание?**
Да. Вызов `self.method()` всегда разрешается по реальному типу объекта (по MRO во время выполнения), а не по типу, где метод вызывается. Поэтому метод базового класса, вызывающий `self.other()`, попадёт в переопределение из подкласса. Это и есть позднее связывание (late binding).

**Q18. Можно ли в Python обращаться к атрибуту класса по имени внутри метода без `self`?**
Нет. Тело класса не образует enclosing-scope для методов. Внутри метода к атрибутам класса обращаются через `self.attr` (с учётом возможного переопределения в экземпляре/подклассе) или явно `ClassName.attr`. Обращение просто по имени даст `NameError`.

**Q19. Почему `@classmethod` предпочтителен для альтернативных конструкторов?**
Потому что он получает `cls` — реальный класс, у которого вызван метод. Если в подклассе вызвать унаследованный classmethod-фабрику, она создаст экземпляр **подкласса**, а не базового класса. Хардкод имени базового класса сломал бы наследование.

## Подводные камни (gotchas)

1. **Изменяемый атрибут класса разделяется всеми экземплярами.** `tricks = []` на уровне класса + `self.tricks.append()` → общий список. Заводите изменяемые данные в `__init__`.

2. **Mutable default-аргумент.** `def f(x, acc=[])` — список создаётся один раз при определении функции и переживает между вызовами. Используйте `acc=None` + `if acc is None: acc = []`.

3. **`UnboundLocalError`.** Любое присваивание имени в функции делает его локальным во всей функции; чтение до присваивания падает. Нужен `global`/`nonlocal`.

4. **`super()` vs прямой вызов родителя.** В множественном наследовании `Parent.__init__(self)` ломает MRO. Используйте `super().__init__()` во всех классах кооперативно.

5. **Тело класса не видно методам по имени.** Внутри метода `return attr` → `NameError`; нужно `self.attr`. Comprehension в теле класса тоже не видит атрибуты класса (кроме самого первого итерируемого).

6. **Итератор одноразовый.** После прохода `for` по генератору/итератору он исчерпан — повторный `for` ничего не даст. Список многоразовый, генератор — нет.

7. **Запись `obj.x = ...` всегда создаёт атрибут экземпляра** и затеняет атрибут класса — даже если вы хотели изменить «общее» значение. Для изменения атрибута класса пишите `ClassName.x = ...`.

8. **Name mangling — не безопасность.** `__attr` доступен как `_Class__attr`. Не полагайтесь на него для сокрытия секретов.

9. **`__slots__` запрещает новые атрибуты и убирает `__dict__`.** Код, ожидающий динамические атрибуты или `vars(obj)`, сломается. При наследовании от класса без slots `__dict__` всё равно появится.

10. **`is` vs `==`.** `is` сравнивает идентичность (один и тот же объект в памяти), `==` — равенство (через `__eq__`). Для сравнения значений используйте `==`; `is` — только для `None`, `True`, `False`, синглтонов.

11. **MRO-конфликт падает при определении класса.** Несовместимый порядок баз → `TypeError: Cannot create a consistent MRO`. Проверяйте порядок базовых классов.

12. **dataclass с изменяемым default.** `tags: list = []` вызовет ошибку; нужен `field(default_factory=list)`.

## Шпаргалка

```python
# --- Области видимости ---
# LEGB: Local -> Enclosing -> Global -> Built-in
global name      # присваивать глобальной переменной модуля
nonlocal name    # присваивать переменной объемлющей функции

# --- Класс и экземпляр ---
class MyClass(Base):
    attr = 0                 # атрибут КЛАССА (общий)
    def __init__(self, x):
        self.x = x           # атрибут ЭКЗЕМПЛЯРА (свой)
    def method(self): ...        # метод экземпляра (self)
    @classmethod
    def cmethod(cls): ...        # метод класса (cls) — для фабрик
    @staticmethod
    def smethod(): ...           # статический — без self/cls

obj = MyClass(1)             # создание экземпляра -> вызов __init__

# --- Наследование ---
class Child(Parent):
    def __init__(self, *a):
        super().__init__(*a)     # кооперативный вызов по MRO
isinstance(obj, MyClass)         # True для класса и подклассов
issubclass(Child, Parent)        # проверка иерархии
Child.__mro__                    # порядок разрешения методов (C3)
Child.mro()                      # то же, метод

# --- Приватность (соглашения) ---
self._internal      # "защищённое" (соглашение)
self.__private      # name mangling -> self._ClassName__private

# --- Итератор (протокол) ---
class It:
    def __iter__(self): return self
    def __next__(self):
        if конец: raise StopIteration
        return значение
```

**MRO (C3) в одной формуле:**
`L[C] = C + merge(L[B1], ..., L[Bn], [B1, ..., Bn])`, где `merge` берёт голову первого списка, не входящую в хвосты остальных.

**Три правила «общего vs своего»:**
- Чтение `obj.x`: экземпляр → класс → базовые (по MRO).
- Запись `obj.x = v`: ВСЕГДА в экземпляр.
- `obj.mutable.append(v)`: мутирует найденный объект (часто общий объект класса!).
