# Глава 12. Виртуальные окружения и пакеты (Virtual Environments and Packages) — конспект и вопросы

## О чём глава

Реальные приложения зависят от сторонних пакетов (библиотек), которые ставятся из интернета. Проблема: разным проектам нужны **разные версии** одних и тех же библиотек. Если ставить всё в системный Python, версии конфликтуют — обновление пакета под проект А ломает проект Б. Решение — **виртуальные окружения**: изолированные «песочницы» со своим набором пакетов под каждый проект. Эта глава учит создавать окружения через `venv` и управлять пакетами через `pip`.

Это одна из самых практичных глав: на собеседовании ожидают, что junior умеет создать окружение, поставить зависимости и зафиксировать их в `requirements.txt`.

## Ключевые концепции (с примерами и выводом)

### Зачем нужны виртуальные окружения

Представим: приложение A требует `requests==2.25`, а приложение B — `requests==2.31`. В одном системном Python обе версии не уживутся: установка второй перезапишет первую. Виртуальное окружение даёт каждому проекту **собственную** копию интерпретатора и папку с библиотеками, полностью изолированную от системы и других проектов.

Выгоды:
- Изоляция зависимостей — у каждого проекта свой набор и версии пакетов.
- Нет конфликтов версий между проектами.
- Не нужны права администратора — пакеты ставятся в папку проекта.
- Воспроизводимость — легко повторить точное окружение на другой машине.

### Создание окружения через python -m venv

```bash
# Создать виртуальное окружение в папке .venv (или tutorial-env)
python3 -m venv .venv

# Структура: создаётся папка с копией интерпретатора и pip
# .venv/
#   bin/        (Scripts/ на Windows) — python, pip, activate
#   lib/        — установленные пакеты
#   pyvenv.cfg  — конфигурация окружения
```

Распространённое соглашение — называть папку `.venv` или `venv` и добавлять её в `.gitignore` (окружение не коммитят, коммитят только `requirements.txt`).

### Активация в разных ОС

После создания окружение нужно **активировать** — тогда команды `python` и `pip` будут указывать на изолированную версию.

```bash
# Linux / macOS (bash / zsh)
source .venv/bin/activate

# Windows (cmd.exe)
.venv\Scripts\activate.bat

# Windows (PowerShell)
.venv\Scripts\Activate.ps1
```

После активации в приглашении терминала появляется префикс `(.venv)`:

```bash
(.venv) $ python --version
# => Python 3.12.0  (версия из окружения)

(.venv) $ which python      # на Windows: where python
# => /path/to/project/.venv/bin/python  (интерпретатор из окружения)
```

Деактивация — командой `deactivate`:

```bash
(.venv) $ deactivate
$            # префикс исчез, вернулись к системному Python
```

### Управление пакетами через pip

`pip` — стандартный менеджер пакетов, ставит библиотеки из PyPI (Python Package Index).

**Установка пакета (последняя версия):**

```bash
(.venv) $ pip install requests
# => Successfully installed requests-2.31.0 ... (со всеми зависимостями)
```

**Установка конкретной версии:**

```bash
(.venv) $ pip install requests==2.25.1     # ровно эта версия
(.venv) $ pip install "requests>=2.20,<3.0" # диапазон версий
```

**Обновление пакета до последней версии:**

```bash
(.venv) $ pip install --upgrade requests
```

**Удаление пакета:**

```bash
(.venv) $ pip uninstall requests
```

**Список установленных пакетов:**

```bash
(.venv) $ pip list
# =>
# Package    Version
# ---------- -------
# certifi    2024.2.2
# requests   2.31.0
# urllib3    2.2.1
```

**Информация о конкретном пакете:**

```bash
(.venv) $ pip show requests
# =>
# Name: requests
# Version: 2.31.0
# Summary: Python HTTP for Humans.
# Location: /path/to/.venv/lib/python3.12/site-packages
# Requires: certifi, charset-normalizer, idna, urllib3
```

### Фиксация зависимостей: freeze и requirements.txt

`pip freeze` выводит все установленные пакеты с точными версиями — это «снимок» окружения для воспроизведения.

```bash
# Сохранить текущие зависимости в файл
(.venv) $ pip freeze > requirements.txt

# Содержимое requirements.txt:
# certifi==2024.2.2
# requests==2.31.0
# urllib3==2.2.1
```

`requirements.txt` коммитят в репозиторий. На другой машине окружение восстанавливают одной командой:

```bash
(.venv) $ pip install -r requirements.txt
# => ставит все пакеты ровно тех версий, что в файле
```

## Подробный разбор по темам

### Полный рабочий цикл проекта

```bash
# 1. Создать окружение
python3 -m venv .venv

# 2. Активировать
source .venv/bin/activate          # Linux/macOS

# 3. Установить нужные пакеты
pip install requests flask

# 4. Зафиксировать зависимости
pip freeze > requirements.txt

# 5. Закоммитить requirements.txt (но НЕ папку .venv)

# --- на другой машине / у коллеги ---
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt    # восстановить точное окружение
```

### Почему .venv не коммитят в git

Папка окружения содержит бинарники интерпретатора и пакеты — она большая, платформо-зависимая (Windows-окружение не запустится на Linux) и легко пересоздаётся из `requirements.txt`. Коммитят только список зависимостей. Папку добавляют в `.gitignore`.

### Точные версии vs диапазоны

- `requests==2.31.0` — ровно эта версия (максимальная воспроизводимость, `pip freeze` пишет так).
- `requests>=2.20` — не ниже указанной (гибче, но риск, что новая версия сломает совместимость).
- `requests~=2.31.0` — совместимая версия (то же, что `>=2.31.0, <2.32.0`).

Для приложений предпочитают точные версии (воспроизводимость), для библиотек — диапазоны (совместимость с проектами пользователей).

### PyPI — откуда берутся пакеты

PyPI (Python Package Index, pypi.org) — центральный репозиторий, откуда `pip` скачивает пакеты. Там сотни тысяч библиотек. `pip install имя` ищет пакет именно там.

### Современная экосистема: poetry, uv, pipenv

Стандартные `venv` + `pip` + `requirements.txt` работают, но у них есть ограничения (нет lock-файла с хешами, ручное управление). В индустрии популярны более продвинутые инструменты:

- **poetry** — управление зависимостями и сборка пакетов; описывает зависимости в `pyproject.toml`, создаёт `poetry.lock` с точными версиями и хешами, сам управляет окружениями.
- **uv** — очень быстрый менеджер пакетов и окружений на Rust (от Astral); drop-in замена pip/venv, в десятки раз быстрее. Набирает популярность как современный стандарт.
- **pipenv** — комбинирует pip и virtualenv, использует `Pipfile`/`Pipfile.lock`.
- **conda** — менеджер окружений и пакетов, популярный в data science (умеет ставить не только Python-пакеты).

Все они решают ту же задачу — изоляция и воспроизводимость, но удобнее и надёжнее (lock-файлы, разрешение конфликтов). Для собеседования достаточно знать, что они существуют и зачем нужны, а базовый `venv` + `pip` понимать обязательно.

### pyproject.toml — современный способ описания проекта

Современные проекты описывают метаданные и зависимости в `pyproject.toml` (стандарт PEP 518/621) вместо `requirements.txt` + `setup.py`. Это единый файл конфигурации, который понимают pip, poetry, uv и др.

## Частые вопросы на собеседовании (Q/A)

**В: Зачем нужны виртуальные окружения?**
О: Чтобы изолировать зависимости проектов. Разным проектам нужны разные версии одних библиотек; в общем системном Python они конфликтуют. Окружение даёт каждому проекту свой изолированный набор пакетов.

**В: Как создать и активировать виртуальное окружение?**
О: Создать — `python -m venv .venv`. Активировать на Linux/macOS — `source .venv/bin/activate`, на Windows — `.venv\Scripts\activate`. Деактивировать — `deactivate`.

**В: Что делает `pip freeze` и зачем `requirements.txt`?**
О: `pip freeze` выводит все установленные пакеты с точными версиями. Сохранив их в `requirements.txt` и закоммитив, можно воспроизвести то же окружение на другой машине через `pip install -r requirements.txt`.

**В: Как установить конкретную версию пакета?**
О: `pip install package==1.2.3` для точной версии или `pip install "package>=1.0,<2.0"` для диапазона.

**В: Нужно ли коммитить папку виртуального окружения в git?**
О: Нет. Она большая, платформо-зависимая и пересоздаётся из `requirements.txt`. Коммитят только файл зависимостей, а папку (`.venv`) добавляют в `.gitignore`.

**В: Что такое PyPI?**
О: Python Package Index (pypi.org) — центральный публичный репозиторий пакетов, откуда `pip` их скачивает.

**В: Чем отличаются `pip list` и `pip freeze`?**
О: Оба показывают установленные пакеты. `pip list` — в табличном человекочитаемом виде, `pip freeze` — в формате `пакет==версия`, готовом для `requirements.txt`.

**В: В чём преимущество poetry/uv над `pip` + `venv`?**
О: Они дают lock-файл с точными версиями и хешами (надёжная воспроизводимость), автоматически разрешают конфликты зависимостей, управляют окружениями и сборкой через один файл (`pyproject.toml`). `uv` к тому же очень быстрый.

**В: Как обновить пакет до последней версии?**
О: `pip install --upgrade имя_пакета` (или `-U`).

**В: Что произойдёт, если ставить пакеты без активного окружения?**
О: Они установятся в системный/пользовательский Python, могут потребовать прав администратора и вызвать конфликты версий между проектами — именно то, чего избегают окружениями.

**В: Как посмотреть зависимости и расположение конкретного пакета?**
О: `pip show имя` — покажет версию, описание, путь установки и список зависимостей (`Requires`).

## Подводные камни (gotchas)

- **Забыли активировать окружение** — `pip install` поставит пакет в систему, а не в проект. Проверяйте префикс `(.venv)` в приглашении или `which python`.
- **Коммит папки `.venv`** в git — раздувает репозиторий и не переносится между ОС. Добавляйте в `.gitignore`.
- **Активация на Windows** отличается: `Scripts\activate`, а не `bin/activate`; в PowerShell может потребоваться разрешение на запуск скриптов (`Set-ExecutionPolicy`).
- **`pip freeze` пишет ВСЕ пакеты**, включая транзитивные зависимости — иногда удобнее вручную поддерживать список прямых зависимостей (или использовать poetry/uv с разделением).
- **Диапазоны версий без верхней границы** (`>=2.0`) могут подтянуть несовместимую новую мажорную версию и сломать проект.
- **Разные `python` и `python3`** — на некоторых системах `python` указывает на Python 2; используйте `python3 -m venv`.
- **Глобальный `pip install`** под sudo может повредить системные пакеты ОС — всегда работайте в окружении.
- **Окружение привязано к версии Python**, с которой создано; смена версии интерпретатора требует пересоздания окружения.

## Шпаргалка

```bash
# Создать окружение
python3 -m venv .venv

# Активировать
source .venv/bin/activate          # Linux / macOS
.venv\Scripts\activate             # Windows (cmd)
.venv\Scripts\Activate.ps1         # Windows (PowerShell)

# Деактивировать
deactivate

# Установить / конкретная версия / диапазон
pip install requests
pip install requests==2.31.0
pip install "requests>=2.20,<3.0"

# Обновить / удалить
pip install --upgrade requests
pip uninstall requests

# Посмотреть пакеты
pip list                           # таблица
pip show requests                  # детали пакета

# Зафиксировать и восстановить зависимости
pip freeze > requirements.txt
pip install -r requirements.txt
```

| Понятие | Суть |
|---|---|
| Виртуальное окружение | Изолированная папка с интерпретатором и пакетами под проект |
| `venv` | Стандартный модуль для создания окружений |
| `pip` | Менеджер пакетов, ставит библиотеки с PyPI |
| `requirements.txt` | Список зависимостей с версиями для воспроизводимости |
| PyPI | Центральный репозиторий пакетов (pypi.org) |
| poetry / uv / pipenv | Современные продвинутые менеджеры зависимостей |
| pyproject.toml | Современный файл конфигурации проекта (PEP 518/621) |

Главное правило: **новый проект — новое окружение**. Коммитим `requirements.txt`, а не `.venv`.
