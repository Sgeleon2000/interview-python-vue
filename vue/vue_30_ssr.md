# Vue 3 — Серверный рендеринг (SSR) — конспект и вопросы

## О чём раздел

По умолчанию Vue-приложение работает как **SPA / CSR** (Client-Side Rendering): сервер отдаёт
пустой `<div id="app">`, а весь HTML строится в браузере на JavaScript. Это плохо для SEO (боты
видят пустую страницу) и даёт долгий **First Contentful Paint** на слабых устройствах/сетях.

**SSR (Server-Side Rendering)** решает это: тот же Vue-компонент рендерится в HTML **на сервере**
и отдаётся клиенту уже готовым. Пользователь сразу видит контент (быстрый FCP), а поисковики
получают полноценную разметку. Затем на клиенте происходит **гидратация** — Vue «оживляет» уже
существующий HTML, навешивает обработчики и делает страницу интерактивной.

Главный «топ-вопрос» этой темы — **hydration mismatch** (несоответствие гидратации): когда HTML,
сгенерированный на сервере, не совпадает с тем, что Vue ожидает построить на клиенте. Ему ниже
посвящён отдельный подробный раздел.

## Ключевые концепции

- **CSR** — рендер в браузере (пустой HTML + JS). **SSR** — рендер HTML на сервере на каждый
  запрос. **SSG** — рендер HTML заранее, на этапе сборки (статические файлы).
- **Гидратация (hydration)** — процесс, в котором клиентский Vue не создаёт DOM заново, а
  «прикрепляется» к серверному HTML: сопоставляет виртуальный DOM с существующими элементами и
  навешивает реактивность и обработчики.
- **Hydration mismatch** — расхождение между серверной и клиентской разметкой; Vue выводит
  предупреждение и перерисовывает несовпавший участок, теряя выгоду SSR.
- **`renderToString`** (из `vue/server-renderer`) — основная функция: превращает инстанс
  приложения в строку HTML на сервере.
- **Изоморфный/универсальный код** — один и тот же код выполняется и на сервере, и в браузере.
  Поэтому нельзя обращаться к `window`/`document` на этапе, который выполняется на сервере.
- **Новый инстанс приложения на каждый запрос** — состояние нельзя шарить между запросами
  (иначе данные одного пользователя утекут другому).
- **Nuxt** — метафреймворк поверх Vue, который берёт на себя весь SSR/SSG/роутинг/сборку.
- **Частичная гидратация / острова (islands)** — гидратировать не всю страницу, а только
  интерактивные части, экономя JS.

## Подробный разбор с примерами кода

### SSR vs SSG vs CSR

| | CSR (SPA) | SSR | SSG |
|---|---|---|---|
| Где рендерится HTML | в браузере | на сервере, на каждый запрос | на сборке, заранее |
| SEO | плохо | хорошо | хорошо |
| FCP / TTFB | медленный FCP | быстрый FCP, TTFB зависит от сервера | очень быстрый (CDN) |
| Свежесть данных | всегда свежие | свежие на каждый запрос | на момент сборки |
| Сложность/стоимость | низкая | высокая (нужен сервер) | средняя |
| Когда использовать | дашборды за логином | новости, e-commerce, динамика | блоги, доки, лендинги |

> Часто комбинируют: статичные страницы — SSG, динамичные — SSR, приватная админка — CSR. Nuxt
> позволяет задавать режим рендеринга по маршрутам (гибридный рендеринг).

### Минимальный SSR вручную: renderToString

Серверная часть рендерит приложение в строку:

```js
// server.js (Node + Express)
import express from 'express'
import { createSSRApp } from 'vue'
import { renderToString } from 'vue/server-renderer'

const server = express()

server.get('*', async (req, res) => {
  // ВАЖНО: новый инстанс приложения НА КАЖДЫЙ запрос — никакого общего состояния
  const app = createSSRApp({
    data: () => ({ count: 1 }),
    template: `<button>count: {{ count }}</button>`,
  })

  // renderToString превращает приложение в HTML-строку
  const html = await renderToString(app)

  res.send(`
    <!DOCTYPE html>
    <html>
      <head><title>SSR</title></head>
      <body>
        <div id="app">${html}</div>
        <script type="module" src="/entry-client.js"></script>
      </body>
    </html>
  `)
})

server.listen(3000)
```

### Гидратация на клиенте

Клиентская точка входа использует `createSSRApp` (а не `createApp`!) и `mount`, который
**гидратирует** существующий HTML, а не создаёт его заново:

```js
// entry-client.js
import { createSSRApp } from 'vue'

const app = createSSRApp({
  data: () => ({ count: 1 }),
  template: `<button @click="count++">count: {{ count }}</button>`,
})

// mount по существующему #app выполнит ГИДРАТАЦИЮ:
// Vue не перестроит DOM, а прикрепится к серверному и навесит обработчики.
app.mount('#app')
```

> Ключевое: компонент-определение на сервере и на клиенте должно быть **одинаковым**. Иначе —
> hydration mismatch.

### Фабрика приложения (universal entry)

Чтобы код переиспользовался на сервере и клиенте без утечки состояния, приложение и его сторы
создают через **фабрику** — функцию, возвращающую свежий инстанс:

```js
// app.js — общий код
import { createSSRApp } from 'vue'
import { createPinia } from 'pinia'
import App from './App.vue'
import { createRouter } from './router'

// Фабрика: вызывается отдельно на каждом запросе сервера и один раз на клиенте
export function createApp() {
  const app = createSSRApp(App)
  const pinia = createPinia()       // СВОЯ Pinia на каждый запрос
  const router = createRouter()     // СВОЙ роутер на каждый запрос
  app.use(pinia)
  app.use(router)
  return { app, router, pinia }
}
```

### Hydration mismatch — главный вопрос собеседования

**Что это.** Vue ожидает, что DOM, который он построит на клиенте при первом рендере, **точно
совпадёт** с HTML, пришедшим с сервера. Если они различаются — это hydration mismatch. В dev-режиме
Vue логирует предупреждение `Hydration node mismatch` и **выбрасывает серверный HTML для несовпавшего
поддерева, перерисовывая его заново** на клиенте. Это значит мерцание, потеря выгоды SSR и иногда
визуальный «скачок» контента.

**Почему возникает.** Любой источник недетерминированности между сервером и клиентом:

```vue
<script setup>
// ❌ Date.now()/Math.random()/new Date() дают РАЗНЫЕ значения на сервере и клиенте
const id = Math.random()                 // mismatch гарантирован
const now = new Date().toLocaleString()  // время сервера ≠ время клиента
</script>
<template>
  <p>{{ now }} — {{ id }}</p>
</template>
```

Типичные причины:
- **Случайность/время:** `Math.random()`, `Date.now()`, `new Date()`, рандомные id.
- **Доступ к `window`/`localStorage` при первом рендере:** на сервере их нет, ветка кода
  расходится (например, `const isMobile = window.innerWidth < 768`).
- **Невалидная вложенность HTML:** браузер «чинит» некорректную разметку (`<div>` внутри `<p>`,
  `<p>` внутри `<p>`, блок внутри `<a>`), и итоговый DOM перестаёт совпадать с серверным.
- **Данные различаются:** сервер отрендерил по одному состоянию стора, а клиент при гидратации
  имеет другое (не передали серверное состояние клиенту — см. ниже).

**Как лечить.**

1. Передавать серверное состояние клиенту, чтобы данные совпали:

```js
// сервер: после рендера сериализуем состояние Pinia в HTML
const state = JSON.stringify(pinia.state.value)
html += `<script>window.__INITIAL_STATE__ = ${state}<\/script>`

// клиент: восстанавливаем ДО mount, чтобы первый рендер совпал с серверным
if (window.__INITIAL_STATE__) {
  pinia.state.value = window.__INITIAL_STATE__
}
```

2. Клиент-специфичное выносить из первого рендера — в `onMounted` (он выполняется **только в
   браузере**, после гидратации):

```vue
<script setup>
import { ref, onMounted } from 'vue'
const isMobile = ref(false)            // на сервере и при гидратации — false (совпадает)
onMounted(() => {
  // выполнится только на клиенте, ПОСЛЕ гидратации — mismatch не возникнет
  isMobile.value = window.innerWidth < 768
})
</script>
```

3. Для заведомо клиентского контента — `<ClientOnly>` (Nuxt) или `<Suspense>` + отрисовка после
   `onMounted`, либо встроенная директива `v-once` для статичного куска.
4. Чинить невалидную вложенность HTML.

### Особенности кода для SSR

```vue
<script setup>
import { onMounted, onBeforeMount } from 'vue'

// ❌ window/document на верхнем уровне setup — упадёт на сервере (их там нет)
// const w = window.innerWidth

// ✅ Обращение к браузерным API — только в onMounted (клиент-онли хук)
onMounted(() => {
  document.title = 'Загружено'
  window.addEventListener('resize', onResize)
})

// onBeforeMount/onMounted и далее НЕ вызываются на сервере.
// На сервере выполняются только setup и beforeCreate/created (sync-часть).
</script>
```

Ключевые правила SSR-кода:
- **Нет `window`, `document`, `navigator`, `localStorage`** на сервере — оборачивайте проверкой
  `typeof window !== 'undefined'` или выносите в `onMounted`.
- **`onMounted` и последующие хуки** не вызываются на сервере — таймеры, подписки, обращения к DOM
  ставьте туда.
- **Никаких побочных эффектов с утечкой:** интервалы/подписки, не очищенные в `onUnmounted`, на
  сервере приведут к утечкам памяти между запросами.
- **Состояние — на каждый запрос своё:** глобальные синглтоны (общий `reactive`-объект, общий
  стор-модуль) шарятся между запросами — данные одного пользователя утекут другому. Поэтому
  Pinia/роутер создаются фабрикой внутри обработчика запроса.
- **Только разрешённые в Node API** в коде, выполняемом на сервере.

### Nuxt как метафреймворк

Писать SSR вручную (сборка двух бандлов, сериализация состояния, роутинг, кеш) — трудоёмко.
**Nuxt** даёт всё из коробки:
- файловый роутинг (`pages/`), авто-импорты компонентов и composables;
- встроенный SSR/SSG/гибридный рендеринг, настраиваемый по маршрутам (`routeRules`);
- `useFetch`/`useAsyncData` — загрузка данных, корректно работающая и на сервере (с передачей
  состояния клиенту, без двойного запроса);
- `<ClientOnly>` для клиент-онли блоков, `useState` для SSR-безопасного состояния;
- управление `<head>` (SEO-метатеги), middleware, серверные API-роуты (`server/`).

```vue
<!-- pages/users.vue (Nuxt) -->
<script setup>
// useAsyncData: на сервере выполнит запрос, отрендерит HTML и передаст данные клиенту,
// на клиенте при гидратации использует уже полученные данные (без повторного запроса)
const { data: users, pending } = await useAsyncData('users', () => $fetch('/api/users'))
</script>
<template>
  <p v-if="pending">Загрузка…</p>
  <ul v-else>
    <li v-for="u in users" :key="u.id">{{ u.name }}</li>
  </ul>
</template>
```

### Частичная гидратация и острова (islands) кратко

Полная гидратация требует отправить клиенту весь JS приложения, даже если интерактивна лишь
малая часть страницы. Идеи экономии:
- **Острова (islands)** — статичный HTML по умолчанию, а интерактивными «оживляются» только
  отдельные компоненты-острова (подход Astro; в Nuxt есть `<NuxtIsland>`/server components).
- **Ленивая гидратация** — гидратировать компонент по условию: при появлении во вьюпорте, по
  взаимодействию, в idle. Во Vue 3.5+ появились хелперы lazy/async hydration
  (`hydrateOnVisible`, `hydrateOnIdle`, `hydrateOnInteraction`) для асинхронных компонентов:

```js
import { defineAsyncComponent, hydrateOnVisible } from 'vue'

// Компонент гидратируется только когда попадёт во вьюпорт — меньше работы при загрузке
const Comments = defineAsyncComponent({
  loader: () => import('./Comments.vue'),
  hydrate: hydrateOnVisible(),
})
```

Цель — отдать пользователю быстрый интерактив там, где он нужен, и не тратить CPU/JS на остальное.

## Полный рабочий пример

Универсальная фабрика + сервер с передачей состояния (упрощённо, без сборщика):

```js
// app.js — общий код (сервер и клиент)
import { createSSRApp } from 'vue'
import { createPinia } from 'pinia'
import App from './App.vue'

export function createApp() {
  const app = createSSRApp(App)
  const pinia = createPinia()
  app.use(pinia)
  return { app, pinia }
}
```

```js
// entry-server.js
import { renderToString } from 'vue/server-renderer'
import { createApp } from './app.js'

export async function render() {
  const { app, pinia } = createApp()
  // здесь могла бы быть предзагрузка данных в стор: await store.fetch()
  const html = await renderToString(app)
  // сериализуем состояние, чтобы клиент стартовал с теми же данными
  const state = JSON.stringify(pinia.state.value)
  return { html, state }
}
```

```js
// entry-client.js
import { createApp } from './app.js'

const { app, pinia } = createApp()
// Восстанавливаем серверное состояние ДО mount — чтобы первый рендер совпал (нет mismatch)
if (window.__INITIAL_STATE__) {
  pinia.state.value = window.__INITIAL_STATE__
}
app.mount('#app')   // гидратация существующего HTML
```

```js
// server.js
import express from 'express'
import { render } from './entry-server.js'

const server = express()
server.get('*', async (req, res) => {
  const { html, state } = await render()
  res.send(`<!DOCTYPE html>
<html><head><title>SSR</title></head>
<body>
  <div id="app">${html}</div>
  <script>window.__INITIAL_STATE__ = ${state}<\/script>
  <script type="module" src="/entry-client.js"></script>
</body></html>`)
})
server.listen(3000)
```

## Частые вопросы на собеседовании (Q/A)

**1. Что такое SSR и зачем он нужен?**
Рендеринг Vue-компонентов в HTML на сервере и отдача готовой разметки. Выгоды: SEO (боты видят
контент), быстрый First Contentful Paint, лучший воспринимаемый перформанс на слабых устройствах.

**2. Что такое гидратация?**
Процесс, в котором клиентский Vue не пересоздаёт DOM, а прикрепляется к уже отрендеренному
сервером HTML: сопоставляет vDOM с существующими узлами и навешивает реактивность и обработчики.

**3. Что такое hydration mismatch и почему это плохо?**
Несовпадение серверного HTML и того, что Vue строит на клиенте. Vue логирует предупреждение и
перерисовывает несовпавший участок заново, теряя выгоду SSR (мерцание, скачок, лишняя работа).

**4. Из-за чего возникает hydration mismatch?**
Недетерминированность между сервером и клиентом: `Math.random()`/`Date.now()`/`new Date()`,
обращение к `window`/`localStorage` в первом рендере, невалидная вложенность HTML (браузер её
«чинит»), разные данные на сервере и клиенте (не передали серверное состояние).

**5. Как избежать hydration mismatch?**
Передавать серверное состояние клиенту (сериализация state) и восстанавливать его до `mount`;
клиент-специфичный код выносить в `onMounted`; использовать `<ClientOnly>`/lazy-гидратацию для
браузерного контента; чинить невалидный HTML.

**6. Чем отличаются SSR, SSG и CSR?**
CSR — HTML строится в браузере (пусто для ботов). SSR — HTML на сервере на каждый запрос (свежие
данные, нужен сервер). SSG — HTML заранее на сборке (статика на CDN, очень быстро, но данные на
момент сборки).

**7. Почему `onMounted` опасен в SSR?**
`onMounted` и последующие lifecycle-хуки **не вызываются на сервере** — только на клиенте. Это
правильное место для DOM/браузерных API, но нельзя рассчитывать, что код в `onMounted` повлияет на
серверный HTML. Серверный рендер видит только setup и `created`.

**8. Почему `window`/`document` нельзя трогать напрямую?**
На сервере (Node) их нет — обращение упадёт. Используйте `typeof window !== 'undefined'` или
выносите в `onMounted`.

**9. Почему нужен новый инстанс приложения на каждый запрос?**
Сервер обрабатывает много пользователей одновременно. Общий инстанс/стор шарил бы состояние между
запросами — данные одного пользователя утекли бы другому. Фабрика создаёт изолированный инстанс
(и Pinia, и роутер) на каждый запрос.

**10. Что делает `renderToString`?**
Из `vue/server-renderer`: асинхронно превращает SSR-инстанс приложения в строку HTML на сервере,
которую затем вставляют в шаблон страницы.

**11. В чём разница между `createApp` и `createSSRApp`?**
`createSSRApp` создаёт приложение в режиме гидратации: на клиенте `mount` не пересоздаёт DOM, а
прикрепляется к серверному. `createApp` — обычный CSR, который перерисовывает всё с нуля.

**12. Что такое Nuxt и зачем он?**
Метафреймворк над Vue: берёт на себя SSR/SSG/гибридный рендеринг, файловый роутинг, авто-импорты,
загрузку данных (`useAsyncData`/`useFetch`) с передачей состояния клиенту, SEO-метатеги, серверные
роуты. Избавляет от ручной настройки двух бандлов и сериализации состояния.

**13. Что такое частичная гидратация и острова?**
Гидратировать не всю страницу, а только интерактивные части. «Острова» — статичный HTML по
умолчанию + точечно оживляемые компоненты. Ленивая гидратация (Vue 3.5+: `hydrateOnVisible`,
`hydrateOnIdle`, `hydrateOnInteraction`) откладывает гидратацию до условия. Цель — меньше JS и CPU.

**14. SSR улучшает TTFB?**
Не обязательно. SSR улучшает **FCP** и SEO, но TTFB может вырасти, потому что серверу нужно время
на рендер. Поэтому SSR комбинируют с кешированием, а статичные страницы делают через SSG (отдаются
с CDN — лучший TTFB).

## Подводные камни (gotchas)

- **`Math.random()`/`new Date()` в шаблоне** → гарантированный hydration mismatch.
- **`window`/`localStorage` в setup на верхнем уровне** → падение на сервере.
- **Не передали серверное состояние клиенту** → данные различаются → mismatch и двойной запрос.
- **Глобальный стор/синглтон вместо фабрики** → утечка состояния между запросами пользователей.
- **Логика в `onMounted`, ожидаемая в серверном HTML** → её там не будет (хук клиент-онли).
- **Невалидная HTML-вложенность** (`<div>` в `<p>`, блок в `<a>`) → браузер «чинит» DOM → mismatch.
- **Не очищенные подписки/таймеры** (`onUnmounted` отсутствует) → утечки памяти на сервере.
- **`createApp` вместо `createSSRApp` на клиенте** → клиент перерисует весь DOM, гидратации не будет.

## Лучшие практики

- Передавайте серверное состояние клиенту и восстанавливайте его **до** `mount`.
- Браузерные API — только в `onMounted`/после проверки `typeof window`.
- Создавайте приложение, Pinia и роутер фабрикой на каждый запрос.
- Избегайте недетерминированности (случайность/время) в рендере; для клиент-онли — `onMounted`/`<ClientOnly>`.
- Пишите валидный HTML, чтобы браузер не «чинил» DOM.
- Для реальных проектов берите Nuxt вместо ручного SSR.
- Выбирайте режим под задачу: лендинги/доки — SSG, динамика/SEO — SSR, приватные дашборды — CSR.
- Рассмотрите ленивую/частичную гидратацию, если страница тяжёлая, а интерактива мало.

## Шпаргалка

```js
// Сервер
import { renderToString } from 'vue/server-renderer'
import { createSSRApp } from 'vue'
const app = createSSRApp(App)           // SSR-режим
const html = await renderToString(app)  // HTML-строка

// Клиент
import { createSSRApp } from 'vue'
createSSRApp(App).mount('#app')         // mount = гидратация (не пересоздаёт DOM)

// Lazy hydration (Vue 3.5+)
import { defineAsyncComponent, hydrateOnVisible, hydrateOnIdle } from 'vue'
defineAsyncComponent({ loader, hydrate: hydrateOnVisible() })
```

**Памятка причин hydration mismatch:** случайность/время · window в первом рендере ·
невалидный HTML · разные данные сервер/клиент.

**Правила одной строкой:**
- SSR → SEO + быстрый FCP; SSG → статика на сборке; CSR → рендер в браузере.
- Гидратация прикрепляется к серверному HTML, а не пересоздаёт его.
- Mismatch = серверный HTML ≠ клиентский → перерисовка → потеря выгоды SSR.
- `window`/`document` и клиент-онли логика — в `onMounted`.
- Свой инстанс приложения/стора на каждый запрос; состояние сериализуем клиенту.
- Для продакшена — Nuxt.
