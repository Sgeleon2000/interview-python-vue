# Vue 3 — Производительность — конспект и вопросы

## О чём раздел

Производительность Vue-приложения складывается из двух разных по своей природе задач:

1. **Производительность загрузки (load performance)** — как быстро пользователь увидит и сможет начать пользоваться приложением при первом заходе. Здесь главное — размер и количество скачиваемых ресурсов (бандлы JS/CSS), время до интерактивности (TTI), First Contentful Paint (FCP), Largest Contentful Paint (LCP).
2. **Производительность обновлений (update performance)** — как быстро интерфейс реагирует на изменения данных (ввод, ответы API, навигация). Здесь главное — стоимость ре-рендеров, размер reactive-графа, эффективность diff/patch виртуального DOM.

Эти задачи требуют разных техник: для загрузки — code splitting, lazy-загрузка, tree-shaking; для обновлений — стабильные ключи, `v-once`/`v-memo`, виртуализация, контроль реактивности. Vue из коробки делает многое (компиляторные оптимизации, кеширование computed), но на больших приложениях знание ручных приёмов критично.

## Ключевые концепции

- **Две метрики**: загрузка (первый рендер) vs обновление (runtime-реактивность). Оптимизируются разными средствами.
- **Code splitting / lazy routes / lazy components** — дробление бандла, чтобы грузить код по требованию.
- **Tree-shaking** — удаление неиспользуемого кода на этапе сборки.
- **Стабильные `key`** — помогают Vue корректно переиспользовать DOM-узлы в списках.
- **`v-once`** — рендер один раз, навсегда статичен.
- **`v-memo`** — мемоизация поддерева по зависимостям, пропуск обновлений.
- **Виртуализация списков** — рендер только видимых элементов.
- **`shallowRef` / `shallowReactive` / `markRaw`** — отключение глубокой реактивности для тяжёлых/неизменяемых данных.
- **`computed`-кеширование** — вычисление пересчитывается только при изменении зависимостей.
- **Props stability** — стабильность пропсов снижает число ре-рендеров дочерних компонентов.
- **`<KeepAlive>`** — сохранение состояния компонента вместо его пересоздания.
- **Профилирование** — Vue DevTools (Components, Timeline) и вкладка Performance в браузере.
- **Компиляторные оптимизации** — static hoisting, patch flags, tree flattening — работают автоматически.

## Подробный разбор с примерами кода

### 1. Оптимизация загрузки

#### Code splitting и lazy-загрузка маршрутов

Вместо того чтобы складывать весь код приложения в один файл, разбиваем его на чанки. Vue Router поддерживает ленивые маршруты через динамический `import()` — Vite/Rollup автоматически выделяют их в отдельные чанки.

```js
// router/index.js
import { createRouter, createWebHistory } from 'vue-router'

const router = createRouter({
  history: createWebHistory(),
  routes: [
    {
      path: '/',
      // Грузится сразу — это начальный экран
      component: () => import('@/views/HomeView.vue'),
    },
    {
      path: '/dashboard',
      // Тяжёлый раздел грузится только при переходе на него
      component: () => import('@/views/DashboardView.vue'),
    },
    {
      path: '/admin',
      // Админка не нужна большинству — отдельный чанк
      component: () => import('@/views/AdminView.vue'),
    },
  ],
})

export default router
```

#### Lazy-загрузка компонентов через defineAsyncComponent

Для тяжёлых компонентов, которые нужны не всегда (модалки, графики, редакторы), используем асинхронную загрузку.

```vue
<script setup>
import { defineAsyncComponent } from 'vue'

// Тяжёлый компонент с графиками — грузится отдельным чанком
const HeavyChart = defineAsyncComponent(() =>
  import('@/components/HeavyChart.vue')
)

// Расширенный вариант с состояниями загрузки/ошибки
const RichEditor = defineAsyncComponent({
  loader: () => import('@/components/RichEditor.vue'),
  // Компонент-заглушка пока грузится
  loadingComponent: () => import('@/components/Spinner.vue'),
  // Компонент при ошибке загрузки
  errorComponent: () => import('@/components/LoadError.vue'),
  // Задержка перед показом загрузчика (мс)
  delay: 200,
  // Таймаут (мс) — после него покажет errorComponent
  timeout: 10000,
})
</script>

<template>
  <!-- Грузится только когда условие истинно -->
  <HeavyChart v-if="showChart" />
</template>
```

#### Tree-shaking и размер бандла

Tree-shaking удаляет код, который нигде не импортируется. Чтобы он работал:

```js
// ХОРОШО: именованный импорт — попадёт только debounce
import { debounce } from 'lodash-es'

// ПЛОХО: импорт всего lodash — в бандл попадёт вся библиотека
import _ from 'lodash'
_.debounce(fn, 300)
```

Правила для эффективного tree-shaking:
- Использовать ESM-версии библиотек (`lodash-es`, а не `lodash`).
- Избегать сайд-эффектов в модулях; помечать `"sideEffects": false` в `package.json` своих пакетов.
- Импортировать только нужные функции, а не весь модуль.

Анализ размера бандла:

```bash
# Визуализатор бандла для Vite
npm install -D rollup-plugin-visualizer
```

```js
// vite.config.js
import { visualizer } from 'rollup-plugin-visualizer'

export default {
  plugins: [
    // Откроет интерактивную карту размеров чанков после сборки
    visualizer({ open: true, gzipSize: true, brotliSize: true }),
  ],
}
```

### 2. Оптимизация обновлений

#### Стабильные `key` в `v-for`

`key` помогает Vue понять, какие узлы переиспользовать при изменении списка. Неправильный ключ (индекс) ломает переиспользование и состояние.

```vue
<script setup>
import { ref } from 'vue'
const users = ref([
  { id: 1, name: 'Анна' },
  { id: 2, name: 'Борис' },
])
</script>

<template>
  <!-- ПЛОХО: индекс как key — при вставке/удалении в начало
       Vue ошибочно переиспользует не те DOM-узлы,
       состояние input'ов "съезжает" -->
  <li v-for="(user, index) in users" :key="index">
    <input :value="user.name" />
  </li>

  <!-- ХОРОШО: стабильный уникальный id из данных -->
  <li v-for="user in users" :key="user.id">
    <input :value="user.name" />
  </li>
</template>
```

#### `v-once` — статичный рендер один раз

Содержимое рендерится один раз и больше никогда не обновляется. Подходит для статики, которая зависит от первоначальных данных.

```vue
<template>
  <!-- Этот блок отрендерится один раз; даже если title изменится,
       DOM здесь не обновится -->
  <header v-once>
    <h1>{{ title }}</h1>
    <p>Версия приложения: {{ appVersion }}</p>
  </header>
</template>
```

#### `v-memo` — мемоизация поддерева (важная тема)

`v-memo` запоминает поддерево и пропускает его обновление, если все значения в массиве зависимостей не изменились. Это как `v-once`, но условный. Основное применение — **огромные списки**, где большинство строк не меняется между рендерами.

Семантика: `v-memo="[dep1, dep2]"` — если все зависимости равны предыдущим (по `===`), Vue полностью пропускает ре-рендер и diff этого поддерева.

```vue
<script setup>
import { ref } from 'vue'

const list = ref(
  Array.from({ length: 10000 }, (_, i) => ({ id: i, active: false }))
)
const selectedId = ref(null)

function select(id) {
  selectedId.value = id
}
</script>

<template>
  <!-- Без v-memo: при смене selectedId Vue заново сравнивает
       все 10000 строк. С v-memo пересчитываются только те,
       у которых изменилось значение isSelected -->
  <div
    v-for="item in list"
    :key="item.id"
    v-memo="[item.id === selectedId]"
    @click="select(item.id)"
  >
    <span :class="{ selected: item.id === selectedId }">
      Элемент #{{ item.id }}
    </span>
  </div>
</template>
```

Как это работает: для каждой строки Vue хранит предыдущий массив `[item.id === selectedId]`. При новом рендере он сравнивает его с новым. Если значение не изменилось — строка пропускается целиком. Так из 10000 строк фактически перерисовываются только 2 (старая выбранная и новая выбранная).

Важные нюансы `v-memo`:
- Массив зависимостей должен **полностью** перечислять всё, что влияет на рендер поддерева. Забытая зависимость = устаревший UI.
- `v-memo` без аргумента (`v-memo="[]"`) эквивалентен `v-once`.
- Не используйте `v-memo` на том же элементе, что и `v-for`, если зависимости не связаны со списком — это легко сделать неправильно.
- Это инструмент микрооптимизации. Применяйте только при доказанной проблеме (профилировщик показывает дорогой ре-рендер списка).

#### Виртуализация длинных списков

Если список содержит тысячи элементов, даже с `key` и `v-memo` рендер всех DOM-узлов дорог. Решение — рендерить только видимые элементы (виртуальный скролл).

```vue
<script setup>
// Библиотека vue-virtual-scroller (TanStack Virtual — альтернатива)
import { RecycleScroller } from 'vue-virtual-scroller'
import 'vue-virtual-scroller/dist/vue-virtual-scroller.css'
import { ref } from 'vue'

const items = ref(
  Array.from({ length: 100000 }, (_, i) => ({ id: i, text: `Строка ${i}` }))
)
</script>

<template>
  <!-- В DOM существуют только видимые ~20 строк,
       остальные переиспользуются при скролле -->
  <RecycleScroller
    class="scroller"
    :items="items"
    :item-size="40"
    key-field="id"
    v-slot="{ item }"
  >
    <div class="row">{{ item.text }}</div>
  </RecycleScroller>
</template>

<style scoped>
.scroller { height: 400px; }
.row { height: 40px; line-height: 40px; }
</style>
```

#### Избегание лишней реактивности: shallowRef / shallowReactive / markRaw (важная тема)

По умолчанию `ref`/`reactive` делают объект **глубоко** реактивным: Vue рекурсивно оборачивает все вложенные свойства в Proxy. Для больших структур (огромные массивы данных, инстансы сторонних библиотек, графовые структуры) это дорого по памяти и по времени.

```vue
<script setup>
import { ref, shallowRef, shallowReactive, markRaw, triggerRef } from 'vue'

// 1. shallowRef — реактивна только ссылка .value, но не внутренности
// Отлично для больших неизменяемых снимков данных
const bigData = shallowRef({ rows: [/* 50000 объектов */] })

// Изменение вложенного поля НЕ вызовет ре-рендер:
bigData.value.rows[0].name = 'x' // реактивности нет!

// Чтобы обновить — заменяем ссылку целиком (иммутабельно):
bigData.value = { rows: [...newRows] }
// либо вручную форсируем обновление:
triggerRef(bigData)

// 2. shallowReactive — реактивны только свойства верхнего уровня
const state = shallowReactive({
  page: 1,          // реактивно
  filters: {},      // ссылка реактивна, но state.filters.x — нет
})

// 3. markRaw — полностью исключает объект из реактивности.
// Применяется к тяжёлым инстансам, которым реактивность не нужна:
// карты (Leaflet/Mapbox), редакторы, WebGL-сцены, чарт-инстансы.
import L from 'leaflet'
const mapState = ref({
  // markRaw защищает инстанс карты от оборачивания в Proxy,
  // иначе Vue попытается сделать реактивными сотни внутренних полей
  map: markRaw(L.map('id')),
})
</script>
```

Правило: если объект большой и/или его внутренности не должны быть реактивными (или вы управляете изменениями иммутабельно), используйте `shallowRef`/`shallowReactive`. Для сторонних инстансов — `markRaw`.

#### Computed-кеширование

`computed` кеширует результат и пересчитывается только при изменении реактивных зависимостей. Это ключевой инструмент производительности — выносите дорогие вычисления из шаблона в `computed`.

```vue
<script setup>
import { ref, computed } from 'vue'

const list = ref([/* много элементов */])
const query = ref('')

// ПЛОХО: метод в шаблоне вызывается на КАЖДЫЙ ре-рендер,
// фильтрация выполняется заново даже если ничего не изменилось.

// ХОРОШО: computed пересчитается только при изменении list или query
const filtered = computed(() =>
  list.value.filter((item) => item.name.includes(query.value))
)
</script>

<template>
  <!-- filtered не пересчитывается, пока list и query не изменятся -->
  <li v-for="item in filtered" :key="item.id">{{ item.name }}</li>
</template>
```

#### Props stability — стабильность пропсов

Если родитель при каждом рендере передаёт дочернему компоненту **новый** объект/массив/функцию, дочерний компонент ре-рендерится без нужды. Стабилизируйте пропсы через `computed`.

```vue
<script setup>
import { ref, computed } from 'vue'

const userId = ref(1)
const theme = ref('dark')

// ПЛОХО: в шаблоне :style="{ color: theme }" создаёт новый объект
// на каждый рендер → дочерний компонент считает пропс изменившимся.

// ХОРОШО: стабилизируем объект через computed —
// новая ссылка только при реальном изменении theme
const styleObject = computed(() => ({ color: theme.value }))
</script>

<template>
  <!-- ChildComponent не ре-рендерится, пока styleObject стабилен -->
  <ChildComponent :style-object="styleObject" :user-id="userId" />
</template>
```

#### `<KeepAlive>` — сохранение состояния компонента

`<KeepAlive>` кеширует инстанс компонента вместо его уничтожения и пересоздания. Это и про UX (сохранённый ввод, позиция скролла), и про производительность (не нужно заново монтировать и запрашивать данные).

```vue
<script setup>
import { ref } from 'vue'
const currentTab = ref('Profile')
</script>

<template>
  <!-- Переключение вкладок сохраняет состояние каждой -->
  <KeepAlive :max="5">
    <component :is="currentTab" />
  </KeepAlive>

  <!-- Можно ограничить include/exclude по имени компонента -->
  <KeepAlive :include="['Profile', 'Settings']">
    <component :is="currentTab" />
  </KeepAlive>
</template>
```

В кешированных компонентах вместо `onMounted`/`onUnmounted` работают хуки `onActivated`/`onDeactivated`:

```js
import { onActivated, onDeactivated } from 'vue'
// Вызывается при возврате на вкладку (компонент уже смонтирован)
onActivated(() => { /* возобновить polling, обновить данные */ })
// Вызывается при уходе с вкладки (без размонтирования)
onDeactivated(() => { /* приостановить polling */ })
```

### 3. Профилирование

#### Vue DevTools

- **Вкладка Components** — дерево компонентов, инспекция props/state.
- **Вкладка Timeline** — события (компонентные рендеры, mutations, событийные хуки) на временной шкале; видно, какие компоненты ре-рендерятся слишком часто.
- **Highlight Updates** (в настройках) — подсветка перерисовываемых компонентов прямо на странице. Если что-то мигает на каждое действие — кандидат на оптимизацию.

#### Браузерный Performance

```js
// main.js — включает интеграцию с вкладкой Performance браузера
import { createApp } from 'vue'
import App from './App.vue'

const app = createApp(App)
// Добавляет user-marks/measures в DevTools → Performance
// (только для dev-режима, в проде включать не нужно)
app.config.performance = true
app.mount('#app')
```

В браузере: DevTools → Performance → Record → выполнить действие → Stop. Смотрите на длинные жёлтые блоки (scripting) и события `Vue render`/`patch`.

### 4. Компиляторные оптимизации Vue (кратко)

Шаблонный компилятор Vue 3 делает много автоматически — понимать полезно для интервью:

- **Static hoisting** — статические узлы (без привязок) поднимаются за пределы render-функции и создаются один раз, а не на каждый рендер.
- **Patch flags** — компилятор помечает динамические узлы флагами (например, "меняется только класс", "меняется только текст"). При обновлении Vue проверяет только помеченные части, а не весь узел — это «информированный» diff вместо полного.
- **Tree flattening (block tree)** — статические поддеревья «сплющиваются», и при обновлении Vue обходит только массив динамических узлов внутри блока, а не всё дерево.
- **Cache handlers** — инлайн-обработчики событий кешируются, чтобы не создавать новую функцию на каждый рендер.

Эти оптимизации — причина, по которой Vue 3 заметно быстрее Vue 2. Они работают автоматически и являются весомым аргументом в пользу шаблонов (компилятор не может так оптимизировать ручные render-функции).

## Полный рабочий пример

```vue
<!-- ProductCatalog.vue — каталог с фильтром, виртуализацией и мемоизацией -->
<script setup>
import { ref, computed, shallowRef, defineAsyncComponent } from 'vue'
import { RecycleScroller } from 'vue-virtual-scroller'
import 'vue-virtual-scroller/dist/vue-virtual-scroller.css'

// Тяжёлая модалка с деталями — ленивая загрузка отдельным чанком
const ProductDetails = defineAsyncComponent(() =>
  import('@/components/ProductDetails.vue')
)

// Большой неизменяемый массив товаров — shallowRef, чтобы Vue
// не оборачивал каждый из 50000 объектов в реактивный Proxy
const products = shallowRef(
  Array.from({ length: 50000 }, (_, i) => ({
    id: i,
    name: `Товар ${i}`,
    price: Math.round(Math.random() * 10000),
    favorite: false,
  }))
)

const query = ref('')
const selectedId = ref(null)
const showDetails = ref(false)

// Дорогая фильтрация кешируется computed —
// пересчитывается только при изменении query или products
const filtered = computed(() => {
  const q = query.value.toLowerCase()
  if (!q) return products.value
  return products.value.filter((p) => p.name.toLowerCase().includes(q))
})

function openDetails(id) {
  selectedId.value = id
  showDetails.value = true
}
</script>

<template>
  <div class="catalog">
    <!-- v-once: заголовок статичен, рендерится один раз -->
    <h1 v-once>Каталог товаров</h1>

    <input v-model="query" placeholder="Поиск..." />

    <!-- Виртуальный скролл: в DOM только видимые строки -->
    <RecycleScroller
      class="scroller"
      :items="filtered"
      :item-size="48"
      key-field="id"
      v-slot="{ item }"
    >
      <!-- v-memo: строка перерисовывается, только если изменилось
           её выделение или цена -->
      <div
        class="row"
        v-memo="[item.id === selectedId, item.price]"
        :class="{ active: item.id === selectedId }"
        @click="openDetails(item.id)"
      >
        <span>{{ item.name }}</span>
        <span>{{ item.price }} ₽</span>
      </div>
    </RecycleScroller>

    <!-- Модалка грузится лениво и только при необходимости -->
    <ProductDetails
      v-if="showDetails"
      :product-id="selectedId"
      @close="showDetails = false"
    />
  </div>
</template>

<style scoped>
.scroller { height: 600px; }
.row {
  height: 48px;
  display: flex;
  justify-content: space-between;
  cursor: pointer;
}
.row.active { background: #e0f0ff; }
</style>
```

## Частые вопросы на собеседовании (Q/A)

**1. В чём разница между производительностью загрузки и обновления? Какими средствами оптимизируется каждая?**
Загрузка — про первый рендер и размер ресурсов: code splitting, lazy routes/components, tree-shaking, минимизация бандла, SSR/SSG. Обновление — про runtime-реактивность: стабильные key, `v-memo`/`v-once`, виртуализация, `computed`-кеширование, контроль реактивности (`shallowRef`), стабильность пропсов.

**2. Зачем нужен стабильный `key` в `v-for` и почему индекс — плохой выбор?**
`key` идентифицирует узел для алгоритма переиспользования DOM. При индексе вставка/удаление в середину списка приводит к тому, что Vue переиспользует не те узлы: состояние компонентов и неуправляемых input'ов «съезжает», возможны баги и лишние patch-операции. Нужен стабильный уникальный id из данных.

**3. Что делает `v-memo` и когда его применять?**
`v-memo="[deps]"` мемоизирует поддерево: если все зависимости равны прежним по `===`, обновление и diff пропускаются полностью. Применяется в очень больших списках, где большинство строк не меняется. Это микрооптимизация — нужна только при доказанной (профилировщиком) проблеме. Если забыть зависимость в массиве — UI устареет.

**4. Чем `v-memo` отличается от `v-once`?**
`v-once` рендерит один раз и никогда не обновляет. `v-memo="[]"` ему эквивалентен. `v-memo="[deps]"` — условная мемоизация: обновляется при изменении зависимостей. То есть `v-once` — частный случай `v-memo` с пустыми зависимостями.

**5. Когда использовать `shallowRef` вместо `ref`?**
Когда значение — большая структура, внутренности которой не нужно делать реактивными, либо вы управляете обновлениями иммутабельно (замена ссылки целиком). Это экономит память и время на создание глубоких Proxy. Минус: мутации вложенных полей не отслеживаются (нужен `triggerRef` или замена `.value`).

**6. Зачем `markRaw` и какой типичный кейс?**
`markRaw` навсегда исключает объект из реактивности. Типичный кейс — инстансы сторонних библиотек (Leaflet/Mapbox-карта, чарт, редактор, WebGL-сцена): у них сотни внутренних полей, оборачивание в Proxy бессмысленно и вредно (ломает методы, тратит ресурсы). Кладём такой инстанс через `markRaw`.

**7. Как работает кеширование `computed` и чем он лучше метода в шаблоне?**
`computed` кеширует результат и пересчитывается только при изменении реактивных зависимостей. Метод в шаблоне вызывается на каждый ре-рендер компонента независимо от того, изменились ли входные данные. Для дорогих вычислений (фильтрация, сортировка, агрегация) `computed` обязателен.

**8. Что такое стабильность пропсов и как лишние объекты вредят производительности?**
Если родитель передаёт новый объект/массив/функцию-литерал при каждом рендере, дочерний компонент видит «изменившийся» пропс и ре-рендерится зря. Стабилизируют через `computed` (новая ссылка только при реальном изменении данных), чтобы дочерний компонент пропускал ненужные обновления.

**9. Что даёт `<KeepAlive>` и какие хуки в нём работают?**
Кеширует инстанс компонента: сохраняется состояние (ввод, скролл), не происходит повторного монтирования и перезапроса данных. Вместо `onMounted`/`onUnmounted` срабатывают `onActivated`/`onDeactivated`. Можно ограничить кеш через `:max`, `:include`, `:exclude`.

**10. Какие компиляторные оптимизации делает Vue 3 автоматически?**
Static hoisting (статические узлы создаются один раз), patch flags (динамические узлы помечаются типом изменения для точечного diff), tree flattening/block tree (обход только динамических узлов), кеширование инлайн-обработчиков. Поэтому шаблоны быстрее ручных render-функций.

**11. Как профилировать производительность Vue-приложения?**
Vue DevTools: вкладка Timeline (видеть частоту/стоимость рендеров), Highlight Updates (подсветка перерисовок). Браузер: `app.config.performance = true` + вкладка Performance (record → действие → анализ scripting/render/patch). Размер бандла — `rollup-plugin-visualizer`.

**12. Как уменьшить размер бандла?**
Lazy routes и async-компоненты (code splitting), tree-shaking (ESM-импорты, `lodash-es`, именованные импорты), удаление неиспользуемых зависимостей, динамический импорт тяжёлых библиотек, анализ через visualizer, gzip/brotli-сжатие на сервере.

## Подводные камни (gotchas)

- **Индекс как `key`** в списках с вставками/удалениями/сортировкой — баги состояния и лишние patch'и.
- **Забытая зависимость в `v-memo`** — поддерево не обновляется при изменении пропущенного значения. Самая частая ошибка.
- **`v-memo` без необходимости** — добавляет накладные расходы на сравнение зависимостей; на простых компонентах может быть медленнее, чем без него.
- **`shallowRef` и неожиданное отсутствие реактивности** — мутация вложенного поля не вызывает обновление; разработчики тратят часы на отладку. Нужно либо заменять `.value`, либо `triggerRef`.
- **`markRaw` на объекте, который потом кладут в реактивное состояние** — Vue не сделает его реактивным; если позже понадобится реактивность вложенных полей — её не будет.
- **Метод вместо `computed`** для дорогих вычислений — пересчёт на каждый рендер.
- **Литералы объектов/массивов в пропсах/стилях** (`:style="{...}"`, `:items="[...]"`) — новая ссылка каждый рендер, лишние обновления дочерних компонентов.
- **`<KeepAlive>` без `:max`** — неконтролируемый рост памяти при большом числе вкладок/маршрутов.
- **Импорт всей библиотеки** (`import _ from 'lodash'`) — раздувает бандл, ломает tree-shaking.
- **Глубокая реактивность огромных массивов** — десятки тысяч Proxy создают лаг при инициализации; нужен `shallowRef`.

## Лучшие практики

1. Сначала измеряйте (DevTools, Performance), потом оптимизируйте — не угадывайте.
2. Lazy-загрузка маршрутов по умолчанию; начальный бандл держите минимальным.
3. Дорогие вычисления — всегда в `computed`, не в методах шаблона.
4. Стабильные уникальные `key` из данных в каждом `v-for`.
5. Для списков > нескольких тысяч элементов — виртуализация.
6. `v-memo`/`v-once` — точечно и только по результатам профилирования.
7. Большие/неизменяемые данные — `shallowRef`/`shallowReactive`; сторонние инстансы — `markRaw`.
8. Стабилизируйте пропсы-объекты через `computed`.
9. `<KeepAlive>` для дорогих в монтировании вкладок/маршрутов с ограничением `:max`.
10. Tree-shaking: ESM-импорты, именованные импорты, анализ бандла визуализатором.
11. Доверяйте компилятору Vue (static hoisting, patch flags) — предпочитайте шаблоны ручным render-функциям.

## Шпаргалка

| Приём | Когда применять |
|-------|-----------------|
| `() => import(...)` в роутере | Разбить бандл по маршрутам |
| `defineAsyncComponent` | Ленивая загрузка тяжёлого компонента |
| ESM / именованные импорты | Включить tree-shaking |
| Стабильный `:key="item.id"` | Любой `v-for` с изменяемым списком |
| `v-once` | Статичный блок, рендер один раз |
| `v-memo="[deps]"` | Огромные списки, пропуск неизменившихся строк |
| `v-memo="[]"` | Эквивалент `v-once` |
| Виртуальный скролл | Списки > ~1000 элементов |
| `shallowRef` | Большие/иммутабельные данные |
| `shallowReactive` | Реактивность только верхнего уровня |
| `markRaw` | Инстансы сторонних библиотек (карты, чарты) |
| `triggerRef` | Форсировать обновление `shallowRef` |
| `computed` | Кешировать дорогие вычисления |
| `computed`-обёртка пропса | Стабилизировать объект-пропс |
| `<KeepAlive :max>` | Сохранять состояние вкладок/маршрутов |
| `onActivated`/`onDeactivated` | Хуки внутри `<KeepAlive>` |
| `app.config.performance = true` | Профиль в браузерном Performance |
| Vue DevTools Timeline | Анализ частоты/стоимости рендеров |

**Главное:** разделяйте оптимизацию загрузки (бандл, lazy, tree-shaking) и обновлений (key, v-memo, виртуализация, контроль реактивности). Измеряйте перед оптимизацией. Vue многое делает сам (computed-кеш, patch flags, static hoisting); ручные приёмы — для доказанных узких мест.
