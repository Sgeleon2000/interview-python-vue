# Vue 3 — Безопасность — конспект и вопросы

## О чём раздел

Безопасность фронтенда на Vue — это прежде всего защита от **XSS (Cross-Site Scripting)** — внедрения вредоносного скрипта, который выполнится в браузере жертвы в контексте вашего сайта. XSS позволяет красть cookie/токены, выполнять действия от имени пользователя, подменять интерфейс.

Vue спроектирован безопасным по умолчанию: текстовая интерполяция (`{{ }}`) и привязка атрибутов автоматически экранируют данные. Проблемы возникают, когда разработчик **сознательно обходит** эти механизмы: `v-html`, динамические `:href`, динамические имена компонентов/тегов из пользовательского ввода, рендеринг пользовательских шаблонов. Раздел разбирает каждый вектор и правильные практики защиты, а также сопутствующие темы: серверную санитизацию, CSP, уязвимости зависимостей.

## Ключевые концепции

- **XSS** — главная угроза фронтенда; цель — выполнить чужой JS в вашем origin.
- **Безопасность по умолчанию** — `{{ }}` и `:attr` экранируют HTML-сущности.
- **`v-html` — опасен**: вставляет сырой HTML без экранирования. Никогда не подавать туда непроверенный ввод.
- **Инъекция через URL** — `:href="userUrl"` может содержать `javascript:`-схему.
- **Инъекция через атрибуты** — пользовательские данные в чувствительных атрибутах.
- **Динамические компоненты/теги** (`<component :is>`) из пользовательского ввода — риск рендера произвольного компонента.
- **Серверная санитизация** — очистка HTML на бэкенде (надёжнее клиентской, но дополняет её).
- **CSP (Content Security Policy)** — заголовок, ограничивающий источники скриптов; глубокая защита.
- **Безопасность шаблонов** — никогда не компилировать строки-шаблоны из пользовательского ввода.
- **Уязвимости зависимостей** — `npm audit`, обновления, минимизация зависимостей.

## Подробный разбор с примерами кода

### 1. Безопасность по умолчанию: интерполяция экранирует

Текстовая интерполяция всегда трактует данные как **текст**, а не как HTML. Это безопасно.

```vue
<script setup>
import { ref } from 'vue'
// Пользователь ввёл вредоносную строку
const userInput = ref('<img src=x onerror="alert(document.cookie)">')
</script>

<template>
  <!-- БЕЗОПАСНО: Vue экранирует символы < > & " '
       На экране появится текст "<img src=x ...>", скрипт НЕ выполнится -->
  <p>{{ userInput }}</p>

  <!-- Привязка атрибута также экранирует значение -->
  <div :title="userInput">Наведи курсор</div>
</template>
```

Под капотом Vue превращает `<` в `&lt;`, `>` в `&gt;` и т.д. Браузер видит текст, а не разметку.

### 2. `v-html` — главный источник XSS (важнейшая тема)

`v-html` вставляет строку как **сырой HTML** в `innerHTML`. Экранирования нет. Любой `<script>`, `onerror`, `onload` и подобное выполнится.

```vue
<script setup>
import { ref } from 'vue'

// Представим, что это пришло из комментария пользователя / API
const comment = ref('<img src=x onerror="fetch(`/steal?c=`+document.cookie)">')
</script>

<template>
  <!-- ОПАСНО: атака выполнится! onerror сработает, cookie утекут.
       НИКОГДА не подавайте в v-html непроверенные данные. -->
  <div v-html="comment"></div>
</template>
```

Замечание про `<script>`: браузер не выполняет `<script>`-теги, вставленные через `innerHTML`, но это **не защита** — атака идёт через атрибуты-обработчики (`onerror`, `onmouseover`, `onload`), `<iframe>`, `<svg>` и т.п. Поэтому `innerHTML` сырого ввода всегда уязвим.

**Когда `v-html` всё же нужен** (например, рендер HTML из CMS/markdown): обязательно **санитизируйте** ввод проверенной библиотекой — `DOMPurify`.

```vue
<script setup>
import { computed } from 'vue'
import DOMPurify from 'dompurify'

const props = defineProps({ rawHtml: String })

// Очищаем HTML: DOMPurify удаляет script, обработчики событий,
// javascript:-ссылки и прочие опасные конструкции,
// оставляя безопасную разметку
const safeHtml = computed(() => DOMPurify.sanitize(props.rawHtml))
</script>

<template>
  <!-- Теперь относительно безопасно: вставляем только очищенный HTML -->
  <div v-html="safeHtml"></div>
</template>
```

Правила для `v-html`:
- Никогда — для пользовательского ввода без санитизации.
- Санитизация — на доверенной библиотеке (DOMPurify), не самописными регулярками.
- Идеально — санитизация и на сервере (при сохранении), и на клиенте (защита в глубину).
- Помните: `v-html` игнорирует Vue-шаблонный синтаксис — внутри него директивы/`{{ }}` не работают.

### 3. Инъекция через URL: `javascript:`-схема

Привязка `:href` экранирует HTML-сущности, но **не** проверяет схему URL. Значение `javascript:alert(1)` приведёт к выполнению кода при клике.

```vue
<script setup>
import { ref, computed } from 'vue'

// Пользователь задал свой сайт в профиле
const userUrl = ref('javascript:alert(document.cookie)')

// ЗАЩИТА: разрешаем только http/https/mailto, иначе обнуляем
const safeUrl = computed(() => {
  try {
    const url = new URL(userUrl.value, window.location.origin)
    const allowed = ['http:', 'https:', 'mailto:']
    return allowed.includes(url.protocol) ? url.href : '#'
  } catch {
    return '#'
  }
})
</script>

<template>
  <!-- ОПАСНО: клик выполнит javascript: -->
  <a :href="userUrl">Профиль (небезопасно)</a>

  <!-- БЕЗОПАСНО: схема проверена -->
  <a :href="safeUrl" rel="noopener noreferrer">Профиль (безопасно)</a>
</template>
```

Дополнительно для внешних ссылок, открываемых в новой вкладке, используйте `rel="noopener noreferrer"`, чтобы целевая страница не получила доступ к `window.opener`.

Та же опасность касается `:src` у `<iframe>`, `<script>` (никогда не делать `:src` скрипта из ввода) и подобных.

### 4. Инъекция через атрибуты

Опасно подставлять пользовательские данные в чувствительные атрибуты, особенно если они объединяются со строками или попадают в обработчики/стили.

```vue
<script setup>
import { ref } from 'vue'
const userStyle = ref('width: 100px') // может прийти вредоносное
</script>

<template>
  <!-- Привязка одного атрибута безопасна (экранируется),
       но избегайте подстановки в атрибуты-обработчики -->
  <div :title="userInput">ok</div>

  <!-- ОПАСНО по смыслу: динамические имена атрибутов из ввода.
       Атакующий может задать имя "onclick" и значение-код -->
  <div v-bind="userProvidedAttrs"></div>
  <!-- если userProvidedAttrs = { onclick: 'alert(1)' } — это инъекция -->
</template>
```

Правило: никогда не передавайте через `v-bind="..."` объект атрибутов, ключи которого контролирует пользователь. Не стройте имена атрибутов/событий из пользовательского ввода.

### 5. Динамические имена компонентов/тегов из ввода

`<component :is="...">` рендерит компонент по имени/объекту. Если имя берётся из пользовательского ввода, злоумышленник может отрендерить компонент, который не должен быть доступен, или (в худшем случае при динамической компиляции) выполнить код.

```vue
<script setup>
import { ref, shallowRef } from 'vue'
import SafeA from './SafeA.vue'
import SafeB from './SafeB.vue'

const userChoice = ref('SafeA')

// ЗАЩИТА: белый список разрешённых компонентов.
// Никогда не резолвим компонент напрямую из строки пользователя
const allowed = { SafeA, SafeB }
const current = shallowRef(allowed['SafeA'])

function choose(name) {
  // Только из белого списка; неизвестное имя игнорируем
  if (name in allowed) current.value = allowed[name]
}
</script>

<template>
  <!-- ОПАСНО: :is из произвольной строки пользователя -->
  <!-- <component :is="userChoice" /> -->

  <!-- БЕЗОПАСНО: :is только из проверенного белого списка -->
  <component :is="current" />
</template>
```

То же касается динамического имени HTML-тега: `<component :is="userTag">` с `userTag = 'script'` или `'iframe'` — потенциальный вектор. Всегда валидируйте через белый список.

### 6. Безопасность шаблонов: не компилировать пользовательский ввод

**Никогда** не передавайте пользовательский ввод как Vue-шаблон. Шаблоны компилируются в JS-код и выполняются — это прямая инъекция кода (эквивалент `eval`).

```js
// КАТАСТРОФА: компиляция строки пользователя как шаблона = выполнение кода
import { compile } from 'vue'

const userTemplate = '{{ constructor.constructor("alert(1)")() }}'
// НИКОГДА так не делать:
const render = compile(userTemplate) // выполнит произвольный JS

// Так же опасно: template-опция компонента из пользовательских данных
const Comp = { template: userTemplate } // НЕТ
```

Vue-выражения внутри шаблонов выполняются как JS. Если пользователь контролирует текст шаблона — он контролирует исполняемый код. Используйте сборку с предкомпилированными шаблонами (рантайм-компилятор в проде лучше вообще не включать). Для пользовательского форматированного контента — markdown + санитизация, а не Vue-шаблоны.

### 7. Серверная санитизация

Клиентская санитизация может быть обойдена (отключённый JS, прямые запросы к API). Поэтому HTML/контент, который где-то будет рендериться как разметка, нужно очищать **на сервере при сохранении** и/или хранить уже безопасным.

```js
// Пример на бэкенде (Node.js) при приёме комментария
import createDOMPurify from 'dompurify'
import { JSDOM } from 'jsdom'

const window = new JSDOM('').window
const DOMPurify = createDOMPurify(window)

function saveComment(rawHtml) {
  // Очищаем перед записью в БД — данные хранятся уже безопасными
  const clean = DOMPurify.sanitize(rawHtml, {
    ALLOWED_TAGS: ['b', 'i', 'a', 'p', 'br', 'ul', 'li'],
    ALLOWED_ATTR: ['href'],
  })
  return db.comments.insert({ html: clean })
}
```

Принцип «защита в глубину»: санитизация на сервере (основная) + на клиенте (дополнительная) + CSP (страховка).

### 8. CSP — Content Security Policy

CSP — HTTP-заголовок, ограничивающий, откуда можно загружать скрипты и можно ли выполнять inline-скрипты. Даже если XSS просочился, строгая CSP может не дать ему выполниться.

```
Content-Security-Policy:
  default-src 'self';
  script-src 'self';
  style-src 'self' 'unsafe-inline';
  img-src 'self' data: https:;
  object-src 'none';
  base-uri 'self';
  frame-ancestors 'none';
```

- `script-src 'self'` — выполнять только свои скрипты, запрещает inline и внешние.
- Важно: Vue в **продакшен-сборке** не использует `eval`/`new Function`, поэтому совместим со строгой CSP **без** `'unsafe-eval'`. Рантайм-компилятор (dev или сборка с компилятором) требует `'unsafe-eval'` — ещё одна причина не включать компилятор в проде.
- `object-src 'none'`, `base-uri 'self'`, `frame-ancestors 'none'` — закрывают частые векторы.

### 9. Уязвимости зависимостей

Большая часть кода фронтенда — это сторонние пакеты. Уязвимость в зависимости = уязвимость в вашем приложении.

```bash
# Проверка известных уязвимостей
npm audit
# Автоисправление совместимых обновлений
npm audit fix
# Просмотр устаревших пакетов
npm outdated
```

Практики:
- Регулярно запускать `npm audit` (или Snyk/Dependabot в CI).
- Минимизировать число зависимостей; избегать пакетов с одним мейнтейнером без обновлений.
- Фиксировать версии через lock-файл, проверять обновления перед мержем.
- Не подключать сторонние скрипты с чужих CDN без `integrity` (SRI) и доверия.

## Полный рабочий пример

```vue
<!-- CommentBlock.vue — безопасный вывод пользовательского контента -->
<script setup>
import { computed } from 'vue'
import DOMPurify from 'dompurify'

const props = defineProps({
  author: { type: String, default: '' },
  // HTML комментария (например, отрендеренный markdown)
  htmlBody: { type: String, default: '' },
  // Ссылка на сайт автора из его профиля
  website: { type: String, default: '' },
})

// 1. Санитизируем HTML перед v-html — удаляем скрипты,
//    обработчики событий, опасные ссылки
const safeBody = computed(() =>
  DOMPurify.sanitize(props.htmlBody, {
    ALLOWED_TAGS: ['b', 'i', 'em', 'strong', 'a', 'p', 'br', 'code'],
    ALLOWED_ATTR: ['href', 'title'],
  })
)

// 2. Проверяем схему URL — отсекаем javascript: и прочее
const safeWebsite = computed(() => {
  if (!props.website) return null
  try {
    const url = new URL(props.website)
    return ['http:', 'https:'].includes(url.protocol) ? url.href : null
  } catch {
    return null
  }
})
</script>

<template>
  <article class="comment">
    <!-- Имя автора — обычная интерполяция, экранируется автоматически -->
    <header>{{ author }}</header>

    <!-- Безопасная ссылка: схема проверена, защита window.opener -->
    <a
      v-if="safeWebsite"
      :href="safeWebsite"
      target="_blank"
      rel="noopener noreferrer"
    >
      Сайт автора
    </a>

    <!-- v-html только с санитизированным контентом -->
    <div class="body" v-html="safeBody"></div>
  </article>
</template>
```

## Частые вопросы на собеседовании (Q/A)

**1. Что такое XSS и почему это главная угроза фронтенда?**
XSS — внедрение вредоносного JavaScript, который выполняется в браузере жертвы в контексте (origin) вашего сайта. Позволяет красть токены/cookie, выполнять действия от имени пользователя, подменять UI. Это самая распространённая клиентская уязвимость.

**2. Почему Vue безопасен по умолчанию?**
Текстовая интерполяция `{{ }}` и привязка атрибутов `:attr` автоматически экранируют HTML-сущности (`<`, `>`, `&`, кавычки). Данные трактуются как текст, а не разметка, поэтому встроенный скрипт не выполнится.

**3. Чем опасен `v-html` и когда его можно использовать?**
`v-html` вставляет сырой HTML в `innerHTML` без экранирования — любой пользовательский ввод с `onerror`/`onload`/`<iframe>` выполнит атаку. Использовать можно только для доверенного или предварительно санитизированного (DOMPurify) контента, в идеале с серверной санитизацией.

**4. Если браузер не выполняет `<script>` из innerHTML, значит `v-html` безопасен?**
Нет. Атака идёт не через `<script>`, а через атрибуты-обработчики (`onerror`, `onmouseover`), `<svg>`, `<iframe>`, `javascript:`-ссылки. Поэтому сырой ввод в `v-html` всегда уязвим; нужна санитизация.

**5. Почему `:href="userUrl"` может быть опасен, ведь атрибуты экранируются?**
Экранирование защищает от HTML-инъекции, но не проверяет схему URL. `javascript:alert(1)` выполнится при клике. Нужно валидировать протокол (разрешать только http/https/mailto) и добавлять `rel="noopener noreferrer"` для внешних ссылок.

**6. Какой риск у `<component :is="userInput">`?**
Пользователь может отрендерить недоступный ему компонент или, при включённой рантайм-компиляции, спровоцировать выполнение кода. Защита — белый список разрешённых компонентов; никогда не резолвить компонент напрямую из строки ввода.

**7. Почему нельзя компилировать пользовательские шаблоны?**
Vue-шаблоны компилируются в исполняемый JS, а выражения внутри `{{ }}` выполняются как код. Передача пользовательского текста в `compile()` или `template`-опцию = произвольное выполнение кода (как `eval`). Для пользовательского контента используют markdown + санитизацию.

**8. Зачем серверная санитизация, если есть клиентская?**
Клиент можно обойти (отключить JS, слать запросы прямо в API). Данные, рендерящиеся как HTML, нужно очищать на сервере при сохранении — это основная линия защиты. Клиентская санитизация и CSP — дополнительные слои (защита в глубину).

**9. Что такое CSP и как она связана с Vue?**
Content-Security-Policy — заголовок, ограничивающий источники скриптов и inline-выполнение. Даже при XSS строгая CSP может заблокировать атаку. Продакшен-сборка Vue не использует `eval`, поэтому совместима с `script-src 'self'` без `'unsafe-eval'`. Рантайм-компилятор требует `'unsafe-eval'` — повод не включать его в проде.

**10. Как защититься от уязвимостей в зависимостях?**
Регулярный `npm audit` (и Dependabot/Snyk в CI), своевременные обновления, минимизация числа зависимостей, lock-файл, SRI (`integrity`) для внешних скриптов, осторожность с малоподдерживаемыми пакетами.

**11. Безопасна ли привязка `v-bind="objectOfAttrs"` с пользовательскими ключами?**
Нет. Если пользователь контролирует ключи объекта, он может задать `onclick` и значение-код. Никогда не строить имена атрибутов/событий из ввода и не передавать пользовательский объект в `v-bind`.

**12. Что такое «защита в глубину» применительно к XSS?**
Несколько независимых слоёв: безопасные дефолты Vue (экранирование) + санитизация на сервере при сохранении + санитизация на клиенте перед `v-html` + строгая CSP + валидация URL/компонентов. Пробой одного слоя не приводит к компрометации.

## Подводные камни (gotchas)

- **`v-html` с пользовательским вводом** — классический XSS. Самая частая ошибка.
- **Иллюзия безопасности `<script>`** — браузер не выполняет вставленный `<script>`, но атрибуты-обработчики и `<iframe>`/`<svg>` выполняются.
- **`:href`/`:src` без проверки схемы** — `javascript:`-URL выполняется при клике.
- **`rel="noopener"` забыт** для `target="_blank"` — целевая страница получает `window.opener`.
- **`<component :is>` из строки ввода** без белого списка.
- **`v-bind="userAttrs"`** с пользовательскими ключами — инъекция обработчиков.
- **Компиляция пользовательских шаблонов** или `template`-опция из ввода — выполнение кода.
- **Только клиентская санитизация** — обходится прямыми запросами к API.
- **Самописная санитизация регулярками** — почти всегда дырявая; нужен DOMPurify.
- **Рантайм-компилятор в проде** — требует `'unsafe-eval'`, ослабляет CSP и раздувает бандл.
- **Игнор `npm audit`** — известные уязвимости в зависимостях остаются в проде.
- **Доверие к данным API** — backend тоже может вернуть непроверенный HTML; санитизируйте перед `v-html`.

## Лучшие практики

1. По умолчанию — интерполяция `{{ }}` и `:attr`; они экранируют, доверяйте им.
2. `v-html` — только для санитизированного (DOMPurify) или полностью доверенного контента.
3. Санитизируйте на сервере при сохранении и на клиенте перед выводом (защита в глубину).
4. Валидируйте схему любых пользовательских URL (`http`/`https`/`mailto`); `rel="noopener noreferrer"` для внешних ссылок.
5. `<component :is>` и динамические теги — только из белого списка.
6. Никогда не компилируйте пользовательский ввод как шаблон; не включайте рантайм-компилятор в проде.
7. Не стройте имена атрибутов/событий из пользовательского ввода; не делайте `v-bind` пользовательского объекта.
8. Внедрите строгую CSP (`script-src 'self'`, `object-src 'none'`).
9. Регулярно проверяйте зависимости (`npm audit`, Dependabot), минимизируйте их число, используйте SRI.
10. Не доверяйте данным от API так же, как пользовательскому вводу.

## Шпаргалка

| Вектор | Риск | Защита |
|--------|------|--------|
| `{{ value }}` | нет (экранируется) | использовать по умолчанию |
| `:attr="value"` | нет (экранируется) | использовать по умолчанию |
| `v-html="raw"` | **XSS** | DOMPurify + серверная санитизация |
| `:href="url"` | `javascript:`-URL | валидация схемы + `rel="noopener noreferrer"` |
| `:src` iframe/script | загрузка чужого кода | белый список доменов |
| `<component :is="x">` | произвольный компонент | белый список |
| `v-bind="userAttrs"` | инъекция обработчиков | не использовать с вводом |
| `compile(userInput)` | выполнение кода | никогда; markdown + санитизация |
| Зависимости | известные CVE | `npm audit`, Dependabot, SRI |
| Любой слой пробит | компрометация | CSP как страховка |

**Главное:** XSS — главная угроза. Vue безопасен по умолчанию (экранирование), опасность появляется при сознательном обходе: `v-html`, `:href` с `javascript:`, динамические компоненты/шаблоны из ввода. Санитизируйте (DOMPurify) на сервере и клиенте, валидируйте URL и имена, держите строгую CSP и следите за зависимостями.
