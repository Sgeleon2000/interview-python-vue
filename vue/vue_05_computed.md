# Vue 3 — Вычисляемые свойства (computed) — конспект и вопросы

## О чём раздел

Вычисляемые свойства (`computed`) — это реактивные значения, производные от других реактивных
данных. Они нужны, чтобы описывать сложную или дорогую логику декларативно, не засоряя шаблон,
и при этом получать автоматическое **кеширование** результата. Это одна из самых частых тем на
собеседованиях: интервьюер почти всегда спрашивает «чем `computed` отличается от метода?» и
«когда `computed`, а когда `watch`?».

В Composition API используется функция `computed()`, в Options API — секция `computed`.

## Ключевые концепции

- `computed()` возвращает реактивную `ref` (`ComputedRef`), доступ к значению — через `.value`
  (в шаблоне `.value` разворачивается автоматически).
- **Кеширование на основе зависимостей**: результат вычисляется один раз и переиспользуется,
  пока не изменится хотя бы одна реактивная зависимость. Это главное отличие от метода.
- Геттер `computed` должен быть **чистой функцией**: без побочных эффектов, без мутаций
  реактивного состояния, без асинхронных операций.
- По умолчанию `computed` доступен только на чтение. Можно сделать **writable computed**,
  передав объект с `get` и `set`.
- `computed` vs `watch`: `computed` — для **производного значения** (что-то возвращает),
  `watch` — для **реакции на изменение** (побочные эффекты: запросы, логирование, навигация).

## Подробный разбор с примерами кода

### Базовый computed (Composition API)

```vue
<script setup>
import { ref, computed } from 'vue'

const firstName = ref('Иван')
const lastName = ref('Петров')

// Геттер. Зависимости (firstName, lastName) отслеживаются автоматически.
const fullName = computed(() => `${firstName.value} ${lastName.value}`)
</script>

<template>
  <!-- В шаблоне .value не нужен -->
  <p>Полное имя: {{ fullName }}</p>
</template>
```

### То же на Options API

```vue
<script>
export default {
  data() {
    return { firstName: 'Иван', lastName: 'Петров' }
  },
  computed: {
    // this автоматически указывает на инстанс компонента
    fullName() {
      return `${this.firstName} ${this.lastName}`
    },
  },
}
</script>
```

### КЕШИРОВАНИЕ — главное отличие от метода

Это топ-вопрос. `computed` запоминает результат и **пересчитывает его только тогда, когда
меняется реактивная зависимость**. Метод же выполняется заново **при каждом ре-рендере**,
даже если входные данные не менялись.

```vue
<script setup>
import { ref, computed } from 'vue'

const list = ref([5, 2, 9, 1, 7])

// computed: отсортирует один раз; пока list не изменится — вернёт кеш
const sortedComputed = computed(() => {
  console.log('computed: пересчёт')
  return [...list.value].sort((a, b) => a - b)
})

// метод: будет вызываться при КАЖДОМ ре-рендере шаблона
function sortedMethod() {
  console.log('method: пересчёт')
  return [...list.value].sort((a, b) => a - b)
}

const counter = ref(0)
</script>

<template>
  <!-- При клике меняется counter, шаблон ре-рендерится -->
  <button @click="counter++">Тик: {{ counter }}</button>

  <!-- sortedComputed НЕ пересчитывается: list не менялся, берётся кеш -->
  <p>{{ sortedComputed }}</p>

  <!-- sortedMethod() пересчитывается на каждый ре-рендер (на каждый клик) -->
  <p>{{ sortedMethod() }}</p>
</template>
```

При нажатии кнопки в консоли появится «method: пересчёт», но не «computed: пересчёт».
Для дорогих операций (сортировка, фильтрация больших списков, парсинг) это критично для
производительности.

> Важно: кеш строится на **реактивных** зависимостях. Если внутри геттера использовать
> нереактивное значение (например, `Date.now()` или обычную переменную модуля), `computed`
> не узнает о его изменении и навсегда «застрянет» на первом результате.

```vue
<script setup>
import { computed } from 'vue'

// АНТИПАТТЕРН: зависимость нереактивна, кеш никогда не обновится
const now = computed(() => Date.now())
// now.value всегда вернёт момент первого вычисления
</script>
```

### Цепочки computed

`computed` может зависеть от другого `computed` — граф зависимостей строится автоматически.

```vue
<script setup>
import { ref, computed } from 'vue'

const price = ref(1000)
const qty = ref(3)

const subtotal = computed(() => price.value * qty.value) // 3000
const withVat = computed(() => subtotal.value * 1.2)      // 3600 (НДС 20%)
const display = computed(() => `${withVat.value.toFixed(2)} ₽`)
</script>
```

### Writable computed (get/set)

По умолчанию присваивание `computed.value = ...` вызовет предупреждение «Write operation
failed: computed value is readonly». Чтобы разрешить запись, передайте объект с `get` и `set`.

```vue
<script setup>
import { ref, computed } from 'vue'

const firstName = ref('Иван')
const lastName = ref('Петров')

const fullName = computed({
  // геттер — как обычно
  get() {
    return `${firstName.value} ${lastName.value}`
  },
  // сеттер — вызывается при fullName.value = '...'
  set(newValue) {
    const parts = newValue.split(' ')
    firstName.value = parts[0] ?? ''
    lastName.value = parts[1] ?? ''
  },
})

fullName.value = 'Пётр Сидоров'
// firstName.value === 'Пётр', lastName.value === 'Сидоров'
</script>
```

Типичный кейс writable computed — двусторонняя привязка `v-model` к производному значению,
особенно для проксирования Vuex/Pinia-состояния в поле формы.

```vue
<script setup>
import { computed } from 'vue'
import { useUserStore } from '@/stores/user'

const store = useUserStore()

// v-model работает через get/set, не мутируя стор напрямую
const name = computed({
  get: () => store.name,
  set: (val) => store.setName(val),
})
</script>

<template>
  <input v-model="name" />
</template>
```

### computed vs watch

```vue
<script setup>
import { ref, computed, watch } from 'vue'

const query = ref('')

// computed: производное значение (ничего «не делает», просто возвращает)
const trimmed = computed(() => query.value.trim().toLowerCase())

// watch: побочный эффект в ответ на изменение (асинхронный запрос)
watch(query, async (newVal) => {
  if (!newVal) return
  const res = await fetch(`/api/search?q=${newVal}`)
  // ...обработка результата
})
</script>
```

Правило: если нужно **получить значение** на основе других — это `computed`.
Если нужно **выполнить действие** при изменении (запрос, таймер, ручная мутация другого
состояния) — это `watch`.

## Полный рабочий пример

Корзина интернет-магазина с фильтром, итогами и редактируемой скидкой через writable computed.

```vue
<script setup>
import { ref, computed } from 'vue'

const items = ref([
  { id: 1, title: 'Клавиатура', price: 3500, qty: 1, inStock: true },
  { id: 2, title: 'Мышь', price: 1500, qty: 2, inStock: true },
  { id: 3, title: 'Монитор', price: 18000, qty: 1, inStock: false },
])

const onlyInStock = ref(false)
const discountPercent = ref(10)

// Производное: отфильтрованный список (кешируется)
const visibleItems = computed(() =>
  onlyInStock.value ? items.value.filter((i) => i.inStock) : items.value
)

// Производное: сумма по видимым товарам
const subtotal = computed(() =>
  visibleItems.value.reduce((sum, i) => sum + i.price * i.qty, 0)
)

// Writable computed: скидка в рублях, можно задать и процент, и сумму
const discountRub = computed({
  get: () => Math.round((subtotal.value * discountPercent.value) / 100),
  set: (rub) => {
    // Пересчитываем процент из введённой суммы скидки
    discountPercent.value = subtotal.value
      ? Math.round((rub / subtotal.value) * 100)
      : 0
  },
})

// Цепочка computed
const total = computed(() => subtotal.value - discountRub.value)

function inc(item) {
  item.qty++
}
</script>

<template>
  <label>
    <input type="checkbox" v-model="onlyInStock" />
    Только в наличии
  </label>

  <ul>
    <li v-for="item in visibleItems" :key="item.id">
      {{ item.title }} — {{ item.price }} ₽ × {{ item.qty }}
      <button @click="inc(item)">+1</button>
    </li>
  </ul>

  <p>Промежуточный итог: {{ subtotal }} ₽</p>

  <label>
    Скидка (₽):
    <!-- v-model к writable computed: пишем сумму, обновляется процент -->
    <input v-model.number="discountRub" type="number" />
  </label>
  <p>Скидка: {{ discountPercent }}% = {{ discountRub }} ₽</p>

  <strong>Итого: {{ total }} ₽</strong>
</template>
```

## Частые вопросы на собеседовании (Q/A)

**1. Чем `computed` отличается от метода?**
`computed` кешируется на основе реактивных зависимостей: пересчитывается только при их
изменении, иначе возвращает запомненный результат. Метод выполняется заново при каждом
ре-рендере. Для дорогих вычислений `computed` существенно экономит ресурсы.

**2. Что такое кеширование `computed` и как оно работает?**
Vue отслеживает, какие реактивные значения читаются внутри геттера, и связывает их с этим
`computed`. Результат хранится; при следующем доступе, если ни одна зависимость не изменилась,
возвращается сохранённое значение без выполнения геттера. При изменении зависимости `computed`
помечается «грязным» (dirty) и пересчитывается лениво — при следующем чтении `.value`.

**3. Когда выбрать метод вместо `computed`?**
Когда нужно действие/значение, зависящее от аргументов (`format(value)`), или результат не
должен кешироваться (например, нужно текущее время при каждом вызове), либо вызов происходит
по событию (`@click="save"`), а не в шаблоне на каждый рендер.

**4. `computed` vs `watch` — в чём разница?**
`computed` — декларативно вычисляет и **возвращает** производное значение (без побочных
эффектов). `watch` — императивно **реагирует** на изменение, выполняя побочные эффекты
(запросы, таймеры, навигация). Если можно выразить через `computed` — почти всегда лучше так.

**5. Можно ли в `computed` делать побочные эффекты или асинхронные операции?**
Нет. Геттер должен быть чистым: только читать данные и возвращать результат. Мутации, запросы,
изменение DOM, `setTimeout` — это `watch` или обработчики событий. Иначе ломается модель
кеширования и появляются трудноотлавливаемые баги.

**6. Что такое writable computed и зачем он нужен?**
Это `computed` с `get` и `set`. Позволяет присваивать значение (`ref.value = x`), которое
сеттер раскладывает на исходные источники. Частый кейс — `v-model` на производном значении или
проксирование стора в форму без прямой мутации.

**7. Почему нельзя мутировать состояние внутри `computed`?**
Геттер может вызываться много раз и в неожиданном порядке (ленивое вычисление, кеш). Мутация
сделает поведение недетерминированным и вызовет лишние реактивные обновления, вплоть до
бесконечных циклов.

**8. Будет ли `computed` пересчитываться, если зависимость нереактивна?**
Нет. Если внутри использовать `Date.now()`, обычную переменную или значение вне реактивной
системы, `computed` не «увидит» его изменения и навсегда останется на первом результате.

**9. Что вернёт `computed` — `ref` или примитив?**
`computed()` возвращает `ComputedRef`. В скрипте читать через `.value`, в шаблоне `.value`
разворачивается автоматически.

**10. Может ли `computed` зависеть от другого `computed`?**
Да. Vue строит граф зависимостей; цепочки и ветвления отслеживаются автоматически, кеш
работает на каждом уровне.

**11. Ленивое или энергичное вычисление у `computed`?**
Ленивое: геттер не выполняется, пока кто-то не прочитает `.value`. Если `computed` нигде не
используется, он не вычисляется вообще, даже при изменении зависимостей.

**12. Как `computed` влияет на производительность списков?**
Фильтрацию/сортировку больших массивов следует выносить в `computed`, а не вызывать как метод
в шаблоне — иначе тяжёлая операция повторяется на каждый ре-рендер. Кеш делает рендер
предсказуемым по стоимости.

**13. Что произойдёт при присваивании в read-only `computed`?**
В dev-режиме Vue выведет предупреждение «computed value is readonly», присваивание не
сработает. Нужно добавить `set` (writable computed).

**14. Можно ли передавать аргументы в `computed`?**
Напрямую нет — `computed` не принимает аргументы. Если нужна параметризация, верните из
`computed` функцию или используйте обычный метод (тогда теряется кеширование).

## Подводные камни (gotchas)

- **Побочные эффекты в геттере**: мутации, запросы, изменение DOM ломают кеш и реактивность.
- **Нереактивные зависимости**: `Date.now()`, `Math.random()`, переменные модуля не отслеживаются
  — `computed` «застывает».
- **Мутация возвращаемого объекта/массива**: возвращать новые ссылки (`[...arr].sort()`), а не
  мутировать исходный массив внутри геттера.
- **Забытый `.value`** в скрипте: `computed` — это `ref`, в JS нужен `.value` (в шаблоне нет).
- **Read-only по умолчанию**: попытка `v-model` на обычном `computed` без `set` приведёт к
  предупреждению.
- **Тяжёлая логика как метод в шаблоне**: `{{ heavyMethod() }}` исполняется на каждый рендер —
  частая причина «тормозов».
- **Бесконечный цикл**: сеттер writable computed, меняющий зависимость геттера так, что геттер
  снова меняется — приведёт к зацикливанию.

## Лучшие практики

- Используйте `computed` для **любого производного значения**, а не вычисляйте его в шаблоне.
- Держите геттер **чистым**: без побочных эффектов, мутаций и асинхронности.
- Возвращайте **новые** структуры данных, не мутируйте исходники (`[...].sort()`, `.map()`).
- Дробите сложную логику на **цепочки маленьких computed** — читаемее и переиспользуемо.
- Для действий и async — используйте `watch` или обработчики событий, не `computed`.
- Writable computed — для `v-model` на производных значениях и проксирования стора.
- Выносите фильтрацию/сортировку списков в `computed` ради кеша и производительности.

## Шпаргалка

```js
import { computed } from 'vue'

// read-only computed (геттер)
const a = computed(() => x.value * 2)
a.value // чтение (в шаблоне без .value)

// writable computed (get/set)
const b = computed({
  get: () => x.value,
  set: (v) => { x.value = v },
})
b.value = 10 // вызывает set

// computed   — производное ЗНАЧЕНИЕ, кешируется, чистый геттер
// method      — вызов на КАЖДЫЙ рендер, кеша нет, можно с аргументами
// watch       — ПОБОЧНЫЙ ЭФФЕКТ при изменении (запрос, таймер, навигация)
```

| Критерий            | computed        | метод           | watch              |
|---------------------|-----------------|-----------------|--------------------|
| Кеширование         | да              | нет             | —                  |
| Возвращает значение | да              | да              | нет                |
| Побочные эффекты    | нет             | да              | да                 |
| Аргументы           | нет             | да              | —                  |
| Когда выполняется   | при чтении (lazy)| на каждый рендер| при изменении      |
```
