# Vue 3 — Композаблы (composables) — конспект и вопросы

## О чём раздел

**Композабл (composable)** — это функция, которая использует Composition API для инкапсуляции и **переиспользования логики с состоянием (stateful logic)**.

Ключевое слово здесь — **stateful**. Если функция просто переиспользует логику без состояния (например, форматирует дату), это обычная утилита (utility function), и для неё Composition API не нужен. Композабл становится композаблом именно тогда, когда внутри него живёт реактивное состояние (`ref`, `reactive`, `computed`), хуки жизненного цикла, watchers и т.п.

Идея: фронтенд-логика часто состоит из «кусков», которые нужно использовать в разных компонентах — отслеживание позиции мыши, работа с localStorage, загрузка данных, дебаунс. Композаблы позволяют вынести такой кусок в отдельную функцию `useSomething()` и подключать её в любом компоненте.

```vue
<script setup>
import { useMouse } from './composables/useMouse'

// Подключаем логику отслеживания мыши одной строкой
const { x, y } = useMouse()
</script>

<template>
  <p>Курсор: {{ x }}, {{ y }}</p>
</template>
```

## Ключевые концепции

- **Композабл = функция на Composition API**, инкапсулирующая stateful-логику.
- **Соглашение об именовании**: имя начинается с `use` (camelCase): `useMouse`, `useFetch`, `useLocalStorage`.
- **Возврат реактивных значений**: композабл обычно возвращает объект с `ref`/`computed`/функциями. Возвращать рекомендуется именно `ref` (а не «развёрнутые» значения), чтобы реактивность сохранялась.
- **Жизненный цикл внутри композабла**: композабл может вызывать `onMounted`, `onUnmounted`, `watch` и т.д. — они привяжутся к компоненту, который его вызвал.
- **Изоляция состояния по умолчанию**: каждый вызов композабла создаёт **свой** независимый экземпляр состояния. Это отличает их от глобальных хранилищ.
- **Композаблы можно вкладывать**: один композабл может вызывать другой.
- **Композаблы заменяют миксины (mixins)** и **renderless-компоненты** как основной способ переиспользования логики во Vue 3.
- **Где вызывать**: только синхронно внутри `<script setup>` / `setup()` или внутри другого композабла (как и любые функции Composition API с хуками жизненного цикла).

## Подробный разбор с примерами кода

### 1. Простейший композабл: useMouse

```js
// composables/useMouse.js
import { ref, onMounted, onUnmounted } from 'vue'

// По соглашению имя начинается с "use"
export function useMouse() {
  // Состояние, инкапсулированное и управляемое композаблом
  const x = ref(0)
  const y = ref(0)

  // Композабл может обновлять своё состояние со временем
  function update(event) {
    x.value = event.pageX
    y.value = event.pageY
  }

  // Композабл может подключаться к жизненному циклу
  // компонента-владельца для настройки и очистки
  onMounted(() => window.addEventListener('mousemove', update))
  onUnmounted(() => window.removeEventListener('mousemove', update))

  // Возвращаем управляемое состояние как результат
  return { x, y }
}
```

Использование:

```vue
<script setup>
import { useMouse } from './composables/useMouse'

const { x, y } = useMouse()
</script>

<template>
  <p>Мышь: {{ x }} / {{ y }}</p>
</template>
```

**Что произошло:**
- Каждый вызов `useMouse()` создаёт **свежие** `x` и `y`.
- `onMounted`/`onUnmounted` привязались к компоненту, который вызвал композабл, — Vue сам подпишет и отпишет слушатель.
- Логику можно использовать в нескольких компонентах, и состояние не будет «смешиваться».

### 2. Композаблы можно вкладывать друг в друга

```js
// composables/useEventListener.js
import { onMounted, onUnmounted } from 'vue'

// Низкоуровневый композабл: подписка на событие с авто-очисткой
export function useEventListener(target, event, callback) {
  onMounted(() => target.addEventListener(event, callback))
  onUnmounted(() => target.removeEventListener(event, callback))
}
```

```js
// composables/useMouse.js (переписанный через useEventListener)
import { ref } from 'vue'
import { useEventListener } from './useEventListener'

export function useMouse() {
  const x = ref(0)
  const y = ref(0)

  // Переиспользуем другой композабл внутри
  useEventListener(window, 'mousemove', (event) => {
    x.value = event.pageX
    y.value = event.pageY
  })

  return { x, y }
}
```

Это показывает важную идею: **композаблы компонуются (composition)** — из мелких строятся крупные.

### 3. useFetch: асинхронная загрузка данных

```js
// composables/useFetch.js
import { ref, watchEffect, toValue } from 'vue'

// url может быть строкой, ref'ом или геттером — принимаем "что угодно"
export function useFetch(url) {
  const data = ref(null)
  const error = ref(null)
  const isLoading = ref(false)

  const fetchData = async () => {
    // сброс состояния перед новым запросом
    data.value = null
    error.value = null
    isLoading.value = true

    try {
      // toValue() разворачивает ref/геттер/обычное значение в значение
      const res = await fetch(toValue(url))
      if (!res.ok) throw new Error(`Ошибка HTTP: ${res.status}`)
      data.value = await res.json()
    } catch (err) {
      error.value = err
    } finally {
      isLoading.value = false
    }
  }

  // watchEffect автоматически перезапустит запрос,
  // если url — это ref/геттер и его значение изменится
  watchEffect(() => {
    fetchData()
  })

  return { data, error, isLoading }
}
```

Использование с реактивным URL:

```vue
<script setup>
import { ref, computed } from 'vue'
import { useFetch } from './composables/useFetch'

const id = ref(1)

// Передаём геттер — при изменении id запрос выполнится заново
const { data, error, isLoading } = useFetch(
  () => `https://api.example.com/posts/${id.value}`
)
</script>

<template>
  <button @click="id++">Следующий пост</button>
  <p v-if="isLoading">Загрузка...</p>
  <p v-else-if="error">Ошибка: {{ error.message }}</p>
  <pre v-else>{{ data }}</pre>
</template>
```

**Важно про `toValue()`**: это рекомендуемый способ нормализации аргументов. Он принимает обычное значение, `ref` или геттер-функцию и возвращает «развёрнутое» значение. Использование внутри `watchEffect`/`computed` делает реактивную зависимость отслеживаемой.

### 4. useLocalStorage: синхронизация состояния с хранилищем

```js
// composables/useLocalStorage.js
import { ref, watch } from 'vue'

export function useLocalStorage(key, defaultValue) {
  // Пытаемся прочитать сохранённое значение
  const stored = localStorage.getItem(key)

  // ref хранит распарсенное значение или дефолт
  const value = ref(stored !== null ? JSON.parse(stored) : defaultValue)

  // Любое изменение value пишем в localStorage
  watch(
    value,
    (newValue) => {
      if (newValue === null || newValue === undefined) {
        localStorage.removeItem(key)
      } else {
        localStorage.setItem(key, JSON.stringify(newValue))
      }
    },
    { deep: true } // deep — чтобы отслеживать изменения внутри объектов/массивов
  )

  return value
}
```

Использование:

```vue
<script setup>
import { useLocalStorage } from './composables/useLocalStorage'

// theme автоматически сохраняется и восстанавливается между перезагрузками
const theme = useLocalStorage('app-theme', 'light')
</script>

<template>
  <button @click="theme = theme === 'light' ? 'dark' : 'light'">
    Тема: {{ theme }}
  </button>
</template>
```

### 5. Соглашения возвращаемого значения

Композабл может вернуть:
- **Объект из нескольких `ref`** — чаще всего. Можно деструктурировать с сохранением реактивности:
  ```js
  const { x, y } = useMouse() // x и y остаются ref'ами
  ```
- **Один `ref`** — когда логично (как `useLocalStorage` выше).
- **`reactive`-объект** — иногда удобно, но при деструктуризации реактивность теряется (нужен `toRefs`).

Рекомендация Vue: возвращать **plain-объект, содержащий `ref`**, чтобы вызывающий код мог деструктурировать без потери реактивности.

## Полный рабочий пример

Композабл `useCounter` с расширенным API + компонент:

```js
// composables/useCounter.js
import { ref, computed } from 'vue'

// options — необязательные настройки, делаем композабл гибким
export function useCounter(initial = 0, options = {}) {
  const { min = -Infinity, max = Infinity, step = 1 } = options

  const count = ref(initial)

  // Производные значения — через computed
  const isMin = computed(() => count.value <= min)
  const isMax = computed(() => count.value >= max)

  function increment() {
    count.value = Math.min(count.value + step, max)
  }
  function decrement() {
    count.value = Math.max(count.value - step, min)
  }
  function reset() {
    count.value = initial
  }
  function set(value) {
    count.value = Math.min(Math.max(value, min), max)
  }

  // Возвращаем понятный, самодокументируемый API:
  // состояние + производные + методы
  return { count, isMin, isMax, increment, decrement, reset, set }
}
```

```vue
<!-- CounterDemo.vue -->
<script setup>
import { useCounter } from './composables/useCounter'

// Два НЕЗАВИСИМЫХ счётчика — у каждого своё изолированное состояние
const a = useCounter(0, { min: 0, max: 10 })
const b = useCounter(5, { step: 5 })
</script>

<template>
  <div>
    <h3>Счётчик A (0..10): {{ a.count }}</h3>
    <button :disabled="a.isMin" @click="a.decrement">−</button>
    <button :disabled="a.isMax" @click="a.increment">+</button>
    <button @click="a.reset">Сброс</button>
  </div>

  <div>
    <h3>Счётчик B (шаг 5): {{ b.count }}</h3>
    <button @click="b.decrement">−5</button>
    <button @click="b.increment">+5</button>
  </div>
</template>
```

Здесь видно главное преимущество композаблов перед миксинами: можно вызвать `useCounter` **дважды в одном компоненте** без коллизий имён — состояние полностью изолировано.

## Частые вопросы на собеседовании (Q/A)

**1. Что такое композабл?**
Функция на Composition API, инкапсулирующая переиспользуемую stateful-логику (состояние, computed, watchers, хуки жизненного цикла). По соглашению имя начинается с `use`.

**2. Чем композабл отличается от обычной утилитной функции?**
Композабл работает с **реактивным состоянием** и/или хуками жизненного цикла Vue, поэтому его можно вызывать только в `setup`-контексте. Утилита без состояния (например, форматтер) — просто функция, и её не нужно делать композаблом.

**3. Почему композаблы лучше миксинов? (топ-вопрос)**
Три ключевые проблемы миксинов, которые решают композаблы:
- **Неясные источники свойств (unclear sources)**: при нескольких миксинах непонятно, какое свойство откуда пришло — нужно лезть в каждый миксин. У композабла источник явный: `const { x } = useMouse()` — сразу видно, что `x` из `useMouse`.
- **Коллизии имён (namespace collisions)**: два миксина могут объявить свойство с одинаковым именем — будет конфликт. С композаблами имена при деструктуризации можно переименовать: `const { x: mouseX } = useMouse()`.
- **Неявная связанность (implicit cross-mixin communication)**: миксины могут молча зависеть друг от друга через общие имена. Композаблы передают данные явно через аргументы и возвращаемые значения.
Плюс композаблы лучше типизируются в TypeScript и поддерживают гибкие аргументы.

**4. Можно ли использовать один композабл несколько раз в компоненте?**
Да. Каждый вызов создаёт **изолированное** состояние. С миксинами так нельзя — один и тот же миксин нельзя «применить дважды» с разными настройками без коллизий.

**5. Где можно вызывать композабл?**
Синхронно в верхнем уровне `<script setup>` / `setup()` или внутри другого композабла. Нельзя вызывать асинхронно (после `await`), в обработчиках событий или условно — иначе сломается привязка хуков жизненного цикла.

**6. Что должен возвращать композабл?**
Обычно plain-объект, содержащий `ref`/`computed`/методы. Это позволяет деструктурировать результат без потери реактивности. Возвращать «развёрнутые» (`.value`) значения не стоит — реактивность пропадёт.

**7. Как принимать аргументы, которые могут быть ref или значением?**
Использовать `toValue(arg)` для нормализации — он развернёт `ref`, вызовет геттер или вернёт обычное значение. Внутри `computed`/`watchEffect` это создаёт реактивную зависимость.

**8. Как композабл управляет жизненным циклом?**
Внутри композабла можно вызывать `onMounted`, `onUnmounted`, `onBeforeUnmount` и др. Они привязываются к компоненту-владельцу. Это позволяет инкапсулировать настройку и очистку (например, подписку/отписку от события) внутри композабла.

**9. Чем композаблы отличаются от renderless-компонентов?**
Renderless-компонент (компонент со слотом, но без собственного шаблона) тоже переиспользует логику, но создаёт лишний экземпляр компонента и накладные расходы рендера, плюс работает только в шаблоне. Композаблы не создают лишних компонентов, дают чистую логику без оверхеда рендера и лучше типизируются. Renderless оправдан, когда нужно переиспользовать ещё и разметку/структуру.

**10. Композаблы дают общее (shared) или изолированное состояние?**
По умолчанию — **изолированное**: состояние создаётся внутри функции при каждом вызове. Чтобы сделать состояние **общим (глобальным)**, нужно вынести `ref` за пределы функции — в модульную область видимости (см. ниже). Для сложного глобального состояния лучше Pinia.

**11. Можно ли использовать композаблы в Options API?**
Да, через `setup()`: вызвать композабл там и вернуть нужные значения, чтобы они стали доступны в `this` и шаблоне.

**12. SSR и композаблы — на что обратить внимание?**
Состояние, вынесенное в модульную область (глобальное), на сервере шарится между запросами — это утечка/кросс-загрязнение данных между пользователями. Поэтому глобальное состояние в композаблах опасно при SSR; используйте Pinia или создавайте состояние на каждый запрос.

## Подводные камни (gotchas)

- **Вызов после `await`**: хуки жизненного цикла, вызванные после первого `await` в `setup`, не привяжутся корректно. Все `onMounted`/`useXxx` с хуками — вызывать синхронно.
- **Возврат развёрнутых значений**: `return { x: x.value }` убивает реактивность. Возвращайте `ref`, не `.value`.
- **Деструктуризация `reactive`**: если композабл возвращает `reactive`-объект, прямая деструктуризация теряет реактивность. Используйте `toRefs`.
- **Случайное общее состояние**: `ref`, объявленный **вне** функции композабла, становится глобальным синглтоном — все компоненты делят его. Иногда это нужно, но часто это баг.
- **Очистка ресурсов**: если композабл подписывается на события/таймеры, обязательно очищайте их в `onUnmounted`, иначе утечки памяти.
- **Условный вызов**: `if (cond) { const { x } = useMouse() }` — нельзя, нарушает правила Composition API.
- **SSR-загрязнение** глобальным состоянием между запросами (см. Q12).
- **`watch` с объектом без `deep`**: при синхронизации объекта (как в `useLocalStorage`) забытый `{ deep: true }` приведёт к тому, что вложенные изменения не отследятся.

## Лучшие практики

- **Имя с `use`** и camelCase: `useMouse`, `useFetch`. Делает код узнаваемым.
- **Принимайте гибкие аргументы**: разрешайте передавать `ref`/геттер/значение и нормализуйте через `toValue()`. Так композабл реактивно реагирует на изменения входов.
- **Возвращайте понятный API**: объект с осмысленными именами `ref`/`computed`/методов. Группируйте: состояние, производные значения, действия.
- **Возвращайте `ref`, а не значения**, чтобы вызывающий код сохранял реактивность при деструктуризации.
- **Инкапсулируйте побочные эффекты**: подписки, таймеры, слушатели — настраивайте и очищайте внутри композабла через хуки жизненного цикла.
- **Композируйте**: стройте крупные композаблы из мелких (как `useMouse` поверх `useEventListener`).
- **Изолированное состояние по умолчанию**; общее состояние выносите осознанно и осторожно (помните про SSR).
- **Документируйте** входы/выходы — особенно если аргументы могут быть реактивными.
- **Для сложного глобального состояния используйте Pinia**, а не «глобальный композабл».
- **Не делайте композаблом то, что состояния не имеет** — это просто утилита.

## Шпаргалка

```js
// Скелет композабла
import { ref, computed, watch, watchEffect, toValue, onMounted, onUnmounted } from 'vue'

export function useThing(input) {
  // 1) Состояние
  const state = ref(null)

  // 2) Производные значения
  const derived = computed(() => /* ... */ null)

  // 3) Методы
  function doSomething() { /* ... */ }

  // 4) Нормализация реактивных аргументов
  watchEffect(() => {
    const value = toValue(input) // ref | геттер | значение -> значение
    // ...
  })

  // 5) Жизненный цикл (настройка/очистка)
  onMounted(() => { /* подписка */ })
  onUnmounted(() => { /* отписка */ })

  // 6) Возврат понятного API из ref'ов
  return { state, derived, doSomething }
}
```

| Аспект | Композаблы | Миксины |
|---|---|---|
| Источник свойства | Явный (деструктуризация) | Неявный |
| Коллизии имён | Нет (можно переименовать) | Да |
| Многократное использование | Да, с изоляцией | Нет |
| Связь между частями | Явная (аргументы/возврат) | Неявная |
| TypeScript | Отличная поддержка | Плохая |

| Сценарий | Что использовать |
|---|---|
| Переиспользование stateful-логики | Композабл (`useXxx`) |
| Логика без состояния | Обычная утилитная функция |
| Переиспользование логики + разметки | Renderless-компонент / слоты |
| Сложное глобальное состояние | Pinia |

- Возвращай `ref`, а не `.value`.
- Нормализуй аргументы через `toValue()`.
- Очищай эффекты в `onUnmounted`.
- Вызывай синхронно в `setup`-контексте.
